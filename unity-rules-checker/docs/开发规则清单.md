## 使用说明

本清单提取自知识库，用于编码时快速查阅。

**严重性定义**:
- **CRITICAL**: 必须遵守，违反会导致严重问题（内存泄漏、崩溃）
- **HIGH**: 强烈建议遵守，违反会显著影响性能或可维护性
- **MEDIUM**: 建议遵守，提升代码质量
- **LOW**: 最佳实践建议，可选优化

**编号规则**:
- 格式：`RULE-<类别>-<序号>`
- 例如：`RULE-GC-001` 表示 GC优化 类别的第1条规则

## 1. GC规避规则（热点路径）

### RULE-GC-001 避免Update中字符串拼接

**条件**: 在 `Update()`, `FixedUpdate()`, `LateUpdate()` 中

**必须**: 使用 `StringBuilder` 或字符串插值 `$""`

**禁止**: 使用 `+` 运算符拼接字符串

**例外**:
- 非热点路径（每帧调用次数<10次）可使用字符串插值
- 一次性拼接操作（如初始化）不受限制
- 编辑器工具代码不受此限制

**原因**: 字符串不可变，每次拼接产生新对象，每帧累积触发GC卡顿

```csharp
// ❌ 错误 - 每帧产生GC
void Update()
{
    string info = "Score: " + score + " Level: " + level;
    text.text = info;
}

// ✅ 正确 - 使用StringBuilder复用
private StringBuilder sb = new();

void Update()
{
    sb.Clear();
    sb.Append("Score: ").Append(score).Append(" Level: ").Append(level);
    text.text = sb.ToString();
}
```

**严重性**: HIGH

**相关**: [最佳实践-GC优化清单](../30_性能优化/内存管理/最佳实践-GC优化清单.md)

---

### RULE-GC-002 缓存WaitForSeconds

**条件**: 在协程中使用 `WaitForSeconds` 时

**必须**: 在类级别缓存 `WaitForSeconds` 对象

**禁止**: 在协程循环中 `new WaitForSeconds()`

**原因**: 每次 `new` 都会在堆上分配内存，循环中累积大量垃圾

```csharp
// ❌ 错误 - 每次循环都new
IEnumerator Loop()
{
    while (true)
    {
        yield return new WaitForSeconds(1f);  // GC!
    }
}

// ✅ 正确 - 缓存复用
private WaitForSeconds waitOneSecond = new(1f);

IEnumerator Loop()
{
    while (true)
    {
        yield return waitOneSecond;  // 无GC
    }
}
```

**严重性**: HIGH

**相关**: [最佳实践-GC优化清单](../30_性能优化/内存管理/最佳实践-GC优化清单.md)

---

### RULE-GC-003 避免Update中new集合

**条件**: 在每帧调用的方法中需要使用集合时

**必须**: 将集合声明为类成员，预先设置容量

**禁止**: 在 `Update` 中 `new List<>()` 或 `new Dictionary<>()`

```csharp
// ❌ 错误 - 每帧分配
void Update()
{
    var list = new List<int>();  // GC!
}

// ✅ 正确 - 缓存集合
private List<int> cachedList = new(100);  // 预设容量

void Update()
{
    cachedList.Clear();  // 不释放内存，复用
}
```

**严重性**: CRITICAL

**相关**: [最佳实践-GC优化清单](../30_性能优化/内存管理/最佳实践-GC优化清单.md)

### RULE-GC-004 避免Update中GetComponent

**条件**: 需要访问同一组件多次时

**必须**: 在 `Awake` 或 `Start` 中缓存组件引用

**禁止**: 在 `Update` 中调用 `GetComponent<T>()`

```csharp
// ❌ 错误 - 每帧都有开销
void Update()
{
    var rb = GetComponent<Rigidbody>();  // 开销!
    rb.velocity = velocity;
}

// ✅ 正确 - 缓存组件
private Rigidbody rb;

void Awake()
{
    rb = GetComponent<Rigidbody>();
}

void Update()
{
    rb.velocity = velocity;
}
```

**严重性**: HIGH

**相关**: [最佳实践-GC优化清单](../30_性能优化/内存管理/最佳实践-GC优化清单.md)

### RULE-GC-005 避免Update中使用LINQ

**条件**: 在每帧调用的方法中

**必须**: 使用 `for` 循环代替 LINQ

**禁止**: 使用 `Where`, `Select`, `ToList` 等 LINQ 方法

**原因**: LINQ 每次调用都会产生闭包和迭代器对象

```csharp
// ❌ 错误 - LINQ产生GC
void Update()
{
    var active = items.Where(x => x.IsActive).ToList();  // GC!
}

// ✅ 正确 - 使用for循环
void Update()
{
    int count = items.Count;
    for (int i = 0; i < count; i++)
    {
        if (items[i].IsActive)
        {
            Process(items[i]);
        }
    }
}
```

**严重性**: CRITICAL

**相关**: [性能数据-foreach-vs-for](../30_性能优化/代码优化/性能数据-foreach-vs-for.md)

---

### RULE-GC-006 Update中遍历List使用for

**条件**: 在 `Update()` 等每帧调用的方法中遍历 `List<T>`

**必须**: 使用 `for` 循环，并缓存 `Count`

**禁止**: 使用 `foreach` 遍历 `List<T>`

**例外**:
- 遍历数组可使用foreach（数组零GC）
- 非热点路径（如每秒执行1次）可使用foreach
- 代码可读性优先的场景（热点路径外）

**原因**: foreach遍历List每帧产生32B GC，60FPS下1分钟累积115KB

```csharp
// ❌ 错误 - 每帧32B GC
void Update()
{
    foreach (var item in itemList)  // GC!
    {
        Process(item);
    }
}

// ✅ 正确 - 零GC
void Update()
{
    int count = itemList.Count;  // 缓存Count
    for (int i = 0; i < count; i++)
    {
        Process(itemList[i]);
    }
}

// ✅ 数组遍历 - foreach也可以（数组零GC）
void Update()
{
    foreach (var item in itemArray)  // 数组零GC
    {
        Process(item);
    }
}
```

**严重性**: HIGH

**相关**: [性能数据-foreach-vs-for](../30_性能优化/代码优化/性能数据-foreach-vs-for.md)

### RULE-GC-007 使用CompareTag代替字符串比较

**条件**: 需要比较GameObject的Tag时

**必须**: 使用 `CompareTag("TagName")`

**禁止**: 使用 `gameObject.tag == "TagName"`

```csharp
// ❌ 错误 - tag属性产生GC
if (gameObject.tag == "Player") { }
```

## 2. 内存管理规则

### RULE-MEM-001 事件订阅必须配对取消

**条件**: 在 `OnEnable` 或 `Start` 中订阅事件时

**必须**: 在 `OnDisable` 或 `OnDestroy` 中取消订阅

**禁止**: 订阅事件后不取消订阅

**例外**:
- 静态事件的订阅者生命周期与整个应用相同时
- 确认事件发布者会在订阅者销毁前自动清理的情况

**原因**: 事件发布者持有订阅者引用，即使销毁GameObject也无法被GC回收

```csharp
// ❌ 错误 - 忘记取消订阅
private void OnEnable()
{
    EventBus.Subscribe<ScoreChangedEvent>(OnScoreChanged);
}

// ✅ 正确 - 配对订阅/取消
private void OnEnable()
{
    EventBus.Subscribe<ScoreChangedEvent>(OnScoreChanged);
}

private void OnDisable()
{
    EventBus.Unsubscribe<ScoreChangedEvent>(OnScoreChanged);
}
```

**严重性**: CRITICAL

**相关**: [踩坑记录-内存泄漏模式](../30_性能优化/内存管理/踩坑记录-内存泄漏模式.md)

---

### RULE-MEM-002 协程必须正确停止

**条件**: 使用 `StartCoroutine` 启动协程时

**必须**: 保存协程引用，在 `OnDisable` 中停止

**禁止**: 启动协程不保存引用，或忘记停止

```csharp
// ❌ 错误 - 没有保存引用
private void Start()
{
    StartCoroutine(SpawnLoop());  // 无法停止
}

// ✅ 正确 - 保存引用并停止
private Coroutine spawnCoroutine;

private void OnEnable()
{
    spawnCoroutine = StartCoroutine(SpawnLoop());
}

private void OnDisable()
{
    if (spawnCoroutine != null)
    {
        StopCoroutine(spawnCoroutine);
        spawnCoroutine = null;
    }
}
```

**严重性**: HIGH

**相关**: [踩坑记录-内存泄漏模式](../30_性能优化/内存管理/踩坑记录-内存泄漏模式.md)

---

### RULE-MEM-003 静态集合必须有清理机制

**条件**: 使用静态集合存储对象引用时

**必须**: 提供 `Remove` 或 `Clear` 方法，定期清理

**禁止**: 只添加不移除

```csharp
// ❌ 错误 - 只增不减
public static class ObjectManager
{
    private static List<GameObject> allObjects = new();

    public static void Register(GameObject go)
    {
        allObjects.Add(go);
    }
    // 没有 Remove 方法！
}

// ✅ 正确 - 配对移除
public static class ObjectManager
{
    private static List<GameObject> allObjects = new();

    public static void Register(GameObject go) => allObjects.Add(go);

    public static void Unregister(GameObject go) => allObjects.Remove(go);

    public static void CleanNulls() => allObjects.RemoveAll(x => x == null);
}
```

**严重性**: HIGH

**相关**: [踩坑记录-内存泄漏模式](../30_性能优化/内存管理/踩坑记录-内存泄漏模式.md)

---

### RULE-MEM-004 DOTween动画必须Kill

**条件**: 使用DOTween创建动画时

**必须**: 保存Tween引用，在 `OnDisable` 中 `Kill()`

**禁止**: 创建动画后不管理生命周期

```csharp
// ❌ 错误 - 动画可能继续运行
private void Start()
{
    transform.DOScale(1.2f, 0.3f).SetLoops(-1);
}

// ✅ 正确 - 手动管理
private Tween scaleTween;

private void OnEnable()
{
    scaleTween = transform.DOScale(1.2f, 0.3f)
        .SetLoops(-1, LoopType.Yoyo)
        .SetAutoKill(false);
}

private void OnDisable()
{
    scaleTween?.Kill();
    scaleTween = null;
}
```

**严重性**: HIGH

**相关**: [踩坑记录-内存泄漏模式](../30_性能优化/内存管理/踩坑记录-内存泄漏模式.md)

## 3. 对象池规则

### RULE-POOL-001 高频创建对象必须池化

**条件**: 当对象满足以下任一条件时：
- 每秒创建/销毁超过 10 次
- 创建成本超过 0.1ms
- 同屏可能出现大量实例（子弹、特效、伤害数字）

**必须**: 使用对象池管理对象生命周期

**禁止**: 直接使用 `Instantiate`/`Destroy` 进行高频创建销毁

```csharp
// ❌ 错误 - 频繁创建销毁
void Fire()
{
    var bullet = Instantiate(bulletPrefab);
    Destroy(bullet);  // 产生垃圾
}

// ✅ 正确 - 使用对象池
private Stack<GameObject> bulletPool = new();

void Fire()
{
    GameObject bullet = bulletPool.Count > 0
        ? bulletPool.Pop()
        : Instantiate(bulletPrefab);
    bullet.SetActive(true);
}

void ReturnBullet(GameObject bullet)
{
    bullet.SetActive(false);
    bulletPool.Push(bullet);
}
```

**严重性**: CRITICAL

**例外**:
- 原型开发阶段（快速验证，可后续优化）
- 创建/销毁频率低于每秒5次
- 对象生命周期不可预测（如玩家动态生成的内容）

**原因**: 对象池避免重复分配内存，减少GC压力

**相关**: [设计原理-对象池本质](../10_架构设计/设计原理-对象池本质.md)

---

### RULE-POOL-002 归还对象时必须重置状态

**条件**: 从对象池归还对象时

**必须**: 调用对象的 `Reset()` 方法重置所有状态

**禁止**: 直接 `SetActive(false)` 归还而不重置

**原因**: 对象可能带有上次使用的残留状态，导致复用时行为异常

```csharp
// ❌ 错误 - 不重置状态
public void Return(GameObject obj)
{
    obj.SetActive(false);
    pool.Push(obj);
    // 下次取出时，速度、目标等状态还在！
}

// ✅ 正确 - 重置状态
public void Return(GameObject obj)
{
    var bullet = obj.GetComponent<Bullet>();
    bullet.Reset();  // 重置速度、目标、伤害等
    obj.SetActive(false);
    pool.Push(obj);
}
```

**严重性**: CRITICAL

**相关**: [设计原理-对象池本质](../10_架构设计/设计原理-对象池本质.md)

---

### RULE-POOL-003 不池化长期存活对象

**条件**: 对象存活时间很长（如玩家角色、Manager）

**禁止**: 池化这类对象

**原因**: 对象池的优势在于"重复使用"，长期存活的对象池化没有意义，反而浪费内存

```csharp
// ❌ 错误 - 玩家不需要池化
public class PlayerPool { ... }  // 毫无意义

// ✅ 正确 - 只池化高频创建销毁的对象
// 子弹、特效、伤害数字、列表项、弹窗
```

**严重性**: MEDIUM

**相关**: [设计原理-对象池本质](../10_架构设计/设计原理-对象池本质.md)

## 4. 架构决策规则

### RULE-ARCH-001 单例仅用于全局服务

**条件**: 考虑使用单例模式时

**必须**: 仅对真正的全局服务使用单例（如 GameManager, AudioManager, SaveSystem）

**禁止**:
- 将游戏实体（Player, Enemy, Bullet）做成单例
- 为了"方便访问"而滥用单例

**例外**:
- 编辑器工具类（EditorWindow相关）可使用单例模式
- 确认全局唯一且生命周期与应用相同的系统

**原因**: 单例创建全局状态，导致隐藏依赖、难以测试、多场景冲突

```csharp
// ❌ 错误 - 滥用单例
public class PlayerSingleton : MonoBehaviour { ... }
public class EnemySingleton : MonoBehaviour { ... }

// ✅ 正确 - 仅全局服务用单例
public class AudioManager : MonoBehaviour
{
    public static AudioManager Instance { get; private set; }

    private void Awake()
    {
        if (Instance != null && Instance != this)
        {
            Destroy(gameObject);
            return;
        }
        Instance = this;
        DontDestroyOnLoad(gameObject);
    }
}
```

**严重性**: HIGH

**相关**: [架构决策-单例vs依赖注入](../10_架构设计/架构决策-单例vs依赖注入.md), [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ARCH-002 避免事件触发事件

**条件**: 在事件处理函数中

**必须**: 使用直接调用处理连锁反应

**禁止**: 在事件处理中发布新事件（事件→事件→事件）

**原因**: 事件链不可预测，可能无限循环，调试困难

```csharp
// ❌ 错误 - 事件风暴
public void OnPlayerDeath(PlayerDeathEvent e)
{
    EventBus.Publish(new ScoreChangedEvent());  // 事件1
}

public void OnScoreChanged(ScoreChangedEvent e)
{
    EventBus.Publish(new AchievementCheckEvent());  // 事件2
}

// ✅ 正确 - 直接调用
public void OnPlayerDeath(PlayerDeathEvent e)
{
    scoreManager.AddDeathScore(e.PlayerId);
    achievementSystem.CheckDeathAchievements(e.PlayerId);
    saveManager.Save();
}
```

**严重性**: HIGH

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ARCH-003 高频对象使用管理器模式

**条件**: 同屏大量同类对象（如100+子弹）

**必须**: 使用单一管理器统一更新

**禁止**: 每个对象都有自己的 `Update` 方法

**原因**: 大量Update调用有Unity内部开销，无法批量优化

```csharp
// ❌ 错误 - 每个物体都有Update
public class Bullet : MonoBehaviour
{
    private void Update()
    {
        transform.position += direction * speed * Time.deltaTime;
    }
}

// ✅ 正确 - 管理器统一更新
public class BulletManager : MonoBehaviour
{
    private List<BulletData> bullets = new();

    private void Update()
    {
        foreach (var bullet in bullets)
        {
            bullet.position += bullet.direction * bullet.speed * Time.deltaTime;
        }
    }
}
```

**严重性**: HIGH

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ARCH-004 缓存FindObject结果

**条件**: 需要引用其他对象时

**必须**: 在 `Start` 或 `Awake` 中查找并缓存

**禁止**: 在 `Update` 中调用 `GameObject.Find`, `FindObjectOfType`

```csharp
// ❌ 错误 - 每帧Find
void Update()
{
    var player = GameObject.Find("Player");  // O(n)!
}

// ✅ 正确 - 缓存引用
private Transform player;

void Start()
{
    player = GameObject.FindWithTag("Player")?.transform;
}

void Update()
{
    if (player != null) { /* ... */ }
}
```

**严重性**: HIGH

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ARCH-005 使用枚举代替字符串标识

**条件**: 需要标识类型、状态、场景等时

**必须**: 使用枚举或常量类

**禁止**: 使用字符串作为标识符

**原因**: 字符串拼写错误不报错，运行时才发现问题

```csharp
// ❌ 错误 - 字符串滥用
public void PlayAnimation(string animName) { }
PlayAnimation("walkk");  // 拼写错误不报错！

// ✅ 正确 - 使用枚举
public enum AnimationType { Idle, Walk, Run, Jump }

public void PlayAnimation(AnimationType anim) { }
PlayAnimation(AnimationType.Walk);  // 类型安全
```

**严重性**: MEDIUM

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ARCH-006 使用SerializeField代替public

**条件**: 需要在Inspector中暴露字段时

**必须**: 使用 `[SerializeField] private`

**禁止**: 使用 `public` 字段

**原因**: public破坏封装，任何地方都能修改，难以追踪变化

```csharp
// ❌ 错误 - 所有字段public
public class Player : MonoBehaviour
{
    public float health;
    public float speed;
}

// ✅ 正确 - SerializeField + 属性
public class Player : MonoBehaviour
{
    [SerializeField] private float maxHealth = 100f;
    [SerializeField] private float speed = 5f;

    public float MaxHealth => maxHealth;
    public float Speed => speed;
}
```

**严重性**: MEDIUM

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

---

### RULE-ARCH-007 第三个类需要相同能力时引入接口

**条件**: 当第三个类需要与已有两个类相同的能力时

**必须**: 提取共同接口

**禁止**:
- 为了"未来可能"而提前抽象（YAGNI）
- 只有一个实现时就创建接口

**原因**: 两个类可能是巧合，三个类是模式（Rule of Three）

```csharp
// ❌ 过早抽象 - 只有一个实现
public interface IDamageable { void TakeDamage(float dmg); }
public class Player : IDamageable { ... }

// ✅ 第三个类出现时再抽象
// 有 Player, Enemy, Destructible 都需要受伤
public interface IDamageable
{
    void TakeDamage(float damage);
    bool IsDead { get; }
}
```

**严重性**: MEDIUM

## 5. 异步编程规则

### RULE-ASYNC-001 避免协程嵌套地狱

**条件**: 需要顺序执行多个异步操作时

**必须**: 使用 `UniTask` 的 `async/await` 或状态机

```csharp
private IEnumerator AttackSequence()

    yield return StartCoroutine(DealDamage());
}

// ✅ 正确 - 使用UniTask
private async UniTask AttackSequence(CancellationToken ct)
{
    await MoveToTarget(ct);
    await PlayAttackAnim(ct);
    await DealDamage(ct);
}
```

**严重性**: MEDIUM

**相关**: [反模式-常见架构陷阱](../10_架构设计/反模式-常见架构陷阱.md)

### RULE-ASYNC-002 异步操作必须支持取消

**条件**: 使用 `async/await` 或协程时

**必须**: 支持 `CancellationToken` 取消

**禁止**: 无限循环的异步操作没有取消机制

```csharp
// ❌ 错误 - 无法取消
private async UniTask LoopForever()
{
    while (true)
    {
        await UniTask.Delay(1000);
        DoSomething();
    }
}

// ✅ 正确 - 支持取消
private async UniTask LoopForever(CancellationToken ct)
{
    while (!ct.IsCancellationRequested)
    {
        await UniTask.Delay(1000, cancellationToken: ct);
        DoSomething();
    }
}
```

**严重性**: HIGH

```csharp
public class Player : MonoBehaviour

}
```

### RULE-REFACTOR-003 嵌套超过3层必须重构

**条件**: 代码嵌套层次超过 3 层（if/for/while嵌套）

**必须**: 使用早返回、提取方法、状态机等重构

**禁止**: 继续加深嵌套

```csharp
// ❌ 错误 - 深层嵌套

    if (item != null)
    {
        if (item.IsActive)
        {
            if (item.Type == ItemType.Weapon)
            {
                if (item.Durability > 0)
                {
                    // 终于到了...
                }
            }
        }
    }
}

// ✅ 正确 - 早返回
public void Process(Item item)
{
    if (item == null) return;
    if (!item.IsActive) return;
    if (item.Type != ItemType.Weapon) return;
    if (item.Durability <= 0) return;

    // 处理逻辑
}
```

### RULE-REFACTOR-004 重构前必须建立测试

**条件**: 计划进行任何非trivial的重构时

**必须**: 先为待重构代码编写测试用例

**禁止**: 没有测试保障就直接重构

**原因**: 重构的定义是"不改变外部行为的前提下改进内部结构"，没有测试无法验证行为是否改变

```csharp

    var enemy = new Enemy();
    int initialHealth = enemy.Health;

    player.Attack(enemy);

    Assert.AreEqual(initialHealth - 10, enemy.Health);
}

// 然后再安全重构Attack方法
```

**严重性**: CRITICAL

**相关**: [架构演进-从Demo到量产](../10_架构设计/架构演进-从Demo到量产.md)

### RULE-REFACTOR-005 每次只改一小部分

**条件**: 进行重构时

**必须**: 每次只改一小部分，改完立即测试

**禁止**: 一次性大规模重构

**原因**: 小步重构容易定位问题，大重构出问题难以排查

```
✅ 正确流程：
1. 建立测试
2. 小改动A → 测试通过 → 提交
3. 小改动B → 测试通过 → 提交
4. 小改动C → 测试通过 → 提交

❌ 错误流程：
1. 建立测试
2. 大改动A+B+C+D
3. 测试失败
4. 不知道哪里出错
```

**严重性**: HIGH

**相关**: [架构演进-从Demo到量产](../10_架构设计/架构演进-从Demo到量产.md)

### RULE-REFACTOR-006 继承→组件化迁移时机

**条件**: 出现以下任一情况时：
- 类爆炸（继承树超过3层）
- 组合爆炸（需要飞行+游泳+潜水的角色）
- 无用继承（NPC继承了attackDamage但不需要）

**必须**: 将继承改为组件组合

**步骤**:
1. 识别现有类中的独立职责
2. 每个职责变成独立组件
3. 定义组件间通信接口
4. 原类只负责协调组件
5. 移除继承关系

```csharp
// ❌ 继承架构问题
Character → CombatCharacter → Player
                              → Enemy
         → NPC (不需要战斗但继承了)

// ✅ 组件化后
Player = Movement + Health + Attack + Jump + Input
Enemy = Movement + Health + Attack + AI
NPC = Movement + Interact (不需要Health也可以)
```

**严重性**: HIGH

**必须**: 在资源不再使用时调用`Addressables.Release(handle)`

```csharp
// ✅ 正确 - 管理Handle生命周期
```

### RULE-RES-005 实例化对象使用对象池

**条件**: 频繁创建销毁的GameObject（子弹、特效、伤害数字）

**必须**: 结合Addressables和对象池

**禁止**: 每次都Addressables.InstantiateAsync

```csharp
// ❌ 错误 - 每次都实例化新对象
async void Fire()
{
    var handle = Addressables.InstantiateAsync("Bullet");
    await handle.Task;
    // 用完Destroy，浪费性能
}

// ✅ 正确 - 预加载+对象池
private Stack<GameObject> bulletPool = new();

async void InitBulletPool()
{
    // 预加载10个子弹到池中
    for (int i = 0; i < 10; i++)
    {
        var handle = Addressables.InstantiateAsync("Bullet", poolContainer);
        var bullet = await handle.Task;
        bullet.SetActive(false);
        bulletPool.Push(bullet);
    }
}

GameObject GetBullet()
{
    if (bulletPool.Count > 0)
    {
        var bullet = bulletPool.Pop();
        bullet.SetActive(true);
        return bullet;
    }
    return null;  // 池空了，等待复用
}
```

**严重性**: HIGH

## 10. 编译期优化规则

### RULE-PERF-001 不需要继承的类使用sealed

**条件**: 类不需要被继承时

**必须**: 标记为`sealed`

**禁止**: 默认允许继承（除非明确需要）

```csharp
// ❌ 错误 - 不需要继承却允许
public class PlayerController { }

// ✅ 正确 - 使用sealed
public sealed class PlayerController
{
    // JIT编译器可以优化虚方法调用
    // 因为知道不会有子类
}

// ✅ 只有明确需要继承时才留出扩展点
public abstract class BaseController { }
```

**严重性**: MEDIUM

**原因**: sealed允许编译器进行更多优化，如虚方法调用内联

**相关**: [性能数据-sealed性能](../30_性能优化/代码优化/性能数据-sealed性能.md)

---

### RULE-PERF-002 避免热点路径使用虚方法

**条件**: 在Update等每帧调用的代码中

**必须**: 避免使用虚方法或接口调用

**禁止**: 热点路径中的虚方法调用

```csharp
// ❌ 错误 - 热点路径使用虚方法
public class Entity
{
    public virtual void Update() { }  // 虚方法调用有开销
}

// ✅ 正确 - 热点代码避免虚方法
public sealed class Entity
{
    public void Update() { }  // 直接调用，可内联
}
```

**严重性**: MEDIUM

**原因**: 虚方法需要查表确定实际实现，无法内联

**相关**: [性能数据-虚方法性能](../30_性能优化/代码优化/性能数据-虚方法性能.md)

---

### RULE-PERF-003 小数据结构使用struct代替class

**条件**: 满足以下所有条件时
- 数据小于16字节
- 不需要继承
- 逻辑上表示单个值
- 不需要频繁装箱拆箱

**必须**: 使用`struct`

**禁止**: 使用class表示小数据结构

```csharp
// ❌ 错误 - 小数据用class
public class IntVector2  // 只有8字节
{
    public int x, y;
}

// ✅ 正确 - 使用struct
public struct IntVector2
{
    public int x, y;
    // 栈上分配，无GC，缓存友好
}
```

**严重性**: MEDIUM

**原因**: struct在栈上分配，无GC压力，缓存命中率高

**相关**: [性能数据-struct-vs-class](../30_性能优化/代码优化/性能数据-struct-vs-class.md)

---

### RULE-PERF-004 使用泛型避免装箱拆箱

**条件**: 需要存储值类型到集合时

**必须**: 使用泛型集合`List<T>`、`Dictionary<TKey,TValue>`

**禁止**: 使用非泛型集合`ArrayList`、`Hashtable`

```csharp
// ❌ 错误 - 装箱拆箱
ArrayList list = new();
list.Add(42);  // int装箱为object
int value = (int)list[0];  // 拆箱

// ✅ 正确 - 使用泛型
List<int> list = new();
list.Add(42);  // 无装箱
int value = list[0];  // 无拆箱
```

**严重性**: HIGH

**原因**: 装箱产生GC分配，拆箱有性能损失

**相关**: [最佳实践-GC优化清单](../30_性能优化/内存管理/最佳实践-GC优化清单.md)

## 11. 代码安全规则

### RULE-SAFE-001 公共方法必须进行null检查

**条件**: 公共API接收可能为null的参数时

**必须**: 在方法开始检查null并抛出ArgumentException

**禁止**: 直接使用可能为null的参数

```csharp
// ❌ 错误 - 没有null检查
public void SetTarget(Transform target)
{
    target.position = newPosition;  // NullReferenceException
}

// ✅ 正确 - null检查
public void SetTarget(Transform target)
{
    if (target == null)
        throw new ArgumentNullException(nameof(target));

    target.position = newPosition;
}
```

**严重性**: HIGH

**原因**: 早期失败比隐藏的NullReferenceException更好调试

---

### RULE-SAFE-002 使用?.操作符处理可能为null的对象

**条件**: 访问可能为null的对象成员时

**必须**: 使用`?.`或`??`操作符

**禁止**: 先null判断再访问成员

```csharp
// ❌ 错误 - 冗长的null检查
if (player != null && player.health != null)
{
    player.health.value = 100;
}

// ✅ 正确 - 使用?.操作符
player?.health?.SetValue(100);

// ✅ 配合??提供默认值
string playerName = player?.name ?? "Unknown";
```

**严重性**: MEDIUM

**原因**: 代码更简洁，减少嵌套

---

### RULE-SAFE-003 避免在Update中抛出异常

**条件**: Update等方法可能遇到错误情况时

**必须**: 使用条件判断避免异常

**禁止**: 使用try-catch捕获Update中的异常

```csharp
// ❌ 错误 - Update中抛异常
void Update()
{
    var enemy = enemies[currentIndex];
    enemy.Attack();  // 可能NullReference或IndexOutOfRange
}

// ✅ 正确 - 条件检查
void Update()
{
    if (currentIndex >= 0 && currentIndex < enemies.Count)
    {
        var enemy = enemies[currentIndex];
        if (enemy != null)
        {
            enemy.Attack();
        }
    }
}
```

**严重性**: CRITICAL

**原因**: 异常处理开销极大，Update中的异常会严重影响帧率

---

### RULE-SAFE-004 使用TryGet模式代替直接访问

**条件**: 方法可能失败时（如查找、解析）

**必须**: 使用`TryXxx(out result)`模式

**禁止**: 抛异常表示失败

```csharp
// ❌ 错误 - 找不到时抛异常
public Player GetPlayer(int id)
{
    if (!players.ContainsKey(id))
        throw new KeyNotFoundException();

    return players[id];
}

// ✅ 正确 - TryGet模式
public bool TryGetPlayer(int id, out Player player)
{
    return players.TryGetValue(id, out player);
}

// 使用时
if (TryGetPlayer(playerId, out var player))
{
    player.DoSomething();
}
```

**严重性**: MEDIUM

**原因**: TryGet模式更符合Unity风格，性能更好
