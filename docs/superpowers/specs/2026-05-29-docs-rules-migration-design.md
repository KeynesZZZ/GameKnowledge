## 3. 执行阶段

### 阶段 0：冻结基线

**目标：** 确认工作区状态，记录当前基线。

**操作：**
```bash
git status --short
git log --oneline -5
```

**验收：** 确认未提交变更来源，记录基线 commit hash。

| `00_元数据与模板/【模板】代码审查清单.md` | 同上 |
| `00_元数据与模板/【模板】代码片段.md` | 同上 |

### 阶段 2：脚本批量迁移

**目标：** 227 个文档的 frontmatter 全部标准化。

**步骤：**

1. **审查迁移脚本**：读取 `migrate_docs_to_latest_rules.py`，确认逻辑与最新规则一致。如不一致，先修复脚本。
2. **运行迁移脚本**：
   ```bash
   python3 UnityKnowledge/scripts/migrate_docs_to_latest_rules.py
   ```
3. **运行元数据校验**：
   ```bash
   python3 UnityKnowledge/scripts/validate_metadata.py --strict
   ```
4. **修复校验错误**：如有错误，针对性修复后重跑。

**脚本负责的修改：**
- 补缺失 frontmatter
- 补齐 9 个必填字段
- 映射旧状态到新状态
- 补 `validation` 默认值
- 补 `related: []`
- 推断 category（根据目录结构）
- 补最小「文档定位」段落（机械模板）
- 补「相关链接」段落（空列表）
- 更新 `updated` 时间戳

**验收：**
```text
validate_metadata.py --strict → 错误: 0, 警告: 0
```

### 阶段 3：Claude 智能补结构

**目标：** 将机械模板内容替换为有意义的定位和链接。

**按目录优先级处理：**

| 优先级 | 目录 | 文档数 | 策略 |
|--------|------|--------|------|
| P0 | `00_元数据与模板` | 19 | 人工级质量，逐文件精修 |
| P1 | `10_架构设计` | 40 | 批量读取，根据内容推断 |
| P2 | `20_核心系统`（含子目录） | 65 | 同上 |
| P3 | `30_性能优化`（含子目录） | 55 | 同上 |
| P4 | 其他（25, 35, 36, 40, 50, 60, 90, 100） | 48 | 同上 |

**每个目录的 Claude 操作：**
1. 批量读取目录内所有文档
2. 识别每个文档的缺失和质量问题
3. 生成有意义的「文档定位」说明（基于文档类型和内容）
4. 填充 `related` 字段和「相关链接」段落（2-3 个相关文档）
5. 写入修改

**质量标准：**
- 文档定位不超过 3 句话，说清「是什么、给谁看、解决什么问题」
- 相关链接至少 2 个，且指向真实存在的文档
- 无机械重复文本

### 阶段 4：修断链

**目标：** 内部链接检查归零。

**步骤：**

1. **运行链接检查**：
   ```bash
   python3 UnityKnowledge/scripts/check_links.py
   ```
2. **Claude 逐条修复**，按断链类型处理：
   - 模板占位链接 → 改成普通文本占位
   - URL 编码路径 → 改为真实文件名
   - 文档改名 → 指向真实存在文档
   - 文档确实缺失 → 创建 Inbox 草稿或改"待整理"文本
3. **循环直到归零**：重跑检查，修复新发现的断链，直到 0 断链。

**验收：**
```text
check_links.py → 断链: 0
```

### 阶段 5：质量修复

**目标：** 修正脚本无法处理的内容质量问题。

**Claude 逐目录扫描，修复以下问题：**

| 问题类型 | 修复方式 |
|----------|----------|
| 标题前缀不匹配 | 根据内容判断正确前缀，统一标题（frontmatter title + H1 + 文件名） |
| H1 与 title 不一致 | 对齐到 frontmatter title |
| 空 wiki-link（`、、`） | 删除或替换为实际链接 |
| `related` 仍为空 | 基于目录上下文填充 |
| `status` 过于乐观 | 降级到合理状态（默认 `待验证`） |
| `validation` 过于乐观 | 降级到合理值（默认 `未经测试`） |
| `相关链接` 段落内容错误 | 清理并填入正确链接 |
| 自引用链接 | 修复为指向其他文档 |

**处理策略：** 按阶段 3 的目录优先级，Claude 批量读取每个目录的文档，一次性识别和修复所有质量问题。

### 阶段 6：最终校验

**目标：** 全库通过所有自动化检查。

**命令：**
```bash
python3 UnityKnowledge/scripts/validate_metadata.py --strict
python3 UnityKnowledge/scripts/check_links.py
```

**验收标准：**

| 检查项 | 目标 |
|--------|------|
| 元数据错误 | 0 |
| 元数据警告 | 0 |
| 内部断链 | 0 |
| 正式文档缺少「文档定位」 | 0 |
| 正式文档缺少「相关链接」 | 0 |

**抽查重点目录：** 00、10、30、20、40，确认：
- 无机械重复内容影响阅读
- 标题前缀合理
- category 符合目录
- status/validation 不过于乐观

## 4. 风险与缓解

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 迁移脚本与最新规则不一致 | 脚本可能引用旧字段名或旧映射 | 阶段 2 前先审查脚本逻辑 |
| Claude 智能填充 related 不准确 | 自动推断的关联可能不够精确 | 后续人工优化高频文档 |
| 大量 diff 难以审阅 | 227 个文件同时修改 | 先看统计摘要，再抽查重点 |
| 修改引入新断链 | 补充的链接可能指向不存在的文档 | 每轮修改后重跑 check_links |
