# 【综述】热更新面试问答

> 热更新是 2026 北京游戏客户端岗的**高频硬性要求**（HybridCLR + Addressables 几乎是标配考点）。本文是 [[【综述】高级Unity游戏开发面试复习地图]] 域三的深入版，与 [[【设计原理】热更新方案对比]]（真相层）严格对齐。`#面试` `#综述`
>
> 核心心法：**代码热更与资源热更是两条正交轴**，别混着讲。选型题答「场景约束 + 权衡」，机制题答「为什么 iOS 不能 JIT」。

---

## 一、概念基础

### Q1. 为什么游戏需要热更新？iOS 为什么会限制？

**要点**：
- **为什么需要**：绕过商店审核（苹果审核数天~数周且可能拒审）、快速修 bug、活动/内容迭代、A/B 测试与运营调优。
- **iOS 限制根因**：iOS 强制 **W^X（Write XOR Execute）内存保护**——一块内存「可写」就「不可执行」，**禁止 JIT**（JIT 需要把生成代码写进可执行内存）。所以 iOS 上代码不能现场编译执行，只能：① 预先 AOT 编译；② 用**解释器**跑字节码（Lua / ILRuntime）；③ HybridCLR 式「AOT 主体 + 解释器补充」。

**高频追问**：为什么 Android 没这问题？→ Android 允许 JIT，Mono/LuaJIT 可用 JIT 加速；但 Android 现也默认 IL2CPP（AOT）。

> 关联：[[【设计原理】热更新方案对比]]

---

### Q2. 代码热更和资源热更是什么关系？

**要点**：**两条正交轴，组合使用**。
- **资源热更**：贴图/模型/配置/预制体/音频等，载体是 **AssetBundle / Addressables**。几乎所有商业游戏都做。
- **代码热更**：C# 逻辑的更新，载体是 **HybridCLR / ILRuntime / Lua（xLua·toLua）**。受 iOS JIT 限制，方案选型更复杂。
- 一套完整热更 = 资源热更（AB/Addressables）+ 代码热更（其一）。两者**独立选型**（如 Addressables + HybridCLR）。

**高频追问**：只更资源不改代码行不行？→ 行，但绝大多数 bug 修复和玩法调整都涉及代码，所以代码热更几乎是刚需。

> 关联：[[【设计原理】热更新方案对比]] [[【教程】打包与热更新]]

## 二、资源热更（AssetBundle / Addressables）

### Q3. AssetBundle 打包策略与分组原则？

**要点**（分组原则，来自 [[【综述】Unity客户端面试题]] §26）：
1. 常更新与不常更新的分开（减小热更包）
2. 需同时加载的资源放一起（提高加载效率）
3. 被多包共享的公共依赖单独成包（避免冗余打包）
4. 同时加载的零碎小资源合并

**高频追问**：依赖冗余是怎么产生的、怎么发现？→ 一个资源被多个 AB 引用且未单独抽包时，会被分别打进每个 AB（体积翻倍）。用 **AssetBundle Analyzer / BuildReport / `AssetDatabase.GetDependencies`** 分析依赖树发现冗余。详见 [[【综述】Unity资源依赖与打包陷阱]]。

> 关联：[[【教程】打包与热更新]] [[【综述】Unity资源依赖与打包陷阱]]

---

### Q4. AB 压缩格式怎么选？LZMA vs LZ4

**要点**：
- **LZMA**：压缩率最高（包最小），但**流式压缩**，读任意资源需整包解压 → 适合**首包 / 整包下发**（省流量，不在运行时读）
- **LZ4 / LZ4HC**：块压缩，支持分块随机访问（seek），解压快 → 适合**热更 / 运行时按需加载**
- **最佳实践**：CDN 传 LZMA 省流量，Unity Caching 下载到本地解压缓存为可随机访问形式，后续 LoadFromFile 走 LZ4 思路——**传输用 LZMA、本地用 LZ4**，体积与加载兼得。

**高频追问**：为什么热更几乎都选 LZ4？→ 运行时按需加载某个资源，LZ4 只解压对应 chunk，LZMA 要从头解整包。

> 关联：[[【综述】Unity客户端面试题]] §26

---

### Q5. Addressables 和裸 AssetBundle 的区别？要注意什么？

**要点**：Addressables 是 AB 之上的**管理框架**，自动处理依赖图、引用计数、异步加载、分组配置，不用手写依赖管理与版本比对。代价是有学习成本、对底层 AB 不透明、早期版本有坑。

**使用注意**（高频追问）：
- **引用计数**：`LoadAssetAsync` +1 / `Release` -1，归零才卸载——漏 Release 泄漏，多 Release 崩溃
- **异步句柄**：注意完成时机与异常；避免 `.Result` 同步阻塞
- **分组即加载单元**：同组资源会一起进内存，按使用场景分组
- **同地址多资源 / Label**：理解 Label 的批量加载语义

> 关联：[[【教程】资源管线-Addressables]] [[【最佳实践】Addressables性能优化]]

## 三、代码热更方案对比（核心考点）

### Q6. 对比 HybridCLR / ILRuntime / xLua / toLua，怎么选？

**要点**（与 [[【设计原理】热更新方案对比]] 对齐）：

| 方案 | 执行机制 | 性能 | 侵入性 / 代价 | 适用 |
|---|---|---|---|---|
| **HybridCLR** | 扩充 il2cpp runtime，AOT 主体 + 解释器补充元数据 | 接近原生 C#（最佳） | 低侵入；代价在**构建管线 + 泛型补充 AOT 元数据** | 想用纯 C# 全量热更、追求性能与工作流统一 |
| **ILRuntime** | C# IL → 自研解释器，独立 AppDomain | 中（低于原生） | 需生成适配器/委托桥接；跨域调用有开销 | C# 热更、不想引入 Lua；老项目多 |
| **xLua** | Lua 热更业务；**Hotfix 可注入式热修已上线 C#** | LuaJIT 快（iOS 解释器慢） | 业务逻辑要拆到 Lua；Hotfix 需打标签 | 腾讯系生态、需要热修 C# + Lua 业务 |
| **toLua#** | 纯 Lua 业务，C#↔Lua 互调 | LuaJIT 快（iOS 解释器慢） | 业务大量迁 Lua；**不可热修 C# 接口** | 老牌重度 Lua 项目 |

**选型决策**（高级岗要有自己判断）：
- 新项目 / 想统一 C# 工作流 / 性能敏感 → **HybridCLR**（2026 主流）
- 老项目已重度 Lua → xLua / toLua
- 想热修线上 C# 但不全量 Lua → xLua Hotfix
- 团队不愿引入 Lua 且可接受解释器性能 → ILRuntime（新项目已被 HybridCLR 挤占）

**高频追问**：
- **HybridCLR 为什么低侵入却要折腾构建管线？** → 它复用 il2cpp，但需要补充 AOT 元数据（处理泛型实例化等 AOT 无法覆盖的部分），构建期要生成并打包这些补充元数据。
- **LuaJIT 在 iOS 为什么慢？** → iOS 禁 JIT，LuaJIT 退化到解释器模式（DyFLO），性能优势消失；基准比较要以 iOS 解释器模式为准。

> 关联：[[【设计原理】热更新方案对比]]

### Q7. 为什么 HybridCLR 能做到「近乎原生 C# 热更」？代价是什么？

**要点**：HybridCLR 没有另起炉灶搞新语言/新虚拟机，而是**扩充 il2cpp 的运行时**，让原本纯 AOT 的 il2cpp 变成「AOT + 解释器」混合：主体热更程序集的 IL 由解释器执行，AOT 部分仍原生跑。因此：可以用全部 C# 特性、工作流与普通 C# 开发几乎一致、性能接近原生。

**代价**：① 构建管线复杂（补充元数据、打包流程）；② 泛型等需要 AOT 元数据补充，配置不对会运行时报错；③ 解释执行的热更代码性能略低于 AOT（但远好于 Lua/ILRuntime）。

> 关联：[[【设计原理】热更新方案对比]]

## 四、tolua / Lua 深入（若项目用 Lua）

### Q8. toLua# 的调用机制？C#↔Lua 边界为什么有开销？

**要点**：toLua# 通过 **Wrap（绑定代码生成）** 把 C# 类/方法包成 Lua 可调用形式，`LuaState` 管理 Lua 虚拟机。C# 调 Lua / Lua 调 C# 都要跨边界：参数/返回值要转换类型（值类型装箱、字符串拷贝等）、压栈弹栈。

**GC 来源**（高频追问）：跨边界传递的值类型装箱、`string` 拷贝、委托/回调包装类生成、临时 table。优化见 [[【笔记】tolua性能与GC优化]]。

> 关联：[[【笔记】tolua入门与调用机制]] [[【笔记】tolua性能与GC优化]]

---

### Q9. tolua 热更落地的高频坑？

**要点**（来自 [[【踩坑】tolua热更新常见坑]]）：Wrap 缺失（C# 反射变更后没重新生成 Wrap）、LuaJIT 平台限制（iOS 无 JIT）、C# 签名变更导致 Lua 调用崩、泛型/ref/协程在 Lua 侧的处理陷阱、C#↔Lua 互相持有导致的内存泄漏。

> 关联：[[【踩坑】tolua热更新常见坑]] [[【实战案例】tolua热更新系统完整落地]]

---

### Q10. Lua 面向对象为什么容易踩坑？

**要点**：Lua OOP 基于 table + metatable `__index`，**核心坑是读写不对称**——`__index` 是**读委托**（读属性沿原型链找），但**写只落在实例本身**。导致「子类写一个同名属性，不会改父类，但读取行为可能与预期不符」。`.` vs `:`（是否自动传 self）、继承链委托方向也易错。

**高频追问**：`table.sort` 是什么排序？→ 快排（非稳定）；大量字符串拼接用什么？→ `table.concat`（字符串不可变，`..` 是 O(n²)）。详见 [[【综述】Unity客户端面试题]] §四。

> 关联：[[【笔记】Lua面向对象深入]] [[【笔记】Lua语法速查]]

## 五、热更工程流程

### Q11. 讲讲一套完整热更的运行时流程？

**要点**（端到端时序，参考 [[【实战案例】tolua热更新系统完整落地]]）：
1. **版本比对**：客户端请求服务器版本清单（manifest），与本地比对，算出需更新的资源/代码包列表
2. **下载**：按列表下载（断点续传、并发、失败重试）
3. **校验**：MD5/Hash 校验完整性，失败重下
4. **加载生效**：资源写入 Caching/本地，代码热更模块 Reload/加载；旧资源清理
5. **回滚**：校验或加载失败要能回滚到上一可用版本

**高频追问**：怎么保证热更失败不把游戏搞挂？→ 双版本/备份目录、加载失败回滚、关键阶段有兜底；灰度发布（先小比例用户）。

> 关联：[[【实战案例】tolua热更新系统完整落地]] [[【教程】打包与热更新]]
