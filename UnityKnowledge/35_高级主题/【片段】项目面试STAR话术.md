# 【片段】项目面试 STAR 话术

> ⚠️ **重要前提**：面试项目故事**必须是你的真实经历，量化指标必须真实**。vault 中这两个项目的复盘目前是骨架（无实测数据），所以本文只提供**基于已记录项目方向的 STAR 叙事骨架 + 技术点 + 高频追问**；凡标 `[待填]` 处请用你自己的真实数据替换。`#面试` `#片段`
>
> 本文是 [[【综述】高级Unity游戏开发面试复习地图]] §十的展开，与 [[【综述】性能优化面试问答]] 配套。

## STAR 方法论（高级岗要点）

- **S**ituation 情境：项目类型、规模、团队、遇到的具体问题（1–2 句，别铺垫太长）
- **T**ask 任务：你的职责 + 目标（**带量化目标**：如「把启动时长从 X 降到 Y」「支撑 N 个平台」）
- **A**ction 行动：你**具体**做了什么决策、用了什么技术、为什么这么选、踩了什么坑（**核心，占 60%**）
- **R**esult 结果：**量化收益**（指标前后对比）+ 沉淀（规则/框架/文档）

**高级岗红线**：
1. **能量化的必须量化**——「优化了启动」是废话，「冷启动 20s→5s，首场景加载 8.5s→1.2s」才是高级表达
2. **讲权衡不讲万能**——「为什么选 A 不选 B、代价是什么」比「我用了 A」值钱十倍
3. **不夸大、不冒领**——没做过的不说；如果是参考方案/团队成果，明确「我负责的是 X」
4. **能被深挖**——每个 Action 都要能扛住 3 层追问（机制 → 权衡 → 踩坑）

## 故事一：休闲游戏云存档系统（架构 / 冲突解决 / 多平台）

> 技术点来源：[[100_项目实战/01_休闲游戏云存档系统/README]] 各章 + [[【实战案例】休闲游戏云存档系统复盘]]。这是高级岗**系统设计 + 工程落地**的好故事。

**S 情境**：一款面向多端的休闲游戏（微信小游戏 / 抖音小游戏 / 海外平台 / App，`[待填：具体平台与 DAU 量级]`），玩家跨设备切换频繁，原有存档方案只在本地，换设备就丢档，客诉集中在「进度丢失」。

**T 任务**：我负责设计并落地一套统一云存档系统，目标是：① 多端数据一致；② 支持离线游玩、上线自动同步；③ 平台差异对业务层透明；④ 妥善处理多端并发修改的冲突。（`[待填：你的具体职责边界、是否主程]`）

**A 行动**（基于 vault 已记录的设计，按你的真实实现调整）：
- **Provider 接口抽象**：用 [[04-Provider接口设计]] 把各平台存档能力抽象成统一接口，业务层只面对 `ISaveProvider`，新增平台只实现接口（开闭原则）。参考 [[03-平台检测系统]] 做运行时平台判定。
- **核心数据结构**：设计带版本号/时间戳的存档结构 [[02-核心数据结构]]，为冲突检测与合并打基础。
- **本地缓存与离线**：[[08-本地缓存与离线]]——离线时写本地，上线后比对版本同步，保证离线可玩。
- **冲突解决策略**：[[09-冲突解决策略]]——这里要能讲清你的策略（最后写赢 LWW / 字段级合并 / 让玩家选？时间戳怎么对齐？），这是面试官最爱深挖的点。
- **统一管理器**：[[10-统一管理器实现]] 收口存档生命周期，对接各 Provider。

**R 结果**：`[待填：换设备丢档客诉下降 X%、同步成功率 Y%、接入新平台从 N 天降到 M 天]`；沉淀为可复用的存档模块 + Provider 模式，后续项目可直接套用。

**🔥 高频追问**：
1. **冲突到底怎么解决的？** → 讲清策略（多数休闲游戏用 LWW + 关键资源字段级合并），时间戳用服务器时间避免客户端时钟漂移；极端冲突回退到「保留两份让玩家选」。
2. **离线玩了很久再上线，数据怎么合？** → 本地版本 vs 云端版本比对，按冲突策略走；离线期间的「增量操作」若记录了 op log 可重放合并。
3. **多平台差异怎么屏蔽的？** → Provider 模式 + 平台检测，业务层无感；微信/抖音/海外的存储 API、鉴权、限额差异都封装在各自 Provider 里。见 [[06-微信小游戏实现]] [[07-抖音小游戏实现]] [[05-海外平台实现]]。
4. **存档防作弊 / 加密吗？** → `[按你的实现答：是否签名/加密、防篡改校验]`
5. **数据量 / 存档体积大了怎么办？** → 增量同步、压缩、分槽、限制单档大小。

**一句话电梯版**（30 秒口述）：「我设计了一套基于 Provider 模式的多端云存档系统，把微信/抖音/海外的平台差异封装在接口实现里，用版本号 + LWW 解决跨设备冲突，离线可玩上线自动同步，让业务层完全不感知平台差异，新平台接入只需实现一个接口。」

## 故事二：休闲游戏框架（架构 / 复用 / 降本）

> 技术点来源：[[100_项目实战/02_休闲游戏框架/README]] + [[【实战案例】休闲游戏框架复盘]]。这是**架构设计与权衡**的好故事。

**S 情境**：团队连续做休闲游戏，每个新项目都从零搭 UI、事件、资源、流程脚手架，重复劳动多、风格不统一、bug 重复犯。

**T 任务**：我主导搭建一套休闲游戏基础框架，目标是降低新项目启动成本、统一 UI/事件/资源/流程能力。`[待填：团队规模、覆盖项目数]`

**A 行动**（按你的真实实现调整）：
- **UI 框架**：[[01-UI框架设计]]——用 MVP / MVVM 分层（参考 [[【架构决策】UI架构-MVP vs MVVM]]），数据与显示分离，支持列表虚拟化、界面栈管理、显隐控制。
- **事件系统**：[[02-事件系统设计]]——类型安全的全局事件总线解耦模块（参考 [[【代码片段】事件总线]]），**重点讲怎么防订阅泄漏**（自动注销、生命周期绑定）。
- **取舍**：明确哪些抽象保留、哪些**不过早抽象**（复盘里提到「抽象过重/过早」是坑——讲一个你砍掉的过度设计，体现克制）。

**R 结果**：`[待填：新项目启动周期从 N 周降到 M 周、复用到 X 款游戏、通用 bug 下降 Y%]`；沉淀为团队规范 + 代码片段库（呼应 [[10_架构设计]] 的设计模式专题）。

**🔥 高频追问**：
1. **为什么不直接用 ET / GameFramework / ILRuntime 框架？** → 讲权衡：自研框架轻量可控、贴合团队休闲游戏场景、学习/维护成本低；现成框架功能多但包袱重、耦合自己的流程难。这是高级岗经典题，必须有自己判断。
2. **UI 框架用的什么架构模式？为什么？** → MVP（View 被动、Presenter 处理逻辑）或 MVVM（数据绑定），讲清你选的理由和代价。见 [[【教程】MVP模式深入讲解]]。
3. **事件系统怎么避免内存泄漏？** → 订阅必须配对取消、与 GameObject 生命周期绑定、用弱引用或统一注销点；讲一个「忘了取消订阅导致泄漏」的踩坑与修复。
4. **模块依赖怎么管？** → 单例 vs 依赖注入（[[【架构决策】单例vs依赖注入]]）、是否有循环依赖、初始化顺序怎么保证。
5. **接入新玩法的扩展性如何？** → 举一个后来加玩法的例子，证明框架扩展点设计合理（而不是到处改核心）。

**一句话电梯版**：「我主导设计了一套休闲游戏框架，用 MVP + 类型安全事件总线统一了 UI 和模块通信，把每个新项目的启动脚手架从重复搭建变成配置接入，并通过订阅生命周期管理根治了事件泄漏类 bug。」

## 通用准备清单

### 技术名词（被问到要能秒答）
Provider 模式、MVP/MVVM、事件总线、对象池、LWW（最后写赢）、增量同步、版本号冲突检测、订阅生命周期、开闭原则。

### 量化指标库（回忆你自己的真实数字，填进 STAR）
- 存档：换设备丢档客诉下降 %、同步成功率、新平台接入天数
- 框架：新项目启动周期 N→M 周、复用项目数、通用 bug 下降 %
- 性能：fps / 启动时长 / 内存 MB / DC 数 / GC B，前后对比

### 被深挖 / 被质疑时的应对
- **没做过的细节被追问**：坦诚「这部分是团队 XX 负责的，我了解的是……」，不要硬编。
- **被问「为什么不用 X 方案」**：先肯定 X 的合理性，再讲你的场景约束（团队/周期/规模）下为什么 Y 更合适——体现权衡而非偏见。
- **被指出坑**：承认 + 讲修正 + 讲沉淀（这是加分项，不是减分项）。
