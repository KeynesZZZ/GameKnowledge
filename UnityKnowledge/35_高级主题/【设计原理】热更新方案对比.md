# 热更新方案对比

> 从**底层执行机制**对比 Unity 主流代码热更方案，服务于技术选型。

## 文档定位

从底层机制角度对比 Unity 主流**代码热更**方案（HybridCLR、ILRuntime、xLua / toLua#、puerts）的原理、优劣势与适用场景，帮助团队做技术选型。本文只讲「为什么、怎么选」，具体「怎么接入」交给周边教程：

- 资源热更的完整流程 → [[../40_工具链/【教程】资源管线-Addressables]]
- HybridCLR 集成代码、版本管理、多平台打包 → [[【教程】打包与热更新]]
- toLua# 绑定机制 / 性能 / 踩坑 → [[tolua专题索引]]

---

## 一、概述

热更新（hot update）指**不重新发布 App（不上架/不强制整包更新）**的情况下，更新游戏内容。它解决三类需求：

1. **活动与内容迭代**：节日活动、新关卡、新卡牌——等不及 1～7 天的审核。
2. **紧急修 bug**：线上崩溃、数值错误，几小时内修复，无需走商店审核。
3. **A/B 与灰度**：分批放量、可回滚的渐进式发布。

但「热更新」被笼统使用时，其实混了**两条正交的轴**。本文的核心修正就是把它们拆开。

---

## 二、两条正交的轴：资源热更 vs 代码热更

```
                    代码热更：否              代码热更：是
              ┌──────────────────────┬──────────────────────┐
  资源热更：否 │  纯整包更新            │  仅换逻辑脚本（极少见） │
              ├──────────────────────┼──────────────────────┤
  资源热更：是 │  只更贴图/配置/关卡/音效 │  完整热更（主流形态）   │
              └──────────────────────┴──────────────────────┘
```

- **资源热更（AssetBundle / Addressables）= 基座层**。任何方案都要先有它：贴图、模型、配置、音频、场景这些「数据」要能从 CDN 下发替换。这是 Unity 官方原生能力，详见 [[../40_工具链/【教程】资源管线-Addressables]]。
- **代码 / 逻辑热更 = 本文对比主体**，也是真正难的部分：如何在不重发 App 的前提下，更新**可执行的逻辑**。

> ⚠️ 旧版本文档把 AssetBundle / Addressables（资源）和 Lua / ILRuntime / HybridCLR（代码）放在同一张表里对比，混淆了这两条轴——它们不是竞争关系，而是「基座 + 上层」的关系。资源热更细节见 Addressables 教程，本文不再展开。

## 三、为什么代码热更难（根因）

理解方案差异前，必须先理解一个**平台根因**，它决定了所有方案的形态：

1. **iOS 禁止 W^X 内存（Writable XOR Executable）**：系统不允许同一块内存「既可写又可执行」。这意味着无法在运行时生成可执行的机器码 → **JIT（即时编译）在 iOS 上被禁用**。
2. **Unity 在 iOS / IL2CPP 走 AOT（Ahead-Of-Time）**：C# 被先编译成 IL，再由 IL2CPP 翻译成 C++、编译成本地机器码。首发包里的代码是「提前编译好」的。
3. **推论**：「下载一个 DLL，运行时 JIT 执行」在 iOS 上**行不通**。

因此在 iOS 上，所有代码热更方案本质上只能落到两类：

- **带一个解释器**，运行时解释字节码 / IL（Lua VM、ILRuntime、HybridCLR 在 IL2CPP 之上挂的解释器模块）；
- **用本就有解释器的脚本语言**承载逻辑（Lua / JS）。

> Android 理论上可以 JIT，但工程上几乎所有团队都走 IL2CPP + 解释器方案，以**保持 iOS / Android 双端一致**，避免维护两套行为。所以下文一律以「iOS 解释器模式」为基准。

## 四、三类方案的执行机制

### 4.1 Lua 系（xLua / toLua# / SLua）

**执行模型**：宿主是 C#（Unity），进程内嵌入一个 **Lua / LuaJIT 虚拟机**。热更逻辑用 Lua 写，跑在 Lua VM 里；C# 与 Lua 通过 **Wrap（构建期生成的绑定胶水代码）+ 栈交互**双向互调。

- **LuaJIT 平台差异（关键）**：Android / Editor / Windows 上 LuaJIT 可开 JIT，Lua 执行接近原生；**iOS 上 JIT 被禁，退回解释器模式**。详见 [[【笔记】tolua性能与GC优化]]「LuaJIT 与平台差异」。
- **xLua 独有的 Hotfix**：构建期用 **IL 注入**往 C# 方法里插跳板，运行时可把某个 C# 方法体**重定向到 Lua 实现** → **能热修已打包进首发包的 C# bug**。这是 xLua 相对 toLua# 的决定性差异。
- **toLua#（tolua）的边界**：只能热更「Lua 逻辑」，**Wrap 是编译期生成的 C# 代码，本身不可热更**；C# 接口签名一变，已发布的旧 Lua 就对不上（呼应 [[【踩坑】tolua热更新常见坑]] 坑 3）。

| | 优点 | 缺点 |
|---|---|---|
| Lua 系 | 成熟、生态广、Lua VM 体积小启动快；跨边界开销可控 | 双语言双工程；跨边界调用有开销 + GC（[[【笔记】tolua性能与GC优化]]）；iOS 无 JIT 性能打折；调试体验弱于 C# |

**适用**：团队已有 Lua 栈 / Lua 强；或已上线项目需要修首发包 C# bug（xLua Hotfix）。

### 4.2 ILRuntime

**执行模型**：一个**纯 C# 实现的 IL 解释器**。热更 DLL 编译成 IL，运行时由 ILRuntime（它自己也是 C#）逐条解释执行；通过**反射 / 委托适配器（Adapter）**与主工程互调。

- **优点**：全 C# 开发、不学 Lua、可与主工程共享代码、接入相对简单。
- **缺点**：**解释执行开销大**（自己解释 IL，比原生慢约一个数量级）；反射式绑定带来跨边界性能与 GC 压力；泛型 / 反射 / `async` 等高级特性支持有限、常需手写 Adapter；调试体验一般。
- **趋势**：曾是 C# 热更的主流方案，但新项目多被 HybridCLR 取代——性能与 C# 特性完整度差距明显。老项目维护成本高时，多维持现状、新模块评估迁移。

**适用**：历史遗留 ILRuntime 项目；或对性能不敏感、想用全 C# 但暂时不上 HybridCLR 的场景。

### 4.3 HybridCLR（华佗 / huatuo）

**执行模型**：**不是独立解释器，而是 IL2CPP 的扩展**。IL2CPP 把 C# 翻成 C++/本地码（AOT），HybridCLR 在它之上为「热更程序集」补充一个解释器模块，让热更 DLL 作为**完整受管代码**加载执行——支持泛型、反射、`async`、多线程等近乎全部 C# 特性，且**近原生性能**。

- **关键工程点：泛型补充 AOT 元数据**。AOT 程序集里用到的泛型，需要构建期生成、运行时 `LoadMetadataForAOTAssembly` 加载「同源映像（homologous images）」，否则泛型实例化会抛 `MissingMethod` 异常。完整集成代码见 [[【教程】打包与热更新]] 的 HybridCLR 段。
- **优点**：全 C#、单语言、近原生性能、C# 特性几乎完整、**侵入低**（无需学新语言，热更程序集与主工程同语言同结构）。
- **缺点**：构建管线复杂（要生成 AOT 元数据、处理泛型补充与剪裁）；IL2CPP 编译 / 打包耗时增加；有一定学习与踩坑成本。

> ✅ **纠偏**：旧版文档写「HybridCLR 侵入性强」**不准确**。相比 Lua 它侵入性反而**更低**（同语言、无脚本层）；真正的代价是**构建管线复杂 + 泛型需补 AOT 元数据**。

**适用**：**当前新项目代码热更的主流推荐**。

### 4.4 补充：puerts

**执行模型**：嵌入 **V8（或 QuickJS）JS 引擎**，热更逻辑用 **TypeScript / JavaScript** 写，常与 React 风格 UI 框架配合。

- **优点**：TS/JS 生态、前端团队友好、可复用 npm 生态。
- **缺点**：JS 引擎体积大（V8）、GC 与 Unity 不同步、跨边界开销、与 C# 工程整合需磨合。
- **适用**：团队 Web / 前端背景强，或要做 TS 驱动的 UI 框架。

## 五、多维对比表

| 维度 | xLua / toLua#（Lua） | ILRuntime | HybridCLR | puerts（TS/JS） |
|------|----------------------|-----------|-----------|-----------------|
| 执行性能 | LuaJIT JIT 强（Android）；iOS 退解释器 | 解释执行，开销大（最慢之一） | **近原生（最快）** | V8 较快，但 GC/跨边界有成本 |
| 开发语言 | Lua + C#（双语言） | 全 C# | 全 C# | TS/JS + C# |
| 学习成本 | 中（Lua + Wrap 机制） | 低–中 | 中（构建管线 + 泛型补充） | 低（前端背景）/ 中 |
| 调试体验 | 一般（Lua 调试器） | 一般 | **好（C# 调试器可直接 attach）** | 一般（devtools 风格） |
| C# 特性完整度 | 低（泛型 / ref / 协程受限） | 中（需手写 Adapter，`async` 弱） | **高（近乎完整）** | 低（JS 语义） |
| iOS 表现 | 无 JIT，解释器模式 | 解释执行 | 近原生 | V8 可用，体积大 |
| 生态 / 维护活跃度 | xLua 腾讯活跃；toLua# 减弱 | 维护放缓 | **高度活跃、增长快** | 活跃（腾讯） |
| 包体影响 | 小（Lua VM 小） | 小–中 | 中（AOT 元数据） | 大（V8） |
| 能否热修**已上线** C# | **xLua Hotfix 可**；toLua# 不可 | 否 | 否 | 否 |

> 注：「能否热修已上线 C#」这一行最需要精确理解。xLua Hotfix 能对**已打包进 AOT 的 C# 方法**做注入式热修（构建期插桩 + 运行时重定向到 Lua）；而 HybridCLR / ILRuntime / puerts 热更的都是**独立的热更程序集 / 脚本里的逻辑**，凡是已编译进首发包 AOT 的 C# 代码无法被「打补丁」，只能通过把逻辑迁移到热更程序集来规避。这是「修已发 bug」场景下 xLua 至今无可替代的原因。

## 六、选型决策矩阵

```
要改的是资源（贴图/配置/关卡/音效）吗？
└─ 是 → Addressables（资源热更即可），不需要代码热更。见 [[../40_工具链/【教程】资源管线-Addressables]]

要改的是逻辑/代码吗？
├─ 新项目、想要全 C# + 近原生性能 + 完整特性
│     └─ HybridCLR（当前首选）
├─ 已有大量 Lua 代码 / 团队 Lua 强
│     └─ xLua（还需热修 C# bug 时）或 toLua#（纯 Lua 逻辑热更）
├─ 已上线、需要修首发包里 C# 的 bug
│     └─ xLua Hotfix（唯一能注入式热修已发布 C#）
├─ 团队 Web/TS 背景、或要做 TS 驱动 UI
│     └─ puerts
└─ 老项目用 ILRuntime、改动成本高
      └─ 维持 ILRuntime；新模块评估迁移 HybridCLR
```

一句话趋势：**新项目默认考虑 HybridCLR；Lua 系靠 xLua Hotfix 在「修已发 bug」场景守住不可替代的一席；ILRuntime 在新项目中逐步淡出。**

## 七、共性工程问题

无论选哪种代码热更方案，都要处理下面这些与方案无关的工程问题：

- **版本管理**：客户端版本 + 资源版本 + 代码版本三套版本号；区分「强制更新」与「可跳过更新」。参考实现见 [[【教程】打包与热更新]]「版本管理」段。
- **DLL / 字节码下发与 CDN**：热更程序集（HybridCLR 的 dll）或 Lua 字节码要随 Addressables / CDN 下发；注意 Lua 字节码跨 LuaJIT 版本、跨 32/64 位字长**不兼容**（[[【踩坑】tolua热更新常见坑]] 坑 4）。
- **HybridCLR 泛型的 AOT 元数据补充**：构建期生成、运行时加载；漏补 → 泛型实例化 `MissingMethod`。必须纳入构建流程并测试覆盖。
- **以 iOS 解释器模式为性能基准**：评估任何方案时，不要拿 Editor / Android 的 JIT / 原生数据外推 iOS（[[【笔记】tolua性能与GC优化]]）。
- **下载加密与完整性校验**：热更包做哈希 / 签名校验防篡改；Lua / dll 可加密下发，客户端解密后加载。
- **灰度与回滚**：分批灰度放量、保留「可回滚到上一可用版本」的能力；客户端持久化当前可用版本标记，加载失败自动回滚。

## 八、最佳实践 DO / DON'T

### DO ✅

- **先分清「资源热更」还是「代码热更」**，再选方案——它们是正交的两层。
- **代码热更性能一律以 iOS 解释器模式为基准**评估。
- **HybridCLR 项目**：确保 AOT 泛型元数据完整补充、纳入 CI 构建流程并加测试覆盖。
- **对外发布的 C# 接口当契约**：破坏性变更带版本兼容，避免 toLua# / 跨版本热更失效。
- **灰度发布 + 回滚预案**：任何一次热更都要能回滚。

### DON'T ❌

- 不要把**资源热更和代码热更混在同一张表**里对比（旧版本文档的错误）。
- 不要拿 **Editor / Android JIT 数据**评估 Lua 系在 iOS 的表现。
- 不要以为 **HybridCLR「侵入性强」**——它侵入低，代价在构建管线与泛型补充。
- 不要在 **toLua# 里假设能热修已发布 C# 方法**——只有 xLua Hotfix 能。
- 不要忘了热更包的**完整性校验与版本兼容**。
