## 一、渲染管线流程

### Q1. 讲讲渲染管线的三大阶段？

**要点**（与 [[【笔记】Unity客户端面试题]] §6 一致）：
- **应用阶段（CPU）**：加载数据到显存 → 设置渲染状态（Shader/光源/材质）→ 发起 DrawCall
- **几何阶段**：顶点着色器（坐标变换：模型→齐次裁剪空间，逐顶点）→ [曲面细分/几何着色器] → 裁剪 → 屏幕映射
- **光栅化阶段**：三角形设置 → 三角形遍历（生成片元）→ 片元着色器（逐片元着色）→ 逐片元操作（模板/深度测试、混合）

**记忆**：应用（CPU 喂数据）→ 几何（决定「在哪」）→ 光栅化（决定「什么颜色 + 能不能看见」）。

> 关联：[[【教程】渲染管线基础]]

---

### Q2. 顶点/片元着色器各能做什么？深度测试在哪一步？

**要点**：
- **顶点着色器**：逐顶点，**必须完成坐标变换**（到齐次裁剪空间），可做逐顶点光照；不能创建/销毁顶点，拿不到顶点间关系
- **片元着色器**：逐片元，决定颜色（纹理采样、光照计算）；不能改片元位置
- **深度测试**：在「逐片元操作」阶段（片元着色器**之后**），决定可见性；Early-Z 在片元着色器**之前**做深度剔除（前提：片元着色器不写深度、关闭 Alpha Test）

**高频追问**：为什么 Early-Z 能省性能？→ 在执行昂贵的片元着色器前就丢弃被遮挡片元。但 Alpha Test / 透明 / 写深度会破坏 Early-Z。

> 关联：[[【教程】HLSL与Shader基础]] [[【教程】URP Lit Shader]]

---

### Q3. 前向渲染 vs 延迟渲染？

**要点**：
- **前向渲染（Forward）**：逐物体逐光源计算光照，物体数 × 光源数 复杂度；支持半透明、MSAA；**光源多了很贵**。移动端主流（URP 默认）。
- **延迟渲染（Deferred）**：先把几何/法线/深度/反射率写到 **G-buffer**，再逐像素统一算光照；光照复杂度与物体数解耦，**支持大量实时光源**；代价：G-buffer 带宽大、不支持半透明（半透明仍需前向）、MSAA 困难、移动端支持差。

**选型**：移动端 / 光源少 → 前向；PC 主机 / 大量光源 → 延迟（或前向+）。

**高频追问**：URP 为什么默认前向？→ 移动端 G-buffer 带宽 + 半透明 + MSAA 限制；URP 也提供 Forward+（分块前向）提升多光源。

> 关联：[[【教程】渲染管线基础]]

## 二、管线选型

### Q4. Built-in / URP / HDRP / SRP 区别与怎么选？

**要点**：
- **Built-in**：老管线，固定，移动端优化弱，逐步淘汰
- **SRP（Scriptable Render Pipeline）**：可编程管线框架，URP/HDRP 都基于它
- **URP**：轻量、快速、**移动端友好**，SRP Batcher 加速；手游首选
- **HDRP**：高端 PC/主机，物理光照、光线追踪，画质最高但重

**选型**：手游/移动+PC 中低端 → **URP**；3A PC/主机 → HDRP；新项目无脑 URP。

> 关联：[[【最佳实践】URP常用配置]] [[【教程】URP Lit Shader]]

---

### Q5. SRP Batcher 是什么？为什么重要？

**要点**：SRP Batcher 是 URP/HDRP 下的批处理优化：**同一 Shader 的不同材质实例**，无需 GPU Instancing 那样要求同网格，就能让 GPU 复用「常量缓冲（CBUFFER）」、跳过重复的 Shader 设置，减少 CPU 渲染状态切换开销。

**前提**：Shader 要兼容 SRP Batcher（用 `CBUFFER_START`/`CBUFFER_END` 包围属性，符合 URP 规范）。

**高频追问**：SRP Batcher 和 GPU Instancing 冲突吗？→ 同一对象一般二选一；SRP Batcher 处理「同 Shader 不同材质/网格」，GPU Instancing 处理「同网格同材质多实例」。

> 关联：[[【代码片段】Shader基础模板]] [[【最佳实践】URP常用配置]]

## 三、Shader 与 HLSL

### Q6. 一个最基本的顶点/片元 Shader 结构？

**要点**：URP 下：
- `Properties`（属性，Inspector 可调）→ `HLSLPROGRAM`/`ENDHLSL`
- 顶点函数：输入位置，乘以 MVP（`UNITY_MATRIX_MVP`）变换到裁剪空间，输出 `SV_POSITION`
- 片元函数：输入插值，采样纹理，输出 `SV_TARGET` 颜色
- 用 `CBUFFER` 包属性以兼容 SRP Batcher；包含 URP shader library（`Core.hlsl` 等）

**高频追问**：让我手写一个？→ 提前练熟：[[【代码片段】Shader基础模板]] 有 URP 模板。能现场写「贴图采样 + tint 颜色」的最简版即可。

> 关联：[[【教程】HLSL与Shader基础]] [[【代码片段】Shader基础模板]] [[【实战案例】自定义Shader]]

### Q7. 描边 / 溶解 / 流光 / 扭曲 大致怎么实现？

**要点**（思路）：
- **描边**：法线外扩法（顶点沿法线膨胀一遍背面渲染纯色，正面叠主体）或后处理法（基于深度/法线边缘检测 Sobel）
- **溶解**：用噪声纹理（或 Dither）与阈值比较，clip 掉 < 阈值的片元，边缘加烧灼颜色
- **流光/扫光**：用 UV 坐标 + 时间函数算一条移动的高亮带，叠加到原色
- **扭曲**：采样时用噪声/法线扰动 UV，再做主纹理采样（水面、热空气）

> 关联：[[【教程】常用Shader技术]] [[【实战案例】自定义Shader]] [[【实战案例】三消游戏视觉效果]]

### Q8. Shader 变体为什么爆炸？怎么治？

**要点**：见 [[【笔记】性能优化面试问答]] Q11——`multi_compile`/`shader_feature` 关键字组合指数增长，导致编译慢、包体大、运行时内存暴涨。治理：`shader_feature_local`、合并关键字、Shader Variant Collection、变体剥离（Stripping）。

## 四、光照与阴影

### Q9. 实时阴影怎么实现的？

**要点**（与 [[【笔记】Unity客户端面试题]] §11 一致）：
- **Pass1 阴影 Pass**：从光源视角渲染，只写深度 → 得 ShadowMap
- **Pass2 主渲染**：片元变换到光源空间，比较其深度与 ShadowMap 对应深度，更远 → 被遮挡 → 在阴影中
- **软化**：PCF（多点采样平均）/ PCSS
- **平行光**：CSM 级联阴影（近处高分辨率、远处低分辨率）

**高频追问**：实时阴影为什么贵？→ 多一遍渲染（Pass1）+ 带宽（ShadowMap 采样）+ PCF 多次采样。移动端常降阴影分辨率、减级联数，或用 baked/贴图圆盘模拟阴影替代。

> 关联：[[【最佳实践】光照与烘焙优化]]

---

### Q10. 移动端光照策略？

**要点**：移动端主光（平行光）用前向渲染 + 简化 BRDF；少用多实时光源；静态光照烘焙到 Lightmap；环境光用 SH（球谐）/ 反射探针；尽量不开实时阴影或大幅降级。

> 关联：[[【最佳实践】光照与烘焙优化]]

## 五、后处理

### Q11. 后处理代价？移动端怎么取舍？

**要点**：后处理（Bloom / DOF / 色调映射 / 抗锯齿 / 屏幕扭曲等）本质是「对整张屏幕的片元再跑一遍或数遍 Shader」——**带宽和 ALU 都很贵**，移动端 GPU 带宽是瓶颈。

**移动端取舍**：开必要的（色调映射、FXAA/TAA 抗锯齿比 MSAA 省），慎用 DOF/Bloom（降分辨率做），注意 LDR/HDR 带宽差异；用 Frame Debugger + Profiler GPU 测每个效果的代价。

> 关联：[[【教程】后处理]]

## 六、ComputeShader / GPU

### Q12. ComputeShader 适合做什么？和普通 Shader 区别？

**要点**：ComputeShader 是**通用 GPU 计算（GPGPU）**，不画像素，而是对任意数据（缓冲、纹理）做并行计算。适合：大规模粒子模拟、流体、植被/草摆动、骨骼蒙皮 GPU 加速、图像处理。区别：它不绑渲染管线阶段，输出到任意 buffer，可读写；编程模型是 thread group / thread。

**高频追问**：什么任务该上 Compute？→ 数据并行、计算量大、规则统一（每个元素独立计算）的任务；不适合需要分支发散严重或需 CPU 读取中间结果的场景（GPU↔CPU 回读慢）。

> 关联：[[【教程】ComputeShader入门]] [[【设计原理】粒子系统GPU加速机制]]

## 七、移动端渲染优化

### Q13. 移动端 GPU 架构（TBDR）对优化有什么影响？

**要点**：移动端 GPU 多为 **TBDR（Tile-Based Deferred Rendering，分块延迟渲染）**——把屏幕分小块（tile），逐块处理，片内高速缓存（tile memory）极快。影响：
- **带宽是瓶颈**（不是填充率），所以要减少纹理采样、降低 overdraw、用压缩格式（ASTC）
- **Alpha blend / Alpha Test 影响大**：blend 占带宽，Alpha Test 破坏 Early-Z（移动端 HSR 隐面剔除失效）
- **深度 PreZ 在移动端收益不如桌面**（TBDR 自带 HSR），盲目开 PreZ 反而可能更慢——这是移动端经典坑

**高频追问**：为什么移动端要特别在意 overdraw？→ TBDR 虽有 HSR，但半透明叠加无法剔除，仍逐层 blend，吃带宽。

> 关联：[[【最佳实践】Overdraw优化实战]] [[【最佳实践】3D渲染优化指南]]

---

### Q14. 渲染优化你一般从哪几方面入手？

**要点**（呼应 [[【笔记】性能优化面试问答]] 域三）：降 DrawCall（合批/SRP Batcher/Instancing）、减 overdraw（遮挡剔除/LOD）、压带宽（纹理压缩 ASTC、关不必要 mipmap）、降 Shader 成本（变体管理、指令数、移动端简化）、光照烘焙替代实时、后处理取舍、Frame Debugger + Profiler GPU 量化定位。

> 关联：[[【教程】Frame Debugger渲染诊断流程]] [[【最佳实践】3D渲染优化指南]] [[【教程】渲染性能优化]]
