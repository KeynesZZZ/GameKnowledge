# 【综述】性能优化面试问答

> 性能优化是高级岗**权重最高、最易拉开差距**的考察域。本文是 [[【综述】高级Unity游戏开发面试复习地图]] 域一的深入版：以「问答 + 高频追问 + 关联笔记」模拟真实面试节奏，每题答到**机制 / 权衡 / 代价**——这是高级与中级的分界。`#面试` `#综述`
>
> 用法：先自答再看要点；高频追问是面试官的下一句，务必能接住。系统学习顺序见 [[【教程】性能优化学习路径]]，全景入口见 [[性能优化专题索引]]。

## 一、CPU 优化

### Q1. DrawCall 是什么？为什么多了会卡？怎么降低？

**要点**：DrawCall 是 CPU 向 GPU 发起的一次绘制命令（指向图元列表，不含材质）。卡顿根因不在 GPU 画不动，而在 **CPU 提交命令 + 切换渲染状态的开销**——每个 DC 前要设置 Shader/材质/贴图等，状态切换成本高，CPU 成为瓶颈。

**降 DC 手段**（按优先级）：
1. **批处理（合批）**：静态批处理（不动合并）、动态批处理（小网格顶点少自动合并）、**GPU Instancing**（同网格同材质多实例）、**SRP Batcher**（URP/HDRP 下同 Shader 实例的常量缓冲复用，不要求同网格）
2. **合图集**：让更多元素共享材质 → 满足合批前提
3. **减少材质数**：合并相似材质

**高频追问**：
- DC 多了怎么定位？→ **Frame Debugger** 逐绘制步看每个 DC 的来源（见 [[【教程】Frame Debugger渲染诊断流程]]）
- SRP Batcher 为什么不要求同网格？→ 它复用的是「同一 Shader 的 CBUFFER 常量」，跳过的是 Shader 设置而非几何合并

> 关联：[[【最佳实践】3D渲染优化指南]] [[【教程】CPU优化技术]]

### Q2. 为什么我的物体没有合批？（合批条件与打断因素）

**要点**：
- **静态批处理**：要求物体勾选 Static（且不影响运行时位置），代价是**内存翻倍**（合并后的大网格 + 原网格都保留）
- **动态批处理**：顶点数有上限（<300 顶点），且每帧 CPU 重新计算变换合并矩阵，**实例多了反而 CPU 开销大**
- **GPU Instancing**：同网格 + 同材质实例，但**不能**用在受不同光源/阴影系数影响的物体上（部分情况）

**打断合批的常见因素**：不同材质/贴图、层级穿插、Mask、阴影、多 Pass Shader、接受光照计算差异。

**高频追问**：为什么 UI 图集也打了，DC 还是多？→ 层级穿插打断；文字字体图集差异；Mask 增加 DC。详见 [[【设计原理】UGUI合批机制深度解析]]、[[【综述】UGUI性能优化实战总览]]。

> 关联：[[【性能数据】UGUI DrawCall影响因素全面测试]] [[【踩坑记录】常见性能陷阱]]

### Q3. Update 里哪些操作最伤性能？怎么优化？

**要点**（热路径红线）：
1. **每帧 GC 分配**：`new`、闭包捕获局部变量、`foreach` 对部分集合装箱（5.5+ 已优化但要警惕）、字符串拼接（`+` / `ToString`）
2. **每帧查找**：`GetComponent` / `FindObjectWithTag` / `Camera.main`（旧版全局查找）——缓存到字段
3. **不必要的计算**：可用协程/事件驱动的逻辑不要放 Update
4. **LINQ**：热路径避免，有 GC 且比手写循环慢

**优化手段**：缓存引用、对象池复用、`StringBuilder` 拼接、用事件/标志位替代轮询、空实现 Update 直接删掉（Unity 会对空 Update 仍有调用开销）。

**高频追问**：怎么知道 Update 在干什么？→ Profiler 的 CPU 模块，按 Hierarchy 深挖某帧；或 `Profiler.BeginSample` 打点。

> 关联：[[【最佳实践】Update优化清单]] [[【踩坑记录】常见性能陷阱]] [[【性能数据】LINQ性能测试]] [[【设计原理】为什么foreach比for慢]]

### Q4. 对象池解决什么问题？什么场景该用 / 不该用？

**要点**：对象池用「空间换时间」复用实例，避免反复 `Instantiate/Destroy`（触发 GC、堆碎片、卡顿），适合**频繁创建销毁**的对象（子弹、特效、UI 列表项、敌人）。

**代价**：常驻内存（对象不释放）、状态重置逻辑、池容量管理。**不该用**的场景：生命周期长、数量稳定的对象（场景建筑、主角）——硬要池化只增加复杂度。

**高频追问**：池对象取出时要做什么？→ 完整重置状态（位置/速度/HP/事件回调清空），否则出现「脏数据」bug。

> 关联：[[【教程】对象池模式]] [[【代码片段】对象池通用实现]] [[【设计原理】对象池本质]] [[【性能数据】对象池vs实例化]]

## 二、内存与 GC

### Q5. Unity 的 GC 机制？什么操作产生 GC？怎么避免？

**要点**：Unity 默认 **Mono/Boehm GC**（标记-清除/压缩，stop-the-world，非分代非压缩）。当 `new` 分配对象、堆不足、达到阈值时触发。GC 时暂停所有线程扫描/压缩托管堆，移动端频繁 GC = 卡顿尖刺。

**产生 GC 的典型操作**：`new` 引用类型、装箱（`int` 传 `object`）、闭包捕获局部变量、`foreach` 装箱（旧版）、`Coroutine yield return new XXX()`、LINQ、反射、字符串拼接。

**避免手段**：对象池、缓存委托、`StringBuilder`、用 `struct` 避免堆分配、避免热路径装箱、`yield` 缓存（`var wfs = new WaitForSeconds(1f)` 不要写在 yield 里）、用 UniTask 替代部分协程。

**高频追问**：Boehm GC 为什么不分代？→ 历史原因 + 跨平台一致性；IL2CPP 也用 Boehm。要分代需换 GC（实验性）。

> 关联：[[【设计原理】GC工作原理深度解析]] [[【最佳实践】GC优化清单]] [[【设计原理】Unity内存管理]]

### Q6. 如何排查内存泄漏？

**要点**：泄漏 = 内存持续增长不回落。排查流程：
1. **Memory Profiler**：在三个时点（如场景刚进 / 玩 10 分钟 / 玩 30 分钟）各拍一次快照
2. **快照对比**： diff 找出「只增不减」的对象类型与引用链
3. **顺引用链定位**：是谁持有了本该释放的对象（常见：静态字段、单例、事件未取消订阅 `+=` 没 `-=`、字典缓存只增不删、` DontDestroyOnLoad` 挂错对象）

**常见泄漏模式**：事件订阅未取消、静态集合累积、`Destroy` 了 GameObject 但 C# 侧还有引用其组件、Resources/AB 未 Unload。

**高频追问**：C#↔Lua 互持泄漏怎么查？→ tolua/xLua 边界 profiler，看是否有 C# 对象被 Lua table 长期持有或反之。

> 关联：[[【实战案例】内存泄漏排查实战]] [[【踩坑记录】内存泄漏模式]] [[【最佳实践】内存分析工具使用指南]]

### Q7. 游戏内存主要由什么占用？怎么压？

**要点**：移动游戏内存大头通常是**资源**——纹理 > 网格 > 动画片段 > 音频 > Shader。

**压缩手段**：
- **纹理**：用 ASTC（移动端首选）/ ETC2，关掉不需要的 mipmap（UI），读/写关闭（Read/Write Enabled 关），按需尺寸不要超采样；详见 [[【性能数据】纹理压缩格式对比]]
- **音频**：长 BGM 用 Streaming（不全进内存），短音效用 Compressed in Memory
- **网格/动画**：压缩网格、压缩动画、启用 Rig 裁剪、限制骨骼数
- **Shader**：变体管理（见 Q11），否则内存爆炸
- **Resources 目录**：移出不用资源（整个 Resources 目录会全打包并常驻索引）

**高频追问**：UI 图集占了很大内存怎么办？→ Sprite Atlas 合理分图集、关闭 Read/Write、TMP 字体 Atlas 优化。

> 关联：[[【最佳实践】资源卸载指南]] [[【性能数据】纹理压缩格式对比]] [[【最佳实践】TextMeshPro性能优化实战]]

---

### Q8. 突发的 GC 尖刺（某一帧卡一下）怎么定位和消除？

**要点**：尖刺 = 某帧突然分配大量对象触发 GC。定位：Profiler CPU 模块看 GC Alloc 列，找到分配最多的那帧那函数。常见元凶：热路径 `foreach`+闭包、每帧字符串拼接、LINQ、反射缓存失效。

**消除**：把分配移出热路径（初始化时预分配 + 复用）、缓存、对象池、`StringBuilder`、用值类型。

**高频追问**：为什么有时候 GC 间隔不固定？→ Boehm 按阈值触发，分配速率高就频繁；增量式 GC（Incremental GC）可把单次停顿分摊到多帧，减轻尖刺——但要测试是否真的更平滑。

> 关联：[[【最佳实践】GC优化清单]] [[【设计原理】GC工作原理深度解析]]

## 三、渲染优化

### Q9. Overdraw 是什么？怎么检测和优化？

**要点**：Overdraw = 同一像素被多次绘制（半透明叠加、被遮挡物仍渲染）。代价是 GPU 片元处理带宽浪费。

**检测**：Scene 视图右上角 Overdraw 模式、Profiler GPU 模块、渲染层级检测。

**优化**：**遮挡剔除（Occlusion Culling）**、合理层级（避免前后大量半透明叠加）、UI 减少全屏半透明大图、LOD、提前深度 PreZ（移动端要权衡）。

> 关联：[[【最佳实践】Overdraw优化实战]] [[【最佳实践】遮挡剔除实战]]

---

### Q10. LOD / 遮挡剔除 / 光照烘焙 各解决什么问题？代价是什么？

**要点**：
- **LOD**：远处用低面数网格 → 省三角形/顶点处理；代价：内存增加（多套网格）、切换突兀（要过渡）
- **遮挡剔除**：跳过被挡住的物体渲染 → 省 DC 与 GPU；代价：烘焙耗时、动态场景效果有限、配置复杂
- **光照烘焙（Lightmap）**：静态光照预计算到贴图 → 省实时计算；代价：不能动、贴图内存、物体必须标 Static

**高频追问**：移动端阴影怎么处理？→ 实时阴影极贵，常降分辨率/减 CSM 级联数、用 baked/模拟阴影（贴图圆盘）替代。

> 关联：[[【最佳实践】LOD系统实战指南]] [[【最佳实践】遮挡剔除实战]] [[【最佳实践】光照与烘焙优化]]

---

### Q11. Shader 变体爆炸是怎么回事？怎么治？

**要点**：Shader 里每个 `multi_compile`/`shader_feature` 关键字组合会生成一份编译版本（变体）。关键字越多变体数**指数增长**，导致：编译慢、包体大、**运行时内存暴涨**（所有变体加载）、SRP Batcher 受影响。

**治理**：用 `shader_feature`（只编用到的）而非 `multi_compile`（全编）、合并关键字、local keywords（`shader_feature_local`，上限更大）、用 **Shader Variant Collection** 只收集/打包实际用到的变体、变体剥离（Stripping）配置。

> 关联：[[【性能数据】Shader性能优化]] [[【教程】HLSL与Shader基础]]

---

### Q12. 粒子系统为什么卡？怎么优化？

**要点**：粒子卡顿多因 **CPU 模拟开销（大量粒子 + 碰撞/子发射器）+ Overdraw（半透明叠加）+ DC 多（多个独立 ParticleSystem）**。

**优化**：控制最大粒子数、关掉不必要的碰撞、用 **GPU Events/粒子**把模拟上 GPU、合并同材质粒子、LOD（远处降粒子数）、屏幕外剔除、用纹理序列帧替代复杂模拟。

> 关联：[[【最佳实践】粒子系统优化]] [[【设计原理】粒子系统GPU加速机制]]

## 四、启动与加载

### Q13. 冷启动慢怎么定位？

**要点**：用 **Unity 启动报告 / Performance Reporting / Profiler** 拆解各阶段：引擎初始化 → 首场景加载 → 资源加载 → 首帧渲染。常见瓶颈：Resources 目录过大（启动时全索引）、首场景资源同步加载、大量 Awake/Start 逻辑、Shader 编译、IL2CPP 首次 JIT 预热。

**优化方向**：首屏只加载必要资源 + 异步 + 预加载、清理 Resources、Shader 预热（Preload Shaders）、分帧初始化逻辑、Addressables 按需加载。

> 关联：[[【实战案例】启动时间优化实战]] [[【设计原理】Unity启动流程深度解析]] [[【最佳实践】资源预加载策略]]

---

### Q14. 首包 / 分包 / 热更包策略？

**要点**：
- **首包**：只放进游戏必需的最小资源（首场景 + UI + 核心），其余走下载
- **分包**：按玩法/场景拆 AB，用到再下，降低首包体积与启动内存
- **热更包**：见 [[【设计原理】热更新方案对比]]，资源热更（AB/Addressables）+ 代码热更（HybridCLR/Lua）
- 压缩：传输 LZMA 省流量、本地缓存转 LZ4 随机访问（见 [[【综述】Unity客户端面试题]] §26）

> 关联：[[【教程】打包与热更新]] [[【综述】Unity资源依赖与打包陷阱]]

---

### Q15. Addressables 异步加载要注意什么？

**要点**：
- **引用计数**：`LoadAssetAsync` +1，`Release` -1，归零才真正卸载——漏 Release 泄漏，多 Release 崩溃
- **异步句柄**：注意完成回调时机、异常处理；避免同步 `.Result` 阻塞
- **内存**：注意同时加载量，大资源及时 Release
- **分组策略**：同组资源会同时加载到内存，按使用频率/场景分组

**高频追问**：Addressables 和裸 AssetBundle 的区别？→ Addressables 是 AB 之上的管理框架，自动处理依赖与引用计数，不用手写依赖图。

> 关联：[[【教程】资源管线-Addressables]] [[【最佳实践】Addressables性能优化]]

## 五、工具与方法论

### Q16. 你常用的性能分析工具？怎么用？

**要点**（必须熟练）：
- **Profiler（CPU/GPU/内存/渲染）**：定位每帧耗时大头，Hierarchy 视图深挖；务必在**真机 + Release 构建**上抓（Editor/Development 版数据不准）
- **Frame Debugger**：逐 DC 渲染诊断，看合批、看每步状态
- **Memory Profiler**（包）：快照对比查泄漏、看对象引用链、看资源内存分布
- **Performance Reporting / 真机日志**：远程抓取线上性能数据
- **Overdraw / Stats 视图**：快速概览

**高频追问**：为什么 Editor 上测的性能不准？→ Editor 有额外开销、不反映 IL2CPP/真机 GPU；Development Build 也比 Release 慢。

> 关联：[[【教程】性能分析工具]] [[【教程】Frame Debugger渲染诊断流程]] [[【最佳实践】内存分析工具使用指南]]

### Q17. 讲一次你做过的性能优化案例（开放题 → STAR）

> 这是高级面试**必问**开放题。面试官想看你的**方法论闭环**：现象 → 量化指标 → 定位工具 → 根因 → 方案 → 验证 → 沉淀。**临场组织不好会大幅失分**。
>
> 用 **STAR** 结构讲：**S**ituation（什么项目、什么问题：如「战斗时偶发掉帧到 30fps」）→ **T**ask（你的职责：定位并优化到 60fps）→ **A**ction（用 Profiler 抓真机帧 → 发现是 XXX 函数每帧 GC + DC 过多 → 方案：对象池 + 合图集 + 静态批处理）→ **R**esult（GC 降为 0、DC 从 180→40、稳定 60fps，已沉淀为团队规则）。
>
> 参考 vault 现成案例组织你的故事：[[【实战案例】UI卡顿优化全流程]]（30→60fps 全流程）、[[【实战案例】内存泄漏排查实战]]（480→120MB）、[[【实战案例】启动时间优化实战]]（20s→5s）。提前准备 2–3 个，见 [[【综述】高级Unity游戏开发面试复习地图]] §十。
