# 【笔记】大规模单位动画方案

> 承 [[【笔记】同屏大规模单位渲染方案]] 的追问：10w 单位、12 种怪物、每怪死亡/move/attack1/attack2/idle 动画，如何实现。本文聚焦「大规模 + 多类型 + 独立动画」这一最硬场景。

## 新增的两个矛盾

10w 渲染与更新的底子（最少 archetype、BRG+DOTS Instancing 合批、Burst job 更新）沿用 [[【笔记】同屏大规模单位渲染方案]]。本题新增：

| 矛盾 | 棘手之处 |
|------|---------|
| 12 种怪物 | 若每种用不同组件 → archetype 爆炸 → batch 爆炸 |
| 每单位独立动画 | `Animator`/`SkinnedMeshRenderer` 是 GameObject 体系，10w 个 Animator 主线程卡死；Entities Graphics 无「ECS 版 Animator」 |

核心思路：**把动画播放整个搬到 GPU（VAT），CPU 只算「该播哪段、播到哪」。**

---

## 一、动画方案选型（核心决策）

| 方案 | 量级上限 | 适用 | 10w 评价 |
|------|---------|------|---------|
| 传统 `Animator`（companion） | 几百 | 主角/Boss | ❌ 10w 主线程爆炸 |
| DOTS Animation（`com.unity.animation`） | 千级 | 少量 ECS 角色真骨骼 | ⚠️ 1.4 期 preview，撑不到 10w |
| Unity 6 Mesh Deformations（蒙皮） | 万级 | 中等规模真骨骼 | ⚠️ CPU 端骨骼更新仍是瓶颈 |
| **GPU 顶点动画（VAT）** | **10w+** | **RTS 大军、群妖** | ✅ **工业标准，CPU 零骨骼** |
| Sprite 序列（2D 俯视） | 10w+ | 2D 风格 | ✅ 2D 更简单 |

**结论：10w 带动画单位 = GPU 顶点动画纹理（VAT）+ ECS 状态机驱动。**

---

## 二、VAT（Vertex Animation Texture）原理

把每帧顶点位置（+法线）烘焙进纹理：**横轴 = 顶点 ID，纵轴 = 帧**。Vertex Shader 按当前帧采样得顶点位置——完全 GPU 驱动，CPU 零骨骼计算。

```
AnimTex:  横 = vertexId / vertexCount
          纵 = frame   / frameCount
每段动画  = 纹理里一段连续帧 (startFrame, frameCount, fps, loop)
```

- **12 怪 × 5 动画 = 60 段**，全部烘焙进一张（或纹理数组），用一张小 LUT 记每段 `(startFrame, frameCount, fps)`。
- ⚠️ VAT 要求**同纹理内 mesh 顶点数一致**；12 种怪物若 mesh 不同，按 mesh 分别建 VAT（或纹理数组按 `monsterType` 索引），按 mesh 种类分 batch（最多 ~12 batch，可接受）。
- per-instance 只传 `(monsterType, animState, animTime)`，全是 DOTS Instancing 的 batch 友好属性，**不拆 batch**。

## 三、ECS 状态机驱动

每单位只存「状态 + 时间」，状态机 Burst job 并行跑，结果通过 `MaterialPropertyOverride` 喂 shader。

```csharp
using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

public enum AnimState : byte { Idle, Move, Attack1, Attack2, Death }

// —— 每单位动画数据（unmanaged）——
public struct UnitAnim : IComponentData {
    public AnimState State;
    public float Time;       // 动画内时间
    public float Speed;
}
public struct MonsterType : IComponentData { public byte TypeId; }  // 0..11

// —— 传 shader 的 per-instance 属性（DOTS Instancing，batch 友好）——
//   [MaterialProperty("_AnimState")] struct AnimStateProp : IComponentData { ... }
//   [MaterialProperty("_AnimTime")]  struct AnimTimeProp  : IComponentData { ... }
// 把 State/Time 写进这些组件，Entities Graphics 自动上传为 GPU 实例属性

// —— 状态机 + 时间推进（Burst 并行，10w 实体 ~1-3ms）——
[BurstCompile]
partial struct AnimStateMachineJob : IJobEntity
{
    public float Dt;
    void Execute(ref UnitAnim anim, in Velocity vel, in Health hp, in AttackSignal atk)
    {
        // 优先级：死亡 > 攻击 > 移动 > 待机
        if (hp.Current <= 0)                       { Set(ref anim, AnimState.Death);   }
        else if (atk.Trigger1)                     { Set(ref anim, AnimState.Attack1); }
        else if (atk.Trigger2)                     { Set(ref anim, AnimState.Attack2); }
        else if (math.lengthsq(vel.Value) > 0.01f) { Set(ref anim, AnimState.Move);    }
        else                                       { Set(ref anim, AnimState.Idle);    }

        anim.Time += Dt * anim.Speed;
    }
    static void Set(ref UnitAnim a, AnimState s) { if (a.State != s) { a.State = s; a.Time = 0; } }
}
```

shader 端据 `_AnimState`/`_AnimTime` + LUT 采样 VAT（伪代码）：

```hlsl
// 1) (monsterType, animState) → LUT 取 (startFrame, fps, loop)
// 2) frame = animTime * fps + startFrame   (loop 则 frac)
float2 uv  = float2(vertexId / _VertCount, frame / _TexFrames);
float3 pos = tex2Dlod(_AnimTex, float4(uv, 0, 0)).xyz;   // VAT 采样
```

## 四、12 种怪物的数据组织（关键避坑）

| 维度 | ✅ 正确 | ❌ 错误 |
|------|--------|--------|
| 类型区分 | `MonsterType`(byte) + `MaterialMeshInfo` 索引 + VAT 偏移 | 每种怪物不同组件 |
| archetype 数 | **尽量 1 个**（所有怪物同组件集） | 12 种 = 12+ archetype |
| 动画数据 | 60 段烘焙进 VAT + LUT | 60 个 AnimatorController |
| per-instance 差异 | `MaterialPropertyOverride`（不拆 batch） | 12 种材质实例 |

理想：**12 怪 5 动画全在 1 个 archetype**，按 mesh 种类只分 ~12 batch。

---

## 五、VAT 制作管线（一次性烘焙，运行时零成本）

1. 每种怪物 mesh：在 DCC/Unity 里把 5 个动画逐帧 `SkinnedMeshRenderer.BakeMesh` 采样成顶点序列；
2. 写 Editor 工具把顶点位置（必要时法线）写进 `Texture2D`（RGBA Float）；
3. 生成 LUT：每段 `(meshId, startFrame, frameCount, fps, loop)`；
4. shader + LUT + VAT 资源打包。

> 工程量在烘焙这一步，但**一次性**。社区有现成 VAT 工具可参考（如 Unity 版 VAT 烘焙脚本）。

---

## 六、性能与避坑

- **CPU**：状态机 + 时间推进 job（10w ~1–3ms，Burst 并行）；攻击/死亡用 `IEnableableComponent` 或事件 buffer 触发，**不每帧扫全场**。
- **GPU**：VAT = vertex shader 多一次纹理采样，10w 低面数无压力；**overdraw**（透明/特效）才是真瓶颈。
- **动画硬切**：`Time=0` 会跳变；要平滑可在 shader 交叉淡入两帧（成本略增）。
- **死亡收尾**：死亡动画播完用 `IEnableableComponent` 软禁用 + 回对象池，**不要 `DestroyEntity`**（结构性变更贵）。
- **2D 俯视**：跳过 VAT，用 sprite 序列 + `MaterialPropertyOverride` 传 frame index，shader 切 UV，简单得多。
- **真骨骼特例**：Boss/主角等少量单位仍可用 Animator（companion）或 DOTS Animation，与大规模 VAT 单位混用。

---

## 速查清单

- [ ] 动画用 VAT（GPU 顶点动画），不要 Animator/SkinnedMeshRenderer 撑 10w
- [ ] 怪物类型用 `MonsterType`(byte) + 索引区分，不用不同组件，保持 archetype 最少
- [ ] 状态机用 `IJobEntity`+Burst+`ScheduleParallel`，优先级 死亡>攻击>移动>待机
- [ ] per-instance 传 `(monsterType, animState, animTime)` 用 `MaterialPropertyOverride`
- [ ] 60 段动画烘焙进 VAT + LUT，一次性烘焙
- [ ] 攻击/死亡用事件触发，不每帧扫全场
- [ ] 死亡播完用 `IEnableableComponent` + 对象池，禁 `DestroyEntity`
