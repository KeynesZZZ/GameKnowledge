# 【笔记】


> 承 [[【笔记】Entities 1.4 与 Entities Graphics 1.4 官方文档]] 的「10w 同屏」追问。聚焦大规模单位这一具体场景的落地方案与避坑。

## 核心判断：瓶颈不在「画」，在「提交」和「更新」

GPU 画 10w 个简单 mesh 毫无压力。会卡死你的是两个 CPU 端瓶颈：

| 瓶颈 | 传统 GameObject 方案 | 后果 |
|------|---------------------|------|
| **DrawCall 提交** | 每单位 1 个 DrawCall | 10w DrawCall → 主线程提交卡死 |
| **每帧数据更新** | 10w 个 `Update()` 单线程 | 移动/AI 吃满 CPU |

方案目标：**把 10w 单位压成个位数 DrawCall，把 10w 次更新塞进 Burst 多核 job**。技术栈 = ECS + Entities Graphics + Jobs/Burst。

---

## 五个关键手段

### 1. 数据层：ECS + 最少 archetype

每单位 = 一个 Entity（`LocalTransform` + `Velocity` + 业务组件，全 unmanaged struct），躺在连续内存的 Archetype Chunk 里，cache 友好。

⚠️ **铁律：archetype 数量压到最少。** Entities Graphics **每个 archetype 生成一个 batch**。给单位加「弓兵/步兵/骑兵」不同组件组合会让 archetype 爆炸。**用 `MaterialMeshInfo` 索引区分共享数据，而不是靠不同组件。** 详见 [[【设计原理】ECS为什么快]] 的 chunk 布局原理。

### 2. 渲染层：Entities Graphics 自动合批（BRG + DOTS Instancing）

- `BatchRendererGroup` 把**同 archetype + 同材质 + 同网格**的实例自动合成 batch。
- **DOTS Instancing** 让 per-instance 属性（颜色/缩放/血条）通过 GPU buffer 按 instance index 寻址，**不拆 batch、不需为每色建新材质**。这是 10w 同种单位能合成个位数 DrawCall 的关键。
- **真正拆 batch 的是**：不同材质资源、不同 mesh、不同 `RenderFilterSettings`（layer/阴影设置）、archetype 不同——**而非 per-instance 属性值不同**（这是 DOTS Instancing 的设计目的，常被误传）。

### 3. 更新层：IJobEntity + Burst + ScheduleParallel

10w 实体位移绝不能主线程 `foreach`，也不能 `EntityManager.SetComponentData`（同步阻塞）。用 `IJobEntity`，Burst 编译后 `ScheduleParallel()` 跑满多核——10w 位移通常 1–2ms。详见 [[【教程】JobSystem详解]]、[[【教程】Burst编译器]]。

### 4. LOD + 剔除：控制 GPU 压力

10w 单位远处占大头，必须降级：`MeshLODGroup` 远处切低面数 mesh 甚至 billboard；配合 Entities Graphics 开箱即用的视锥剔除。**LOD 屏幕百分比范围要设准**，太宽 GPU 剔不掉。

### 5. 避开两个隐形杀手

- **别频繁结构性变更**：10w 实体每帧增删 / 改 `ISharedComponentData` 会触发 chunk 迁移，开销巨大。单位「死/活」用 `IEnableableComponent`（bit 开关，无结构性变更）+ 对象池，而非 `DestroyEntity`/`CreateEntity`。
- **别每帧改共享组件**：`ISharedComponentData` 适合**静态分组**（同材质归簇），不适合每帧变化的值。

## 资源组织与加载

> 10w 单位共用同一种 mesh/材质时直接引用即可。但实际项目有多种单位类型（步兵、弓兵、骑兵……12 种怪），如何组织、加载、切换 mesh/材质是工程落地的核心问题。

### Mesh / Material 的两种持有模式

```
模式 A：SubScene + Baking（推荐，静态/可预览）
  ┌──────────────────────────────────────────────────┐
  │ SubScene                                         │
  │  ├─ GameObject(prefab)                           │
  │  │   ├─ MeshFilter  → mesh                       │
  │  │   └─ MeshRenderer → material(s)               │
  │  │   └─ UnitAuthoring (MonoBehaviour)            │
  │  └─ Baking Pipeline 自动转换                      │
  │      → Entity + MaterialMeshInfo + 渲染组件       │
  └──────────────────────────────────────────────────┘
  ✅ 编辑器可预览
  ✅ Unity 自动管 LOD Group → MeshLODGroup baking
  ✅ 材质/mesh 直接拖拽，无加载代码
  ❌ 所有资源打包进 SubScene，不能运行时热替换

模式 B：运行时注册（推荐，动态/热更场景）
  ┌──────────────────────────────────────────────────┐
  │ 启动时（主线程，一次性）                           │
  │  1. Addressables 加载 mesh/material              │
  │  2. 构建 RenderMeshArray(mats[], meshes[])       │
  │  3. RenderMeshUtility.AddComponents(proto, ...)   │
  │  4. Instantiate(proto, 10w)                      │
  └──────────────────────────────────────────────────┘
  ✅ 资源走 Addressables，可热更/按需加载
  ✅ 运行时可注册新的 mesh/材质（切武器/装备）
  ❌ 需要手写加载逻辑
```

### 模式 B 完整代码：Addressables 加载 + 注册

```csharp
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;

/// <summary>启动时加载所有单位资源，注册到 ECS 渲染系统。</summary>
public class UnitAssetLoader : MonoBehaviour
{
    [Header("12 种怪 × (Mesh + Material) 的 Addressable 引用")]
    public AssetReferenceT<Mesh>[]     MeshRefs;
    public AssetReferenceT<Material>[] MaterialRefs;

    private async void Start()
    {
        // 1) 并行加载所有 mesh 和材质（Addressables 内部自动并行）
        var meshTasks     = new Task<Mesh>[MeshRefs.Length];
        var materialTasks = new Task<Material>[MaterialRefs.Length];
        for (int i = 0; i < MeshRefs.Length; i++)
            meshTasks[i] = MeshRefs[i].LoadAssetAsync<Mesh>().Task;
        for (int i = 0; i < MaterialRefs.Length; i++)
            materialTasks[i] = MaterialRefs[i].LoadAssetAsync<Material>().Task;

        await Task.WhenAll(meshTasks);
        await Task.WhenAll(materialTasks);

        // 2) 提取结果
        var meshes    = new Mesh[MeshRefs.Length];
        var materials = new Material[MaterialRefs.Length];
        for (int i = 0; i < meshTasks.Length; i++)     meshes[i]    = meshTasks[i].Result;
        for (int i = 0; i < materialTasks.Length; i++)  materials[i] = materialTasks[i].Result;

        // 3) 交给 ECS 初始化系统
        var em = World.DefaultGameObjectInjectionWorld.EntityManager;
        InitializeUnits(em, materials, meshes);
    }

    void InitializeUnits(EntityManager em, Material[] mats, Mesh[] meshes)
    {
        // 构建 prototype：用 RenderMeshArray 传入全部材质/mesh
        var proto = em.CreateEntity();
        var desc = new RenderMeshDescription(shadowCastingMode: ShadowCastingMode.Off);
        var rma  = new RenderMeshArray(mats, meshes);    // 12 材质 + 12 mesh

        // 默认指向第 0 组（运行时可改 MaterialMeshInfo 切换）
        RenderMeshUtility.AddComponents(proto, em, desc, rma,
            MaterialMeshInfo.FromRenderMeshArrayIndices(0, 0));

        em.AddComponentData(proto, new LocalToWorld { Value = float4x4.identity });
        em.AddComponentData(proto, new LocalTransform());
        em.AddComponentData(proto, new Velocity());
        em.AddComponentData(proto, new UnitTypeIndex { Value = 0 });
        em.AddComponent<UnitTag>(proto);

        // 批量克隆
        var units = em.Instantiate(proto, 100_000);

        // 后续 Job 设置每个单位的 MaterialMeshInfo → 切换外观
        // （见下方「运行时切换外观」）

        em.DestroyEntity(proto);  // 删除原型，只保留克隆体
    }
}
```

### 运行时切换外观（MaterialMeshInfo 索引）

```csharp
/// <summary>每个单位的 MonsterType byte 值映射到 RenderMeshArray 的索引。</summary>
/// <remarks>
/// RenderMeshArray 在注册时按顺序传入 mats[] / meshes[]：
///   索引 0 = 步兵 (mat[0], mesh[0])
///   索引 1 = 弓兵 (mat[1], mesh[1])
///   索引 2 = 骑兵 (mat[2], mesh[2])
///   ...
///
/// MaterialMeshInfo 存储 (materialIndex, meshIndex)。
/// 改变这两个值 = 切换单位的 mesh 和材质 → Entities Graphics 自动重新分 batch。
/// </remarks>
[BurstCompile]
partial struct SetAppearanceJob : IJobEntity
{
    void Execute(ref MaterialMeshInfo mmi, in UnitTypeIndex type)
    {
        // 怪物类型 byte → mesh/材质索引（一对一映射，或查表）
        mmi = MaterialMeshInfo.FromRenderMeshArrayIndices(type.Value, type.Value);
    }
}

// 调用：在生成后一次性设置
new SetAppearanceJob().ScheduleParallel(state.Dependency);
```

```
⚠️ MaterialMeshInfo 切换的代价：
  改 MaterialMeshInfo 值 → Entities Graphics 在下一帧重新分 batch
  → 同帧内大量切换会导致 batch 重排序开销
  解决：
  1. 初始化时一次性设置好，不要每帧切
  2. 如果必须切（换装备/进化），分帧分散，每帧切 1000 个
  3. 切 mesh 比 切材质 贵（LOD 也要重新匹配）
```

### per-instance 差异：DOTS Instancing 材质属性

```
同 batch 内的 per-instance 差异（颜色、血量、选中高亮、动画时间）
→ 不能用 MaterialPropertyBlock（ECS 不支持）
→ 不能建新材质实例（拆 batch）
→ 用 [MaterialProperty] 标注的 IComponentData：

  // 组件定义
  [MaterialProperty("_BaseColor")] public struct UnitColor    : IComponentData { public float4 Value; }
  [MaterialProperty("_AnimTime")]  public struct AnimTimeProp : IComponentData { public float Value; }
  [MaterialProperty("_Selected")]  public struct SelectedProp : IComponentData { public float Value; }

  // Entities Graphics 自动把这些组件的值上传到 GPU buffer
  // Shader 按 instance index 读取（DOTS Instancing）
  // → 不拆 batch！

  // Shader 端只需确保启用 DOTS Instancing：
  // #pragma multi_compile _ DOTS_INSTANCING_ON
  // UNITY_DOTS_INSTANCING_START
  // UNITY_DOTS_INSTANCED_PROP(float4, _BaseColor)
  // UNITY_DOTS_INSTANCING_END
  // col = UNITY_ACCESS_DOTS_INSTANCED_PROP(_BaseColor);
```

### 多类型单位的 batch 分析

```
假设 12 种怪，每种同 mesh + 同材质：

  archetype 相同（所有单位共享同一套组件）
  但 MaterialMeshInfo 不同 → 按 mesh/材质组合分 batch

  实际 batch 数 = min(12, 可见 mesh 种类数)

  如果同屏 12 种全可见 → 12 个 DrawCall（仍可接受）
  如果远处全是 LOD2 同一种低精度 mesh + 同材质 → 1 个 DrawCall

结论：DrawCall 数 = 可见的 (mesh, material) 组合数，
      与单位总数（10w）无关。
```

### LOD 资源策略

```
每类单位准备 3 级 LOD mesh：

  LOD0 (< 40px)：高精度 500-2000 三角面
  LOD1 (40-150px)：中精度 100-300 三角面
  LOD2 (> 150px)：低精度 10-50 三角面

Batch 影响：
  同一种单位，LOD0 和 LOD1 使用不同 mesh → 不同 batch
  10w 单位 × 3 LOD → 最多 36 个 batch（12 种 × 3 LOD）
  但远处大量单位共用 LOD2 → 大部分压到同一 batch

内存影响：
  每种单位 3 个 mesh + 3 张 VAT 纹理
  12 种 × 3 = 36 mesh + 36 VAT 纹理
  → VAT RGBAFloat 2048×64 = 4MB/张 × 36 = ~144MB（可接受，必要时降 Half）
```

### Addressables 集成要点

```
资源组织（Addressables Group）：
  UnitMeshes/      ← 12 种怪 × 3 LOD = 36 mesh
  UnitMaterials/   ← 12 材质（DOTS Instancing shader）
  VATTextures/     ← 12 种 × N 动画 = ~36-72 张纹理

加载策略：
  1. 启动时同步加载核心资源（Addressables.LoadScene + preload）
  2. 可选：分包加载（不同关卡/地图 → 不同怪组合）
  3. 释放：切关时 Addressables.Release(mesh/material)

⚠️ 注意：
  - RenderMeshArray 持有的 mesh/material 引用 = Addressables 的 Asset 句柄
  - 释放前必须确保没有 Entity 引用（先 DestroyEntity → 再 Release）
  - 否则 batch 引用已释放资源 → 渲染崩溃
```

### SubScene 与 Addressables：能配合用吗？

> **短答案：原生不兼容，但可以分工配合。**

SubScene 和 Addressables 是两套**独立的打包管线**，不能直接用 Addressables API 加载 SubScene：

```
SubScene（Entities）                    Addressables
──────────────                         ──────────────
Editor 阶段把 GameObject 烘焙成         按标签/分组打包 AssetBundle
二进制 Entity 数据（EntityScene）        运行时按引用异步加载 Asset

两套独立的序列化格式：
  SubScene → Entity Scene 二进制流       Addressables → 标准 AssetBundle

❌ 同一份 mesh/材质同时被 SubScene 引用和 Addressables 打包
   → 资源在包体中被重复包含（体积膨胀）
```

这是 Unity 官方确认的长期限制（截至 Entities 1.4 / Unity 6 仍未原生集成）。

#### SubScene 自己的加载/流式系统

SubScene 不需要 Addressables——它**内置了一套完整的场景流式加载**，基于 `SceneSystem`：

```csharp
// SubScene 运行时加载（Entities 1.4）
[BurstCompile]
public partial struct SceneLoadSystem : ISystem
{
    private Entity m_SceneEntity;

    public void OnUpdate(ref SystemState state)
    {
        // 异步加载 SubScene（通过 SceneGUID 引用）
        if (/* 满足加载条件 */)
        {
            m_SceneEntity = SceneSystem.LoadSceneAsync(
                state.WorldUnmanaged,
                sceneGUID,
                new SceneSystem.LoadParameters
                {
                    Flags = SceneLoadFlags.BlockOnImportInEditor
                }
            );
        }

        // 卸载
        if (/* 满足卸载条件 */)
        {
            SceneSystem.UnloadScene(state.WorldUnmanaged, m_SceneEntity);
        }
    }
}
```

```
SubScene Section 流式加载（内置分块，类似 Addressables 分组）：

  SubScene
  ├─ Section 0（始终加载：地形 + 核心 prefab）
  ├─ Section 1（按需加载：东区区域）
  ├─ Section 2（按需加载：西区区域）
  └─ Section 3（按需加载：Boss 房间）

  // 单独加载某个 Section
  SceneSystem.LoadSceneAsync(world, sceneGUID, loadParam, sectionIndex: 1);
```

> Editor 里在 SubScene Inspector 中划分 Section。每个 Section 独立打包、独立加载——这是 SubScene 原生的流式加载能力，不需要 Addressables。

#### 两者分工配合

```
┌──────────────────────────────────────────────────────────┐
│                    实际项目资源分层                         │
│                                                           │
│  SubScene 管（SceneSystem 加载）：  Addressables 管：      │
│  ─────────────────────────────     ──────────────        │
│  ✓ 烘焙好的单位 prefab              ✓ DLC/热更新 mesh     │
│  ✓ 静态场景（地形/建筑）            ✓ UI 图集/音效/配置    │
│  ✓ LOD Group → MeshLODGroup        ✓ 脚本/数据表          │
│  ✓ Entity 初始数据                                         │
│                                                           │
│  加载：SceneSystem                  加载：Addressables     │
│  合批：Baking 自动完成              合批：RenderMeshArray  │
└──────────────────────────────────────────────────────────┘
```

#### 典型 10w 单位项目的资源分配

```
资源类型              → 放哪里        → 怎么加载
───────────────────────────────────────────────────────
12 种怪 prefab        → SubScene      → Baking 自动处理
(含 mesh + 材质 + LOD)                 → SceneSystem.LoadSceneAsync

VAT 纹理              → SubScene 内    → Baking 时自动引用
                      （或独立 Addressable）→ 大文件可单独管

UI / 音效 / 配置表    → Addressables  → Addressables.LoadAsset

DLC 新怪物            → Addressables  → 运行时 RenderMeshArray 注册
（发版后新增的）         （mesh + 材质）    （模式 B）
```

#### DLC/热更新场景的注意事项

```
运行时通过 Addressables 下载新 mesh/材质 → 必须走模式 B（运行时注册），
完全绕过 SubScene：

  1. Addressables.LoadAssetAsync<Mesh>()     → 下载 mesh
  2. Addressables.LoadAssetAsync<Material>() → 下载材质
  3. new RenderMeshArray(materials, meshes)  → 注册
  4. EntityManager.CreateEntity() + AddComponents → 从零建 Entity
  5. Instantiate → 批量克隆

  ⚠️ 这种 Entity 不能从 SubScene prefab 克隆（SubScene 的 mesh 走自己打包管线）
     → 所有组件手动挂载（LocalTransform + MaterialMeshInfo + 业务组件）
     → 详见上文「模式 B 完整代码」
```

#### 速查对比

| 维度 | SubScene (SceneSystem) | Addressables |
|------|----------------------|--------------|
| **加载 API** | `SceneSystem.LoadSceneAsync` | `Addressables.LoadAssetAsync` |
| **打包格式** | Entity Scene 二进制 | AssetBundle |
| **流式加载** | Section 分块（内置） | 标签/分组（手动管） |
| **Baking** | Editor 自动烘焙 | 无 |
| **热更新** | ❌ 不能运行时下载新 SubScene | ✅ 天然支持 |
| **DLC** | ❌ 需客户端自带 | ✅ 按需下载 |
| **资源去重** | ❌ 与 Addressables 包会重复 | — |
| **适用** | 静态场景 + 预制单位 | 动态加载 / DLC / UI / 音效 |

> **实践建议**：10w 单位的核心渲染（mesh / 材质 / VAT）走 SubScene + Baking；热更新和 DLC 走 Addressables + 运行时 RenderMeshArray 注册。两者各管各的资源，不要交叉引用同一份 mesh/材质。

### 内存预算

| 资源类型 | 单份大小 | 12 种 × 3 LOD | 说明 |
|----------|----------|---------------|------|
| Mesh (LOD0) | ~100-500KB | ~6MB | 高精度 |
| Mesh (LOD1) | ~20-80KB | ~2.4MB | 中精度 |
| Mesh (LOD2) | ~5-10KB | ~360KB | 低精度 |
| 材质 | ~16KB | ~576KB | shader + 参数 |
| VAT 纹理 (RGBAFloat 2048×64) | ~4MB | ~48-144MB | 主要内存消耗 |
| **合计** | | **~60-160MB** | 取决于动画数量 |

> VAT 纹理是大头。移动端可降 RGBAHalf（精度够用）或压缩到 BC6H 格式，内存可减半。详见 [[【笔记】大规模单位动画方案]]。

## 最小骨架（1.4 现代 API）

```csharp
using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Rendering;
using Unity.Transforms;
using UnityEngine.Rendering;

public struct Velocity : IComponentData { public float3 Value; }
public struct UnitTag   : IComponentData { }          // tag，便于 query

// —— 运行时建 prototype（主线程，一次性）——
Entity BuildPrototype(EntityManager em, Material mat, Mesh mesh)
{
    var proto = em.CreateEntity();
    var desc  = new RenderMeshDescription(shadowCastingMode: ShadowCastingMode.Off);
    var rma   = new RenderMeshArray(new[] { mat }, new[] { mesh });
    RenderMeshUtility.AddComponents(proto, em, desc, rma,
        MaterialMeshInfo.FromRenderMeshArrayIndices(0, 0));   // 全部同 batch
    em.AddComponentData(proto, new LocalToWorld());
    em.AddComponent<Velocity>(proto);
    em.AddComponent<UnitTag>(proto);
    return proto;
}
// 一次批量克隆 10w（内部已优化，远快于循环单建）
Entity[] units = em.Instantiate(proto, 100_000);

// —— 系统：10w 位移，Burst + 多核并行 ——
[BurstCompile]
public partial struct UnitMovementSystem : ISystem
{
    [BurstCompile] public void OnUpdate(ref SystemState state)
        => new MoveJob { Dt = SystemAPI.Time.DeltaTime }.ScheduleParallel();

    [BurstCompile]
    partial struct MoveJob : IJobEntity
    {
        public float Dt;
        void Execute(ref LocalTransform t, in Velocity v, in UnitTag _)
            => t.Position += v.Value * Dt;
    }
}
```

> per-instance 差异（如颜色）用 `MaterialPropertyOverride`（DOTS Instancing，batch 友好），不要为此建新材质。

> 完整可跑的工程骨架（含 Authoring/Baker、Spawn、VAT 动画系统）见 [[【实战案例】10w单位渲染与动画最小Demo]]。

```csharp
// === 组件定义（全 unmanaged struct）===

public struct Velocity      : IComponentData { public float3 Value; }

public struct UnitTypeIndex : IComponentData { public byte Value; }  // 0..11，索引区分类型

[MaterialProperty("_BaseColor")] public struct UnitColor : IComponentData { public float4 Value; }

public struct UnitAlive : IComponentData, IEnableableComponent { }

```
❌ 错误做法（archetype 爆炸）：
   弓兵 → {Bow, RangeAttack}
   步兵 → {Sword, MeleeAttack}
   → N 种组件组合 = N 种 archetype = N 个 batch 起步

✅ 正确做法（统一 archetype）：
   全部单位 → {UnitTag, Velocity, UnitTypeIndex, UnitColor, UnitAlive}
   类型差异靠 MaterialMeshInfo 索引（byte → mesh/材质查表）

  → 启动时 Addressables 加载 → RenderMeshArray 注册 → Instantiate

### 第 4 步：Burst Job 并行更新

```csharp
[BurstCompile]
public partial struct UnitMovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
        => new MoveJob { Dt = SystemAPI.Time.DeltaTime }.ScheduleParallel();

    [BurstCompile]
    partial struct MoveJob : IJobEntity
    {
        public float Dt;
        void Execute(
            ref LocalTransform transform,
            in Velocity velocity,
            in UnitAlive _    // disabled 的 entity 自动跳过
        )
        {
            transform.Position += velocity.Value * Dt;
            if (math.lengthsq(velocity.Value) > 0.001f)
            {
                float3 dir = math.normalizesafe(velocity.Value);
                transform.Rotation = quaternion.LookRotation(dir, math.up());
            }
        }
    }
}
```

### 第 5 步：LOD + 阴影配置

```
LOD 配置（Baking 自动处理 或 代码添加）：
  LOD0: 屏幕占比 > 0.4  → 高精度 mesh（500-2000 面）
  LOD1: 0.1 ~ 0.4       → 中精度 mesh（100-300 面）
  LOD2: < 0.1           → 低精度 mesh（10-50 面）

阴影策略：
  10w 单位全投影 → GPU 深度 Pass 爆炸
  ✅ RenderMeshDescription(shadowCastingMode: Off)  → 关投影
  ✅ 或仅 LOD0 投影（近处少数单位），LOD1/LOD2 不投影
```

### 第 6 步：Profiler 验证

```
上线前在目标机用 Profiler + Frame Debugger 检查：

Profiler CPU:
  ├─ UnitMovementSystem        ~1-2ms（Burst 并行）
  ├─ EntitiesGraphicsSystem     ~1-3ms（culling + batching）
  ├─ 无主线程热点（EntityManager.SetComponentData 禁用）
  └─ 无结构性变更尖刺（CreateEntity/DestroyEntity）

Frame Debugger:
  ├─ DrawCall 数 = 个位数 ~ 36（mesh 种类 × LOD 数）
  ├─ 每 batch 实例数 > 1000（否则 archetype/mesh 分裂）
  └─ 无 transparent/backbuffer 多余 Pass

Profiler GPU:
  ├─ Vertex shader（含 VAT 采样）开销
  └─ Overdraw（透明叠加、粒子）→ 主要瓶颈转移到这里
```

## 工程避坑

| 坑 | 现象 | 解决 |
|---|---|---|
| archetype 爆炸 | DrawCall 数不降 | 用 `UnitTypeIndex byte` + MaterialMeshInfo 索引，不用不同组件区分类型 |
| 频繁创建/销毁 | 帧率暴跌 | `IEnableableComponent` + 对象池，不做结构性变更 |
| 改 ISharedComponentData | 帧率暴跌 | 共享组件只做静态分组，不每帧改 |
| 阴影性能 | GPU 帧时间 >10ms | 10w 单位关阴影或仅 LOD0 投影 |
| Overdraw | GPU 填充率满 | 激进 LOD + 透明粒子限量 + 分辨率缩放 |
| main thread 阻塞 | 主线程 >16ms | 所有 10w 级操作必须在 Job 里 |
| MaterialPropertyBlock | 不生效 | ECS 不支持 MPB，必须用 DOTS Instancing 的 `[MaterialProperty]` |
| MaterialMeshInfo 大量切换 | batch 重排序卡顿 | 初始化设好不每帧切；必须切时分帧（每帧 1000 个） |
| Addressables 释放后渲染崩溃 | 粉屏 / mesh 丢失 | 先 DestroyEntity → 再 Addressables.Release |
| LOD 切换闪烁 | 远处单位抖动 | 设 LOD cross-fade 或放宽切换阈值 |

## 性能预算（经验估算，需实测）

| 项 | 10w 简单 mesh（<100 三角面） |
|----|------------------------------|
| CPU 位移更新（Burst 并行） | ~1–2ms |
| DrawCall 数 | 个位数（同 batch） |
| 主要瓶颈转移 | **overdraw**（透明/粒子）、LOD 切换闪烁、动画、AI 决策 |

## 进阶：GPU-driven（>10w 时）

规模继续上探（几十万）时，连 Burst job 的 CPU 更新都嫌贵：用 **Compute Shader 在 GPU 端算位置**，C# 只喂初始数据（或读回），CPU 近乎零更新。这是 Entities Graphics 之外的进阶路线，适合 RTS 大军、粒子云等超大规模场景。

## 何时 ECS 反而不是最优解

若 10w 单位是**纯 2D sprite / 粒子**，`Graphics.RenderMeshInstanced` 或 VFX Graph 可能比完整 ECS 栈更省事。ECS 的优势在单位有**独立逻辑/状态**时才体现；纯视觉粒子用 GPU 粒子系统更直接。

## 速查清单

- [ ] archetype 数量压到最少（用 MaterialMeshInfo 索引区分，不靠不同组件）
- [ ] 单位位移用 IJobEntity + Burst + ScheduleParallel，禁主线程 foreach / EntityManager.SetComponentData
- [ ] per-instance 差异用 `[MaterialProperty]` IComponentData（DOTS Instancing），不建新材质
- [ ] 静态单位走 SubScene + Baking；DLC/热更走 Addressables + RenderMeshArray 注册
- [ ] SubScene 用 `SceneSystem.LoadSceneAsync` 加载（Section 分块流式），不用 Addressables
- [ ] SubScene 和 Addressables 不交叉引用同一份 mesh/材质（否则包体重复）
- [ ] 运行时切 MaterialMeshInfo 分帧分散（每帧 ≤1000），避免 batch 重排序
- [ ] 设 MeshLODGroup + 准确的 LOD 屏幕百分比范围（激进 LOD 省填充率）
- [ ] 死活用 IEnableableComponent + 对象池，禁频繁 CreateEntity/DestroyEntity
- [ ] 阴影按需关（RenderMeshDescription 的 ShadowCastingMode），或仅 LOD0 投影
- [ ] URP 渲染路径必须 Forward+（Entities Graphics 不支持 Forward 非+）
- [ ] 释放资源前先 DestroyEntity，再 Addressables.Release
- [ ] 上线前 Profiler 实测：关注 overdraw、LOD 切换、batch 数、无主线程热点
