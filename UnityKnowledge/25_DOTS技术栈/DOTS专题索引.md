## DOTS三大支柱

### 1. ECS (Entity Component System)

**核心理念**：
- Entity = ID
- Component = 纯数据
- System = 纯逻辑

**优势**：
```
├─ 数据与逻辑分离
├─ 高性能缓存友好
├─ 易于并行处理
└─ 内存布局优化
```

**文档**：
- [[【教程】ECS架构入门]] ← ECS入门
- [[设计原理-ECS为什么快]] ← ECS原理

---

### 2. C# Job System

**核心理念**：
- 多线程任务
- 安全的并行编程
- 无GC数据结构

**优势**：
```
├─ 多线程并行计算
├─ 避免主线程阻塞
├─ 自动负载均衡
└─ 安全的数据访问
```

**文档**：
- [[【教程】JobSystem详解]] ← Job System完整教程

---

### 3. Burst Compiler

**核心理念**：
- 基于LLVM的编译器
- C#转高性能本地代码
- SIMD优化

**优势**：
```
├─ 性能提升10-100倍
├─ 低功耗
└─ 跨平台
```

**文档**：
- [[【教程】Burst编译器]] ← Burst完整教程

## 使用场景

### 适合DOTS的场景

```
✅ 高性能计算
├─ 大量相同对象（粒子、敌人、子弹）
├─ 复杂计算（路径查找、物理模拟）
└─ 需要并行处理的场景

✅ 内存敏感
├─ 移动平台
├─ 大量对象
└─ 内存有限

✅ CPU密集型
├─ 物理模拟
├─ AI行为
└─ 寻路算法
```

### 不适合DOTS的场景

```
❌ 简单逻辑
└─ 性能要求不高

❌ 快速原型
└─ 开发速度优先

❌ UI/脚本逻辑
└─ DOTS不适合UI

❌ 需要拖拽配置
└─ Inspector依赖的功能
```

## 常见问题

### Q1: DOTS一定比OOP快吗？

**A**: 不一定

```
DOTS优势的场景：
├─ 大量相同对象（>1000个）
├─ 复杂计算
└─ CPU密集型

OOP优势的场景：
├─ 对象数量少（<100个）
├─ 简单逻辑
└─ 快速开发

结论：根据场景选择
```
