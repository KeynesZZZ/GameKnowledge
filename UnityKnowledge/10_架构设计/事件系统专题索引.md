## 关键概念

### 事件系统核心思想

```
传统方式（紧耦合）：
A模块 ──直接调用──> B模块
  ↑ 如果B变化，A也要改

事件方式（解耦）：
A模块 ──发布事件──> Event Bus ──订阅──> B模块
                       ↑
                    C模块也在订阅
                    A完全不知道B、C的存在
```

### 事件系统的优势

✅ **解耦**：模块间无需直接依赖
✅ **扩展**：新增订阅者无需修改发布者
✅ **灵活**：运行时动态订阅/取消订阅
✅ **清晰**：事件流向明确，易于追踪

### 事件系统的代价

❌ **复杂度**：比直接调用复杂
❌ **调试**：事件链路追踪困难
❌ **性能**：有轻微性能开销
❌ **内存**：忘记取消订阅会导致内存泄漏

## 方案对比

### 三种事件方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **C# event** | 简单、性能好 | 紧耦合、不支持序列化 | 单个类内部通信 |
| **UnityEvent** | Inspector配置、支持序列化 | 性能差、装箱拆箱 | 跨脚本通信、UI交互 |
| **事件总线** | 完全解耦、灵活 | 复杂、调试困难 | 跨模块通信、大型项目 |

详细对比：[[架构决策-事件系统实现对比]]

## 常见问题

### Q1: 事件和委托的区别？

**A**:
- **委托（Delegate）**：类型安全的函数指针
- **事件（Event）**：基于委托的发布订阅模式，是对委托的封装

事件是对委托的封装，提供了更安全的访问控制。

参见：[[设计原理-事件驱动本质]]

### Q2: UnityEvent和C# event哪个好？

**A**: 各有优劣：
- **UnityEvent**：适合Inspector配置、序列化场景
- **C# event**：适合代码中调用、性能要求高

参见：[[架构决策-事件系统实现对比]]

### Q3: 如何避免内存泄漏？

**A**: 订阅事件时必须在合适时机取消订阅：
```csharp
private void OnEnable()
{
    EventBus.Subscribe<GameEvent>(OnGameEvent);
}

private void OnDisable()
{
    EventBus.Unsubscribe<GameEvent>(OnGameEvent);
}
```

### Q4: 事件可以传递参数吗？

**A**: 可以。事件总线支持泛型参数：
```csharp
public class PlayerEvent
{
    public int PlayerId;
    public float Health;
}

EventBus.Publish<PlayerEvent>(new PlayerEvent { ... });
```

参见：[[代码片段-事件总线]]

### Q5: UnityEvent的性能问题？

**A**: UnityEvent存在装箱拆箱问题，性能比C# event差。在高频调用场景（如每帧调用）建议使用C# event。

参见：[[../../20_核心系统/26_UI系统/【源码解析】Unity事件系统实现机制]]

## 实战案例

### 案例1：玩家事件系统

```csharp
// 定义事件
public class PlayerEvent
{
    public int Level;
    public float Exp;
}

// 发布事件
EventBus.Publish<PlayerEvent>(new PlayerEvent { Level = 5, Exp = 1000f });

// 订阅事件
EventBus.Subscribe<PlayerEvent>(OnPlayerLevelUp);

private void OnPlayerLevelUp(PlayerEvent evt)
{
    Debug.Log($"Player leveled up to {evt.Level}");
}
```

### 案例2：UI事件系统

```csharp
// 按钮点击
public void OnButtonClick()
{
    EventBus.Publish("ButtonClick", this);
}

// UI管理器订阅
EventBus.Subscribe("ButtonClick", OnButtonClicked);
```

### 案例3：跨场景通信

```csharp
// 使用DontDestroyOnLoad的事件管理器
public class GlobalEventManager : MonoBehaviour
{
    void Awake() => DontDestroyOnLoad(this);
}
```

## 最佳实践

### ✅ DO

- 使用强类型事件（避免魔法字符串）
- 及时取消订阅（避免内存泄漏）
- 事件命名清晰（PlayerDied而不是OnEvent）
- 提供事件文档（列出所有事件及其参数）

### ❌ DON'T

- 不要在Update中发布事件（性能差）
- 不要在事件处理中发布新事件（导致事件风暴）
- 不要忘记取消订阅（内存泄漏）
- 不要过度使用事件（简单场景直接调用更好）
