# 事件系统实现对比

> Unity中多种事件系统实现方式的对比分析

## 概述

事件系统是解耦组件通信的核心机制，本篇对比Unity中常见的实现方式。

## 实现方式

### 1. C# 原生事件

```csharp
public class EventProvider
{
    public event Action<int> OnScoreChanged;

    public void UpdateScore(int score)
    {
        OnScoreChanged?.Invoke(score);
    }
}

public class EventConsumer
{
    public EventConsumer(EventProvider provider)
    {
        provider.OnScoreChanged += HandleScoreChanged;
    }

    private void HandleScoreChanged(int score)
    {
        Debug.Log($"Score: {score}");
    }
}
```

**优点**: 类型安全、零分配
**缺点**: 需要手动取消订阅、强引用

### 2. UnityEvent

```csharp
public class UnityEventExample : MonoBehaviour
{
    [SerializeField] private UnityEvent<int> onScoreChanged;

    public void TriggerScore(int score)
    {
        onScoreChanged.Invoke(score);
    }
}
```

**优点**: Inspector可配置、支持持久化监听
**缺点**: 每次Invoke有分配、性能较低

### 3. ScriptableObject事件

```csharp
[CreateAssetMenu(fileName = "GameEvent", menuName = "Events/GameEvent")]
public class GameEvent : ScriptableObject
{
    private readonly List<GameEventListener> listeners = new();

    public void Raise()
    {
        for (int i = listeners.Count - 1; i >= 0; i--)
        {
            listeners[i].OnEventRaised();
        }
    }

    public void RegisterListener(GameEventListener listener)
    {
        listeners.Add(listener);
    }

    public void UnregisterListener(GameEventListener listener)
    {
        listeners.Remove(listener);
    }
}
```

**优点**: 架构解耦、可跨场景使用
**缺点**: 需要额外的Listener组件

### 4. EventBus（全局事件总线）

```csharp
public static class EventBus
{
    private static readonly Dictionary<Type, List<Delegate>> events = new();

    public static void Subscribe<T>(Action<T> callback) where T : struct
    {
        var type = typeof(T);
        if (!events.ContainsKey(type))
        {
            events[type] = new List<Delegate>();
        }
        events[type].Add(callback);
    }

    public static void Publish<T>(T eventData) where T : struct
    {
        var type = typeof(T);
        if (events.TryGetValue(type, out var callbacks))
        {
            foreach (var callback in callbacks)
            {
                ((Action<T>)callback).Invoke(eventData);
            }
        }
    }
}

// 使用结构体避免分配
public struct ScoreChangedEvent
{
    public int Score;
}
```

**优点**: 完全解耦、类型安全
**缺点**: 需要管理订阅生命周期

## 性能对比

| 实现方式 | Invoke开销 | 内存分配 | 类型安全 |
|----------|------------|----------|----------|
| C# Event | 最低 | 无 | ✅ |
| UnityEvent | 较高 | 有 | ✅ |
| SO Event | 中等 | 无 | ❌ |
| EventBus | 中等 | 无 | ✅ |

## 最佳实践

### DO ✅

- 使用结构体作为事件数据避免GC
- 在OnDisable/OnDestroy时取消订阅
- 优先使用类型安全的事件系统

### DON'T ❌

- 不要在事件处理中抛出异常
- 不要创建循环事件依赖
- 不要忘记取消订阅导致内存泄漏

## 选择指南

| 场景 | 推荐方案 |
|------|----------|
| Inspector配置 | UnityEvent |
| 跨系统通信 | EventBus |
| 配置驱动 | ScriptableObject Event |
| 高频事件 | C# 原生事件 |
