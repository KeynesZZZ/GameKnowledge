# 【最佳实践】单例模式

> 核心价值：在需要全局唯一访问时，提供简洁、线程安全的单例实现

## 概述

单例模式是最常用的设计模式之一，在Unity中有多种实现方式，各有优劣。

## 实现方式对比

### 1. 懒汉式（非线程安全）

```csharp
public class LazySingleton
{
    private static LazySingleton instance;

    public static LazySingleton Instance
    {
        get
        {
            if (instance == null)
            {
                instance = new LazySingleton();
            }
            return instance;
        }
    }
}
```

**优点**: 简单、延迟初始化
**缺点**: 非线程安全
**适用**: Unity主线程环境

### 2. 饿汉式

```csharp
public class EagerSingleton
{
    private static readonly EagerSingleton instance = new EagerSingleton();

    public static EagerSingleton Instance => instance;

    private EagerSingleton() { }
}
```

**优点**: 线程安全、实现简单
**缺点**: 启动时初始化
**适用**: 初始化成本低、必定使用的单例

### 3. MonoSingleton

```csharp
public class MonoSingleton<T> : MonoBehaviour where T : MonoSingleton<T>
{
    private static T instance;
    private static readonly object lockObj = new object();

    public static T Instance
    {
        get
        {
            if (instance == null)
            {
                lock (lockObj)
                {
                    if (instance == null)
                    {
                        var go = new GameObject($"[{typeof(T).Name}]");
                        instance = go.AddComponent<T>();
                        DontDestroyOnLoad(go);
                    }
                }
            }
            return instance;
        }
    }

    protected virtual void Awake()
    {
        if (instance == null)
        {
            instance = this as T;
            DontDestroyOnLoad(gameObject);
        }
        else if (instance != this)
        {
            Destroy(gameObject);
        }
    }
}
```

**优点**: 支持Inspector配置、生命周期管理
**缺点**: 需要MonoBehaviour
**适用**: GameManager、AudioManager等

### 4. ScriptableObject单例

```csharp
[CreateAssetMenu(fileName = "GameConfig", menuName = "Config/GameConfig")]
public class GameConfig : ScriptableObject
{
    private static GameConfig instance;

    public static GameConfig Instance
    {
        get
        {
            if (instance == null)
            {
                instance = Resources.Load<GameConfig>("GameConfig");
            }
            return instance;
        }
    }
}
```

**优点**: 可视化配置、支持热更新
**缺点**: 需要Resources加载
**适用**: 配置类单例

## 最佳实践

### DO ✅

- 使用泛型基类减少重复代码
- 在Awake中处理重复实例
- 考虑使用Dependency Injection替代
- 为MonoBehaviour单例添加DontDestroyOnLoad

### DON'T ❌

- 不要在单例构造函数中调用Unity API
- 不要过度使用单例导致全局依赖
- 不要忘记处理场景切换时的生命周期

## 使用场景

| 场景 | 推荐实现 | 理由 |
|------|----------|------|
| GameManager | MonoSingleton | 需要MonoBehaviour生命周期 |
| 配置管理 | ScriptableObject | 可视化配置 |
| 工具类 | 静态类或懒汉式 | 无状态管理 |
| 网络管理 | MonoSingleton | 需要协程支持 |
