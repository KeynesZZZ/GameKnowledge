## 一、问题背景：没有对象池会怎样？

### 1.1 频繁实例化的代价

```csharp
// 射击游戏中的子弹
public class Bullet : MonoBehaviour
{
    private void Update()
    {
        transform.position += transform.forward * speed * Time.deltaTime;
    }

    private void OnTriggerEnter(Collider other)
    {
        // 击中后销毁
        Destroy(gameObject);  // ← 这里发生了什么？
    }
}

// 发射子弹
public void Fire()
{
    var bullet = Instantiate(bulletPrefab, spawnPoint.position, spawnPoint.rotation);
    // ↑ 这里又发生了什么？
}
```

**看似简单的 Instantiate/Destroy，背后发生了什么？**

### 1.2 内存分配过程

```

│                 Instantiate 的幕后操作                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 托管堆分配                                              │
│     └─ new GameObject() → 堆上分配内存                      │
│     └─ 所有 MonoBehaviour 组件 → 各自分配                   │
│     └─ 所有字段（数组、字符串等）→ 递归分配                 │
│                                                             │
│  2. Unity内部操作                                           │
│     └─ C++ 层创建 GameObject 结构                          │
│     └─ Transform 层级计算                                   │
│     └─ 渲染组件注册                                         │
│     └─ 物理组件注册                                         │
│                                                             │
│  3. 脚本初始化                                              │
│     └─ Awake() 调用                                         │
│     └─ OnEnable() 调用                                      │
│                                                             │
│  时间成本：0.1ms ~ 几毫秒（取决于复杂度）                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Destroy 的真正代价

```
┌─────────────────────────────────────────────────────────────┐
│                 Destroy 的幕后操作                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 标记为"待销毁"（不是立即删除！）                        │
│                                                             │
│  2. 当前帧结束才真正销毁                                    │
│     └─ 为什么？避免中途修改导致的问题                       │
│                                                             │
│  3. 产生的"垃圾"                                            │
│     └─ 托管堆上的引用 → 变成垃圾                           │
│     └─ 但内存不会立即释放！                                │
│                                                             │
│  4. GC（垃圾回收）会在某时刻触发                            │
│     └─ 暂停所有线程                                         │
│     └─ 标记-清除-压缩                                       │
│     └─ 恢复线程                                             │
│     └─ 这就是"卡顿"的来源！                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、对象池的核心原理

### 2.1 本质思想：空间换时间

```
传统方式：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   需要 → 创建 → 使用 → 销毁 → GC → 需要 → 创建 → ...       │
│          ↑              ↑                                   │
│        分配成本        回收成本                             │
│                                                             │
│   每次都有成本！                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

对象池方式：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   启动时：预创建 N 个对象                                   │
│           ┌───┬───┬───┬───┬───┐                            │
│           │ 0 │ 1 │ 2 │ 3 │ 4 │  ← 池中待用                │
│           └───┴───┴───┴───┴───┘                            │
│                                                             │
│   运行时：                                                  │
│   需要 → 取出 → 使用 → 归还 → 需要 → 取出 → ...            │
│          ↑         ↑                                        │
│        O(1)     O(1)                                        │
│                                                             │
│   几乎零成本！                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 对象池的数据结构

```csharp
public class SimpleObjectPool
{
    // 为什么用 Stack 而不是 List？
    // - Stack: Push/Pop 都是 O(1)，且不产生GC
    // - List: Add/RemoveAt(end) 也是 O(1)，但 RemoveAt(middle) 是 O(n)

    private readonly Stack<GameObject> pool = new Stack<GameObject>();
    private readonly GameObject prefab;

    public GameObject Get()
    {
        if (pool.Count > 0)
        {
            // 复用已有对象
            var obj = pool.Pop();  // O(1)
            obj.SetActive(true);
            return obj;
        }
        else
        {
            // 池中无可用对象，创建新的
            return Instantiate(prefab);
        }
    }

    public void Return(GameObject obj)
    {
        obj.SetActive(false);
        pool.Push(obj);  // O(1)
    }
}
```

### 2.3 为什么 SetActive(false) 而不是真的删除？

```
┌─────────────────────────────────────────────────────────────┐
│                   SetActive(false) 的效果                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 保留的内容：                                            │
│     ├─ GameObject 引用（托管堆上不被GC）                   │
│     ├─ 所有组件及其状态                                     │
│     └─ 已分配的内存                                         │
│                                                             │
│  ❌ 停止的行为：                                            │
│     ├─ Update() 不再调用                                    │
│     ├─ 渲染（不在视野内）                                   │
│     ├─ 物理模拟                                             │
│     └─ 协程暂停                                             │
│                                                             │
│  本质：对象还在，只是"休眠"                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、对象池的权衡

### 3.1 收益

| 收益 | 说明 |
|------|------|
| **避免GC** | 对象常驻内存，不产生垃圾 |
| **分配快速** | Get/Return 都是 O(1) |
| **帧率稳定** | 没有突发的创建/销毁开销 |
| **内存可预测** | 可以预估最大内存占用 |

### 3.2 代价

| 代价 | 说明 |
|------|------|
| **内存占用** | 对象常驻内存，即使不用 |
| **复杂度增加** | 需要管理池的生命周期 |
| **状态重置** | 归还时必须重置状态 |
| **调试困难** | 对象不是新建的，可能有残留状态 |

### 3.3 什么时候不该用对象池

```csharp
// ❌ 不适合池化的场景

// 1. 对象很少创建（如 Boss）
//    - 创建一次，用完就没了
//    - 池化反而浪费内存

// 2. 对象状态极其复杂
//    - 重置成本 > 创建成本
//    - 容易出现状态污染

// 3. 内存极其紧张
//    - 移动端低内存设备
//    - 需要及时释放内存

// 4. 对象存活时间很长
//    - 如玩家角色
//    - 池化没有意义
```

### 3.4 适合池化的场景

```csharp
// ✅ 适合池化的场景

// 1. 频繁创建销毁
//    - 子弹、特效、伤害数字
//    - 列表项、弹窗

// 2. 创建成本高
//    - 复杂UI
//    - 带大量组件的对象

// 3. 数量可控
//    - 同屏最多N个子弹
//    - 列表最多显示M项
```

## 四、深入理解：GC 的工作原理

### 4.1 为什么 GC 会导致卡顿

```
┌─────────────────────────────────────────────────────────────┐
│                    C# 垃圾回收机制                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  代际假说 (Generational Hypothesis):                        │
│  ├─ 大部分对象很快变成垃圾（短命）                          │
│  └─ 少部分对象长期存活（长寿）                              │
│                                                             │
│  Unity 使用分代 GC (Generational GC):                       │
│                                                             │
│  ┌─────────────────────────────────────────┐                │
│  │ Gen 0 (最年轻)                          │                │
│  │ └─ 新创建的对象                         │                │
│  │ └─ 回收频率：高                         │                │
│  │ └─ 回收速度：快                         │                │
│  ├─────────────────────────────────────────┤                │
│  │ Gen 1 (中间)                            │                │
│  │ └─ 存活一次 GC 的对象                   │                │
│  │ └─ 回收频率：中                         │                │
│  ├─────────────────────────────────────────┤                │
│  │ Gen 2 (最老)                            │                │
│  │ └─ 长期存活的对象                       │                │
│  │ └─ 回收频率：低                         │                │
│  │ └─ 回收速度：慢！ ← Full GC             │                │
│  └─────────────────────────────────────────┘                │
│                                                             │
│  对象池的对象 → Gen 2（长期存活）→ 几乎不参与 GC            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 对象池如何避免 GC

```
无对象池时：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  创建 ──► Gen 0 ──► (很快变成垃圾) ──► GC 回收              │
│  创建 ──► Gen 0 ──► (很快变成垃圾) ──► GC 回收              │
│  创建 ──► Gen 0 ──► (很快变成垃圾) ──► GC 回收              │
│                                                             │
│  GC 频繁触发！                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘

有对象池时：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  启动 ──► 创建对象 ──► Gen 2（长期存活）                    │
│                                                             │
│  运行时：                                                   │
│  取出 ──► 使用 ──► 归还 ──► 取出 ──► 使用 ──► 归还         │
│                                                             │
│  没有新对象创建 = 没有垃圾 = GC 不触发                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五、常见误区

### 5.1 误区：对象池一定比 Instantiate 快

```
真实情况：

首次获取（池为空）：
├─ Instantiate + 池初始化
└─ 比直接 Instantiate 更慢！

后续获取：
├─ 只是 SetActive(true)
└─ 比 Instantiate 快很多

结论：对象池的优势在于"重复使用"
```

### 5.2 误区：所有东西都应该池化

```
真实情况：

池化的代价：
├─ 内存常驻
├─ 代码复杂度
└─ 状态管理

判断标准：
├─ 每秒创建/销毁超过 10 次？→ 考虑池化
├─ 创建成本超过 0.1ms？→ 考虑池化
└─ 对象状态简单？→ 适合池化
```

### 5.3 误区：忘了重置状态

```csharp
// ❌ 危险：不重置状态
public void Return(GameObject obj)
{
    obj.SetActive(false);
    pool.Push(obj);
    // 如果对象上有 "速度"、"目标" 等状态
    // 下次取出来时，这些状态还在！
}

// ✅ 正确：重置状态
public void Return(GameObject obj)
{
    var bullet = obj.GetComponent<Bullet>();
    bullet.Reset();  // 重置速度、目标等

    obj.SetActive(false);
    pool.Push(obj);
}
```

### 对象池本质速查

```
┌─────────────────────────────────────────────────────────────┐
│                    对象池本质                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  核心思想：空间换时间                                       │

│  解决的问题：                                               │
│  └─ 频繁创建/销毁导致的 GC 卡顿                             │

│  代价：                                                     │
│  └─ 内存常驻、代码复杂、状态管理                            │
│                                                             │
│  适用场景：                                                 │
│  └─ 频繁创建销毁 + 状态简单 + 数量可控                      │
│                                                             │
│  不适用场景：                                               │
│  └─ 很少创建 + 状态复杂 + 内存紧张                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
