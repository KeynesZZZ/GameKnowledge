> 核心问题：没有设计模式会怎样？设计模式能解决什么问题？

## 文档定位

本文档从**底层机制角度**深入讲解为什么要用设计模式的本质原理。

**相关文档**：[[【设计原理】为什么要用设计模式]]

## 一、核心问题：没有设计模式会怎样？

### 1.1 代码腐化的过程

```
项目时间线
────────────────────────────────────────────────────────>

第1周:  代码清晰，功能简单
        └─ Player.Move() → 完成

第1月:  开始出现重复
        └─ Player.Move()
        └─ Enemy.Move()    ← 复制粘贴
        └─ NPC.Move()      ← 又复制

第3月:  到处修改
        └─ 修改移动逻辑需要改3个地方
        └─ 改了Player忘记改Enemy
        └─ Bug层出不穷

第6月:  不敢修改
        └─ "这段代码别动，动了就崩"
        └─ 新功能只能绕着写
        └─ 技术债务累积
```

### 1.2 本质问题：耦合

```csharp
// ❌ 高耦合代码：改动一处，影响多处
public class Player : MonoBehaviour
{
    private void Update()
    {
        // 移动逻辑直接写在Player里
        float h = Input.GetAxis("Horizontal");
        transform.position += new Vector3(h, 0, 0) * speed * Time.deltaTime;

        // 动画逻辑也在这里
        animator.SetFloat("Speed", Mathf.Abs(h));

        // 音效也在这里
        if (Mathf.Abs(h) > 0.1f && !audioSource.isPlaying)
        {
            audioSource.Play();
        }

        // 这意味着：
        // 1. 想改移动方式？改Player
        // 2. 想改动画触发？改Player
        // 3. 想改音效逻辑？还是改Player
        // 4. 想复用移动逻辑给Enemy？复制粘贴
    }
}
```

**问题本质**：
- **职责不清**：一个类做了太多事
- **难以复用**：逻辑被锁死在具体类里
- **难以测试**：依赖太多，无法隔离测试
- **难以扩展**：加功能意味着改代码

## 二、设计模式解决了什么？

### 2.1 设计模式的本质

```
┌─────────────────────────────────────────────────────────────┐
│                    设计模式本质                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   设计模式 = 前人总结的「解耦方案」                         │
│                                                             │
│   ┌─────────────┐         ┌─────────────┐                  │
│   │  问题场景    │  ───►  │  标准解法    │                  │
│   └─────────────┘         └─────────────┘                  │
│                                                             │
│   对象创建太分散？    →    工厂模式                         │
│   状态切换太混乱？    →    状态模式                         │
│   通知关系太复杂？    →    观察者模式                       │
│   功能扩展太困难？    →    策略模式                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 解耦后的代码

```csharp
// ✅ 低耦合代码：职责分离，易于扩展

// 移动逻辑独立
public class MovementSystem
{
    public void Move(Transform target, Vector3 direction, float speed)
    {
        target.position += direction * speed * Time.deltaTime;
    }
}

// 动画逻辑独立
public class AnimationController
{
    public void UpdateMovementAnimation(Animator animator, float speed)
    {
        animator.SetFloat("Speed", speed);
    }
}

// 音效逻辑独立
public class AudioController
{
    public void PlayMovementSound(AudioSource source, bool isMoving)
    {
        if (isMoving && !source.isPlaying)
            source.Play();
        else if (!isMoving && source.isPlaying)
            source.Stop();
    }
}

// Player只负责协调
public class Player : MonoBehaviour
{
    private MovementSystem movement;
    private AnimationController animation;
    private AudioController audio;

    private void Update()
    {
        float h = Input.GetAxis("Horizontal");

        // 委托给各个系统
        movement.Move(transform, new Vector3(h, 0, 0), speed);
        animation.UpdateMovementAnimation(animator, Mathf.Abs(h));
        audio.PlayMovementSound(audioSource, Mathf.Abs(h) > 0.1f);
    }
}
```

**改进效果**：
- **职责清晰**：每个类只做一件事
- **易于复用**：MovementSystem可以给Enemy、NPC用
- **易于测试**：可以单独测试每个系统
- **易于扩展**：加新功能不需要改现有代码

## 三、设计模式的核心原理

### 3.1 设计原则（SOLID）

```
┌─────────────────────────────────────────────────────────────┐
│                    SOLID 设计原则                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  S - 单一职责原则 (Single Responsibility)                   │
│      └─ 一个类只做一件事                                    │
│                                                             │
│  O - 开闭原则 (Open/Closed)                                 │
│      └─ 对扩展开放，对修改关闭                              │
│                                                             │
│  L - 里氏替换原则 (Liskov Substitution)                     │
│      └─ 子类可以替换父类                                    │
│                                                             │
│  I - 接口隔离原则 (Interface Segregation)                   │
│      └─ 接口要小而专一                                      │
│                                                             │
│  D - 依赖倒置原则 (Dependency Inversion)                    │
│      └─ 依赖抽象，不依赖具体                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 原则与模式的对应

| 问题 | 违反的原则 | 解决的模式 |
|------|-----------|-----------|
| 一个类做太多事 | 单一职责 | 外观模式、命令模式 |
| 加功能必须改代码 | 开闭原则 | 策略模式、工厂模式 |
| 创建对象太复杂 | 依赖倒置 | 工厂模式、依赖注入 |
| 对象间通知太乱 | 依赖倒置 | 观察者模式、事件总线 |
| 状态切换是if-else | 开闭原则 | 状态模式 |
| 需要撤销操作 | 单一职责 | 命令模式 |

## 四、设计模式的代价

### 4.1 不是所有问题都需要模式

```
┌─────────────────────────────────────────────────────────────┐
│                    模式的代价                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 使用模式的收益：                                        │
│     ├─ 代码更易维护                                        │
│     ├─ 更容易扩展                                          │
│     ├─ 更容易测试                                          │
│     └─ 团队协作更容易                                      │
│                                                             │
│  ❌ 使用模式的代价：                                        │
│     ├─ 类数量增加                                          │
│     ├─ 理解成本上升                                        │
│     ├─ 简单问题复杂化                                      │
│     └─ 可能过度设计                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 什么时候不该用模式

```csharp
// ❌ 过度设计：为简单逻辑创建复杂模式

// 只有一个Enemy类型，却创建了工厂
public interface IEnemyFactory
{
    IEnemy Create(EnemyType type);
}

public class EnemyFactory : IEnemyFactory
{
    public IEnemy Create(EnemyType type)
    {
        return type switch
        {
            EnemyType.Basic => new BasicEnemy(),  // 就这一种！
            _ => throw new ArgumentException()
        };
    }
}

// ✅ 适度设计：简单场景用简单方案
public class EnemySpawner : MonoBehaviour
{
    [SerializeField] private GameObject enemyPrefab;

    public void Spawn()
    {
        Instantiate(enemyPrefab);
    }
}
```

**判断标准**：

| 场景 | 建议 |
|------|------|
| 原型/Demo | 最简单实现，不用模式 |
| 小型项目（<1万行） | 关键模块用模式 |
| 中型项目（1-10万行） | 核心架构用模式 |
| 大型项目（>10万行） | 全面使用模式 |

## 五、Unity中常用的模式

### 5.1 模式使用频率

```
高频使用（每个项目都用）
├─ 单例模式 ─── GameManager, AudioManager
├─ 对象池 ───── 子弹、特效、UI
├─ 观察者/事件 ─ UI更新、游戏事件
└─ 状态机 ──── 角色状态、游戏流程

中频使用（根据需求使用）
├─ 命令模式 ── 输入处理、技能系统
├─ 策略模式 ── AI行为、技能效果
├─ 工厂模式 ── 敌人生成、道具生成
└─ 外观模式 ── 复杂子系统封装

低频使用（特殊场景）
├─ 迭代器 ─── 自定义集合
├─ 备忘录 ─── 存档系统
├─ 责任链 ─── 伤害计算
└─ 访问者 ─── 复杂数据处理
```

### 5.2 Unity内置的模式

| 模式 | Unity中的体现 |
|------|--------------|
| **组合模式** | GameObject + Component |
| **观察者模式** | UnityEvent, Action |
| **策略模式** |不同的Shader, Physics Material |
| **工厂模式** | Instantiate(), ScriptableObject |
| **单例模式** | FindObjectOfType (伪单例) |
| **状态模式** | Animator Controller |

## 六、学习路径建议

### 6.1 循序渐进

```
第一阶段：理解问题
└─ 先写出能运行的代码
└─ 感受代码变乱的过程
└─ 理解"为什么需要模式"

第二阶段：掌握基础模式
└─ 单例（但要理解滥用的问题）
└─ 对象池
└─ 观察者/事件
└─ 状态机

第三阶段：理解原则
└─ 学习SOLID原则
└─ 理解"为什么这个模式这样设计"
└─ 开始能判断"该不该用模式"

第四阶段：灵活运用
└─ 不拘泥于标准模式
└─ 根据实际需求变通
└─ 代码简洁 + 易维护
```

### 6.2 推荐学习顺序

```
1. 先体验痛苦
   └─ 写一个不用模式的项目
   └─ 体会扩展困难

2. 学习最常用的3个
   └─ 对象池（性能）
   └─ 事件系统（解耦）
   └─ 状态机（流程控制）

3. 理解设计原则
   └─ SOLID原则
   └─ 为什么这些原则重要

4. 按需学习其他模式
   └─ 遇到问题时再学
   └─ 带着问题学效率更高
```
