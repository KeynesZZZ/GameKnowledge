# 【反模式】常见架构陷阱

> 核心价值：识别陷阱，避免踩坑，提升代码质量

## 一、什么是反模式？

```
┌─────────────────────────────────────────────────────────────┐
│                    反模式定义                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  反模式 = 看似解决了问题，实际上制造更多问题的"常见错误"   │
│                                                             │
│  特点：                                                     │
│  ├─ 初看起来是合理的解决方案                                │
│  ├─ 短期内可能有效                                          │
│  ├─ 长期会带来更大问题                                      │
│  └─ 在项目中反复出现                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、常见反模式

### 2.1 单例滥用

```csharp
// ❌ 反模式：什么都做成单例
public class PlayerSingleton : MonoBehaviour { ... }
public class EnemySingleton : MonoBehaviour { ... }
public class BulletSingleton : MonoBehaviour { ... }
public class UISingleton : MonoBehaviour { ... }
public class ConfigSingleton : MonoBehaviour { ... }

// 问题：
// 1. 全局状态混乱
// 2. 无法测试
// 3. 隐藏依赖
// 4. 多场景冲突

// ✅ 正确做法：按需使用
// 只有真正的全局服务才用单例
// 如：GameManager, AudioManager, SaveSystem
```

### 2.2 事件风暴

```csharp
// ❌ 反模式：事件触发事件
public void OnPlayerDeath(PlayerDeathEvent e)
{
    EventBus.Publish(new ScoreChangedEvent());  // 事件1
}

public void OnScoreChanged(ScoreChangedEvent e)
{
    EventBus.Publish(new AchievementCheckEvent());  // 事件2
}

public void OnAchievementCheck(AchievementCheckEvent e)
{
    EventBus.Publish(new SaveGameEvent());  // 事件3
}

// 问题：
// 1. 调用链不可预测
// 2. 可能无限循环
// 3. 调试困难

// ✅ 正确做法：直接调用或命令模式
public void OnPlayerDeath(PlayerDeathEvent e)
{
    scoreManager.AddScore(score);
    achievementSystem.CheckAchievements();
    saveSystem.Save();
}
```

### 2.3 Update泛滥

```csharp
// ❌ 反模式：每个物体都有Update
public class Bullet : MonoBehaviour
{
    private void Update()
    {
        transform.position += direction * speed * Time.deltaTime;
    }
}

// 1000个子弹 = 1000个Update调用
// 问题：
// 1. 大量Unity内部开销
// 2. 难以统一管理
// 3. 无法批量优化

// ✅ 正确做法：管理器模式
public class BulletManager : MonoBehaviour
{
    private List<Bullet> bullets = new List<Bullet>();

    private void Update()
    {
        foreach (var bullet in bullets)
        {
            bullet.transform.position += bullet.direction * bullet.speed * Time.deltaTime;
        }
    }
}
```

### 2.4 GameObject.Find每帧

```csharp
// ❌ 反模式：Update里Find
public class Enemy : MonoBehaviour
{
    private void Update()
    {
        var player = GameObject.Find("Player");  // 每帧都搜索！
        if (player != null)
        {
            // 追踪玩家
        }
    }
}

// 问题：GameObject.Find 是 O(n) 操作
// 100个敌人 × 每帧 = 性能灾难

// ✅ 正确做法：缓存引用
public class Enemy : MonoBehaviour
{
    private Transform player;

    private void Start()
    {
        player = GameObject.FindWithTag("Player").transform;
    }

    private void Update()
    {
        if (player != null)
        {
            // 追踪玩家
        }
    }
}
```

### 2.5 字符串做所有事

```csharp
// ❌ 反模式：字符串滥用
public void PlayAnimation(string animName)
{
    animator.Play(animName);  // 拼写错误不报错！
}

public void LoadScene(string sceneName)
{
    SceneManager.LoadScene(sceneName);  // 运行时才发现错误
}

public void GetItem(string itemId)
{
    // 字符串比较，没有类型安全
}

// ✅ 正确做法：枚举或常量
public enum AnimationType { Idle, Walk, Run, Jump }
public enum SceneType { Menu, Game, Pause }

public void PlayAnimation(AnimationType anim)
{
    animator.Play(anim.ToString());
}
```

### 2.6 协程嵌套地狱

```csharp
// ❌ 反模式：协程嵌套
private IEnumerator AttackSequence()
{
    yield return StartCoroutine(MoveToTarget());
    yield return StartCoroutine(PlayAttackAnim());
    yield return StartCoroutine(DealDamage());
    yield return StartCoroutine(PlayEffect());
    yield return StartCoroutine(ReturnToPosition());
}

// 问题：
// 1. 难以取消
// 2. 难以调试
// 3. 状态不清晰

// ✅ 正确做法：状态机或async/await
private async UniTask AttackSequence()
{
    await MoveToTarget();
    await PlayAttackAnim();
    await DealDamage();
    await PlayEffect();
    await ReturnToPosition();
}
```

### 2.7 过度抽象

```csharp
// ❌ 反模式：为了"未来可能"而抽象
public interface IEntity
{
    int Id { get; }
}

public interface IMovingEntity : IEntity
{
    void Move(Vector3 direction);
}

public interface IAttackingEntity : IMovingEntity
{
    void Attack();
}

public interface IDamageableEntity : IEntity
{
    void TakeDamage(float damage);
}

// 只有一个实现
public class Player : MonoBehaviour, IAttackingEntity, IDamageableEntity
{
    // ...
}

// 问题：
// 1. 过度设计
// 2. 增加复杂度
// 3. YAGNI (You Aren't Gonna Need It)

// ✅ 正确做法：需要时再抽象
// 先写具体实现，发现需要复用/扩展时再抽象
```

### 2.8 Inspector公开所有

```csharp
// ❌ 反模式：所有字段public
public class Player : MonoBehaviour
{
    public float health;
    public float maxHealth;
    public float speed;
    public float jumpForce;
    public int level;
    public int experience;
    public List<Item> inventory;
    // ... 所有都public

// 问题：
// 1. 破坏封装
// 2. 任何地方都能修改
// 3. 难以追踪变化

// ✅ 正确做法：SerializeField + 属性
public class Player : MonoBehaviour
{
    [SerializeField] private float maxHealth = 100f;
    [SerializeField] private float speed = 5f;

    private float currentHealth;

    public float Health => currentHealth;
    public float MaxHealth => maxHealth;

    public void TakeDamage(float damage)
    {
        currentHealth = Mathf.Max(0, currentHealth - damage);
    }
}
```

## 三、反模式速查表

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| **单例滥用** | 全局状态混乱 | 按需使用，考虑依赖注入 |
| **事件风暴** | 调用链不可控 | 直接调用或命令模式 |
| **Update泛滥** | 性能开销大 | 管理器统一更新 |
| **每帧Find** | O(n)开销 | 缓存引用 |
| **字符串滥用** | 无类型安全 | 使用枚举/常量 |
| **协程嵌套** | 难以控制 | 状态机或async/await |
| **过度抽象** | 复杂度增加 | 需要时再抽象 |
| **Inspector全开** | 破坏封装 | SerializeField + 属性 |

## 四、如何识别反模式

### 4.1 代码异味

```
当你看到以下情况，警惕反模式：

├─ 一个类超过 500 行
├─ 方法超过 50 行
├─ 嵌套超过 3 层
├─ 参数超过 5 个
├─ 一个文件有 10 个类
├─ Update 里有 GameObject.Find
├─ 字符串到处传递
├─ 注释说"TODO: 以后优化"
└─ 修改一个bug产生两个新bug
```

### 4.2 自检问题

```
问自己：
├─ 这个设计是解决问题还是制造问题？
├─ 如果现在不做，以后会很麻烦吗？
├─ 新人能理解这段代码吗？
├─ 修改这个会影响其他地方吗？
├─ 这个代码能复用吗？
└─ 如果需求变了，改起来容易吗？
```
