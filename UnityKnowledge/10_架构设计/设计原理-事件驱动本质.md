# 设计原理 - 事件驱动本质

> 理解事件驱动架构的耦合解耦原理 `#设计原理` `#事件系统` `#解耦`

## 问题背景：直接调用的问题

### 场景：玩家受到伤害

```csharp
// ❌ 直接调用：紧耦合
public class Player : MonoBehaviour
{
    [SerializeField] private HealthBar healthBar;
    [SerializeField] private AchievementSystem achievementSystem;
    [SerializeField] private QuestSystem questSystem;
    [SerializeField] private SoundManager soundManager;
    [SerializeField] private EffectManager effectManager;
    [SerializeField] private AnalyticsManager analyticsManager;
    // ... 更多依赖

    public void TakeDamage(int damage)
    {
        currentHealth -= damage;

        // 直接调用各个系统
        healthBar.UpdateHealth(currentHealth, maxHealth);
        achievementSystem.OnPlayerDamaged(damage);
        questSystem.CheckDamageObjective(damage);
        soundManager.PlaySound("hurt");
        effectManager.SpawnEffect("blood", transform.position);
        analyticsManager.LogEvent("player_damaged");

        // 每次新增功能都要改这里！
    }
}
```

## 事件驱动的本质

### 核心思想

```
┌─────────────────────────────────────────────────────────────┐
│                     事件驱动本质                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   "Don't call us, we'll call you"                          │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                                                     │  │
│   │   直接调用：A ──────► B                             │  │
│   │              A 必须知道 B 的存在                    │  │
│   │                                                     │  │
│   │   事件驱动：A ──► 事件 ──► B                        │  │
│   │                   ▲                                 │  │
│   │                   │                                 │  │
│   │                   └── C, D, E...                    │  │
│   │                                                     │  │
│   │   A 只需要知道"发生了什么事"                        │  │
│   │   不需要知道"谁会处理"                              │  │
│   │                                                     │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 重构后

```csharp
// ✅ 事件驱动：解耦
public class Player : MonoBehaviour
{
    public void TakeDamage(int damage)
    {
        currentHealth -= damage;

        // 只发布事件，不关心谁处理
        EventBus.Publish(new PlayerDamagedEvent
        {
            Damage = damage,
            CurrentHealth = currentHealth,
            Position = transform.position
        });
    }
}

// 各系统独立订阅
public class HealthBar : MonoBehaviour
{
    private void OnEnable() => EventBus.Subscribe<PlayerDamagedEvent>(OnPlayerDamaged);
    private void OnDisable() => EventBus.Unsubscribe<PlayerDamagedEvent>(OnPlayerDamaged);

    private void OnPlayerDamaged(PlayerDamagedEvent e)
    {
        UpdateDisplay(e.CurrentHealth);
    }
}

public class AchievementSystem : MonoBehaviour
{
    private void OnEnable() => EventBus.Subscribe<PlayerDamagedEvent>(OnPlayerDamaged);
    private void OnDisable() => EventBus.Unsubscribe<PlayerDamagedEvent>(OnPlayerDamaged);

    private void OnPlayerDamaged(PlayerDamagedEvent e)
    {
        CheckDamageAchievement(e.Damage);
    }
}

// 新增系统不需要改Player！
public class NewFeature : MonoBehaviour
{
    private void OnEnable() => EventBus.Subscribe<PlayerDamagedEvent>(OnPlayerDamaged);
    // ...
}
```

│ HealthBar Achiev Quest Sound Effect Analytics NewFeature   │

## 观察者模式 vs 发布订阅模式

### 两种模式对比

```
┌─────────────────────────────────────────────────────────────┐
│                   观察者模式                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Subject ──────────► Observer                             │
│     │                     │                                 │
│     │    直接通知         │                                 │
│     └─────────────────────┘                                 │
│                                                             │
│   特点：                                                    │
│   • Subject知道Observer的存在                               │
│   • 同步调用                                                │
│   • 一对多关系                                              │
│                                                             │
│   Unity例子：                                               │
│   • C# delegate/event                                       │
│   • UnityEvent                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  发布订阅模式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Publisher ──► Event Channel ──► Subscriber               │
│                     │                                       │
│              中间人/经纪人                                   │
│                                                             │
│   特点：                                                    │
│   • Publisher不知道Subscriber                               │
│   • 通过中间人解耦                                          │
│   • 可以异步                                                │
│   • 可以持久化                                              │
│                                                             │
│   Unity例子：                                               │
│   • EventBus / MessageBus                                   │
│   • ScriptableObject事件                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 选择指南

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| **UI按钮点击** | 观察者 | 简单直接，一对一 |
| **全局游戏事件** | 发布订阅 | 解耦多个系统 |
| **技能触发效果** | 发布订阅 | 效果可插拔 |
| **属性变化通知UI** | 观察者 | 紧密关联 |
| **跨场景通信** | 发布订阅 | 生命周期独立 |

## 事件驱动的代价

### 隐藏的成本

```
┌─────────────────────────────────────────────────────────────┐
│                   事件驱动的代价                            │
├──────────────────────┬──────────────────────────────────────┤
│        代价          │              说明                    │
├──────────────────────┼──────────────────────────────────────┤
│  流程不直观          │  看代码不知道谁会响应事件            │
│  调试困难            │  事件链追踪复杂                      │
│  性能开销            │  事件分发、委托调用有成本            │
│  内存泄漏风险        │  忘记取消订阅                        │
│  时序问题            │  事件处理顺序不确定                  │
│  过度使用            │  简单调用也用事件，增加复杂度        │
└──────────────────────┴──────────────────────────────────────┘
```

### 调试困难示例

```csharp
// 直接调用：调用栈清晰
A.Method() → B.Method() → C.Method()
// 调试时F11一步步进入

// 事件驱动：调用栈断裂
A.Publish(event) → ??? → B.Handler() → ??? → C.Handler()
// 中间发生了什么？不知道！
```

### 内存泄漏示例

```csharp
// ❌ 忘记取消订阅
public class Enemy : MonoBehaviour
{
    private void Start()
    {
        EventBus.Subscribe<PlayerDamagedEvent>(OnPlayerDamaged);
        // 忘记在OnDestroy取消订阅！
    }

    // Enemy被销毁后，EventBus还持有它的引用
    // → Enemy无法被GC回收 → 内存泄漏！
}

// ✅ 正确做法
public class Enemy : MonoBehaviour
{
    private void OnEnable() => EventBus.Subscribe<PlayerDamagedEvent>(OnPlayerDamaged);
    private void OnDisable() => EventBus.Unsubscribe<PlayerDamagedEvent>(OnPlayerDamaged);
}
```

## 事件风暴问题

### 问题：事件过多导致混乱

```
┌─────────────────────────────────────────────────────────────┐
│                     事件风暴                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Event A ──► Handler X ──► Event B ──► Handler Y ──► ...  │
│       │                          │                          │
│       │                          ▼                          │
│       │                     Event C ──► Handler Z           │
│       │                          │                          │
│       ▼                          ▼                          │
│   Event D                   Event E                         │
│       │                          │                          │
│       └──────────┬───────────────┘                          │
│                  ▼                                          │
│            事件循环/死锁                                     │
│                                                             │
│   问题：                                                    │
│   • 谁触发了什么？不知道                                    │
│   • 执行顺序是什么？不确定                                  │
│   • 性能瓶颈在哪？难以定位                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 解决方案

public class EventHistory

## 何时使用事件驱动

### 决策树

```
┌─────────────────────────────────────────────────────────────┐
│                    是否使用事件驱动？                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 调用者和被调用者是否需要解耦？                          │
│     └─ 否 → 直接调用                                       │
│                                                             │
│  2. 是否有多个监听者？                                      │
│     └─ 否 → 考虑直接调用或回调                             │
│                                                             │
│  3. 监听者是否会变化（运行时增减）？                        │
│     └─ 否 → 考虑直接依赖                                   │
│                                                             │
│  4. 是否需要跨场景/跨生命周期通信？                         │
│     └─ 是 → 事件驱动                                       │
│                                                             │
│  5. 是否需要记录/重放事件？                                 │
│     └─ 是 → 事件驱动                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 游戏中的实际场景

| 场景 | 使用事件？ | 理由 |
|------|-----------|------|
| **玩家死亡** | ✅ 是 | 多系统响应：UI、音效、成就、存档 |
| **敌人攻击** | ✅ 是 | 效果可插拔，便于扩展 |
| **获取物品** | ✅ 是 | 背包、任务、成就都需要知道 |
| **移动控制** | ❌ 否 | 输入→移动，直接调用更简单 |
| **物理碰撞** | ❌ 否 | Unity已有事件系统（OnCollision） |
| **简单的UI更新** | ❌ 否 | 直接赋值更清晰 |

## 最佳实践

### 1. 事件命名规范

```csharp
// ✅ 清晰的事件命名
public struct PlayerDiedEvent { }
public struct EnemySpawnedEvent { public EnemyType Type; }
public struct ItemCollectedEvent { public ItemData Item; }

// ❌ 模糊的事件命名
public struct UpdateEvent { }  // 更新什么？
public struct DataEvent { }    // 什么数据？
```

### 2. 事件数据结构

```csharp
// ✅ 使用struct避免GC
public struct DamageEvent
{
    public int SourceId;
    public int TargetId;
    public int Amount;
    public DamageType Type;
}

// ❌ 使用class产生GC
public class DamageEvent  // 每次new都会分配堆内存
{
    // ...
}
```

### 3. 订阅生命周期管理

```csharp
// ✅ 使用OnEnable/OnDisable配对
private void OnEnable() => EventBus.Subscribe<Event>(Handler);
private void OnDisable() => EventBus.Unsubscribe<Event>(Handler);

// ❌ 使用Start/OnDestroy（可能导致内存泄漏）
private void Start() => EventBus.Subscribe<Event>(Handler);
private void OnDestroy() => EventBus.Unsubscribe<Event>(Handler);
// 如果对象被禁用但没有销毁，仍然会收到事件！
```

### 4. 避免事件中执行耗时操作

```csharp
// ❌ 事件处理中执行耗时操作
private void OnPlayerDied(PlayerDiedEvent e)
{
    // 同步加载场景会阻塞所有事件处理！
    SceneManager.LoadScene("GameOver");
}

// ✅ 延迟处理
private void OnPlayerDied(PlayerDiedEvent e)
{
    // 标记状态，下一帧处理
    isPlayerDead = true;
}

private void Update()
{
    if (isPlayerDead)
    {
        isPlayerDead = false;
        StartCoroutine(LoadGameOverScene());
    }
}
```

## 总结

```
┌─────────────────────────────────────────────────────────────┐
│                     事件驱动本质总结                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：直接调用 → 紧耦合 → 难以扩展、难以测试              │
│                                                             │
│  本质：发布者不关心订阅者，通过中间人解耦                   │
│                                                             │
│  代价：流程不直观、调试困难、性能开销、内存泄漏风险         │
│                                                             │
│  适用：                                                    │
│  • 多系统响应同一事件                                       │
│  • 需要解耦的模块                                           │
│  • 跨场景/跨生命周期通信                                    │
│                                                             │
│  不适用：                                                   │
│  • 简单的一对一调用                                         │
│  • 调用关系固定的场景                                       │
│  • 需要明确执行顺序的场景                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
