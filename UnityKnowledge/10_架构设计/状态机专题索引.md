# 状态机专题索引

> 状态机模式的完整知识体系：从理论到实践

## 专题概览

状态机（Finite State Machine, FSM）是游戏开发中管理**对象状态和状态转换**的核心技术，通过定义状态和转换规则，让对象行为更清晰、更易维护。

## 关键概念

### 状态机核心思想

```
传统方式（if-else混乱）：
if (state == "idle") {
    if (input == "attack") { ... }
    else if (input == "move") { ... }
}
else if (state == "attack") {
    if (input == "hit") { ... }
}
// 状态越多，逻辑越复杂

状态机方式（清晰明确）：
State ──(Event)──> Transition ──> State
  ↑                        ↓
  └────────────────────────┘
  每个状态独立，转换规则清晰
```

### 状态机的优势

✅ **清晰**：状态和转换规则一目了然
✅ **易维护**：新增状态不影响现有状态
✅ **可扩展**：灵活添加新状态和转换
✅ **防错**：避免非法状态转换

### 状态机的要素

- **状态（State）**：对象的某种模式或形态
- **事件（Event）**：触发状态转换的条件
- **转换（Transition）**：从一个状态到另一个状态
- **动作（Action）**：进入/退出状态时执行的操作

## 通用状态机 vs Animator状态机

### 通用状态机

**特点**：
- 纯C#代码实现
- 完全自定义控制
- 适用于游戏逻辑状态
- AI行为、敌人状态、游戏流程等

**优势**：
- 灵活性高
- 性能好
- 易于调试

**适用场景**：
- 敌人AI状态（巡逻、追击、攻击、死亡）
- 游戏流程状态（菜单、游戏、暂停、结算）
- UI状态（显示、隐藏、动画）

**实现**：
- [[设计原理-状态机本质]]
- [[教程-状态机模式]]
- [[代码片段-有限状态机]]

### Animator状态机

**特点**：
- Unity内置系统
- 可视化编辑
- 针对动画优化
- 与动画系统深度集成

**优势**：
- 可视化状态流转
- 动画混合和过渡
- 与Animation Clip集成
- 支持Blend Tree

**适用场景**：
- 角色动画状态（Idle、Walk、Run、Attack）
- 动画过渡和混合
- 复杂动画逻辑

**实现**：
- [[../../20_核心系统/21_动画系统/【设计原理】Animator状态机]]

## 常见问题

### Q1: 什么时候用通用状态机，什么时候用Animator？

**A**: 判断标准：

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 角色动画 | Animator | 动画混合、可视化、与动画系统集成 |
| 敌人AI | 通用状态机 | 游戏逻辑、灵活性、性能 |
| UI动画 | Animator | 简单动画状态、可视化 |
| 游戏流程 | 通用状态机 | 业务逻辑、代码控制 |

### Q2: 状态机和Switch语句的区别？

**A**:
- **Switch语句**：代码组织方式，仍在Update中判断
- **状态机**：设计模式，状态独立管理、转换规则清晰

状态机是对Switch的封装和提升。

参见：[[设计原理-状态机本质]]

### Q3: 如何避免状态爆炸？

**A**:
- 使用层次状态机（HSM）
- 将大状态拆分为小状态
- 使用状态组合
- 考虑其他模式（如行为树）

### Q4: Animator状态机性能如何？

**A**: Animator有一定性能开销，但Unity已优化：
- 状态机本身开销很小
- 动画混合有一定开销
- 建议复用AnimatorController
- 避免过多复杂状态

参见：[[../../20_核心系统/21_动画系统/【最佳实践】Animator状态机最佳实践]]

### Q5: 如何在状态机中传递数据？

**A**: 多种方式：
- 使用全局状态管理器
- 状态机参数（Animator）
- 状态基类中的公共属性
- 事件系统传递

## 实战案例

### 案例1：敌人AI状态机

```csharp
public enum EnemyState
{
    Patrol,
    Chase,
    Attack,
    Dead
}

public class EnemyFSM : MonoBehaviour
{
    private EnemyState currentState;

    public void ChangeState(EnemyState newState)
    {
        // 退出当前状态
        OnExitState(currentState);

        // 切换状态
        currentState = newState;

        // 进入新状态
        OnEnterState(newState);
    }

    private void OnEnterState(EnemyState state)
    {
        switch (state)
        {
            case EnemyState.Patrol:
                // 初始化巡逻
                break;
            case EnemyState.Chase:
                // 初始化追击
                break;
            // ...
        }
    }
}
```

### 案例2：Animator状态机配置

```
Animator Controller结构：

Base Layer
├─ Idle
├─ Walk
├─ Run
├─ Attack
└─ Die

Parameters:
- Speed (Float)
- IsGrounded (Bool)
- AttackTrigger (Trigger)

Transitions:
- Idle → Walk (Speed > 0.1)
- Walk → Run (Speed > 5.0)
- Any → Attack (AttackTrigger)
- Any → Die (Health <= 0)
```

### 案例3：层次状态机

```
主状态机：
├─ 移动状态
│  ├─ 行走
│  ├─ 跑步
│  └─ 跳跃
├─ 战斗状态
│  ├─ 攻击
│  ├─ 防御
│  └─ 受击
└─ 死亡状态

每个子状态可以有自己的状态机
```

## 最佳实践

### ✅ DO

- 明确定义状态和转换规则
- 使用状态机模式管理复杂状态逻辑
- 动画相关优先使用Animator
- 游戏逻辑优先使用通用状态机
- 使用状态基类封装公共逻辑

### ❌ DON'T

- 不要在Update中用大量if-else判断状态
- 不要让状态机过于复杂（考虑层次状态机）
- 不要混用状态机和全局变量管理状态
- 不要在状态机中处理业务逻辑（应该调用其他系统）

## 进阶阅读

### 层次状态机（HSM）
- 状态可以包含子状态
- 减少状态数量
- 更好的组织结构

### 并发状态机
- 多个状态机并行运行
- 适用于复杂对象
- 注意状态冲突

### 下推自动机
- 用于UI导航
- 历史记录管理
- 堆栈式状态管理
