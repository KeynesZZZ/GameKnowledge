## 二、DrawCall优化

### 2.1 什么是DrawCall

```
DrawCall定义：
├─ CPU命令GPU绘制对象
├─ 包含：网格、材质、Shader参数
├─ 每次DrawCall有CPU开销
└─ DrawCall越多，CPU负担越重

性能影响：
├─ 每次DrawCall：约0.01-0.1ms CPU时间
├─ 100个DrawCall：约1-10ms
└─ 1000个DrawCall：可能成为瓶颈
```

---

### 2.2 DrawCall优化策略

#### 策略1：静态批处理（Static Batching）

```csharp
// 场景设置
GameObject obj1 = GameObject.CreatePrimitive(PrimitiveType.Cube);
obj1.isStatic = true;  // 标记为静态

GameObject obj2 = GameObject.CreatePrimitive(PrimitiveType.Sphere);
obj2.isStatic = true;  // 标记为静态

// Unity自动合并这些对象的网格
// 减少DrawCall
```

**优点**：
- 自动合并，无需手动操作
- 显著减少DrawCall
- 适合不移动的对象

**缺点**：
- 合并后无法移动
- 增加内存占用
- 不适合大量对象

---

#### 策略2：动态批处理（Dynamic Batching）

```csharp
// 小网格自动合并
// 条件：
// 1. 顶点数<900（移动平台）
// 2. 相同Material
// 3. 相同Shader

GameObject enemy1 = Instantiate(enemyPrefab);
GameObject enemy2 = Instantiate(enemyPrefab);
// Unity自动批处理，减少DrawCall
```

**优点**：
- 对象可以移动
- 自动处理

**缺点**：
- 有顶点数限制
- CPU开销（合并计算）
- 移动平台限制更严

---

#### 策略3：GPU Instancing（推荐）

```csharp
// Shader中使用Instancing
// instancingShader.shader
Shader "Custom/InstancingShader"
{
    SubShader
    {
        // 启用Instancing
        InstancingMode = InstancingMode.NoInstance  // 禁用
        InstancingMode = InstancingMode.AllInstances  // 启用

        Pass
        {
            // 使用实例ID
            UNITY_INSTANCING_BUFFER_START(int, instanceID)
            UNITY_INSTANCING_BUFFER_END(int, instanceID)

            // Shader代码...
        }
    }
}

// C#中使用
Graphics.DrawMeshInstanced(mesh, 0, material, matrices);
```

**优点**：
- 最优性能
- 支持大量对象
- 无内存开销

**缺点**：
- Shader需要支持
- 需要特定材质属性

### 2.4 SetPass Call：比 DrawCall 更底层的指标

> DrawCall 是「画」的命令，SetPass Call 是「切换渲染状态」的命令。**SetPass Call 才是真正限制合批的瓶颈。**

#### 什么是 SetPass Call

```
渲染一帧时，CPU 对 GPU 下达两类核心命令：

  SetPass Call（设通道）           DrawCall（画）
  ──────────────────              ────────────
  "接下来用这个 Shader 的          "把顶点/三角形
   第 N 个 Pass，                  按刚才设好的通道
   绑定这些纹理/参数"              画出来"

  开销来源：                       开销来源：
  Shader 切换 → GPU 状态重置       顶点数据提交
  纹理/CBuffer 绑定                （现代 GPU 上很轻）
  混合/深度测试模式切换

  一次 SetPass 之后可以跟多次 DrawCall（= 一个 Batch）
```

**一句话：SetPass Call 是切换渲染状态的命令。每次 Shader/Pass/材质切换 → 新的 SetPass → 新的 Batch → 无法合批。**

#### SetPass vs DrawCall vs Batch 三者关系

```
Frame Debugger 逐条展开：

  SetPass #1 (Shader A, Pass 0)         ← SetPass Call ①
  ├─ Draw Mesh (Cube)                    ← DrawCall ①
  └─ Draw Mesh (Sphere)                  ← DrawCall ②   → 同一个 Batch

  SetPass #2 (Shader A, Pass 1, Shadow)  ← SetPass Call ②
  └─ Draw Mesh (Cube + Sphere)           ← DrawCall ③

  SetPass #3 (Shader B, Pass 0)         ← SetPass Call ③
  └─ Draw Mesh (Plane)                   ← DrawCall ④

统计：
  SetPass Calls = 3      ← = Batch 数
  Draw Calls    = 4

关系：一个 Batch = 一次 SetPass + N 次 DrawCall
      SetPass Call 数 = Batch 数（URP 下）
```

#### 什么情况会产生 SetPass Call

```
① Shader 变化（最贵）
   Shader A → Shader B → GPU 切换整个着色器程序 → pipeline state 重置

② Pass 变化（同一 Shader 内）
   多 Pass Shader 的每个 Pass 都是一次 SetPass
   例：ForwardAdd Pass（逐像素光源）、Shadow Caster Pass

③ 材质参数变化（中等）
   同 Shader 但不同材质实例 → 纹理/CBuffer 绑定变化
   SRP Batcher 可消除此类开销

④ 渲染类型切换
   Opaque → Transparent → 需要切换混合模式 → 新 SetPass
```

#### 常见 SetPass 杀手

**杀手 1：多光源 → 多 Pass**

```
URP 附加光源（Additional Lights）：

  方向光（主光源）：1 个 Pass
  每个附加点光源：+1 个 Pass（Per Vertex 或 Per Pixel）

  场景 8 个点光源 + 1 方向光：
    Built-in Forward：理论 = 1 + 8 = 9 SetPass/物体
    URP Forward+：附加光合到 1 Pass → 1 + 1 = 2 SetPass

  → Forward+ 路径是多光源场景的必选项
```

**杀手 2：阴影 → ShadowCaster Pass**

```
投射阴影的物体 → 额外跑 ShadowCaster Pass：

  正常渲染：1 SetPass（物体）
  投射阴影：+1 SetPass（Shadow Map）

  1000 个投射阴影的物体 → 额外 1000 SetPass
  → 大规模单位（10w）必须关阴影或仅 LOD0 投影
```

**杀手 3：材质实例过多**

```
100 个物体用同一材质 → 1 SetPass（可合批）
100 个物体用 100 个材质 → 100 SetPass（无法合批）

  常见原因：
  - 运行时 new Material(source) 创建材质副本
  - 脚本动态改 renderer.material 属性（自动 clone 材质）
  → 用 MaterialPropertyBlock 或 SharedMaterial 替代
```

**杀手 4：多 Pass 特效 Shader**

```
自定义多 Pass Shader（描边、溶解、全屏后效）：

  Shader "Custom/Outline" {
      Pass 0 { /* 正常渲染 */ }     ← SetPass 1
      Pass 1 { /* 描边 */ }         ← SetPass 2
      Pass 2 { /* 溶解边缘 */ }     ← SetPass 3
  }

  N 个同种物体用同 3 Pass → 仍 3 SetPass（同 Shader 可合批）
  但 N 个不同 Shader → N × 3 SetPass
```

#### 怎么查 SetPass Call

```
方法 1：Game 视图 Stats 面板
  → 显示 Batches / SetPass Calls / Draw Calls
  → 快速看全局，超标第一时间发现

方法 2：Frame Debugger（Window → Analysis → Frame Debugger）
  → 逐 SetPass 步进，看每个 Pass 画了什么
  → "Why isn't this batched?" 提示合并失败原因
  → 排查 SetPass 爆炸的最佳工具

方法 3：Profiler → Rendering
  → 搜索 "Render.Render" → 看 SetPass 调用栈
  → 定位是哪个 Pass 贵

方法 4：ProfilerRecorder API（2022+）
  → 编程抓取 Batches / SetPass Calls 指标做自动化监控
```

### 2.5 SRP Batcher：减少 SetPass 的利器

> URP 默认开启的合批优化。核心思想：**同 Shader 的不同材质不再需要重新 SetPass，只更新 CBuffer 常量。**

#### 原理

```
传统管线（无 SRP Batcher）：
  物体 A（Shader X, Mat 1）→ SetPass → DrawCall
  物体 B（Shader X, Mat 2）→ SetPass → DrawCall   ← 材质不同 → 重新 SetPass
  物体 C（Shader X, Mat 3）→ SetPass → DrawCall
  → 3 SetPass Calls

SRP Batcher（URP 默认开启）：
  物体 A/B/C 都用同一 Shader（兼容 SRP Batcher）
  → 不重新 SetPass！只更新 CBUFFER 里的 per-material 常量
  → 1 次 SetPass + 3 次 CBuffer 更新 + 3 DrawCall
  → SetPass Calls 从 3 降到 1
```

#### 兼容条件

```
Shader 必须使用 CBUFFER 包裹材质属性：

  CBUFFER_START(UnityPerMaterial)
      float4 _BaseColor;
      float  _Cutoff;
      // ... 所有材质参数
  CBUFFER_END

✅ 满足条件：
  - 所有材质属性在 UnityPerMaterial CBUFFER 内
  - 不使用 MaterialPropertyBlock（URP 下会 break SRP Batcher）
  - URP / HDRP（Built-in RP 不支持）

❌ 破坏 SRP Batcher：
  - 材质属性散落在 CBUFFER 外
  - 使用 MaterialPropertyBlock
  - Shader 中使用 multi_compile 导致变体差异
```

> URP 内置 Shader（Lit/Unlit/SimpleLit）默认兼容 SRP Batcher。自定义 Shader 需手动加 CBUFFER。

#### 四种合批方案对比

```
                    SetPass 开销         合批能力               适用
  ─────────────     ──────────────       ────────────────       ──────────────
  传统渲染           每次材质切换         不同材质 → 拆           基线
  GPU Instancing     一次 SetPass         同 mesh + 同材质 → 合   大量相同物体
  SRP Batcher        一次 SetPass         同 Shader → 合          URP 通用（默认开）
  DOTS Instancing    一次 SetPass         同 mesh/材质 → 合       10w 级 ECS

优先级：SRP Batcher（通用打底）> GPU Instancing（大量同类）> DOTS Instancing（超大规模）
```

#### 实测对比（1000 个同 Shader 不同参数的物体）

```
                      SetPass    DrawCall    Batch
  无 SRP Batcher:      ~1000      ~1000       ~1000
  开 SRP Batcher:        ~1-3      ~1000       ~1-3
  开 GPU Instancing:     ~1-3       ~1-3       ~1-3
  DOTS Instancing:        ~1         ~1          ~1
```

### 2.6 SetPass Call 优化策略

```
优先级（从高到低）：

  ① 减少材质/Shader 种类（根治）
     → 合并纹理图集，用 UV 区分而非不同材质
     → 统一用同一套 Shader（URP Lit/Unlit），参数区分外观

  ② 开 SRP Batcher（URP 默认开）
     → 确保自定义 Shader 兼容 CBUFFER_START(UnityPerMaterial)
     → 避免 MaterialPropertyBlock（URP 下会 break）

  ③ 关掉不必要的阴影（大规模场景）
     → ShadowCaster Pass 是 SetPass 翻倍的最大来源
     → 远处单位关投影或仅 LOD0 投影

  ④ 减少附加光源 / 切 Forward+ 路径
     → Forward+ 把多光源合到少数 Pass
     → 附加光源越多 → SetPass 越多

  ⑤ GPU Instancing / DOTS Instancing
     → 同 mesh 同材质 → 一次 SetPass 画完所有实例

  ⑥ LOD 降级减少参与渲染的物体数
     → 远处物体用同一低精度 mesh → 同 Batch
```

### 3.1 LOD原理

```
┌─────────────────────────────────────────────────────────────┐
│                      LOD原理                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  距离远 → 使用低精度模型（减少三角形）                       │

│  距离近 → 使用高精度模型                                    │
│                                                             │
│  效果：                                                     │
│  ├─ 远处对象看不见细节，使用低模型                           │
│  ├─ 近处对象需要细节，使用高模型                             │
│  └─ 整体三角形数量大幅减少                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 LOD实现

```csharp
using UnityEngine;

public class LODController : MonoBehaviour
{
    public LOD[] lods;

    [System.Serializable]
    public class LOD
    {
        public float distance;
        public GameObject[] renderers;
    }

    void Update()
    {
        float distance = Vector3.Distance(Camera.main.transform.position, transform.position);

        for (int i = 0; i < lods.Length; i++)
        {
            bool shouldShow = distance < lods[i].distance;

            foreach (var renderer in lods[i].renderers)
            {
                renderer.SetActive(shouldShow);
            }
        }
    }
}

// 使用Unity LOD Group组件
LODGroup lodGroup = gameObject.AddComponent<LODGroup>();
LOD[] lods = new LOD[3];

lod0: 0-50% distance → high quality model
lod1: 50-80% distance → medium quality model
lod2: 80-100% distance → low quality model
```

### 3.3 LOD优化建议

```
LOD级别建议：
├─ LOD0（近）：100%三角形
├─ LOD1（中）：50%三角形
├─ LOD2（远）：25%三角形
└─ LOD3（极远）：10%三角形或隐藏

切换距离建议：
├─ 小场景：5-10米
├─ 中场景：10-30米
└─ 大场景：30-100米

优化技巧：
✅ 使用3-5个LOD级别
✅ 平滑过渡（淡入淡出）
✅ 结合遮挡剔除
❌ 不要过度LOD（增加内存）
```

## 四、遮挡剔除

### 4.1 视锥体剔除

```
Unity自动剔除：
├─ 视锥体外的对象不渲染
├─ 相机看不到的对象不渲染
└─ 自动处理，无需手动操作

启用方式：
Camera.main.useOcclusionCulling = false;  // 禁用
Camera.main.useOcclusionCulling = true;   // 启用
```

---

### 4.2 遮挡剔除（Occlusion Culling）

```
什么是遮挡剔除？
├─ 被其他物体遮挡的对象不渲染
├─ 即使在视锥体内也不渲染
└─ 手动配置遮挡数据

设置步骤：
1. 烘焙遮挡数据
   Window → Rendering → Occlusion Culling → Bake

2. 配置遮挡参数
   Inspector → Occlusion Area

3. 运行时剔除
   自动生效

适用场景：
✅ 复杂室内场景
✅ 城市环境
✅ 密集建筑群
❌ 开阔场景（效果有限）
```

## 五、GPU Instancing实战

### 5.1 大量相同对象优化

```csharp
using UnityEngine;

public class GrassRenderer : MonoBehaviour
{
    public int grassCount = 10000;
    public Mesh grassMesh;
    public Material grassMaterial;

    private Matrix4x4[] matrices;
    private MaterialPropertyBlock props;

    void Start()
    {
        matrices = new Matrix4x4[grassCount];
        props = new MaterialPropertyBlock();

        // 初始化草的位置
        for (int i = 0; i < grassCount; i++)
        {
            Vector3 position = new Vector3(
                Random.Range(-50f, 50f),
                0,
                Random.Range(-50f, 50f)
            );

            matrices[i] = Matrix4x4.TRS(
                position,
                Quaternion.identity,
                Vector3.one
            );
        }
    }

    void Update()
    {
        // 一次DrawCall渲染所有草
        Graphics.DrawMeshInstanced(
            grassMesh,
            0,
            grassMaterial,
            matrices,
            grassCount,
            props
        );
    }
}
```

**性能对比**：
- 不使用Instancing：10000个DrawCall
- 使用Instancing：1个DrawCall
- **性能提升：10000倍**

### 6.2 移动端Shader优化

```hlsl
// ❌ 复杂Shader（移动端性能差）
Shader "Custom/ComplexShader"
{
    SubShader
    {
        Pass
        {
            CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag

            // 复杂光照计算
            #include "UnityStandardCore.cginc"

            // 多次纹理采样
            sampler2D _MainTex;
            sampler2D _BumpMap;
            sampler2D _EmissionMap;
            sampler2D _DetailMask;

            // 大量计算
            // ...
            ENDCG
        }
    }
}

// ✅ 优化Shader（移动端）
Shader "Mobile/Diffuse"
{
    SubShader
    {
        Pass
        {
            CGPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            #pragma multi_compile_fog

            // 简化光照
            #include "UnityCG.cginc"

            // 减少纹理采样
            sampler2D _MainTex;

            // 简化计算
            struct appdata
            {
                float4 vertex : POSITION;
                float2 uv : TEXCOORD0;
            };

            struct v2f
            {
                float4 pos : SV_POSITION;
                float2 uv : TEXCOORD0;
                UNITY_FOG_COORDS(1)
            };

            v2f vert (appdata v)
            {
                v2f o;
                o.pos = UnityObjectToClipPos(v.vertex);
                o.uv = v.uv;
                UNITY_TRANSFER_FOG(o, o.pos);
                return o;
            }

            fixed4 frag (v2f i) : SV_Target
            {
                fixed4 col = tex2D(_MainTex, i.uv);
                UNITY_APPLY_FOG(i.fogCoord, col);
                return col;
            }
            ENDCG
        }
    }
}
```

## 八、常见问题

### Q1: SetPass Call 和 DrawCall 有什么区别？

**A**: SetPass Call 是切换渲染状态（Shader/Pass/材质），DrawCall 是实际绘制。

```
SetPass Call = "换笔"    → 切换 Shader/材质/Pass，GPU 状态重置
DrawCall     = "画线"    → 提交顶点数据，GPU 绘制

一次 SetPass 之后可以跟多次 DrawCall（= 一个 Batch）
→ 真正限制合批的是 SetPass Call，不是 DrawCall

Frame Debugger 里：
  每个 "Draw Mesh (batched)" = 1 DrawCall
  每次 Shader/Pass 切换 = 1 SetPass Call
```

---

### Q2: DrawCall越少越好吗？

**A**: 不是，要权衡

```
DrawCall过少的问题：
├─ 过度批处理增加内存
├─ 合批对象无法灵活管理
└─ 可能增加CPU开销

合理范围：
├─ 移动平台：50-100
├─ PC平台：200-500
└─ 关键是稳定，不是越少越好
```

---

### Q3: SRP Batcher 开了但 SetPass 没降？

**A**: 大概率是 Shader 不兼容或用了 MaterialPropertyBlock。

```
排查步骤：
1. Frame Debugger → 看是否有 "Node Batcher" 标签
   → 有 = SRP Batcher 生效
   → 无 = 被跳过

2. 检查 Shader 兼容性：
   Inspector → 选中 Shader → 看 "SRP Batcher" 是否 ✅
   → ❌ → 材质属性没在 CBUFFER_START(UnityPerMaterial) 内

3. 检查 MaterialPropertyBlock：
   → URP 下 MPB 会 break SRP Batcher
   → 改用 GPU Instancing + MaterialPropertyBlock（不冲突）
   → 或用 DOTS Instancing（ECS 场景）
```

---

### Q4: LOD会增加内存吗？

**A**: 会，但可以接受

```
LOD内存开销：
├─ 需要存储多个模型
├─ 内存增加约50-100%
└─ 但GPU性能提升更大

权衡：
- 内存便宜，GPU性能宝贵
- 性能提升值得内存开销
```
