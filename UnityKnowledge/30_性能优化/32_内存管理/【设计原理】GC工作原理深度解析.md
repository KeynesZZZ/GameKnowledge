## 一、GC基础概念

### 1.1 什么是GC

```
┌─────────────────────────────────────────────────────────────┐
│                    GC（Garbage Collection）                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  定义：自动内存管理系统                                       │
│  ├─ 自动分配内存                                            │
│  ├─ 自动回收不再使用的内存                                   │
│  └─ 减轻开发者负担                                          │
│                                                             │
│  Unity使用的GC：                                            │
│  ├─ Mono GC（IL2CPP之前）                                   │
│  ├─ Boehm GC（默认）                                        │
│  └─ Incremental GC（增量GC）                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 1.2 托管堆 vs 原生堆

```
内存布局：
┌─────────────────────────────────────────────────────────────┐
│                        内存                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  托管堆（Managed Heap）                                      │
│  ├─ 由GC管理                                               │
│  ├─ 存储托管对象（C#对象）                                 │
│  ├─ 自动分配和回收                                         │
│  └─ 大小：几十MB到几百MB                                   │
│                                                             │
│  原生堆（Native Heap）                                      │
│  ├─ 不由GC管理                                            │
│  ├─ 存储原生对象（Texture、Mesh等）                         │
│  └─ 需要手动释放                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、Unity GC的代际机制

### 2.1 三代GC

```
┌─────────────────────────────────────────────────────────────┐
│                  Unity GC代际机制                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第0代（Generation 0）                                      │
│  ├─ 新分配的对象                                           │
│  ├─ 生存时间短                                             │
│  ├─ GC频率：最高                                           │
│  └─ 回收速度：最快                                         │
│                                                             │
│  第1代（Generation 1）                                      │
│  ├─ 从第0代晋升的对象                                      │
│  ├─ 经过一次GC仍然存活                                     │
│  ├─ GC频率：中等                                           │
│  └─ 回收速度：中等                                         │
│                                                             │
│  第2代（Generation 2）                                      │
│  ├─ 从第1代晋升的对象                                      │
│  ├─ 生存时间长                                             │
│  ├─ GC频率：最低                                           │
│  └─ 回收速度：最慢                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 2.2 对象晋升过程

```
对象生命周期：

1. 分配（Allocation）
   ├─ 对象在第0代分配
   └─ 位置：托管堆末尾

2. GC检查（GC Check）
   ├─ 第0代空间不足时触发
   ├─ 检查第0代对象引用
   └─ 标记存活对象

3. 晋升或回收（Promote or Collect）
   ├─ 存活对象 → 晋升到第1代
   └─ 未引用对象 → 回收内存

4. 重复过程
   ├─ 第1代对象 → 第2代
   └─ 第2代对象 → 长期存活

示例：
var obj = new MyClass();  // 第0代
// 第一次GC，obj存活 → 晋升到第1代
// 第二次GC，obj存活 → 晋升到第2代
```

---

### 2.3 为什么需要代际机制

```
代际假设（Generational Hypothesis）：
├─ 大多数对象寿命很短                                         │
├─ 少数对象寿命很长                                          │
└─ 分代管理可以提升性能                                      │

性能提升：
├─ 第0代GC：快速扫描，回收大部分垃圾                          │
├─ 第1代GC：较少触发，扫描更多对象                           │
├─ 第2代GC：很少触发，全面扫描                               │
└─ 平均性能：优于单代GC                                      │

数据：
├─ 第0代GC：约1-5ms                                         │
├─ 第1代GC：约5-20ms                                        │
└─ 第2代GC：约50-200ms                                      │
```

## 三、GC触发条件

### 3.1 触发条件

```
┌─────────────────────────────────────────────────────────────┐
│                    GC触发条件                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  条件1：代际空间不足                                        │
│  ├─ 第0代达到阈值（通常几MB）                              │
│  ├─ 第1代达到阈值（通常几十MB）                            │
│  └─ 第2代达到阈值（通常几百MB）                            │
│                                                             │
│  条件2：手动调用                                            │
│  ├─ System.GC.Collect()                                    │
│  ├─ Resources.UnloadUnusedAssets()                         │
│  └─ SceneManager.LoadScene()                                │
│                                                             │
│  条件3：系统内存压力                                        │
│  ├─ 移动平台内存告警                                       │
│  ├─ 系统内存不足                                           │
│  └─ Unity主动触发GC                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 3.2 GC阈值

```csharp
// Unity GC的代际阈值（示例，实际值可能不同）
第0代阈值：约2-5MB
├─ 每分配这么多内存触发一次GC
└─ iOS/Android通常较小

第1代阈值：约10-20MB
├─ 从第0代晋升的对象累积到这么多触发GC
└─ 触发频率低于第0代

第2代阈值：约100-200MB
├─ 从第1代晋升的对象累积到这么多触发GC
└─ 触发频率最低，但耗时最长
```

## 四、GC工作流程

### 4.1 GC的四个阶段

```
┌─────────────────────────────────────────────────────────────┐
│                    GC工作流程                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段1：暂停（Stop The World）                               │
│  ├─ 暂停所有托管线程                                        │
│  ├─ 冻结内存状态                                            │
│  └─ 准备开始GC                                              │
│           ↓ 约0.1-1ms                                       │
│                                                             │
│  阶段2：标记（Mark）                                         │
│  ├─ 从根对象（Root）开始扫描                                │
│  ├─ 标记所有可达对象                                        │
│  ├─ 递归标记引用关系                                        │
│  └─ 未标记对象 = 垃圾                                       │
│           ↓ 主要耗时部分                                    │
│                                                             │
│  阶段3：压缩（Compact）[可选]                               │
│  ├─ 移动存活对象到内存一端                                 │
│  ├─ 消除内存碎片                                            │
│  └─ 更新对象引用                                            │
│           ↓ 取决于碎片程度                                  │
│                                                             │
│  阶段4：回收（Sweep）                                        │
│  ├─ 释放未标记对象的内存                                    │
│  ├─ 更新代际指针                                            │
│  └─ 恢复线程执行                                            │
│           ↓ 约0.1-1ms                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 4.2 根对象（Root）

```
什么是根对象？
├─ GC标记的起点                                              │
├─ 被认为是"总是存活"的对象                                  │
└─ 包括：                                                    │
   ├─ 静态字段（static fields）                              │
   ├─ 栈上的局部变量（local variables）                       │
   ├─ 当前活动线程的栈                                        │
   └─ GC句柄（GCHandle）                                      │

示例：
public class GameManager
{
    private static GameManager instance;  // 根对象
    private List<Enemy> enemies;  // 通过instance可达
}

GC从instance开始标记，enemies也会被标记为存活
```

## 五、GC性能影响

### 5.1 GC卡顿原因

```
为什么GC会导致卡顿？

1. Stop The World
   ├─ GC期间暂停所有托管线程
   ├─ 游戏逻辑停止执行
   └─ 表现为帧率下降或冻结

2. 标记阶段耗时
   ├─ 需要遍历所有对象
   ├─ 对象越多，耗时越长
   └─ 可能耗时几十到几百毫秒

3. 压缩阶段耗时
   ├─ 需要移动大量内存
   ├─ 更新所有对象引用
   └─ 可能导致严重卡顿
```

---

### 5.2 GC性能数据

```
不同代际的GC耗时：

第0代GC：
├─ 耗时：1-5ms
├─ 触发频率：每秒多次
├─ 影响程度：小
└─ 对60FPS影响：可接受

第1代GC：
├─ 耗时：5-20ms
├─ 触发频率：每分钟几次
├─ 影响程度：中
└─ 对60FPS影响：可能掉帧

第2代GC：
├─ 耗时：50-200ms
├─ 触发频率：几分钟一次
├─ 影响程度：大
└─ 对60FPS影响：严重卡顿
```

---

### 5.3 GC导致的性能问题示例

```
场景：每帧分配1KB内存

计算：
├─ 每帧分配：1KB
├─ 60FPS分配：60KB/秒
├─ 第0代阈值：2MB
├─ 触发频率：2MB / 60KB = 约33秒一次
└─ 影响：每33秒一次GC，可接受

场景：每帧分配100KB内存

计算：
├─ 每帧分配：100KB
├─ 60FPS分配：6MB/秒
├─ 第0代阈值：2MB
├─ 触发频率：2MB / 6MB = 每0.33秒一次
└─ 影响：每0.33秒一次GC，严重卡顿
```

## 六、减少GC触发的策略

### 6.1 核心策略

```
┌─────────────────────────────────────────────────────────────┐
│              减少GC触发的核心策略                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  策略1：减少分配                                            │
│  ├─ 避免在Update中new对象                                   │
│  ├─ 避免在Update中new集合                                   │
│  ├─ 避免字符串拼接                                         │
│  └─ 避免使用LINQ                                            │
│                                                             │
│  策略2：复用对象                                            │
│  ├─ 使用对象池                                             │
│  ├─ 复用临时对象                                           │
│  └─ 预分配集合                                             │
│                                                             │
│  策略3：使用值类型                                          │
│  ├─ struct代替class（适用时）                               │
│  ├─ 避免装箱拆箱                                           │
│  └─ 使用数组代替List（适用时）                              │

│  ├─ 使用StringBuilder                                      │
```

### 6.2 具体技巧

#### 技巧1：复用集合

```csharp
// ❌ 错误：每帧new List
void Update()
{
    var enemies = new List<Enemy>();
    // ...
}

// ✅ 正确：复用List
private List<Enemy> enemies = new List<Enemy>();

void Update()
{
    enemies.Clear();
    // ...
}
```

#### 技巧2：使用对象池

```csharp
// ❌ 错误：频繁Instantiate和Destroy
void SpawnBullet()
{
    var bullet = Instantiate(bulletPrefab);
    // ...
}

// ✅ 正确：使用对象池
void SpawnBullet()
{
    var bullet = bulletPool.Get();
    // ...
    bulletPool.Return(bullet);
}
```

#### 技巧3：避免装箱拆箱

```csharp
// ❌ 错误：装箱拆箱
void ProcessValue(int value)
{
    object obj = value;  // 装箱
    int result = (int)obj;  // 拆箱
}

// ✅ 正确：使用泛型
void ProcessValue<T>(T value) where T : struct
{
    // 无装箱拆箱
}
```

## 八、常见问题

### Q1: 手动调用GC好吗？

**A**: 通常**不推荐**手动调用GC

```csharp
// ❌ 不推荐：每帧调用GC
void Update()
{
    System.GC.Collect();  // 每帧GC，性能灾难
}

// ✅ 正确：只在必要时调用
void OnApplicationPause(bool pause)
{
    if (pause)
    {
        // 应用进入后台时GC
        System.GC.Collect();
    }
}
```

**原因**：
- 手动GC成本高
- GC会自动优化
- 频繁手动GC反而降低性能

---

### Q2: IL2CPP会影响GC吗？

**A**: IL2CPP不会改变GC机制，但可以减少GC

```
Mono vs IL2CPP：

Mono：
├─ 使用Mono运行时
├─ JIT编译
├─ GC开销较大
└─ 适合开发阶段

IL2CPP：
├─ AOT编译
├─ 代码性能更好
├─ GC机制相同
├─ 但GC分配可能减少（编译器优化）
└─ 适合发布版本
```

**结论**：IL2CPP可以间接改善GC性能

---

### Q3: 增量GC（Incremental GC）有什么用？

**A**: 增量GC可以减少单次GC卡顿

```
普通GC：
├─ 一次性完成所有GC工作
├─ 可能耗时50-200ms
└─ 导致明显卡顿

增量GC：
├─ 分多次完成GC工作
├─ 每次只执行一小部分
├─ 单次耗时：几毫秒
└─ 卡顿不明显

开启方式：
Player Settings → Other Settings → Incremental GC
```
