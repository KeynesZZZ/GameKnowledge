前言

在手机游戏研发期，发热问题一直都是一个顽疾，对于经验不足的测试朋友来说，发热问题的优化一直都是一个难点，今天就来捋一下针对这块性能的优化，简单的分享一些我的经验和一些已有的手段。希望能起到抛砖引玉的作用。

本文档的目标很明确：帮开发者和 QA 同学搞清楚手机为什么会发热、怎么定位发热的根源、以及用什么方法把温度降下来。每一章我都尽量给出具体的操作步骤和可量化的标准。争取给大家一份有用的干货。

**适用对象** ：使用 Unity 或 Cocos的手游开发者，尤其是初中级开发者。如果你刚入行不久，接到一个发热优化的任务不知道从哪下手，这篇指南也能提供基础学习的内容。如果是经验丰富的老手，也非常欢迎各位佬指错纠正。

**文档结构** ：

- 第一章讲清楚发热的物理原理和手机降频机制
- 第二章给出一套系统的定位方法论，教你找到问题出在哪（含 GPU 性能计数器实战用法）
- 第三章聚焦渲染优化，这是 GPU 发热的主要来源（含渲染分辨率优化、后处理效果优化、Shader 复杂度与 GPU 深度性能分析）
- 第四章讲测试验证，确保优化措施真的生效

### 第一章：手机为什么会发热？

### 1.1 发热的物理原理

先说一个最基本的常识：能量不会凭空消失。

手机电池输出的电能，经过芯片处理后，绝大部分最终都变成了热能。这跟电暖器的道理是一样的，只不过芯片不是故意发热，它是在运算的过程中”顺带”产生了热量。

![](https://pic3.zhimg.com/v2-3102325aaa21ebcc38178af950e9821c_1440w.jpg)

手机游戏发热的原理

手机里最大的发热源是 SoC（System on Chip）。游戏画面绘制过程中，CPU 负责游戏逻辑运算、物理模拟、AI 决策、脚本执行、绘制指令调用，GPU 负责画面渲染、光照计算、后处理特效。哥俩一齐开动，当数据量较小还好，一旦数据量上去了，功耗飙升，芯片温度跟着往上走，散热能力跟不上时，芯片直接进入降频模式，帧率下降以及手感烫手。

电脑遇到这个问题怎么办？装个风扇，热量被气流带走。但手机没有风扇，它只能靠机身外壳被动散热。就像一间没有窗户的房间，人在里面运动，热量只能透过墙壁慢慢散出去，房间里温度上升的速度远比散热的速度快。热量在机身内部不断积累，温度就上去了。

整个过程可以简化为三步：

```
高性能运算 → 芯片温度升高 → 机身表面温度升高
   (起因)       (内部)         (玩家感受到的)
```

### 1.2 手机游戏的三大发热源

手机游戏的发热不是由单一原因造成的。以下是三大发热源及其在总功耗中的占比：

| 发热源 | 占比 | 具体内容 |
| --- | --- | --- |
| CPU 计算 | 30-40% | 游戏逻辑、物理模拟、AI 行为树、脚本执行、垃圾回收（GC） |
| GPU 渲染 | 40-50% | 顶点处理、像素填充、Shader 计算、后处理特效（Bloom、DOF 等） |
| 内存带宽 | 10-20% | 纹理采样、帧缓冲读写、资源加载与卸载 |

GPU 渲染是最大的发热源。这意味着如果你的游戏发热严重，首先要怀疑的就是渲染管线出了问题。

> 一个实用的判断方法：如果你的游戏画面很复杂但逻辑很简单（比如跑酷游戏），那发热大概率来自 GPU；如果画面不复杂但同屏有很多角色在跑 AI（比如策略游戏），那 CPU 可能是主因。如果你的游戏画面简单、逻辑也简单，但画面有较多特效，可能是全透或者大面积半透效果多，那么可能是 overdraw 较高导致发热。

### 1.3 温度阈值与降频机制

手机不是到了极限才会变卡。它有一套自我保护机制，叫 **Thermal Throttling** （热降频）。

通俗地说，手机内部有温度传感器，它一直在监测芯片温度。一旦温度超过了某个警戒线，手机操作系统会主动降低芯片的运行频率。频率低了，功耗就小了，产热也就少了，温度就能慢慢降回来。但代价是性能下降，你的游戏帧率会跟着掉。

这就像汽车的限速器。发动机温度太高，系统自动限制油门，车速就上不去了。

以下是典型的温度阈值与对应行为：

| 温度范围 | 手机行为 | 玩家感受 |
| --- | --- | --- |
| 35-40°C | 正常运行 | 无明显感知 |
| 40-45°C | 开始轻微降频 | 手感微温，性能影响极小 |
| 45-50°C | 明显降频 | 手感发热，可能出现卡顿 |
| 50-55°C | 大幅降频 | 明显卡顿，帧率骤降 |
| \>55°C | 极端降频或强制关闭应用 | 手机发烫，可能闪退 |

详细的数据可以参考下面两个链接：

[post.smzdm.com/p/ae64m5](https://link.zhihu.com/?target=https%3A//post.smzdm.com/p/ae64m55m/)

[toutiao.com/article/764](https://link.zhihu.com/?target=https%3A//www.toutiao.com/article/7643426848545735220/%3F%26source%3Dm_redirect)

不同芯片平台的降频策略有差异。高通骁龙的降频相对温和，会逐级降低频率；联发科天玑在中低端芯片上降频更激进一些；苹果 A 系列因为芯片面积大、能效比高，通常能撑更久，但一旦触发降频，幅度也很大。

> ⚠️ 不要只看帧率来判断发热。有些游戏在降频后帧率变化不明显（比如锁 30fps 的游戏），但芯片温度已经很高了。一定要同时监控温度数据。

### 1.4 发热对游戏体验的影响链

发热不是孤立的问题，它引发的连锁反应最终会伤害玩家体验：

```
发热
 ↓
系统降频保护
 ↓
芯片计算能力下降
 ↓
帧率下降 / 帧时间不稳定
 ↓
画面卡顿、操作延迟
 ↓
操作不跟手，操作失误增加
 ↓
烫手，操作体验极差，耗电量快
 ↓
玩家挫败感上升
 ↓
关闭游戏 / 卸载
```

这条链路里，开发者能控制的环节是从”发热”到”帧率下降”这一段。只要发热不严重，降频就不会触发，后面的连锁反应也就不会发生。

### 第二章：发热问题定位方法论

### 2.1 定位思路总览

发热的本质就是一句话：芯片太忙了，忙到产热超过了散热能力。

但”忙”的原因可能不同。在动手优化之前，必须先搞清楚是 CPU 忙、GPU 忙、还是两者都忙。不同的问题根源对应不同的优化方向。如果方向搞错了，花大量时间优化了 CPU，结果真正的瓶颈在 GPU，那就白费力气了。

以下是排查路径的决策树：

```
游戏发热严重
     │
     ├─── 打开 Unity Profiler，观察 CPU vs GPU 耗时
     │
     ├─── CPU 耗时 > 16ms（60fps 标准）
     │      │
     │      ├─── Rendering 占比高 ──→ Draw Call 过多 → 进入 3.1 节
     │      ├─── Scripts 占比高 ───→ 脚本效率问题 → 检查 Update/协程/GC
     │      ├─── Physics 占比高 ───→ 物理模拟过重 → 简化碰撞体/降低频率
     │      └─── Animation 占比高 ──→ 动画开销大 → 启用骨骼压缩/减少骨骼数
     │
     └─── GPU 耗时 > 16ms
            │
            ├─── 画面复杂度高 ─────→ 面数过高 → 进入 3.2 节
            ├─── 阴影开销大 ──────→ 阴影设置不合理 → 进入 3.3 节
            ├─── 像素处理量大 ────→ 渲染分辨率过高 → 进入 3.4 节
            ├─── 后处理特效多 ────→ 精简后处理 → 进入 3.5 节
            └─── Shader 复杂度高 → GPU 深度分析 → 进入 3.6 节
```

> 16ms 这个数字怎么来的？60fps 意味着每帧必须在 1⁄60 ≈ 16.67ms 内完成。如果 CPU 或 GPU 单独的耗时就已经超过 16ms，那一帧的总耗时一定会超标。如果是 30fps 的目标，阈值就是 33ms。

### 2.2 使用 Unity Profiler 定位

Unity Profiler 是排查性能问题的第一工具。以下是具体使用方法。

**CPU Profiler：**

1. 打开方式： `Window → Analysis → Profiler`
2. 点击左上角的 `Record` 按钮开始录制
3. 在设备上操作你想要分析的场景
4. 停止录制，查看时间线

重点关注这几个模块：

| 模块 | 含义 | 正常范围（60fps） |
| --- | --- | --- |
| Rendering | 渲染相关 CPU 开销（Draw Call 准备等） | < 4ms |
| Scripts | C# 脚本执行耗时 | < 6ms |
| Physics | 物理引擎计算 | < 2ms |
| Animation | 骨骼动画计算 | < 2ms |
| GC Alloc | 垃圾回收内存分配 | 尽量接近 0 |

Profiler 界面的 Hierarchy 视图会列出所有函数调用的耗时。点击 `Total` 列排序，排在最上面的就是最耗时的函数。

```
Profiler 数据示例（简化版）：

  Total ms   |  Self ms  |  函数名
  ---------- | --------- | ---------------------------
    14.2     |    0.3    |  Camera.Render
    12.8     |    0.1    |    Drawing
     8.5     |    8.5    |      Batch.Draw
     3.1     |    3.1    |    Shadows.Draw
     5.6     |    5.6    |  UILateProcessing
     4.2     |    4.2    |  PlayerUpdate.Update
```

在这个例子里， `Batch.Draw` 占了 8.5ms，说明 Draw Call 是主要瓶颈。 `PlayerUpdate.Update` 占了 4.2ms，这个脚本也值得优化。

**GPU Profiler：**

GPU Profiler 的使用方式与 CPU Profiler 类似，但需要切换到 GPU 模块。它能帮你看到渲染管线每个阶段的耗时：

- `Shadow` ：阴影渲染
- `Render` ：主渲染 pass
- `PostProcess` ：后处理

如果 `Shadow` 占了 GPU 总时间的 30% 以上，阴影优化就是你的首要任务。

**Memory Profiler：**

内存问题不一定直接导致发热，但频繁的垃圾回收（GC）会占用 CPU 时间，间接增加发热。使用 Memory Profiler 拍摄内存快照，对比不同时间点的快照，查找持续增长的资源，那很可能就是泄漏。

### 2.3 使用 Frame Debugger 分析渲染

Frame Debugger 是一个逐帧查看渲染过程的工具。它的作用是让你看到每一帧到底画了什么、画了多少次。

**使用步骤：**

1. `Window → Analysis → Frame Debugger`
2. 点击 `Enable` ，游戏会暂停在当前帧
3. 使用左右箭头逐步查看每个 Draw Call
4. 左侧列表显示所有渲染事件的顺序

**关注以下问题：**

- **不必要的渲染** ：一面墙后面有个物体，玩家根本看不到，但它仍然在被渲染。这说明遮挡剔除没有生效。
- **重复渲染** ：同一个物体因为某些原因被渲染了多次。这通常是因为脚本重复添加了渲染指令，或者相机设置有问题。
- **高 Overdraw 区域** ：同一个像素被多个半透明物体反复绘制。Frame Debugger 中的 Overdraw 视图会用颜色标注重绘次数，红色表示重绘次数很高。

大体的效果如下面的图（图片为 AI 生成），可以帮助大家理解：

![](https://pic4.zhimg.com/v2-ec2715b022d332dc7290748f54677ceb_1440w.jpg)

冗余渲染/overdraw 更加容易导致发热

### 2.4.1 GPU 性能计数器实战：用外部工具看 GPU “内部温度计”

上面列了一堆工具，但知道名字和会用是两回事。当你已经确认 GPU 是发热瓶颈后，需要用这些工具深入 GPU 内部看具体数据。以下是最关键的 GPU 性能计数器及其判断标准：

**高通设备（Snapdragon Profiler / PerfDog）：**

| 计数器 | 含义 | 关注范围 | 判断标准 |
| --- | --- | --- | --- |
| GPU Frequency | GPU 当前运行频率 | 200-1000 MHz | 长期处于最大频率的 80% 以上 = GPU 满载，发热风险高 |
| ALU Utilization | ALU 计算利用率 | 0-100% | \> 60% = Shader 计算重，需要简化 Shader |
| GPU Load | GPU 总体负载 | 0-100% | \> 85% 持续 = GPU 几乎满载，发热不可避免 |
| Shader Instructions / Cycles | Shader 执行的指令/周期数 | \- | 值越大 Shader 越复杂 |

**Mali 设备（Arm Mobile Studio / Streamline）：**

| 计数器 | 含义 | 关注范围 | 判断标准 |
| --- | --- | --- | --- |
| GPU Frequency | GPU 当前频率 | \- | 长期满频 = 持续高负载 |
| Arithmetic Utilization | 算术单元利用率 | 0-100% | \> 60% = 计算瓶颈 |
| Fragment Cycles | Fragment 阶段周期数 | \- | 占总 GPU Cycles > 70% = 像素处理瓶颈 |
| Vertex Cycles | Vertex 阶段周期数 | \- | 占总 > 40% = 顶点处理瓶颈 |
| L2 Cache Miss Rate | 缓存未命中率 | 0-100% | \> 20% = 带宽瓶颈，纹理/渲染目标过大 |
| External Bandwidth | 外部内存带宽 | \- | 持续接近峰值 = 带宽瓶颈 |

**Apple 设备（Xcode Metal System Trace）：**

| 计数器 | 含义 | 关注范围 | 判断标准 |
| --- | --- | --- | --- |
| GPU Time | GPU 每帧耗时 | \- | \> 16ms = GPU 瓶颈 |
| ALU Utilization | 实际计算利用率 | 0-100% | \> 70% = Shader 计算密集 |
| ALU Limiter | 含等待的 ALU 利用率 | 0-100% | Limiter 高但 Utilization 低 = 带宽瓶颈 |
| Texture Sample Requests | 纹理采样请求数 | \- | 过多 = 减少纹理采样次数 |
| Vertex Load | 顶点处理负载 | \- | 过高 = 减面 / LOD |

**快速判断模板：**

```
观察 GPU Frequency
  ├── 长期满频 → GPU 严重过载 → 必须降负载（分辨率/后处理/Shader）
  ├── 波动频繁 → 负载不均匀 → 检查是否有间歇性重载（如特效爆发）
  └── 稳定在中低频 → GPU 负载可控 → 发热主因可能在 CPU 或带宽

观察 ALU Utilization
  ├── > 60% → Shader 太重 → 简化数学运算、减少复杂数学函数
  ├── 30-60% → 正常 → 继续看其他指标
  └── < 30% 但 GPU 仍然热 → 带宽瓶颈 → 减少纹理采样、降低纹理分辨率

观察 Fragment Cycles 占比
  ├── > 70% → 像素处理瓶颈 → 降渲染分辨率、简化 Fragment Shader、减 Overdraw
  └── < 50% + Vertex Cycles > 40% → 顶点瓶颈 → 减面、LOD
```

> 这些外部工具的数据比 Unity Profiler 精确得多，因为它们直接读取 GPU 硬件寄存器，而不是通过引擎层估算。如果条件允许，发热优化的深度分析 **必须使用平台原生工具** ，Unity Profiler 只能做初步定位。

### 第三章：渲染优化 — 解决 GPU 发热

渲染是手机游戏最大的性能开销来源。前面提到过，GPU 渲染通常占手机总功耗的 40-50%。如果能把 GPU 的负担降下来，发热问题就能缓解一大半。

本章从 Draw Call、面数、阴影三个维度讲解优化方法，每个方法都会给出具体的操作步骤。

### 3.1 Draw Call 优化

### 3.1.1 什么是 Draw Call？

想象你请了一个画师来画一栋房子的正面图。房子有墙壁、窗户、门、屋顶、花园篱笆。如果每画一个部分你都要重新告诉他”用什么颜色、用什么笔刷、画在哪里”，那光是沟通就要花很多时间。

Draw Call 就是这种”指令”。CPU 向 GPU 下达一次渲染指令，告诉它”用什么材质、画哪个 Mesh、在哪个位置”。指令下达本身需要时间，GPU 每次接收指令也需要准备。Draw Call 越多，CPU 和 GPU 的沟通成本就越高。

在移动端，Draw Call 的开销比 PC 上大得多。因为移动端 CPU 和 GPU 共享内存，每次状态切换（切换材质、切换 Shader）都会带来额外的延迟。

### 3.1.2 Draw Call 分档标准

不同性能的设备对 Draw Call 的承受能力不同。以下是参考标准：

| 设备档次 | Draw Call 标准 | [SetPass Call](https://zhida.zhihu.com/search?content_id=275895867&content_type=Article&match_order=1&q=SetPass+Call&zhida_source=entity) 标准 |
| --- | --- | --- |
| 低端（骁龙 660 及以下） | < 300 | < 100 |
| 中端（骁龙 778G / 天玑 1100） | < 500 | < 150 |
| 高端（骁龙 8 Gen 系列 / A 系列） | < 800 | < 200 |

SetPass Call 是 Draw Call 中一个值得单独关注的指标。它表示 Shader/材质切换的次数。每次切换材质，GPU 都需要重新配置渲染管线，这个开销比普通 Draw Call 还要大。如果你的 SetPass Call 是 258，意味着 GPU 每帧要切换 258 次材质状态，这在移动端是完全不可接受的。

### 3.1.3 静态批处理（Static Batching）

**原理** ：把标记为 Static 的物体合并成一个大的 Mesh，这样原本需要多次 Draw Call 才能渲染的物体，现在一次就搞定了。

打个比方，原本画 100 块砖头，要下 100 次指令。静态批处理相当于提前把这 100 块砖拼成一面墙，以后画这面墙只需要一次指令。

**操作方法** ：

1. 在 Hierarchy 中选中不移动的物体
2. 在 Inspector 右上角勾选 `Static` （或者只勾选 `Batching Static` ）
3. Unity 会在构建时自动将这些物体合并

**前提条件** ：合并的物体必须使用相同的材质。材质不同就没法合并。

| 优点 | 缺点 |
| --- | --- |
| 大幅减少 Draw Call | 合并后的 Mesh 会额外占用内存 |
| 对代码零侵入，无需改动逻辑 | 只适用于不移动的物体 |
| 配置简单，一键勾选 | 合并后的 Mesh 无法单独剔除或排序 |

**适用场景** ：建筑、地形、装饰物、路灯、围栏等不会移动的静态物体。

> ⚠️ 静态批处理会增加内存占用，因为它会为合并后的 Mesh 创建一份额外的顶点数据。如果场景非常大、静态物体非常多，需要关注内存增量。

### 3.1.4 GPU Instancing

**原理** ：如果一个 Draw Call 可以渲染多个相同 Mesh 的物体，那就不需要为每个物体单独发一个 Draw Call。GPU Instancing 就是让一个 Draw Call 同时渲染多个形状相同但位置、颜色不同的物体。

就像复印机：你设定好模板，然后告诉它”印 100 份，每份放在不同位置”。只需要下达一次指令。

**操作方法** ：

1. 选中材质
2. 在材质面板底部勾选 `Enable GPU Instancing`
![](https://pic3.zhimg.com/v2-3ed1925e2d457d08e99f02cd0cbb7b18_1440w.jpg)

**前提条件** ：物体的 Mesh 必须相同，材质必须相同（但可以通过 MaterialPropertyBlock 修改部分属性），Shader 必须支持 Instancing。

**适用场景** ：树木、草地、石头、重复出现的物品。

**代码示例** ：

```
using UnityEngine;

public class InstancingExample : MonoBehaviour
{
    public Mesh mesh;
    public Material material;
    private MaterialPropertyBlock mpb;
    private Matrix4x4[] matrices;
    private Vector4[] colors;

    // 同一帧最多渲染 1023 个实例（API 限制）
    private const int MAX_INSTANCES = 1023;

    void Start()
    {
        mpb = new MaterialPropertyBlock();
        matrices = new Matrix4x4[MAX_INSTANCES];
        colors = new Vector4[MAX_INSTANCES];

        for (int i = 0; i < MAX_INSTANCES; i++)
        {
            // 随机位置
            Vector3 pos = new Vector3(
                Random.Range(-50f, 50f),
                0f,
                Random.Range(-50f, 50f)
            );
            matrices[i] = Matrix4x4.TRS(pos, Quaternion.identity, Vector3.one);
            colors[i] = new Vector4(
                Random.Range(0.6f, 1f),
                Random.Range(0.6f, 1f),
                Random.Range(0.6f, 1f),
                1f
            );
        }
    }

    void Update()
    {
        mpb.SetVectorArray("_Color", colors);
        Graphics.DrawMeshInstanced(mesh, 0, material, matrices, MAX_INSTANCES, mpb);
    }
}
```

这段代码用一次 Draw Call 渲染了 1023 个实例。相比传统方式（1023 个 Draw Call），性能提升是巨大的。

### 3.1.5 材质合并与纹理图集

Draw Call 的一个主要来源是材质切换。场景中有 50 种不同的材质，GPU 就需要切换 50 次状态。如果把多个小纹理合并成一张大纹理（图集），让多个物体共享同一个材质，Draw Call 自然就降下来了。

**原理** ：把 N 张小图拼成 1 张大图，所有使用这张大图的物体可以共享同一个材质，从而减少材质切换次数。

**工具选择** ：

- TexturePacker：功能强大的第三方工具，支持多种打包算法
- Unity Sprite Atlas：Unity 内置的图集工具，适合 2D 项目
- 自定义打包工具：对于 3D 项目的场景物体纹理，通常需要自己写打包脚本

**最佳实践** ：

- 按场景或模块打包图集，不要把整个游戏的所有纹理打成一个超大图集
- 单张图集分辨率建议不超过 2048x2048，移动端超过 4096x4096 的纹理在很多设备上会降低采样效率
- 图集之间留出 2-4 像素的间距，避免纹理 bleed（颜色渗透）

> ⚠️ 图集并不是越大越好。超大图集在纹理采样时会增加 cache miss 的概率，反而可能降低渲染效率。合理规划图集的大小和分组才是关键。

### 3.1.6 SRP Batcher（URP/HDRP 项目）

如果项目使用的是 Unity 的 Scriptable Render Pipeline（URP 或 HDRP），SRP Batcher 是一个值得开启的优化。

**原理** ：传统渲染流程中，每次切换材质都需要重新上传 Shader 数据到 GPU。SRP Batcher 改变了这个流程，它把所有兼容的材质数据按类型分组，GPU 只需要按顺序读取数据即可，不再需要频繁的状态切换。

**开启方法** ：

1. 找到你的 URP Asset
2. `Advanced → SRP Batcher → 勾选 Enable`

**兼容性要求** ：使用的 Shader 必须兼容 SRP Batcher。内置的 Lit 和 Unlit Shader 默认兼容，自定义 Shader 需要检查是否包含 `CBUFFER` 块。

**性能提升预期** ：通常可以减少 20-40% 的 CPU 渲染耗时。效果在 Draw Call 多、材质种类多的场景中最明显。

> 开启 SRP Batcher 之后，Frame Debugger 里会显示 “SRP Batch” 字样，你可以看到每个 Batch 合并了多少个 Draw Call。如果某些 Draw Call 没有被合并，说明它们的 Shader 不兼容，需要修改。

### 3.2 面数优化

### 3.2.1 为什么面数影响发热？

GPU 渲染画面的第一步是处理顶点。每个三角形的三个顶点都要经过顶点着色器（Vertex Shader）的计算，包括坐标变换、骨骼蒙皮、顶点动画等。三角形越多，顶点着色器的工作量就越大，GPU 的计算负载就越高，产热自然就多。

一个面数 50 万的场景和一个面数 10 万的场景，GPU 顶点处理阶段的耗时可能差 3-5 倍。对于移动设备来说，这个差距足以决定游戏是否流畅。

### 3.2.2 面数分档标准

| 设备档次 | 场景总面数上限 | 单角色面数上限 |
| --- | --- | --- |
| 低端 | < 10 万 | < 3,000 |
| 中端 | < 30 万 | < 8,000 |
| 高端 | < 50 万 | < 15,000 |

> ⚠️ 场景总面数是指同屏可见面数，不是整个场景资源的总面数。一个场景可能包含 200 万面的资源，但如果同一时刻玩家只能看到 20 万面，那实际渲染负担就是 20 万面。

### 3.2.3 LOD（Level of Detail）系统

LOD 是面数优化最核心的技术。它的思路很简单：远处的物体玩家看不清细节，就没必要用高精度模型渲染，换成简化版就行。

**LOD 级别设计** ：

| LOD 级别 | 距离范围 | 面数比例 | 说明 |
| --- | --- | --- | --- |
| LOD0 | 近距离（0-15m） | 100% | 原始高精度模型 |
| LOD1 | 中距离（15-40m） | 50% | 中等简化模型 |
| LOD2 | 远距离（40-80m） | 20% | 高度简化模型 |
| LOD3 | 极远（>80m） | Billboard | 替代为面片/广告牌 |

**Unity 中的配置方法** ：

1. 准备好不同精度的模型（可用 Simplygon、InstaLOD 或 Blender 的 Decimate 工具生成简化版）
2. 将不同精度的模型拖入 LOD Group 组件的对应层级
3. 调整每个 LOD 级别的切换距离

**LOD 切换跳变问题** ：

当模型从一个 LOD 级别切换到另一个时，玩家可能会注意到模型突然变了个样子（”跳变”）。解决方法是在 LOD Group 中启用 `Cross Fade` （交叉渐变），让两个级别之间有一个短暂的混合过渡，视觉上会更平滑。

### 3.2.4 遮挡剔除（Occlusion Culling）

即使用了 LOD，如果场景里有很多被遮挡的物体（比如墙后面的家具），它们虽然玩家看不到，但 GPU 可能仍然在处理。

遮挡剔除的原理就是：不渲染被其他物体完全挡住的物体。

**配置方法** ：

1. 选中能遮挡其他物体的大型物体（墙壁、山体、建筑），勾选 `Occluder Static`
2. 选中可能被遮挡的物体，勾选 `Occludee Static`
3. `Window → Rendering → Occlusion Culling`
4. 点击 `Bake` ，Unity 会预计算遮挡关系

**适用条件** ：封闭或半封闭场景效果最好（室内、城市街道、峡谷）。开阔场景（草原、海面）基本没有遮挡关系，开了也没用。

**预期收益** ：对于遮挡关系丰富的场景，通常可以减少 20-40% 的渲染负载。

### 3.3 阴影优化

阴影是移动端的性能杀手之一。实时阴影需要额外的渲染 Pass：从光源视角渲染一次深度图，再在主渲染中使用这张深度图计算阴影。这意味着阴影的开销几乎等于多渲染一整遍场景。

对于面数高、物体多的场景，阴影的开销可能占到 GPU 总时间的 30-50%。

以下是阴影优化的主要策略：

| 优化手段 | 具体操作 | 性能收益 |
| --- | --- | --- |
| 减少阴影距离 | 从 100m 降至 30-50m | 减少 30-50% 阴影开销 |
| 减少级联数 | 从 4 级降至 2 级 | 减少 50% 阴影 Pass |
| 降低阴影分辨率 | 从 2048 降至 1024 | 减少显存和带宽占用 |
| 烘焙静态阴影 | 使用 Lightmap 代替实时阴影 | 完全消除运行时阴影开销 |
| 关闭低端设备阴影 | 低画质档位不渲染阴影 | 大幅降低 GPU 负载 |

**不同设备档位的阴影策略建议** ：

| 设备档次 | 阴影距离 | 级联数 | 分辨率 | 建议 |
| --- | --- | --- | --- | --- |
| 低端 | 不开启阴影 | \- | \- | 使用烘焙阴影或假阴影（blob shadow） |
| 中端 | 30-40m | 2 级 | 1024 | 适度阴影，兼顾性能 |
| 高端 | 50-70m | 2-3 级 | 1024-2048 | 完整阴影体验 |

> 假阴影（Blob Shadow）是一个实用的低成本方案：在角色脚下放一个半透明的圆形贴图，跟随角色移动。虽然不是真实阴影，但视觉上够用，而且几乎零开销。 角色独立阴影：有些项目要求主角阴影要更清晰，可以把主角的阴影独立出来做成独立阴影，单独给主角一个 shadow map。这样角色阴影会更清晰，而场景中其他物品则使用一张级联阴影，糊一点也能接受。

### 3.4 渲染分辨率优化

### 3.4.1 为什么渲染分辨率对发热影响巨大？

在前面的章节中我们讲了 Draw Call、面数、阴影对发热的影响，但有一个因素我们还没深入讨论，它可能是所有优化手段中 **投入产出比最高** 的——渲染分辨率。

渲染分辨率决定了 GPU 每帧需要处理多少个像素（Fragment）。分辨率越高，像素越多，Fragment Shader 需要执行的次数就越多，GPU 的计算量和带宽消耗都随之增长。这是一个 **平方级** 的关系：分辨率从 1080p 降到 720p，像素数减少了 56%，但 GPU 的 Fragment 阶段负载也跟着减少 56%。

打个比方：渲染分辨率就像画布的尺寸。一块 2m × 2m 的画布和一块 1m × 1m 的画布，画同样内容所需要的时间和颜料差别是巨大的。降低渲染分辨率就是换一块更小的画布来画。

以下是不同渲染分辨率下 GPU 工作量的量化对比：

| 渲染分辨率 | 像素总数 | 相对 1080p 像素比 | GPU Fragment 负载变化 | 预估功耗影响 |
| --- | --- | --- | --- | --- |
| 1920×1080（1080p） | 2,073,600 | 100% | 基准 | 基准 |
| 1600×900（900p） | 1,440,000 | 69% | ↓ 31% | ↓ 20-25% |
| 1280×720（720p） | 921,600 | 44% | ↓ 56% | ↓ 35-45% |
| 960×540（540p） | 518,400 | 25% | ↓ 75% | ↓ 50-60% |

> ⚠️ 功耗下降幅度小于像素减少幅度，因为 GPU 还有固定开销（顶点处理、驱动层调度、带宽基础消耗等）。但分辨率仍然是调节 GPU 发热 **最有效的单一旋钮** ，比任何 Shader 优化或 Draw Call 优化都来得快。

### 3.4.2 渲染分辨率与原生分辨率的区别

这里需要区分两个概念：

- **原生分辨率（Screen Resolution）** ：手机屏幕的物理像素数，比如 1080×2400。这是硬件决定的，无法改变。
- **渲染分辨率（Render Resolution）** ：游戏实际渲染画面的分辨率。渲染完后，再通过 upscale（放大）到原生分辨率显示给玩家。

渲染分辨率可以低于原生分辨率。比如手机屏幕是 1080p 的，但游戏只渲染 720p 的画面，然后放大到 1080p 显示。画面会稍微模糊一点，但 GPU 负载大幅降低。

大多数手游玩家其实很难区分 900p 和 1080p 的画质差异（尤其是在手机 6 英寸左右的屏幕上）。但对 GPU 来说，这两者的工作量差了 30%+。

### 3.4.3 分档分辨率标准

| 设备档次 | 原生分辨率 | 建议渲染分辨率 | Render Scale（URP） | 说明 | |———-|———–|—————|——————– -|——| | 低端（骁龙 660 及以下） | 1080p | 540p-720p | 0.5x-0.67x | 画质损失明显但可接受，优先保帧率 | | 中端（骁龙 778G / 天玑 1100） | 1080p | 720p-900p | 0.67x-0.83x | 平衡画质与性能 | | 高端（骁龙 8 Gen 系列 / A 系列） | 1080p-1440p | 900p-1080p | 0.83x-1.0x | 画质优先，温度高时动态降低 |

在 Unity URP 中，Render Scale 的设置路径是： `UniversalRenderPipelineAsset → Quality → Render Scale` 。这个值直接控制渲染分辨率相对于屏幕分辨率的倍数。

### 3.4.4 动态分辨率（Dynamic Resolution）

固定分辨率的问题在于：场景复杂度是动态变化的。主城空旷时 1080p 也能跑满 60fps，但团战时 720p 都卡。动态分辨率就是让渲染分辨率根据当前 GPU 负载自动调整——GPU 忙的时候降低分辨率，GPU 空闲时恢复分辨率。

**Unity URP 动态分辨率示例：**

```
using UnityEngine;
using UnityEngine.Rendering;
using UnityEngine.Rendering.Universal;

public class DynamicResolutionController : MonoBehaviour
{
    [Header("目标帧时间")]
    [SerializeField] private float targetFrameTimeMs = 16.67f; // 60fps
    [SerializeField] private float minScale = 0.5f;
    [SerializeField] private float maxScale = 1.0f;
    [SerializeField] private float adjustSpeed = 0.01f;

    private float currentScale = 1.0f;
    private float frameTimeSmoothed;

    void Start()
    {
        currentScale = maxScale;
        ApplyScale(currentScale);
    }

    void Update()
    {
        // 平滑帧时间，避免单帧波动
        float rawFrameTime = Time.unscaledDeltaTime * 1000f;
        frameTimeSmoothed = Mathf.Lerp(frameTimeSmoothed, rawFrameTime, 0.1f);

        if (frameTimeSmoothed > targetFrameTimeMs * 1.2f)
        {
            // 帧时间超标，降低分辨率
            currentScale = Mathf.Max(minScale, currentScale - adjustSpeed);
        }
        else if (frameTimeSmoothed < targetFrameTimeMs * 0.8f)
        {
            // 帧时间富裕，缓慢恢复分辨率
            currentScale = Mathf.Min(maxScale, currentScale + adjustSpeed * 0.5f);
        }

        ApplyScale(currentScale);
    }

    void ApplyScale(float scale)
    {
        var urpAsset = GraphicsSettings.currentRenderPipeline as UniversalRenderPipelineAsset;
        if (urpAsset != null)
        {
            urpAsset.renderScale = scale;
        }
    }
}
```

**进阶：温度驱动的自适应分辨率**

比帧时间驱动更进一步的做法，是直接根据设备温度来调整分辨率。温度是发热的最终指标，温度驱动的策略更直接：

| 电池温度 | 渲染分辨率策略 | 其他联动操作 |
| --- | --- | --- |
| < 38°C | 满分辨率（1.0x） | 全部效果正常 |
| 38-42°C | 轻微降低（0.85x） | 后处理降级 |
| 42-45°C | 中度降低（0.7x） | 关闭部分后处理、降低阴影 |
| \> 45°C | 大幅降低（0.5x） | 关闭全部后处理、关闭阴影 |

> 主流手游（原神、王者荣耀等）普遍采用了动态分辨率 + 温度联动的策略。玩家感知不到分辨率的微小变化，但能明显感受到手机不那么烫了。

### 3.5 后处理效果优化

### 3.5.1 后处理为什么是发热重灾区？

后处理（Post-Processing）是在整个场景渲染完成后，对最终画面进行的全屏图像处理。包括 Bloom（泛光）、DOF（景深）、SSAO（屏幕空间环境光遮蔽）、Motion Blur（运动模糊）、Tonemapping（色调映射）等效果。

后处理之所以特别耗电发热，原因有三：

1. **全屏像素遍历** ：每个后处理效果至少需要一个全屏 Pass，意味着屏幕上的 **每一个像素** 都要执行一遍 Fragment Shader。1080p 分辨率下就是 200 万个像素。
2. **多次纹理采样** ：Bloom 需要多次降采样和升采样（通常 4-6 次），DOF 需要采样深度缓冲和散焦计算，SSAO 需要在多个方向采样深度。每次采样都消耗带宽。
3. **复杂数学运算** ：模糊算法（高斯、Kawase）、光照计算（SSAO 的球面积分近似）、色彩空间转换等，需要大量 ALU 计算。

简化为链路：

```
后处理效果开启
  → 额外的全屏渲染 Pass（每个效果 1-6 个 Pass）
  → GPU Fragment Shader 全屏计算
  → 大量纹理采样（带宽消耗）
  → 大量 ALU 计算（模糊、光照、色彩转换）
  → GPU Cycle 大幅增加
  → GPU 频率被 DVFS 拉高
  → 功耗超线性增长（P ∝ f × V²）
  → 发热加剧
```

### 3.5.2 后处理效果 GPU 耗时排名

基于移动端实测数据，以下排名从最费到最省（1080p 分辨率参考值）：

| 后处理效果 | 典型 GPU 耗时（移动端） | 发热影响 | 核心开销来源 |
| --- | --- | --- | --- |
| 景深（DOF） | 3-6ms | ⭐⭐⭐⭐⭐ | 多次纹理采样 + 散焦计算 + CoC（Circle of Confusion），极耗 ALU 和带宽 |
| 屏幕空间反射（SSR） | 3-5ms | ⭐⭐⭐⭐⭐ | Ray Marching 多步采样 + 逐像素光线追踪近似 |
| SSAO | 2-4ms | ⭐⭐⭐⭐ | 多次深度缓冲采样 + 球面积分近似计算 |
| Bloom | 1.5-3ms | ⭐⭐⭐⭐ | 多次降采样 + 高斯模糊 + 合成 |
| 运动模糊（Motion Blur） | 1.5-3ms | ⭐⭐⭐ | 速度缓冲生成 + 方向模糊 |
| MSAA 4x | 2-4ms | ⭐⭐⭐⭐ | 采样量翻 4 倍，带宽和计算同时增加 |
| MSAA 2x | 1-2ms | ⭐⭐⭐ | 采样量翻倍 |
| FXAA | 0.5-1.5ms | ⭐⭐ | 全屏像素遍历 + 边缘检测，纯 ALU |
| 色调映射（Tonemapping） | 0.5-1ms | ⭐⭐ | 简单数学运算（曲线映射），开销可控 |
| 色彩校正（Color Grading） | 0.3-0.8ms | ⭐⭐ | LUT（查找表）采样，较快 |
| 镜头畸变（Lens Distortion） | 0.3-0.5ms | ⭐ | UV 偏移计算，简单 |
| 暗角（Vignette） | 0.1-0.3ms | ⭐ | 极简的距离衰减，几乎零开销 |
| 色差（Chromatic Aberration） | 0.2-0.4ms | ⭐ | RGB 三通道偏移采样 |

> ⚠️ 以上耗时数据基于 1080p 渲染分辨率。如果渲染分辨率降到 720p，所有后处理耗时大约降低 50-60%。这也再次说明分辨率对发热的杠杆效应。

一个残酷的现实：如果你的游戏开启了 DOF + SSAO + Bloom + Motion Blur，光后处理一项就可能吃掉 8-16ms 的 GPU 时间。在 60fps（16.67ms 预算）下，后处理就把帧预算吃完了，GPU 只能全速满载运行，发热不可避免。

### 3.5.3 后处理分档策略

不是所有设备都配得上全套后处理。以下是分档建议：

| 设备档次 | 建议开启 | 建议关闭 | 说明 |
| --- | --- | --- | --- |
| 低端 | Vignette、Color Grading（仅 LUT）、Tonemapping（ACES 简化版） | DOF、SSAO、SSR、Bloom、MSAA、Motion Blur | 只保留零成本/极低成本的效果 |
| 中端 | \+ FXAA、Bloom（简化版，降采样 3 级）、Tonemapping | DOF、SSAO、SSR、MSAA 4x、Motion Blur | Bloom 用低分辨率计算 |
| 高端 | \+ SSAO（低采样数）、MSAA 2x、Motion Blur（低采样） | DOF（仅限过场动画）、SSR | 追求画质但仍有底线 |

> DOF（景深）在移动端几乎永远不应该在 gameplay 中开启。它是后处理里最耗性能的效果之一，而且手游的视觉焦点通常不需要景深来引导。只在剧情过场等可控场景中使用，或者用贴图模拟的假 DOF（Blur叠加）替代。

### 3.5.4 Bloom 专项优化

Bloom 是手游中使用最广泛的后处理效果，但也是发热重灾区之一。以下是 Bloom 的专项优化建议：

**优化手段一：减少降采样级数**

Bloom 的实现原理是对画面进行多次降采样（提取高亮区域）→ 多次高斯模糊 → 合成回原图。降采样级数直接决定了计算量：

| 降采样级数 | GPU 耗时（移动端参考） | 画质效果 | 建议 |
| --- | --- | --- | --- |
| 6 级（默认） | 2.5-3ms | 细腻 | 仅高端设备 |
| 4 级 | 1.5-2ms | 较好 | 中高端推荐 |
| 3 级 | 1-1.5ms | 可接受 | 中低端推荐 |
| 2 级 | 0.5-1ms | 粗糙 | 低端设备 |

**优化手段二：降低 Bloom 计算分辨率**

Bloom 不需要在全分辨率下计算。用 1⁄2 甚至 1⁄4 分辨率做 Bloom 计算，再 upscale 回来，视觉差异极小：

```
// URP Bloom 分辨率控制示例
// 在 URP Asset 中设置 Render Scale 或使用自定义 Renderer Feature
// 推荐 Bloom 在 0.5x 分辨率下计算
```

**优化手段三：提高亮度阈值**

Bloom 阈值（Threshold）决定了哪些像素参与泛光计算。提高阈值意味着更少的像素需要处理：

- 默认阈值 0.8 → 大量高亮像素参与 → 计算量大
- 调高到 1.2 → 只有极强的光源参与 → 计算量大幅减少
- 调高到 1.5 → 几乎只有 UI 光效和技能特效参与 → 最低开销

**优化手段四：使用 Separable Blur（分离式模糊）**

2D 高斯模糊的计算量是 O(N²)（N 是卷积核大小）。分离式模糊把它拆成水平和垂直两个 1D Pass，计算量降为 O(2N)。对于 9×9 的卷积核，从 81 次运算降到 18 次。

URP 内置的 Bloom 已经使用了分离式模糊，但如果你的项目有自定义 Bloom 实现，务必确认使用了这个优化。

### 3.5.5 后处理优化的总原则

1. **后处理是可选项，不是必选项** 。移动端的后处理应该”能省则省”，不是”能加则加”。
2. **用 cheaper 的替代方案** ：Color Grading 的 LUT 查找远比运行时色彩空间转换便宜；假 DOF（径向模糊）远比真 DOF 便宜；Blob Shadow 远比实时阴影便宜。
3. **分辨率是后处理的第一杠杆** ：降低渲染分辨率 20%，所有后处理耗时自动减少约 40%（因为像素数减少约 36%）。
4. **后处理是发热的”放大器”** ：它本身不是最大的发热源，但它会让 GPU 在最高负载的那段帧时间里变得更忙，直接推高 GPU 频率，触发更早的 Thermal Throttling。

### 3.6 Shader 复杂度与 GPU 深度性能分析

前面几节我们讲了 Draw Call、面数、阴影、分辨率、后处理这些”宏观”优化。但有时候你做了所有这些优化，GPU 仍然很热、帧率仍然上不去。这时候就需要深入到 GPU 内部，看看 Shader 到底有多”重”，GPU 到底在忙什么。

本节将带你理解几个关键概念： **GPU Clock（时钟频率）** 、 **ALU（算术逻辑单元）利用率** 、 **Shader 复杂度指标** 、 **GPU Cycle（执行周期）** ，以及它们和发热之间的关系。

### 3.6.1 GPU Clock 与发热：不只是”频率高就热”这么简单

在前面的章节中我们提到了 Thermal Throttling（热降频），知道手机温度高了会降低芯片频率。但你可能没想过： **GPU 的频率不是固定的，它一直在动态调整** 。

这个机制叫 **DVFS（Dynamic Voltage and Frequency Scaling，动态电压频率调节）** 。

**DVFS 的工作原理：**

手机 GPU 有一张频率/电压对照表（OPP 表，Operating Performance Points）。GPU 调度器每隔几毫秒采样一次 GPU 利用率，根据利用率决定把频率调高还是调低：

```
GPU 利用率 > 70%（持续）→ 调度器认为负载重 → 提升频率和电压 → 性能提升但功耗飙升
GPU 利用率 < 30%（持续）→ 调度器认为负载轻 → 降低频率和电压 → 省电但性能下降
```

这里的关键是功耗公式： **P = α × C × f × V²**

- P 是功耗（瓦特），直接决定发热量
- f 是频率
- V 是电压
- 频率升高时，电压也必须升高才能保证芯片稳定工作

这意味着 **频率升高 1 倍，功耗不是升高 1 倍，而是升高 2-3 倍** （因为电压也在增加）。这就是为什么你的 Shader 越复杂，GPU 就越被迫跑在高频，发热就越严重——不是线性增长，是超线性增长。

**主流移动 GPU 频率范围参考：**

| GPU | 最低频率 | 最高频率 | 满载功耗参考 | 典型搭载芯片 |
| --- | --- | --- | --- | --- |
| Adreno 610 | 100 MHz | 600 MHz | ~2W | 骁龙 665 |
| Adreno 618 | 200 MHz | 700 MHz | ~3W | 骁龙 750G |
| Adreno 640 | 133 MHz | 585 MHz | ~3.5W | 骁龙 845 |
| Adreno 730 | 200 MHz | 900 MHz | ~5.5W | 骁龙 8 Gen1 |
| Adreno 750 | 200 MHz | 1000 MHz | ~7W | 骁龙 8 Gen3 |
| Mali-G78 | 153 MHz | 850 MHz | ~4.5W | 天玑 9000 |
| Mali-G715 | 200 MHz | 950 MHz | ~5W | 天玑 9200 |
| Apple GPU（A17 Pro） | ~200 MHz | ~1400 MHz | ~6W | A17 Pro |

> 注意高端 GPU 的频率范围跨度极大（200MHz 到 1000MHz+）。这意味着同一个游戏，如果你的 Shader 简单，GPU 可能只跑在 400MHz 就够了，功耗很低；如果你的 Shader 复杂，GPU 被迫拉到 900MHz，功耗可能翻 3 倍。 **Shader 复杂度直接决定了 GPU 在哪个频点工作，从而直接决定了发热量。**

**热反馈环路：**

这是一个恶性循环：

```
Shader 复杂度高
  → GPU 需要更多时间处理每帧
  → GPU 利用率升高
  → DVFS 提升频率和电压
  → 功耗超线性增长（P ∝ f × V²）
  → 温度上升
  → 触发 Thermal Throttling
  → GPU 被强制降频
  → 帧率骤降
```

优化的核心思路就是： **在 GPU 还没被迫拉高频率之前，就把 Shader 的工作量降下来。让 GPU 在低频就能跑满帧率，发热自然就低。**

### 3.6.2 ALU 利用率：衡量 GPU “算力消耗”的核心指标

**什么是 ALU？**

ALU（Arithmetic Logic Unit，算术逻辑单元）是 GPU 内执行数学运算的核心部件。加、减、乘、除、sin、cos、pow、dot、normalize——所有 Shader 里的数学运算最终都在 ALU 上执行。

**什么是 ALU 利用率？**

ALU 利用率 = GPU 在每个时钟周期里 ALU 实际在做计算的时间占比。

- 0% = ALU 完全空闲（GPU 没有计算任务）
- 100% = ALU 每个周期都在做计算（GPU 计算能力被完全吃满）

**为什么 ALU 利用率对发热很重要？**

ALU 利用率直接反映了 Shader 的计算强度。利用率越高，说明你的 Shader 数学运算越密集，GPU 被迫在更高频率下运行，发热越严重。

| ALU 利用率 | 状态 | 含义 | 对发热的影响 |
| --- | --- | --- | --- |
| < 30% | 空闲或带宽瓶颈 | GPU 算力没用满，发热主因不在 Shader 计算 | 发热较低或瓶颈在内存带宽 |
| 30-60% | 正常 | Shader 计算负载适中 | 发热可控 |
| 60-80% | 偏高 | Shader 计算密集，是发热的重要贡献者 | 需要关注，建议优化 |
| \> 80% | 危险 | Shader 计算极重，GPU 全速满载运行 | 发热严重，必须优化 |

**如何读取 ALU 利用率：**

| 工具 | 对应计数器 | 平台 |
| --- | --- | --- |
| Snapdragon Profiler | ALU Utilization | 高通设备 |
| Arm Mobile Studio / Streamline | Mali Shader Core: Arithmetic Utilization | Mali 设备 |
| Xcode Metal System Trace | ALU Utilization + ALU Limiter | Apple 设备 |

> Apple GPU 有两个相关指标： **ALU Utilization** （实际做计算的时间）和 **ALU Limiter** （包含计算 + 等待时间）。如果 Limiter 很高但 Utilization 很低，说明 GPU 在等待数据（带宽瓶颈），而不是在计算。优化方向应该转向减少纹理采样而非简化数学运算。

### 3.6.3 Shader 复杂度量化分析

知道了 ALU 利用率之后，下一步是 **量化你的 Shader 到底有多复杂** 。不能只凭感觉，需要有具体的指标。

**业界标准的 Shader 分析维度（Mali Offline Compiler 模型）：**

ARM 的 Mali Offline Compiler 提供了四个维度的 Shader 复杂度分析，这也是移动端 GPU 分析的业界参考标准：

| 维度 | 缩写 | 含义 | 直观理解 | 优化方向 |
| --- | --- | --- | --- | --- |
| Arithmetic | A | 算术指令周期数 | Shader 里做了多少数学运算 | 减少复杂数学函数、简化公式 |
| Load/Store | LS | 内存读写周期数 | Shader 读写临时变量多少次 | 减少临时变量、优化寄存器使用 |
| Varying | V | 插值变量周期数 | 从顶点到像素传递了多少数据 | 减少 varying 变量数量 |
| Texture | T | 纹理采样周期数 | 采样了几次纹理 | 减少采样次数、确保使用 mipmap |

这四个维度的值加起来，就是 Shader 的总执行周期（Total Cycles）。哪个维度的值最高，就说明 Shader 在那个方向是瓶颈。

**使用 Mali Offline Compiler 分析 Shader：**

```
malisc -V Mali-G78 -c Fragment -s frag_shader.frag
```

输出示例：

```
A       LS       V       T     Stmts    Cycles
  12.50    2.00    1.00    3.00      15     18.50

Work registers:    6
Uniform registers: 12
Stack spilling:    0
16-bit arithmetic: 45%
```

解读：

- **A = 12.50** ：算术周期 12.5 个，是最大瓶颈 → 减少数学运算
- **T = 3.00** ：纹理采样 3 次，可接受
- **Cycles = 18.50** ：总周期 18.5，中等复杂度
- **Stack spilling = 0** ：没有寄存器溢出，好的信号

**Shader 性能预算建议：**

| 指标 | 低端设备 | 中端设备 | 高端设备 |
| --- | --- | --- | --- |
| Fragment Shader 算术周期（A） | < 32 cycles | < 64 cycles | < 128 cycles |
| 纹理采样次数（T） | ≤ 2 次 | ≤ 3 次 | ≤ 4 次 |
| 工作寄存器数 | ≤ 4 | ≤ 8 | ≤ 12 |
| 数学复杂度 | 仅加减乘 | 允许 sin/cos、lerp | 允许 pow、normalize、dot |
| 动态分支 | 禁止 | 仅 uniform 分支 | 可用但避免嵌套 |

> ⚠️ 以上是 **单个** Fragment Shader 的预算。如果你的画面有 Overdraw（同一像素被多次绘制），实际 GPU 工作量 = 单次 Shader 开销 × Overdraw 倍数。一个 32 cycles 的 Shader 在 3x Overdraw 区域里实际消耗 96 cycles。

**Shader 复杂度与 GPU Clock 的实测关系：**

这是一个很多人忽略的关键关系。下面是一个实际测试的案例：

同一个场景，只改变 Fragment Shader 的指令数：

| Fragment Shader 指令数 | GPU 频率（稳定态） | GPU 功耗（估算） | 帧时间 | 发热感知 |
| --- | --- | --- | --- | --- |
| 50 条（简化版） | 400 MHz | ~1.5W | 10ms | 温热 |
| 100 条（标准版） | 550 MHz | ~2.8W | 14ms | 发热 |
| 150 条（复杂版） | 700 MHz | ~4.5W | 22ms | 明显发烫 |

指令数从 50 增加到 150（3 倍），功耗从 1.5W 增加到 4.5W（3 倍），但因为频率同时升高导致电压也升高，实际功耗增长可能超过 3 倍。这就是 DVFS 机制让 Shader 复杂度对发热的影响被”放大”了。

### 3.6.4 GPU Cycle 分析：深入 GPU 内部

**什么是 GPU Cycle？**

GPU Cycle 是 GPU 执行一个操作所需的最小时钟周期。一帧画面从提交到渲染完成，经历了很多阶段的 Cycle：

```
CPU 提交渲染指令
  → 顶点处理（Vertex Cycles）：顶点着色器、坐标变换、骨骼蒙皮
  → 图元装配（Tiler Cycles / Geometry Cycles）：三角形裁剪、剔除
  → 像素处理（Fragment Cycles）：Fragment Shader 执行、纹理采样
  → 后处理（PostProcess Cycles）：全屏效果
  → 帧输出
```

**如何获取 GPU Cycle 数据：**

| 工具 | 可获取的 Cycle 数据 | 平台 |
| --- | --- | --- |
| Snapdragon Profiler | GPU Activity Cycles、Shader Instruction Cycles | 高通 |
| Arm Streamline | Vertex Cycles、Fragment Cycles、Tiler Cycles | Mali |
| Xcode Metal System Trace | Vertex Time、Fragment Time、Tile Time | Apple |
| RenderDoc | Shader 执行统计（Pixel Shader Invocations） | 通用 |

**用 Cycle 数据定位 GPU 发热瓶颈：**

```
第一步：用外部工具抓取一帧的 GPU Cycle 分布

第二步：分析 Cycle 占比
  ├── Fragment Cycles > 70% → 像素处理瓶颈
  │   → 降渲染分辨率
  │   → 简化 Fragment Shader
  │   → 减少后处理效果
  │   → 降低 Overdraw
  │
  ├── Vertex Cycles > 40% → 顶点处理瓶颈
  │   → 减少场景面数
  │   → 配置 LOD
  │   → 简化骨骼数量
  │   → 优化蒙皮网格
  │
  └── Tiler Cycles / Geometry Cycles 异常高 → 几何处理瓶颈
      → 检查是否有超高面数的单个物体
      → 检查是否有过小的三角形（像素覆盖率低）
      → 启用背面剔除
```

> 移动端 GPU 大多采用 Tile-Based Rendering（TBR）架构。这意味着 GPU 先把画面分成小块（Tile），在每个 Tile 内完成所有渲染操作。Tiler Cycles 异常高通常意味着几何数据量太大，或者 Tile 配置不合理。减少顶点数和三角形数是直接的优化手段。

### 3.6.5 GPU 深度分析的实战流程

把本节所有工具和概念串起来，完整的 GPU 发热深度分析流程如下：

```
1. Unity Profiler 确认 GPU 是瓶颈（GPU time > 16ms）
      ↓
2. Unity GPU Profiler 查看 GPU 各阶段耗时分布
   ├── Shadow > 30% → 进入 3.3 节阴影优化
   ├── Render > 50% → 继续深入
   └── PostProcess > 30% → 进入 3.5 节后处理优化
      ↓
3. 用平台工具（Snapdragon Profiler / Arm Streamline / Xcode）抓取详细数据
   ├── 查看 GPU Frequency → 是否长期处于高频率？
   ├── 查看 ALU Utilization → Shader 计算是否过重？
   ├── 查看 Fragment / Vertex / Tiler Cycles 分布 → 瓶颈在哪？
   └── 查看 Bandwidth Utilization → 是否带宽瓶颈？
      ↓
4. 根据分析结果选择优化方向
   ├── ALU 高 + Fragment Cycles 高 → 简化 Shader（参考 3.6.3 预算表）
   ├── ALU 低 + Bandwidth 高 → 减少纹理采样、降低纹理分辨率
   ├── Vertex Cycles 高 → 减面、LOD（参考 3.2 节）
   └── GPU Frequency 长期满频 → 综合降负载（分辨率 + 后处理 + Shader）
      ↓
5. 用 Frame Debugger / RenderDoc 定位具体的"重"Shader
   ├── 找到耗时最长的 Draw Call
   ├── 检查其 Fragment Shader 的指令数和采样次数
   └── 对照 3.6.3 的预算表判断是否超标
      ↓
6. 优化后重新抓取数据，对比 GPU Frequency 和 ALU Utilization 的变化
```

这个流程从宏观（Unity Profiler）到微观（平台工具的 Cycle 和 ALU 数据），逐层深入，最终定位到具体的 Shader 或渲染阶段，是专业级 GPU 发热分析的完整方法论。

### 4.2 测试流程

1. **准备阶段** ：将设备静置 10 分钟，确保初始温度低于 30°C
2. **记录初始数据** ：FPS、CPU 温度、GPU 温度、电池温度、机身表面温度
3. **启动游戏** ：进入目标测试场景
4. **持续运行** ：保持游戏运行 30 分钟，玩家角色保持正常活动（跑图、战斗等）
5. **数据采集** ：每 5 分钟记录一次所有数据
6. **结束测试** ：保存完整数据
7. **对比分析** ：与优化前的基准数据对比

测试场景应该覆盖游戏的核心玩法：主城/大厅、战斗场景、UI 密集界面、大地图跑图。不同场景的负载特征不同，需要分别测试。

### 第五章：CPU 优化 — 解决 CPU 发热

GPU 发热靠渲染管线优化来压制，CPU 发热则要从代码层面动刀。CPU 发热的根源通常不是某一段代码写得”差”，而是大量微小的低效操作每帧重复执行，累加起来吃满了时间预算。下面按模块逐一拆解。

---

### 5.1 脚本优化

### 避免在 Update 中频繁计算

Update 每帧执行一次。60fps 的游戏，一秒就是 60 次。一段在 Update 里跑 1ms 的代码，一秒就吃掉 60ms，占了 30fps 帧预算（33ms）的将近两倍。这是最常见的 CPU 发热源头。

**反模式示例：**

```
void Update()
{
    // 每帧拼接 string，产生大量 GC 分配
    scoreText.text = "Score: " + score.ToString();
    
    // 每帧执行 LINQ 查询，产生 GC 和 CPU 开销
    var enemies = FindObjectsOfType<Enemy>()
        .Where(e => e.isActiveAndEnabled)
        .OrderBy(e => e.distance)
        .ToList();
    
    // 每帧 new 新对象
    var damageInfo = new DamageInfo();
    CalculateDamage(damageInfo);
}
```

**修正后：**

```
private StringBuilder _sb = new StringBuilder(32);
private List<Enemy> _cachedEnemies = new List<Enemy>(64);
private DamageInfo _reusableDamage = new DamageInfo();

void Start()
{
    // 预缓存敌人列表，后续增量更新
    FindObjectsOfType<Enemy>(_cachedEnemies);
}

void Update()
{
    // 用 StringBuilder 拼接，零 GC
    _sb.Clear();
    _sb.Append("Score: ");
    _sb.Append(score);
    scoreText.text = _sb.ToString();
    
    // 复用已缓存的列表，不做 LINQ
    for (int i = 0; i < _cachedEnemies.Count; i++)
    {
        if (!_cachedEnemies[i].isActiveAndEnabled) continue;
        // 直接处理...
    }
    
    // 复用对象
    _reusableDamage.Reset();
    CalculateDamage(_reusableDamage);
}
```

> 一条判断原则：Update 里出现 `new` 、 `+` 拼接字符串、 `LINQ` 、 `GetComponent` ，都是需要排查的对象。

### 使用对象池（Object Pool）

`Instantiate` 和 `Destroy` 并不只是 C# 层面的 new 和 delete。Unity 内部要分配 native 内存、初始化组件、注册到各个子系统。频繁创建销毁会带来两个问题：CPU 尖峰和 GC 压力。对象池是最直接的解法。

**简易对象池实现：**

```
public class GameObjectPool
{
    private readonly Queue<GameObject> _pool;
    private readonly GameObject _prefab;
    private readonly Transform _parent;

    public GameObjectPool(GameObject prefab, int initialSize, Transform parent)
    {
        _prefab = prefab;
        _parent = parent;
        _pool = new Queue<GameObject>(initialSize);

        for (int i = 0; i < initialSize; i++)
        {
            var obj = Object.Instantiate(_prefab, _parent);
            obj.SetActive(false);
            _pool.Enqueue(obj);
        }
    }

    public GameObject Get(Vector3 position, Quaternion rotation)
    {
        GameObject obj;
        if (_pool.Count > 0)
        {
            obj = _pool.Dequeue();
        }
        else
        {
            obj = Object.Instantiate(_prefab, _parent);
        }
        obj.transform.SetPositionAndRotation(position, rotation);
        obj.SetActive(true);
        return obj;
    }

    public void Return(GameObject obj)
    {
        obj.SetActive(false);
        obj.transform.SetParent(_parent);
        _pool.Enqueue(obj);
    }
}
```

适用场景很广：子弹、伤害数字、粒子特效、怪物刷新、UI 列表项，甚至网络消息对象都可以池化。

> ⚠️ 对象池归还时记得重置对象状态（血量、速度、动画状态等），否则下次取出来会带着脏数据。

### 缓存组件引用

`GetComponent` 在 Unity 内部要做类型查找，不是简单的字典查询。每帧调用就是纯浪费。

```
// 反模式
void Update()
{
    var rb = GetComponent<Rigidbody>();
    rb.AddForce(force);
}

// 正确做法
private Rigidbody _rb;

void Awake()
{
    _rb = GetComponent<Rigidbody>();
}

void Update()
{
    _rb.AddForce(_force);
}
```

`GetComponentInChildren` 和 `GetComponentsInChildren` 更慢，因为要遍历子节点树。这类调用更要放在 `Awake` 或 `Start` 里。

### 避免不必要的协程

协程不是免费的。每个协程都是一个小状态机，Unity 每帧要遍历所有活跃协程并检查 yield 条件。几十个协程同时跑，开销可观。

**更好的替代方案：**

| 场景 | 协程写法 | 替代方案 |
| --- | --- | --- |
| 延迟执行 | yield return new WaitForSeconds(1f) | 用计时器变量在 Update 里倒计时 |
| 等待帧结束 | yield return new WaitForEndOfFrame() | 缓存 WaitForEndOfFrame 实例复用 |
| 异步加载 | yield return SceneManager.LoadSceneAsync | 使用 async/await + UniTask |

```
// 如果只是延迟一段时间，用 timer 比协程便宜
private float _timer;

void Update()
{
    _timer -= Time.deltaTime;
    if (_timer <= 0f)
    {
        DoSomething();
        _timer = interval;
    }
}
```

> UniTask 是 Unity 社区广泛使用的 async/await 方案，性能比协程好，GC 更少。项目里如果有大量异步逻辑，值得引入。

### 5.2 物理优化

物理引擎（PhysX）在固定时间步长里跑，每次更新都要做碰撞检测、刚体积分、约束求解。场景里刚体越多、碰撞体越复杂，CPU 负担越重。

**降低物理更新频率：**

Edit → Project Settings → Time → Fixed Timestep

| 设置值 | 频率 | 适用场景 |
| --- | --- | --- |
| 0.02（默认） | 50Hz | 大多数游戏默认 |
| 0.033 | 30Hz | 对物理精度要求不高的游戏 |
| 0.05 | 20Hz | 休闲游戏、卡牌游戏 |

> ⚠️ 降低频率后，快速移动的物体可能出现穿透。需要配合增大 Collider 或开启 CCD（Continuous Collision Detection）。

**简化碰撞体：**

Mesh Collider 是物理优化的头号敌人。一个复杂 Mesh Collider 的计算量是 Box Collider 的几十倍。

| 碰撞体类型 | 计算成本 | 适用场景 |
| --- | --- | --- |
| Box / Sphere | 极低 | 大多数场景 |
| Capsule | 低 | 角色胶囊体 |
| Convex Mesh | 中 | 需要精确碰撞的物体 |
| Mesh Collider（非凸） | 高 | 地形、静态环境（仅用于烘焙） |

规则：能用 Box + Sphere 组合近似的，绝不用 Mesh Collider。

**碰撞矩阵优化：**

在 Edit → Project Settings → Physics → Layer Collision Matrix 中，只勾选需要碰撞的层对。默认所有层互相碰撞，但很多组合永远不会发生（比如怪物和怪物之间、特效和地形之间）。把不必要的勾选去掉，物理引擎直接跳过这些对的检测。

**禁用休眠对象的物理：**

刚体静止一段时间后会自动进入 Sleep 状态，不再参与物理计算。但有些情况会打断休眠（比如微小力、精度误差）。可以通过 `Rigidbody.Sleep()` 手动强制休眠，也可以通过 `sleepThreshold` 调高休眠阈值。

**物理对发热的影响量化：**

在一个包含 200 个刚体、300 个碰撞体的战斗场景中实测：

| 物理耗时占比 | 场景描述 |
| --- | --- |
| 5-8% | 优化良好，碰撞矩阵精简，简单碰撞体 |
| 15-20% | 未优化，大量 Mesh Collider，碰撞矩阵全开 |
| 25%+ | 严重问题，物理帧率过高，刚体未休眠 |

### 5.3 动画优化

骨骼动画的 CPU 开销来自两部分：骨骼矩阵计算（每个骨骼每帧要做矩阵插值和乘法）和蒙皮（将顶点绑定到骨骼）。角色越多、骨骼越密，CPU 越累。

**骨骼数量建议：**

| 角色类型 | 建议骨骼上限 | 说明 |
| --- | --- | --- |
| 主角 | ≤ 80 | 需要高质量面部和手指动画 |
| 主要 NPC | ≤ 50 | 省略手指骨骼 |
| 杂兵/NPC | ≤ 30 | 简化骨骼，可用 Simple Animation |

**Animation Culling：**

Animator 组件有个 `Culling Mode` 属性：

- `Always Animate` ：始终播放，即使角色不在视野内（默认值，最费）
- `Cull Update Transform` ：角色不可见时，只播放动画状态机，不更新骨骼变换
- `Cull Completely` ：角色不可见时，完全停止动画

对于大部分游戏， `Cull Update Transform` 是最佳平衡。主城中同时存在 50 个 NPC，可能只有 10 个在屏幕内。把另外 40 个的骨骼计算省下来，CPU 瞬间轻松一半。

**Animation LOD：**

Unity 2020+ 支持 Animation LOD。远处角色的动画可以降低更新频率：近处每帧更新，远处每 2 帧甚至每 4 帧更新一次，视觉差异不明显，CPU 开销大幅下降。

**优化 Animator Controller：**

避免 Animator Controller 层级过深。每多一层，状态机就要多执行一遍。非必要不要超过 3 层。另外，减少 Transition 数量，复杂的 Transition 图会让状态机每帧做更多条件判断。

**动画压缩设置：**

Animation Compression 选 `Optimal` ，关键帧减少曲线精度。对于不重要的动画（比如杂兵的待机），可以适当降低精度。主角的动画建议保留更高精度，避免出现可见的抖动。

### 5.4 GC（垃圾回收）优化

GC 是 CPU 发热的隐藏杀手。Unity 使用 Boehm GC，执行时会暂停所有托管线程（Stop-The-World）。一次 GC 可能暂停 5-30ms，取决于堆大小和碎片程度。频繁 GC 意味着 CPU 反复做无用功，用户感受到的就是卡顿和发热。

**常见 GC 分配源及解决方案：**

| GC 来源 | 问题原因 | 解决方案 |
| --- | --- | --- |
| string 拼接 | \+ 运算符每次创建新 string | 用 StringBuilder，预分配容量 |
| new 数组/列表 | 每次扩容复制数据 | 预分配足够容量，或用对象池 |
| Box 装箱 | 值类型转 object（如 Hashtable） | 用泛型容器 List\<T>、Dictionary<TKey, TValue> |
| 闭包/Lambda | 捕获变量会生成隐藏类 | 减少不必要的闭包，改用方法引用 |
| yield return new | 每次创建新的 YieldInstruction | 缓存实例：WaitForSeconds \_wfs = new WaitForSeconds(1f) |
| params 参数 | 编译器生成临时数组 | 改用显式重载，避免 params |
| Enum.ToArray() | 装箱 + 数组分配 | 缓存结果，不要每次调用 |

**GC 监控方法：**

```
// 在游戏运行时监控 GC 频率
long _lastGCCount = 0;
float _gcCheckInterval = 60f; // 每 60 秒统计一次

IEnumerator MonitorGC()
{
    while (true)
    {
        _lastGCCount = GC.CollectionCount(0);
        yield return new WaitForSeconds(_gcCheckInterval);
        long delta = GC.CollectionCount(0) - _lastGCCount;
        Debug.Log($"[GC Monitor] 过去 {_gcCheckInterval}s 内发生 {delta} 次 GC");
    }
}
```

**建议标准：** GC 频率 ≤ 2 次/分钟。如果超过 5 次/分钟，说明有持续分配需要排查。超过 10 次/分钟，发热问题会很严重。

> Unity 2022+ 支持 Incremental GC（增量 GC），把一次大暂停拆成多次小暂停。但总量不变，减少的是卡顿感，不是 CPU 总工作量。真正的解法还是减少 GC 分配。

### 第六章：实战案例分析

案例是最好的老师。下面四个案例都来自真实项目，每个都走完”症状 → 定位 → 解决 → 验证”的完整流程。

### 6.1 案例一：主城挂机发热

**症状：** 玩家反馈在主城挂机 5-10 分钟后，手机背面明显发烫。主城没有战斗、没有复杂的 AI，按理说不应该这么热。

**定位过程：**

1. 连接 Unity Profiler，在主城场景运行 5 分钟。发现 Rendering 模块耗时占比 60%，CPU 等待 GPU 占了大部分帧时间。
2. 打开 Frame Debugger，逐 Draw Call 检查。发现主城中几百个建筑、路灯、装饰物，每个都是独立的 Draw Call。原因是这些物体没有被标记为 Static，无法参与静态批处理。
3. 进一步检查材质，发现同类物体使用了不同材质实例（材质球被复制了多份），导致无法合并。

**解决方案：**

- 将所有不移动的建筑物、地面、装饰物标记 Static，启用 Static Batching
- 合并同类物体的材质：所有木质结构用一个材质，所有石质结构用一个材质
- 检查材质实例化问题，确保运行时不会误调用 `renderer.material` （正确做法是 `renderer.sharedMaterial` ）

**效果：**

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| Draw Calls | 940 | 380 |
| SetPass Calls | 310 | 95 |
| 5分钟温度上升 | 9°C | 4°C |

**关键经验：** 主城是发热重灾区。场景复杂、玩家停留时间长，任何渲染低效都会被持续放大。主城优化应该是最高优先级。

---

### 6.2 案例二：多人同屏战斗卡顿发热

**症状：** 10v10 团战时帧率从 60fps 骤降到 20fps，手机 3 分钟内明显发烫。

**定位过程：**

1. CPU Profiler 显示 Animation.Update 和 SkinnedMeshRenderer.Render 占比合计超过 50%。
2. 检查角色模型，发现所有角色（包括远处的小兵）都使用 LOD0 模型，骨骼数 80+，面数 5000+。
3. Animator 的 Culling Mode 全部设为 Always Animate，离屏角色也在做完整动画计算。

**解决方案：**

- 配置角色 LOD：0-15m 使用 LOD0（5000面，80骨骼），15-30m 使用 LOD1（2000面，40骨骼），30m+ 使用 LOD2（800面，20骨骼）
- Animator Culling Mode 改为 `Cull Update Transform`
- 根据设备档次动态限制同屏角色数：低端最多 10 个，中端 15 个，高端 20 个
- 远处角色的 AI 更新频率降低（每 5 帧更新一次而非每帧）

**效果：**

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| 10人同屏 FPS | 22-28fps | 45-55fps |
| Animation 耗时 | 12ms | 5ms |
| 5分钟温度上升 | 10°C | 5°C |

**关键经验：** 同屏大量角色时，LOD 和动画裁剪是第一优先级。不做这两项，其他优化都是杯水车薪。

---

### 6.3 案例三：技能特效密集场景发热

**症状：** 释放大招时手机瞬间发烫，帧率掉到个位数。持续 2-3 秒后恢复正常。

**定位过程：**

1. 在大招释放瞬间抓取 GPU Profiler 数据，粒子渲染耗时占总 GPU 时间的 70%。
2. 开启 Overdraw 视图，发现特效覆盖区域 Overdraw 达到 8-10x。大量半透明粒子层层叠叠。
3. 检查粒子系统配置：单个大招特效包含 6 个独立的 Particle System，合计 1200 个粒子。

**解决方案：**

- 粒子合并：将 6 个 Particle System 中效果相近的合并为 2-3 个，粒子总数从 1200 降至 300
- 所有粒子材质开启 GPU Instancing
- 制定粒子数量上限规范：单个特效 ≤ 100 粒子，同屏特效总粒子数 ≤ 2000
- Overdraw 控制标准：特效区域 ≤ 4x
- 低端设备特效降级：粒子数减半、移除次级特效、降低发射速率

**效果：**

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| 特效期间 FPS | 8-12fps | 30-40fps |
| Overdraw（特效区域） | 8-10x | 3-4x |
| 粒子总数 | 1200 | 300 |
| GPU 耗时（特效帧） | 45ms | 18ms |

**关键经验：** 特效是瞬间发热的罪魁祸首。美术追求视觉效果时容易堆粒子数量，必须给特效制定明确的性能预算。

---

### 6.4 案例四：长时间游戏渐进发热

**症状：** 游戏前 10 分钟温度正常，之后持续上升。30 分钟后手机明显发烫。切换场景后短暂恢复，然后再次恶化。

**定位过程：**

1. 连续运行 30 分钟，每隔 5 分钟用 Memory Profiler 打一次快照。发现总内存从 320MB 持续增长到 520MB。
2. 对比快照，发现一个 `Dictionary<string, AssetBundle>` 持续增长：加载过的 AssetBundle 引用从未被移除。
3. 另一个增长源是 UI 列表：每次打开背包界面都创建新的列表项，关闭时没有销毁，只是隐藏。
4. 内存持续增长 → GC 可回收堆变小 → GC 更频繁触发 → CPU 额外负担 → 发热

**解决方案：**

- 修复 AssetBundle 引用泄漏：场景切换时调用 `AssetBundle.Unload(true)` 释放不再需要的资源
- UI 列表改用对象池：隐藏时归还池子，不再 destroy
- 排查所有 `Dictionary` 和 `List` ：只加不删的集合都是潜在泄漏源
- 每个场景切换时主动调用 `Resources.UnloadUnusedAssets()` 和 `GC.Collect()`

**效果：**

| 指标 | 优化前 | 优化后 |
| --- | --- | --- |
| 30分钟内存增长 | +200MB | +15MB |
| GC 频率 | 8次/分钟 | 1.5次/分钟 |
| 30分钟温度上升 | 12°C | 5°C |

**关键经验：** 渐进发热几乎一定是内存泄漏。养成定期对比内存快照的习惯，新功能上线前跑 30 分钟内存趋势测试。

### 第八章：常见问题与专家建议

### 8.1 常见问题

**Q1: 静态批处理后内存暴增怎么办？**

静态批处理的原理是把所有标记 Static 的物体的几何数据合并成大 Mesh。如果一个场景里有 500 个 Static 物体，每个 1000 面，合并后就是一个 50 万面的大 Mesh，要额外占用内存。解法是只对关键的、确实需要合批的物体标记 Static。小型装饰物、远景物体可以标记，大型独特建筑标记意义不大。每次批处理后监控内存变化，增量超过 20MB 就需要重新评估。

**Q2: GPU Instancing 为什么不生效？**

按这个检查清单逐一排查：

1. Shader 是否包含 `#pragma multi_compile_instancing` 编译指令
2. 材质 Inspector 中是否勾选了 Enable GPU Instancing
3. 代码中是否使用了 `MaterialPropertyBlock` 设置属性（直接修改 `renderer.material` 会破坏 Instancing）
4. 物体的 Mesh 和材质是否相同（不同 Mesh 无法 Instance）
5. 光照模式是否兼容（某些光照条件下 Instancing 会被强制打断）

**Q3: LOD 切换时有明显跳变怎么解决？**

三个手段配合使用：

- 调整距离阈值：把 LOD 级别之间的距离拉开，避免在玩家频繁经过的距离上切换
- 开启 Cross Fade 过渡：Unity LOD Group 支持 `Cross Fade` 模式，切换时两个 LOD 淡入淡出
- 增加中间 LOD 级别：如果 LOD0 到 LOD1 差异太大，插入一个 LOD0.5 级别

**Q4: 遮挡剔除效果不好，几乎没减少 DC？**

常见原因：

- 物体没有标记 Static（Occludee 和 Occluder 都要勾选）
- 场景不够封闭，遮挡剔除对开放世界效果有限
- Bake 的 Cell Size 太大或太小，需要根据场景调整
- 远处的天空盒、地形不会被遮挡剔除处理

遮挡剔除最适合室内场景和城市街巷这种有大量遮挡物的场景。开放大世界更适合用 LOD + 远距离裁剪。

**Q5: 优化后画质下降太多，美术和玩家都不满意？**

三个原则：

- 保留关键视觉元素：角色、UI、主要交互物体的画质优先保证
- 用法线贴图补偿细节：低面模型 + 高质量法线贴图，视觉效果接近高面模型
- 分设备差异化：高端设备保留全部效果，中端去掉后处理，低端再降纹理和粒子。玩家不会拿着高端机和低端机对比画质

### 8.2 专家建议

### 8.2.1 开发规范

性能问题防大于治。规范定在前面，后期省很多力气。

**建模规范：**

| 物体类型 | 面数上限 | 骨骼上限 | 说明 |
| --- | --- | --- | --- |
| 主角 | 5000 | 80 | 含手指和面部骨骼 |
| 重要 NPC | 3000 | 50 | 不含手指 |
| 杂兵/NPC | 1500 | 30 | 可用简化骨骼 |
| 场景小物体 | 500 | 无 | 草丛、石头等 |
| 场景大物体 | 2000 | 无 | 建筑、桥梁等 |

**贴图规范：**

- 尺寸必须是 2 的幂次方（256、512、1024、2048）
- 移动端使用 ASTC（iOS + Android 均支持）或 ETC2 压缩格式
- 主角贴图 1024×1024，NPC 512×512，场景物体 256×256
- 法线贴图比漫反射贴图小一级

**Shader 规范：**

- 单个 Shader 纹理采样不超过 4 次
- 避免在 Fragment Shader 中做分支判断
- 为低端设备准备 Simplified 版本 Shader
- 使用 Shader Variant Collection 管理变体，避免运行时编译卡顿

### 8.2.2 评审机制

| 环节 | 检查内容 | 频率 |
| --- | --- | --- |
| 资源入库 | 面数、贴图大小、Shader 复杂度 | 每次提交 |
| 代码评审 | Update 优化、内存管理、GC 分配 | 每次 MR |
| 性能报告 | FPS、DC、内存、温度 | 每周自动生成 |
| 性能会议 | 趋势分析、瓶颈定位、优化排期 | 每月 |
| 全面优化 | 回归测试、多设备验证、竞品对比 | 每季度 |

### 8.2.3 分层策略

不同设备不应该跑同一套画质配置。建议做三档：

| 档位 | 代表设备 | 策略 |
| --- | --- | --- |
| 高端 | iPhone 13+、骁龙 8 Gen1+ | 保留全部效果 |
| 中端 | iPhone X、骁龙 778G | 去掉后处理、降低阴影质量 |
| 低端 | iPhone 8、骁龙 660 及以下 | 关闭阴影、降分辨率、简化特效 |

更进一步，可以做自适应画质：监控设备温度，温度超过阈值时自动降级画质参数。温度回落后逐步恢复。这比固定分档更智能，也是主流手游的标配做法。

### 8.2.4 持续优化文化

性能优化不是项目末期的突击任务，是贯穿开发全程的日常习惯。

- **建立性能基线** ：每个版本记录关键性能指标（FPS、DC、内存、温度），画趋势图。指标恶化时能第一时间发现
- **经验文档化** ：每个优化案例记录”问题 → 原因 → 方案 → 效果”，归入团队知识库。新同事遇到类似问题时能快速查阅
- **定期团队分享** ：每月组织一次性能优化经验分享会，讨论近期遇到的问题和解决方案
- **工具自动化** ：把性能测试接入 CI/CD 流程，每次构建自动跑基准场景，指标超过阈值就发告警

### D. 温度参考标准

| 状态 | CPU/GPU 温度 | 电池温度 | 体感 |
| --- | --- | --- | --- |
| 正常 | 40-50°C | 35-40°C | 温热，正常游戏体验 |
| 警告 | 50-60°C | 40-45°C | 明显发热，部分设备开始降频 |
| 危险 | \> 60°C | \> 45°C | 烫手，系统强制降频/降亮度，需要立即干预 |

### E. 性能预算分配（30fps = 33ms/帧）

| 模块 | 时间预算 | 占比 | 说明 |
| --- | --- | --- | --- |
| 渲染（GPU） | 15-20ms | 45-60% | 最大的预算消耗者 |
| 游戏逻辑（CPU） | 5-10ms | 15-30% | AI、输入、网络同步 |
| 物理 | 3-5ms | 10-15% | 碰撞检测、刚体模拟 |
| 动画 | 3-5ms | 10-15% | 骨骼计算、蒙皮 |
| 系统/GC | 2-3ms | 6-9% | GC 回收、系统调用 |

> 如果某个模块超出预算，不要急着砍功能。先看能不能把预算从其他模块匀过来。比如物理简单的场景，可以把物理预算让给渲染。

### F. 内存预算参考

| 资源类型 | 预算 | 说明 |
| --- | --- | --- |
| 纹理 | 150-200MB | 占比最大，严格控制贴图尺寸和格式 |
| 网格 | 50-100MB | 包括角色、场景、碰撞体 |
| 音频 | 30-50MB | 背景音乐流式加载，音效预加载 |
| 动画 | 20-40MB | Animation Clip 数据 |
| 其他 | 30-50MB | Shader、Scriptable Object、配置数据 |
| 合计 | 300-450MB | 低端设备应靠近下限 |
