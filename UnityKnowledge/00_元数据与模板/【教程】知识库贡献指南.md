# 知识库贡献指南

> 将知识沉淀嵌入开发流程，使其成为开发的一部分，而非额外负担

## 一、贡献流程

### 1.1 新建文档流程

```
1. 选择模板 → 从 00_元数据与模板/ 中选择合适的模板
2. 填写元数据 → 必填字段全部填写，关系型字段尽量填写
3. 编写内容 → 遵循文档定位指南中的写作要点
4. 本地校验 → 运行 python scripts/validate_metadata.py
5. 提交PR → 使用下方PR模板
6. 代码审查 → 至少一名团队成员Review
7. 合并 → 审查通过后合并到main分支
```

### 1.2 修改文档流程

```
1. 更新内容 → 修改文档正文
2. 更新元数据 → 修改 updated 字段为当前时间
3. 检查关联 → 如果修改影响其他文档，同步更新
4. 本地校验 → 运行校验脚本
5. 提交PR → 说明修改原因
```

## 二、PR模板

提交新文档或修改时，PR描述应包含以下内容：

```markdown
## 变更类型

- [ ] 新增文档
- [ ] 修改现有文档
- [ ] 修复链接/元数据
- [ ] 模板/规范更新

## 变更说明

简要描述本次变更的内容和原因。

## 文档类型

- [ ] 使用了正确的模板（模板名称：___）
- [ ] 标题包含类型前缀（如【代码片段】）
- [ ] 必填元数据字段已填写
- [ ] 关系型元数据已填写（prerequisite/depends_on等）

## 质量检查

- [ ] 已运行 `python scripts/validate_metadata.py` 通过
- [ ] 已运行 `python scripts/check_links.py` 通过
- [ ] 代码片段可编译/运行
- [ ] 内部链接指向正确的文档

## 特殊审查要求

- [ ] 架构决策（ADR）→ 需要架构师 Approve
- [ ] 重大变更 → 需在团队频道通知
- [ ] 性能数据 → 需附带测试环境和可复现步骤
```

## 三、定期回顾机制

### 3.1 季度知识库健康度回顾

每季度（1月/4月/7月/10月的第一周）进行一次知识库健康度回顾。

**回顾清单**：

```markdown
## 季度回顾 - YYYY年Q?

### 1. 过时文档扫描
- [ ] 运行 `python scripts/validate_metadata.py --strict`
- [ ] 检查 status 为"已过时"的文档，确认是否删除或更新
- [ ] 检查 unity_version 是否与当前项目使用版本匹配

### 2. 验证程度审查
- [ ] 筛选 validation 为"未经测试"的文档
- [ ] 安排验证或降级其 status
- [ ] 将经过新项目验证的文档升级 validation 等级

### 3. 新增内容评估
- [ ] 本季度新增了哪些文档？
- [ ] 是否有新的架构决策需要记录？
- [ ] 是否有新的踩坑记录需要沉淀？

### 4. 链接健康度
- [ ] 运行 `python scripts/check_links.py`
- [ ] 修复所有断链

### 5. 索引更新
- [ ] 运行 `python scripts/generate_index.py`
- [ ] 检查生成的知识图谱是否合理

### 回顾结论
- 本季度知识库健康度评分：___/10
- 需要优先处理的问题：
  1. ...
  2. ...
- 下季度计划新增的内容：
  1. ...
  2. ...
```

### 3.2 月度快速检查

每月最后一个工作日，花15分钟做快速检查：

- 本月新增的文档是否都有完整元数据？
- 是否有文档需要更新 status？
- 是否有新的踩坑记录需要记录？

## 四、知识守护者角色

### 4.1 角色定义

指定1-2名核心架构师作为**知识库维护负责人（Knowledge Guardian）**。

### 4.2 职责

| 职责 | 频率 | 说明 |
|------|------|------|
| **审核PR** | 每次提交 | 确保新文档符合规范，元数据完整 |
| **驱动季度回顾** | 每季度 | 组织并主持季度回顾会议 |
| **维护模板** | 按需 | 根据团队反馈优化模板和流程 |
| **解答疑问** | 按需 | 帮助团队成员正确使用知识库 |
| **监控健康度** | 每月 | 运行自动化脚本，跟踪知识库质量 |

### 4.3 轮换机制

- 每半年轮换一次，确保知识不集中在个人
- 交接时需完成一次完整的知识库回顾
- 新任守护者需阅读 `00_元数据与模板/` 下所有规范文档

## 五、文档生命周期管理

### 5.1 何时创建文档

| 触发事件 | 应创建的文档类型 |
|----------|-----------------|
| 做了一个技术选型 | 【架构决策】 |
| 解决了一个棘手Bug | 【踩坑记录】 |
| 写了一段可复用代码 | 【代码片段】 |
| 做了性能优化 | 【性能数据】+ 【最佳实践】 |
| 完成了一个项目/模块 | 【实战案例】 |
| 深入研究了某个系统 | 【源码解析】或【设计原理】 |

### 5.2 何时更新文档

- Unity大版本升级后，检查 `unity_version` 相关文档
- 项目中验证了某个方案后，升级 `validation` 等级
- 发现文档中的错误或过时信息
- 有了更好的实践经验

### 5.3 何时归档文档

- 技术方案被新方案取代 → `status: 已过时`，在 `supersedes` 中标注新文档
- Unity API废弃 → `status: 已过时`
- 项目不再维护相关功能 → `status: 已归档`

## 六、快速参考

### 常用命令

```bash
# 校验元数据
python scripts/validate_metadata.py

# 严格模式校验
python scripts/validate_metadata.py --strict

# 检查链接
python scripts/check_links.py

# 生成索引
python scripts/generate_index.py

# 只生成Markdown索引
python scripts/generate_index.py --format markdown
```

### 文件命名规范

```
【类型前缀】主题名称.md

示例：
【代码片段】对象池通用实现.md
【架构决策】单例vs依赖注入.md
【踩坑记录】UGUI DrawCall飙升.md
```

### 元数据快速模板

```yaml
---
title: 【类型】标题
tags: [标签1, 标签2, 标签3]
category: 一级分类/二级分类
created: 2026-03-06 10:00
updated: 2026-03-06 10:00
description: 一句话描述
status: 已实施
validation: 项目实战
prerequisite: []
depends_on: []
related: []
---
```
