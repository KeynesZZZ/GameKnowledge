# 现有文档规则迁移规划

> 目标：在可回滚、可验证、可重复执行的前提下，把现有 UnityKnowledge 文档迁移到最新规则。

## 一、迁移目标

本次迁移要让旧文档统一符合以下规则：

- 使用统一 YAML Frontmatter
- 使用 13 种文档类型
- 使用轻量生命周期：`草稿` / `待验证` / `已验证` / `已过时` / `已归档`
- 补齐 `status`、`validation`、`related`
- 每篇正式文档具备 `文档定位` 和 `相关链接`
- 模板不再传播旧 category、旧 status、旧标签写法
- 内部链接检查为 0 断链
- `create-doc` skill 与 `00_元数据与模板` 保持一致

## 二、迁移原则

### 1. 先保护，再迁移

迁移前必须确认当前工作区状态，避免覆盖用户未提交内容。

```bash
git status --short
```

如果已有大量未提交变更，优先建立迁移分支或先提交当前基线。

### 2. 机器只做确定性修改

脚本适合做：

- 补缺失字段
- 旧状态映射
- 旧 category 修正
- 缺失章节补最小结构
- 去掉模板中的占位断链

脚本不适合做：

- 判断文章真实质量
- 自动填写复杂 `prerequisite`
- 自动合并重复文档
- 大幅重写正文

### 3. 一次性迁移，分层验证

可以一次性修改所有文档，但验证必须分层：

1. 元数据校验
2. 链接校验
3. 结构抽样
4. 重点目录人工复核

## 三、迁移范围

### 包含

```text
UnityKnowledge/**/*.md
UnityKnowledge/scripts/*.py
.claude/skills/create-doc.md
```

### 排除

```text
UnityKnowledge/.obsidian/
UnityKnowledge/scripts/__pycache__/
README.md（根目录，除非明确需要）
AIWorkflowKnowledge/（单独规划）
```

## 四、迁移规则

### Frontmatter 必填字段

```yaml
---
title:
tags:
category:
created:
updated:
description:
status:
validation:
related:
---
```

### 状态映射

| 旧状态 | 新状态 |
|--------|--------|
| `提议` | `草稿` |
| `讨论中` | `草稿` |
| `已采纳` | `待验证` |
| `已实施` | `待验证` |
| `已拒绝` | `已归档` |
| `已验证` | `已验证` |
| `已过时` | `已过时` |
| `已归档` | `已归档` |

团队 ADR 文档可以保留旧扩展状态，但普通教程、代码片段、踩坑记录优先使用轻量状态。

### validation 默认值

| 文档类型 | 默认 validation |
|----------|-----------------|
| `【实战案例】` | `项目实战` |
| `【性能数据】` | `Demo验证` |
| `【验证报告】` | `Demo验证` |
| `【模板】` | `Demo验证` |
| 其他类型 | `未经测试` |

### category 修正

| 旧写法 | 新写法 |
|--------|--------|
| `代码片段/Unity` | `架构设计/代码片段` |
| 缺失 category | 根据目录和标题前缀推断 |

## 五、执行阶段

### 阶段 0：冻结和备份

目标：确保迁移可回滚。

操作：

```bash
git status --short
git branch codex/docs-rules-migration
```

验收：

- 已确认当前未提交变更来源
- 已切换到迁移分支
- 知道如何回滚

### 阶段 1：修规则源头

目标：确保新文档不再继续变坏。

处理文件：

- `UnityKnowledge/00_元数据与模板/元数据规范.md`
- `UnityKnowledge/00_元数据与模板/文档定位指南.md`
- `UnityKnowledge/00_元数据与模板/文档结构规范.md`
- `UnityKnowledge/00_元数据与模板/文档生命周期.md`
- `UnityKnowledge/00_元数据与模板/文档健康检查清单.md`
- `UnityKnowledge/00_元数据与模板/【模板】*.md`
- `.claude/skills/create-doc.md`

验收：

- 文档类型口径统一为 13 种
- 模板不含旧 category
- YAML 标签示例不带 `#`
- 模板占位链接不会被 link checker 识别为断链

### 阶段 2：批量迁移元数据

目标：让所有正式文档通过元数据校验。

建议使用迁移脚本：

```bash
python3 UnityKnowledge/scripts/migrate_docs_to_latest_rules.py
```

脚本职责：

- 补齐缺失 frontmatter
- 补齐必填字段
- 映射旧状态
- 补 `validation`
- 补 `related: []`
- 更新 `updated`

验收：

```bash
python3 UnityKnowledge/scripts/validate_metadata.py --strict
```

期望结果：

```text
错误: 0
警告: 0
```

### 阶段 3：批量补结构

目标：让旧文档至少具备最新结构底线。

最低要求：

- 有 `文档定位`
- 有 `相关链接`
- 不强行补无价值章节

处理策略：

- 缺少 `文档定位`：补一段最小定位说明
- 缺少 `相关链接`：补空列表或链接到对应规范
- 模板文件：占位链接使用普通文本或代码格式，不使用 `[[ ]]`

验收：

- 抽查每个一级目录至少 3 篇
- 没有明显机械重复到影响阅读的段落

### 阶段 4：修复断链

目标：内部链接检查归零。

检查命令：

```bash
python3 UnityKnowledge/scripts/check_links.py
```

断链处理规则：

| 断链类型 | 处理方式 |
|----------|----------|
| 模板占位链接 | 改成普通文本占位 |
| URL 编码路径 | 改为真实文件名 |
| 文档改名 | 指向真实存在文档 |
| 文档确实缺失 | 改成“待整理”文本，或创建 Inbox 草稿 |

验收：

```text
断链: 0
```

### 阶段 5：重点目录复核

目标：避免“校验通过但内容质量下降”。

优先复核：

1. `00_元数据与模板`
2. `10_架构设计`
3. `30_性能优化`
4. `20_核心系统`
5. `40_工具链`

抽查项：

- 标题前缀是否合理
- category 是否符合目录
- status 是否过于乐观
- validation 是否过于乐观
- 机械补充的 `文档定位` 是否需要人工优化

## 六、验收标准

### 必须通过

```bash
python3 UnityKnowledge/scripts/validate_metadata.py --strict
python3 UnityKnowledge/scripts/check_links.py
```

### 结果要求

| 检查 | 目标 |
|------|------|
| 元数据错误 | 0 |
| 元数据警告 | 0 |
| 内部断链 | 0 |
| 正式文档缺少 `文档定位` | 0 |
| 正式文档缺少 `相关链接` | 0 |

## 七、风险与回滚

### 主要风险

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 批量脚本误判标题类型 | 文件名和内容不一致 | 按目录抽查，必要时手动修 |
| status 过于乐观 | 自动设成已验证会误导 | 默认使用 `待验证` 或 `未经测试` |
| 文档定位过于机械 | 批量补的句子可读性一般 | 后续人工优化高频文档 |
| 大量 diff 难审 | 一次性修改 200+ 文档 | 先看统计，再抽查重点目录 |

### 回滚方式

如果迁移结果不可接受：

```bash
git restore UnityKnowledge .claude/skills/create-doc.md
```

如果已经分支提交，则回退该提交。

## 八、最终产出

迁移完成后应产出：

- 全库已迁移文档
- 迁移脚本
- 校验通过记录
- 残留人工复核清单
- 后续每周维护流程

## 九、推荐执行顺序

```text
1. 确认工作区和分支
2. 修 00_元数据与模板 和 create-doc
3. 执行批量迁移脚本
4. 跑 strict 元数据校验
5. 修所有断链
6. 抽查重点目录
7. 生成迁移报告
8. 提交迁移结果
```
