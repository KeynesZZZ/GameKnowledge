# 元数据规范

> UnityKnowledge 文档 YAML Frontmatter 轻量规范，确保 Obsidian 本地检索和 Dify 同步可用，同时避免为了填字段而降低写作意愿。

## 文档定位

本文档定义 UnityKnowledge 新文档的最小 Frontmatter 要求。旧文档可以保留已有字段；复杂架构、性能、项目复盘文档可以按需补充关系型元数据、状态生命周期和版本兼容性。

---

## 必填字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `title` | string | 文档标题，带类型前缀 | `【笔记】对象池为什么能减少GC` |
| `tags` | array | 标签数组，至少2个 | `[C#, Unity, 性能优化]` |
| `created` | date | 创建日期 | `2026-06-17` |
| `description` | string | 一句话描述 | `C#泛型对象池的基础实现，减少GC分配` |

## 可选字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `category` | string | 分类路径 | `架构设计/代码片段` |
| `updated` | date | 重要更新日期 | `2026-06-17` |
| `unity_version` | string | 适用的Unity版本 | `2022.3+` |
| `cover` | string | 封面图路径 | `./attachments/cover.png` |
| `complexity` | string | 复杂度 | `简单` / `中等` / `复杂` |
| `use_frequency` | string | 使用频率 | `高频` / `中频` / `低频` |
| `dependencies` | array | 依赖项 | `[DOTween, UniTask]` |
| `status` | string | 文档状态（见下方生命周期） | `草稿` / `待验证` / `已验证` / `已过时` / `已归档` |
| `validation` | string | 验证程度 | `未经测试` / `Demo验证` / `项目实战` / `多项目验证` |
| `author` | string | 作者来源（防幻觉的关键标记） | `human` / `llm` |
| `sources` | array | 结论来源（综述页必填，引用即契约） | `["【笔记】对象池", "外部文章A"]` |

## 关系型元数据（按需）

用于表达文档之间的**结构化关系**，适合架构决策、系统设计、项目实战等高价值文档。普通笔记不需要为了完整性硬填。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `prerequisite` | array | 前置知识，阅读本文前应先阅读的文档 | `["【设计原理】状态机本质"]` |
| `depends_on` | array | 依赖系统，本文内容依赖的系统/模块文档 | `["【系统架构】场景管理"]` |
| `is_example_for` | array | 本文是某篇文档的实战案例 | `["【教程】状态机模式"]` |
| `refutes` | array | 本文反驳/纠正了某个观点 | `["【踩坑】常见架构陷阱#Update泛滥"]` |
| `supersedes` | array | 本文取代了哪些旧文档 | `["【架构决策】旧版单例方案"]` |
| `related` | array | 相关文档（无方向性的弱关联） | `["【笔记】foreach vs for"]` |

### 关系型元数据示例

```yaml
---
title: 【实战案例】三消游戏架构
tags: [Unity, 架构, 三消, 实战案例]
category: 架构设计/实战案例
created: 2026-01-10 14:00
updated: 2026-03-06 10:00
description: 三消+Roguelike完整架构实现，涵盖核心算法、UI、战斗系统
prerequisite:
  - "【设计原理】策略模式本质"
  - "【教程】事件系统"
depends_on:
  - "【系统架构】场景管理"
  - "【代码片段】对象池通用实现"
is_example_for:
  - "【教程】状态机模式"
related:
  - "【笔记】三消核心算法Benchmark"
status: 已验证
validation: 项目实战
---
```

## 状态与生命周期（按需）

### status 状态流转

个人知识库优先使用轻量生命周期：

```
Inbox → 草稿 → 待验证 → 已验证 → 已过时 → 已归档
```

| 状态 | 含义 | 适用场景 |
|------|------|----------|
| `Inbox` | 临时记录，尚未分类 | 刚捕获的问题、想法、资料 |
| `草稿` | 已有结构，但内容未完整 | 初版教程、未整理的踩坑记录 |
| `待验证` | 结论合理，但缺少实践验证 | 需要测试或项目验证的方案 |
| `已验证` | 经过实战验证，可信赖 | 上线运行且效果达标 |
| `已过时` | 因技术演进不再适用 | Unity大版本升级导致API变更 |
| `已归档` | 保留但不再维护 | 历史参考文档 |

团队协作或 ADR 文档可额外使用：`提议`、`讨论中`、`已采纳`、`已实施`、`已拒绝`。

### validation 验证程度

| 验证级别 | 含义 | 可信度 |
|----------|------|--------|
| `未经测试` | 纯理论分析或经验总结 | ⭐ |
| `Demo验证` | 在独立Demo中测试通过 | ⭐⭐ |
| `项目实战` | 在至少一个正式项目中使用 | ⭐⭐⭐⭐ |
| `多项目验证` | 在多个项目中反复验证 | ⭐⭐⭐⭐⭐ |

## 适用场景元数据（按需）

用于实现**智能推荐**：新项目启动时，可根据项目属性自动筛选适用的架构模式与最佳实践。

```yaml
applicability:
  project_scale: ["小型原型", "中型团队", "大型量产"]
  project_type: ["3D动作", "2D策略", "联网多人", "休闲", "三消"]
  performance_critical: true
```

| 字段 | 类型 | 说明 | 可选值 |
|------|------|------|--------|
| `applicability.project_scale` | array | 适用项目规模 | `小型原型` / `中型团队` / `大型量产` |
| `applicability.project_type` | array | 适用游戏类型 | `3D动作` / `2D策略` / `联网多人` / `休闲` / `三消` / `Roguelike` / `卡牌` / `SLG` |
| `applicability.performance_critical` | bool | 是否性能敏感 | `true` / `false` |

## 版本兼容性元数据（按需）

避免"知识过期"，Unity大版本升级或包更新时可快速扫描受影响文档。

```yaml
unity_version: "2022.3+"
package_versions:
  - "com.unity.entities": "1.0.0"
  - "com.unity.burst": "1.8.0"
  - "VContainer": "1.1.0"
```

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `unity_version` | string | 适用的Unity最低版本 | `2022.3+` |
| `package_versions` | array | 依赖的包及版本 | `["com.unity.dots: 1.0.0"]` |

## 标题命名规范

### 类型前缀

新文档优先使用以下轻量类型（前 4 种为人写；`【笔记】` 为 LLM 维护的综合页，`author` 必须为 `llm`）：

| 文档类型 | 标题前缀 | 示例 |
|----------|----------|------|
| 笔记 | `【笔记】` | `【笔记】对象池为什么能减少GC` |
| 踩坑 | `【踩坑】` | `【踩坑】UGUI DrawCall飙升` |
| 复盘 | `【复盘】` | `【复盘】休闲游戏云存档系统` |
| 片段 | `【片段】` | `【片段】对象池通用实现` |
| 综述 | `【笔记】` | `【笔记】Unity性能优化全景` |

> `【笔记】` 是 LLM 维护的综合页（跨文档提炼），`author` 必须为 `llm`，且 `sources` 必填、指向它综合了哪些 `author: human` 或已验证笔记 / 外部原文。这是"引用即契约"的落地。

旧文档和复杂专题可以继续使用以下类型，不需要为了统一而批量重命名：

| 文档类型 | 标题前缀 | 示例 |
|----------|----------|------|
| 代码片段 | `【代码片段】` | `【代码片段】对象池通用实现` |
| 最佳实践 | `【最佳实践】` | `【最佳实践】依赖注入使用指南` |
| 踩坑记录 | `【踩坑】` | `【踩坑】UGUI DrawCall飙升` |
| 性能数据 | `【笔记】` | `【笔记】foreach vs for` |
| 设计原理 | `【设计原理】` | `【设计原理】ECS为什么快` |
| 架构决策 | `【架构决策】` | `【架构决策】单例vs依赖注入` |
| 系统架构 | `【系统架构】` | `【系统架构】场景管理` |
| 实战案例 | `【实战案例】` | `【实战案例】三消游戏架构` |
| 教程 | `【教程】` | `【教程】DOTween快速入门` |
| 源码解析 | `【笔记】` | `【笔记】碰撞检测模式对比` |

```

# 按时间范围搜索
created:>=2024-03-01
```

## Dify 同步兼容性

### 向量化处理

YAML frontmatter 会被 Dify 自动识别：
- `title` → 文档标题
- `tags` → 元数据标签
- `description` → 文档摘要

### 分段建议

Dify 索引参数配置：
- 分段方式: 自动分段
- 分段标识符: `\n\n\n`
- 最大分段长度: 500 tokens
- 重叠长度: 50 tokens

---

## 迁移检查清单

重构文档时，确保：

- [ ] 添加最小 YAML frontmatter
- [ ] 标题添加类型前缀
- [ ] tags 数组至少包含 2 个标签
- [ ] created 使用日期格式
- [ ] description 简洁准确（建议 50 字以内）
