# 文档生命周期

> 生命周期的目标是让文档可信、可维护、不过期。

## 推荐状态流

个人知识库优先使用这条轻量状态流：

```text
Inbox → 草稿 → 待验证 → 已验证 → 已过时 → 已归档
```

| 状态 | 含义 | 处理方式 |
|------|------|----------|
| `Inbox` | 临时记录，尚未分类 | 每周整理，移动到正式目录或删除 |
| `草稿` | 已有结构，但内容未完整 | 补齐定位、核心内容和相关链接 |
| `待验证` | 结论合理，但缺少实践验证 | 增加测试、项目案例或来源 |
| `已验证` | 经过 Demo 或项目验证 | 正常使用，定期复查 |
| `已过时` | 因版本或实践变化不再推荐 | 标注原因，链接替代文档 |
| `已归档` | 保留历史价值，不再维护 | 不作为默认推荐结果 |

## 团队 ADR 扩展状态

如果文档是架构决策，可以使用扩展状态：

```text
提议 → 讨论中 → 已采纳 → 已实施 → 已验证
              ↓
            已拒绝
```

这组状态只推荐用于 `【架构决策】` 和团队协作流程，不要求普通教程、代码片段、踩坑记录使用。

## validation 字段

| 验证级别 | 含义 |
|----------|------|
| `未经测试` | 纯理论、经验判断或外部资料整理 |
| `Demo验证` | 在独立 Demo 或小样例中验证 |
| `项目实战` | 在至少一个正式项目中使用 |
| `多项目验证` | 多个项目反复验证，可信度高 |

## 更新规则

- 修改正文结论时，必须更新 `updated`
- 标记 `已过时` 时，必须说明过时原因
- `已归档` 文档不再作为最佳实践引用
- 超过 180 天未更新的高频文档，应进入复查清单

## 文档定位

本文档定义了文档从 Inbox 到已归档的完整状态流转规则，以及 validation 字段的四级可信度标准。适合在创建新文档时确定初始状态，或在定期回顾时判断文档是否需要升级或归档。
