## 二、架构一致性检查

> 确保代码符合团队已采纳的架构决策

- [ ] **依赖注入**：新增的类是否通过DI获取依赖，而非直接 `new` 或使用单例？
  - 关联：→ [[【架构决策】单例vs依赖注入]]
- [ ] **事件系统**：模块间通信是否使用EventBus，而非直接引用？
  - 关联：→ [[【最佳实践】事件系统使用指南]]
- [ ] **对象池**：频繁创建/销毁的对象是否使用对象池？
  - 关联：→ [[【代码片段】对象池通用实现]]
- [ ] **状态管理**：新增的状态逻辑是否使用状态机模式？
  - 关联：→ [[【教程】状态机模式]]
- [ ] **分层架构**：代码是否放在正确的层级（Model/View/Presenter）？
  - 关联：→ [[【系统架构】UI架构]]
- [ ] **命名规范**：类名、方法名、变量名是否符合团队命名规范？

---

## 三、性能检查

> 避免引入性能问题

- [ ] **GC分配**：Update/FixedUpdate中是否有 `new`、字符串拼接、LINQ、装箱？
  - 关联：→ [[【最佳实践】GC优化清单]]
- [ ] **缓存**：是否缓存了 `GetComponent`、`Find` 等昂贵调用的结果？
  - 关联：→ [[【踩坑记录】Unity常见性能陷阱]]
- [ ] **协程**：是否避免了每帧 `StartCoroutine`？是否使用 UniTask 替代？
  - 关联：→ [[【架构决策】协程vs UniTask]]
- [ ] **集合操作**：大集合是否避免了 `foreach`（使用 `for` 或 Span）？
  - 关联：→ [[【性能数据】foreach vs for]]
- [ ] **物理查询**：是否使用了 NonAlloc 版本的物理查询？
  - 关联：→ [[【最佳实践】物理系统性能优化]]

---

## 四、内存安全检查

- [ ] **资源释放**：加载的资源是否有对应的卸载逻辑？
  - 关联：→ [[【踩坑记录】内存泄漏模式]]
- [ ] **事件解注册**：注册的事件是否在 OnDestroy/OnDisable 中解注册？
- [ ] **引用清理**：是否避免了静态引用持有 MonoBehaviour？
- [ ] **纹理/材质**：是否避免了运行时创建未释放的材质实例？

---

## 五、可维护性检查

- [ ] **单一职责**：新增的类是否只做一件事？超过300行的类是否需要拆分？
- [ ] **注释**：公共API是否有XML注释？复杂逻辑是否有行内注释？
- [ ] **魔法数字**：是否将硬编码数字提取为常量或配置？
- [ ] **错误处理**：是否有适当的空检查、边界检查、异常处理？
- [ ] **可测试性**：核心逻辑是否可以脱离MonoBehaviour进行单元测试？

---

## 六、平台兼容性检查

- [ ] **平台差异**：是否考虑了目标平台的限制（如WebGL不支持多线程）？
  - 关联：→ [[【教程】WebGL限制与方案]]
- [ ] **分辨率适配**：UI是否在不同分辨率下正常显示？
- [ ] **输入方式**：是否同时支持触屏和键鼠输入？
