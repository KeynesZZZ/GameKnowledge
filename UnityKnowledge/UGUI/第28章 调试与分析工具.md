### 28.1 Frame Debugger

Frame Debugger 是 Unity 提供的逐帧渲染分析工具，可以查看当前帧中所有渲染指令的执行过程。对于 UGUI 而言，它是分析 Canvas 合批、Draw Call 数量以及渲染顺序最重要的工具之一。

通过 Frame Debugger，开发者可以清楚地看到一个界面最终生成了多少个 Draw Call，以及每个 Draw Call 对应的 UI 元素和材质信息。

打开 Frame Debugger 的方式如下：

**Window → Analysis → Frame Debugger**

打开后点击 **Enable** 按钮，Unity 会暂停当前帧的渲染流程，并允许开发者逐条查看渲染命令。

在 UGUI 分析过程中，Frame Debugger 主要用于观察以下内容：

### 28.1.1 查看 UI Draw Call

当 Canvas 开始渲染时，UGUI 会将多个 Graphic 组件生成的网格数据提交给渲染系统。

如果多个 UI 元素满足合批条件，则会被合并到同一个 Draw Call 中。

如果合批失败，则会产生额外的 Draw Call。

通过 Frame Debugger 可以直接查看每个 Draw Call 对应的渲染对象，从而快速定位导致批处理失效的 UI 元素。

### 28.1.2 分析 Canvas 合批

UGUI 的性能优化很大程度上依赖于 Canvas Batch。

在 Frame Debugger 中选中某个 Draw Call 后，可以查看其使用的：

- Material
- Texture
- Shader
- Vertex Count
- Triangle Count

如果相邻 UI 元素使用了不同材质或纹理，通常会导致新的 Batch 被创建。

因此 Frame Debugger 是分析 Batch 断裂原因最直接的工具。

### 28.1.3 查看渲染顺序

UGUI 的渲染顺序通常由层级顺序决定。

在 Frame Debugger 中可以看到所有 UI 的实际提交顺序，从而验证界面显示结果是否符合预期。

当出现遮挡异常、层级错误或者排序问题时，可以通过 Frame Debugger 快速定位原因。

### 28.1.4 分析 Mask 开销

Mask 和 RectMask2D 都会影响 UI 渲染流程。

特别是 Mask 组件，会引入额外的 Stencil 操作。

在 Frame Debugger 中可以看到 Mask 产生的额外 Draw Call 和 Stencil Pass，从而评估其对性能的影响。

### 28.1.5 常见分析场景

实际开发中，Frame Debugger 最常用于以下问题排查：

- Draw Call 数量异常增多
- Canvas 合批失效
- UI 渲染顺序错误
- Mask 导致性能下降
- 材质或纹理切换过于频繁

对于 UGUI 性能优化而言，Frame Debugger 是最基础也是最常用的分析工具。很多 Canvas 合批问题和 Draw Call 问题，都可以通过它快速定位和验证。

### 28.2 UI Profiler

Profiler 是 Unity 最重要的性能分析工具之一，而 UI Profiler 则专门用于分析 UGUI 的运行状态。通过 UI Profiler，可以观察界面在 CPU 侧的开销情况，以及布局重建、顶点重建和批处理等关键数据。

打开方式如下：

**Window → Analysis → Profiler**

进入 Profiler 后选择 **UI** 模块，即可查看当前界面的相关统计信息。

### 28.2.1 UI Profiler 的主要指标

UI Profiler 会显示当前界面的多个统计数据，其中比较重要的包括：

这些数据能够反映当前界面的整体运行状态。

例如 Vertices 过高通常意味着界面包含大量图片、文本或复杂图形，而 Rebuild 次数过高则可能意味着界面正在频繁刷新。

### 28.2.2 分析 Layout Rebuild

布局系统是 UGUI 中常见的性能瓶颈之一。

当 RectTransform、LayoutGroup 或 ContentSizeFitter 相关属性发生变化时，系统会触发布局重建。

如果界面中频繁出现 Layout Rebuild，通常说明布局组件正在不断重新计算。

常见原因包括：

- 持续修改尺寸
- 动态增删节点
- ContentSizeFitter 与 LayoutGroup 组合使用
- ScrollView 内容频繁变化

当 Layout Rebuild 数量持续增加时，应优先检查布局系统的使用情况。

### 28.2.3 分析 Graphic Rebuild

Graphic Rebuild 表示 UI 顶点数据正在重新生成。

当以下情况发生时，Graphic 会被标记为 Dirty：

- 修改 Image Sprite
- 修改颜色
- 修改文本内容
- 修改顶点相关属性

在 Canvas 更新阶段，这些 Graphic 会重新生成 Mesh 数据。

如果 Graphic Rebuild 数量较高，通常意味着界面正在频繁刷新。

### 28.2.4 分析 Batch 数据

Batch 数量直接影响 UI 渲染性能。

正常情况下，同一个 Canvas 中满足条件的 UI 元素会被合并到较少的 Batch 中。

如果 Batch 数量突然增加，通常说明出现了以下情况：

- 材质切换
- 纹理切换
- Mask 插入
- 独立 Canvas 过多

此时可以结合 Frame Debugger 进一步分析具体原因。

### 28.2.5 CPU Timeline 分析

除了 UI 模块外，Profiler 的 Timeline 视图也是分析 UGUI 的重要工具。

在 Timeline 中，经常能够看到以下函数：

- Canvas.BuildBatch
- Canvas.SendWillRenderCanvases
- LayoutRebuilder.Rebuild
- Graphic.Rebuild

这些函数反映了 UI 系统在当前帧中的实际执行过程。

如果某个函数占用了大量 CPU 时间，则说明对应阶段可能存在性能问题。

### 28.2.6 UI 性能分析思路

使用 UI Profiler 分析 UGUI 时，通常遵循以下流程：

1. 观察 CPU 总耗时。
2. 查看 Layout Rebuild 次数。
3. 查看 Graphic Rebuild 次数。
4. 检查 Batch 和 Vertices 数量。
5. 结合 Timeline 定位具体开销来源。
6. 使用 Frame Debugger 验证渲染结果。

通过 UI Profiler，可以快速判断性能问题究竟来自布局系统、图形重建还是渲染阶段，为后续优化提供明确方向。

### 28.3 RenderDoc 分析 UI

Frame Debugger 可以帮助我们查看 UGUI 的渲染流程和 Batch 情况，但它只能观察 Unity 视角下的渲染结果。如果需要进一步分析 GPU 实际执行的渲染命令、顶点数据、纹理资源以及 Shader 状态，则需要借助 RenderDoc。

RenderDoc 是目前最常用的图形调试工具之一，支持 DirectX、Vulkan、OpenGL 等多种图形接口，能够捕获某一帧的完整渲染数据，并对每个 Draw Call 进行深入分析。

对于 UGUI 而言，RenderDoc 主要用于分析实际 GPU 渲染过程。

### 28.3.1 捕获渲染帧

安装 RenderDoc 后，可以通过 Unity 的 RenderDoc 集成功能捕获当前渲染帧。

当游戏运行时，点击 Capture Frame 按钮，RenderDoc 会记录当前帧所有渲染指令。

捕获完成后，可以在 RenderDoc 中查看该帧的完整渲染过程。

### 28.3.2 查看 Draw Call

在 Event Browser 窗口中，可以看到当前帧执行的所有 Draw Call。

每个 UI Batch 最终都会对应一个或多个 GPU Draw Call。

通过查看这些 Draw Call，可以分析：

- 实际提交次数
- 顶点数量
- 三角形数量
- 渲染目标
- Shader 使用情况

这些数据能够帮助开发者验证 UGUI 的合批结果是否符合预期。

### 28.3.3 查看 Mesh 数据

选中某个 Draw Call 后，可以打开 Mesh Viewer。

Mesh Viewer 会显示当前 Draw Call 使用的顶点数据。

对于 UGUI 而言，可以看到：

- 顶点位置
- UV 坐标
- 顶点颜色
- 三角形索引

这有助于分析 UI 网格生成是否正确，以及复杂控件是否产生了过多顶点。

### 28.3.4 查看纹理资源

在 Texture Viewer 中，可以查看当前 Draw Call 使用的纹理。

UGUI 中常见的纹理包括：

- Sprite Atlas
- UI 图片纹理
- Font Atlas
- RenderTexture

通过观察纹理切换情况，可以分析 Batch 被拆分的原因，以及图集是否发挥了应有作用。

### 28.3.5 查看 Shader 状态

RenderDoc 可以显示当前 Draw Call 的完整渲染状态。

包括：

- Shader
- Material 参数
- Blend State
- Depth State
- Stencil State

对于使用 Mask 的界面，可以重点观察 Stencil Buffer 的变化情况，从而理解 Mask 的实际工作过程。

### 28.3.6 分析过度绘制

UI 通常采用透明渲染。

当多个半透明元素互相覆盖时，GPU 可能会重复渲染同一个像素区域。

这种现象称为 Overdraw（过度绘制）。

通过 RenderDoc 的纹理和渲染结果分析功能，可以观察界面中的大量重叠区域，从而发现潜在的 FillRate 问题。

在移动平台上，过度绘制往往比 Draw Call 更容易成为性能瓶颈。

### 28.3.7 RenderDoc 与 Frame Debugger 的区别

Frame Debugger 和 RenderDoc 都能够查看 Draw Call，但关注点并不相同。

Frame Debugger 更适合分析 Canvas 合批、Batch 拆分和渲染顺序。

RenderDoc 更适合分析 GPU 状态、Mesh 数据、纹理资源以及 Shader 执行过程。

在实际项目中，两者通常配合使用。首先利用 Frame Debugger 定位 UGUI 的渲染问题，再通过 RenderDoc 深入分析 GPU 侧的实际开销，从而获得更加完整的性能分析结果。

### 28.4 自定义工具

前面介绍的 Frame Debugger、Profiler 和 RenderDoc 能够帮助我们从不同角度分析 UGUI 的运行情况，但这些工具更多是在问题出现之后进行分析。

在大型项目中，开发人员往往更希望能够快速发现问题，而不是等到性能下降后再逐步排查。因此，很多团队都会基于 UGUI 源码中的关键节点开发专门的调试工具，用于实时监控界面状态。

这些工具通常不会参与正式发布，而是在开发阶段帮助程序和 UI 工程师快速定位问题。

本节将结合前面章节分析过的 UGUI 源码，实现几个常见的 UI 调试工具。

### 28.4.1 为什么需要自定义工具

Unity 官方工具虽然功能强大，但仍然存在一些局限。

例如，当 Profiler 中出现大量 Canvas.BuildBatch 时，我们知道界面正在频繁重建，但无法直接知道究竟是哪个界面触发了重建。

当 Batch 数量突然增加时，我们知道发生了合批失效，但无法立即定位到具体对象。

当某个界面持续产生 Graphic Rebuild 时，我们知道存在 Dirty 操作，但无法直接知道是谁触发了 Dirty。

因此，在实际项目中，开发团队通常会针对这些高频问题开发专用调试工具。

常见工具包括：

- Canvas 重建监控器
- Graphic Dirty 监控器
- RaycastTarget 检查工具
- Batch 分析工具
- UI 层级分析工具

这些工具虽然实现简单，却能够显著提升 UI 性能问题的定位效率。

### 28.4.2 Canvas 重建监控器

在第 18 章中我们已经分析过 Canvas 更新流程。

每当一帧即将开始渲染时，Unity 会触发：

```csharp
Canvas.willRenderCanvases
```

事件。

UGUI 的布局重建、Graphic 重建以及 Batch 构建都会发生在这一阶段。

因此，我们可以利用这个事件统计 Canvas 的重建频率。

下面实现一个简单的监控器。

```csharp
using UnityEngine;

public class CanvasRebuildMonitor : MonoBehaviour
{
    private int rebuildCount;
    private float timer;

    private void OnEnable()
    {
        Canvas.willRenderCanvases += OnWillRenderCanvases;
    }

    private void OnDisable()
    {
        Canvas.willRenderCanvases -= OnWillRenderCanvases;
    }

    private void OnWillRenderCanvases()
    {
        rebuildCount++;
    }

    private void Update()
    {
        timer += Time.deltaTime;

        if (timer >= 1f)
        {
            Debug.Log($"Canvas Rebuild Count : {rebuildCount}");

            rebuildCount = 0;
            timer = 0f;
        }
    }
}
```

将脚本挂到场景中的任意对象上即可运行。

正常情况下，静态界面的重建次数应当比较稳定。

如果发现数字持续增长，则说明某些 UI 正在频繁触发重建，需要进一步分析布局系统或者 Graphic 系统。

该工具虽然简单，但能够快速判断项目是否存在异常刷新问题。

### 28.4.3 Graphic Dirty 监控器

在第 15 章中我们分析过 Graphic 的 Dirty 机制。

当 Graphic 被修改时，会调用以下函数：

```csharp
SetVerticesDirty();

SetMaterialDirty();

SetLayoutDirty();
```

随后在 Canvas 更新阶段重新生成数据。

如果某个 UI 每帧都调用这些函数，就会不断触发重建。

为了找到这些频繁刷新的对象，可以继承 Image 并监控 Dirty 调用。

```csharp
using UnityEngine;
using UnityEngine.UI;

public class DebugImage : Image
{
    public override void SetVerticesDirty()
    {
        Debug.Log($"{name} SetVerticesDirty");

        base.SetVerticesDirty();
    }

    public override void SetMaterialDirty()
    {
        Debug.Log($"{name} SetMaterialDirty");

        base.SetMaterialDirty();
    }

    public override void SetLayoutDirty()
    {
        Debug.Log($"{name} SetLayoutDirty");

        base.SetLayoutDirty();
    }
}
```

运行后，当对象发生变化时即可在控制台看到日志。

例如：

```csharp
HPBar SetVerticesDirty
HPBar SetVerticesDirty
HPBar SetVerticesDirty
HPBar SetVerticesDirty
```

如果日志每帧都在打印，则说明该控件正在持续刷新。

在实际项目中，经常能够通过这种方式发现：

- 每帧更新文本
- 每帧修改颜色
- 每帧修改尺寸
- 无意义的属性赋值

等问题。

对于复杂界面而言，这是定位 Graphic Rebuild 来源最直接的方法之一。

### 28.4.4 RaycastTarget 检查工具

RaycastTarget 是 UGUI 中最容易被忽视的性能问题之一。

很多美术制作界面时会默认保留：

```csharp
raycastTarget = true;
```

实际上只有需要接收点击事件的控件才需要开启射线检测。

大量无意义的 RaycastTarget 会增加 GraphicRaycaster 的检测开销。

因此很多项目都会开发专门的检查工具。

下面给出一个简单实现。

```csharp
using UnityEngine;
using UnityEngine.UI;

public class RaycastTargetChecker : MonoBehaviour
{
    [ContextMenu("Check RaycastTarget")]
    private void Check()
    {
        Graphic[] graphics = FindObjectsOfType<Graphic>(true);

        foreach (Graphic graphic in graphics)
        {
            if (!graphic.raycastTarget)
                continue;

            if (graphic.GetComponent<Button>())
                continue;

            Debug.LogWarning(
                $"Unused RaycastTarget : {graphic.name}",
                graphic.gameObject);
        }
    }
}
```

执行后会扫描场景中的所有 Graphic。

对于开启了 RaycastTarget，但又不包含 Button 的对象，会输出警告信息。

例如：

```csharp
Unused RaycastTarget : Background
Unused RaycastTarget : Frame
Unused RaycastTarget : Decoration
```

开发人员可以根据输出结果逐个检查并关闭无意义的射线检测。

在包含数千个 UI 元素的大型界面中，这种优化往往能够显著降低点击检测开销。

### 28.4.5 UI Batch 分析工具

Batch 问题通常是 UI 优化中的重点。

然而 Unity 并没有提供专门的 Batch 风险检测工具。

因此很多项目会实现简单的统计工具，对界面中的资源进行分析。

例如统计：

- Canvas 数量
- Material 数量
- Texture 数量
- Mask 数量

如果同一个界面中存在大量材质和纹理，则往往意味着合批效果较差。

简单示例：

```csharp
Graphic[] graphics = FindObjectsOfType<Graphic>(true);

HashSet<Material> materials = new HashSet<Material>();

foreach (Graphic graphic in graphics)
{
    materials.Add(graphic.material);
}

Debug.Log($"Material Count : {materials.Count}");
```

通过这些统计数据，可以在开发阶段提前发现潜在的 Batch 问题，而不必等到进入 Frame Debugger 后再逐步分析。

### 28.4.6 小结

Unity 提供的官方工具能够帮助我们分析 UGUI 的运行结果，而自定义工具则能够帮助我们快速定位问题来源。

在实际项目中，两者往往配合使用。

首先利用 Profiler、Frame Debugger 或 RenderDoc 发现性能异常，再通过自定义工具定位具体对象和触发原因。

对于大型 UI 项目而言，一套完善的调试工具往往比单纯的性能优化经验更加重要。合理利用 UGUI 源码中的关键节点构建调试系统，能够显著提高问题定位效率，并降低后期维护成本。
