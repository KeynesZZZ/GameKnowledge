在实际项目中，UI 很少只停留在“能显示”的阶段。随着产品质量要求不断提升，大量视觉特效会逐渐加入到界面系统中，例如描边、阴影、渐变、发光、模糊、遮罩、粒子、动态材质、软遮挡以及各种动画效果。这些特效能够显著提升界面的视觉表现力，但与此同时，也会给 [UGUI](https://zhida.zhihu.com/search?content_id=275964760&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 渲染体系带来额外负担。

很多开发者在项目初期往往只关注功能实现，而忽略了 UI 特效背后的渲染成本。当项目进入中后期时，便会逐渐出现如下问题：

- [DrawCall](https://zhida.zhihu.com/search?content_id=275964760&content_type=Article&match_order=1&q=DrawCall&zhida_source=entity) 数量急剧增加
- Canvas 重建频率过高
- UI 渲染耗时上升
- GPU FillRate 压力暴涨
- 手机上出现严重掉帧
- 界面滑动时产生卡顿
- Overdraw 过高导致发热

这些问题的根源，往往并不在复杂逻辑本身，而是隐藏在一个个看似普通的 UI 特效之中。

UGUI 本质上仍然是 Mesh 渲染系统。所有 Image、Text、RawImage、Mask 等组件，最终都会被转换为顶点数据并提交给 CanvasRenderer，再由 Canvas 统一参与批处理与渲染。因此，任何会改变顶点、材质、Stencil、裁剪状态或者渲染顺序的 UI 特效，都有可能破坏原有批处理结构。

很多时候，一个简单的 Outline 组件，就可能让原本一次提交完成的界面变成数十次 DrawCall；一个嵌套过深的 Mask，可能直接导致 GPU FillRate 翻倍；一个看似普通的发光特效，也可能因为材质实例化而彻底打断动态合批。

更重要的是，UI 性能问题通常具有极强的“隐蔽性”。

3D 场景中的性能问题往往容易观察，例如模型过多、阴影过重、粒子数量巨大等；而 UI 性能问题则常常隐藏在复杂层级、透明叠加以及特效组件内部。尤其是在移动平台上，UI 往往会成为整帧渲染中最容易被忽视，却又最致命的性能瓶颈。

本章将从性能角度重新审视 UI 特效的实现方式，重点分析特效对批处理、材质系统、Mask 机制以及顶点结构的影响，并结合 Overdraw 等核心问题，建立一套完整的 UI 特效性能分析方法。在此基础上，总结出一系列可落地的优化策略，为后续大规模 UI 系统的性能控制提供理论支撑。

### 19.1 对批处理的影响

在 UGUI 渲染体系中，批处理是影响 UI 性能最核心的机制之一。所谓批处理，本质上就是将多个 UI 元素合并为一次 GPU 提交，从而减少 DrawCall 数量，降低 CPU 与 GPU 之间的通信开销。

UGUI 的所有 Graphic 组件最终都会生成 Mesh 数据，例如 Image 会生成矩形网格，Text 会生成字符网格，而各种 UI 特效则会进一步修改这些顶点数据。Canvas 在渲染阶段会遍历所有 UI 元素，并尝试将满足条件的对象合并到同一个 Batch 中统一提交给 GPU。

如果多个 UI 元素使用相同材质、相同纹理、相同 Shader，并且渲染状态完全一致，那么 Unity 就能够将它们合并为一次 DrawCall。这也是 UGUI 能够在复杂界面下依然维持较高性能的重要原因。

很多开发者认为只要使用同一个图集就一定能够合批，但实际上 UGUI 的 Batch 合并条件远比想象中严格。只要渲染状态出现变化，当前 Batch 就会被立即中断，Unity 必须重新开启新的 DrawCall。

最常见的 Batch 中断因素包括：

- Material 不同
- Texture 不同
- Shader 不同
- Stencil 状态不同
- RectMask2D 裁剪状态不同
- Canvas 不同
- 渲染顺序发生变化
- Additional Shader Channels 不一致

因此，UGUI 批处理的本质并不是“多个对象一起渲染”，而是“渲染状态连续的顶点合并”。

而 UI 特效最容易破坏的，恰恰就是这种渲染状态连续性。

### 19.1.1 UI 特效为什么容易破坏批处理

大多数 UI 特效都会修改 Mesh、材质、Shader 参数或者 Stencil 状态，这些变化都会直接影响 Batch 合并。

例如 Outline 和 Shadow 会额外复制顶点，从而增加 Mesh 数据量；Mask 会修改 Stencil 缓冲状态；某些发光、渐变或者模糊特效则会动态修改材质参数。这些行为都会导致原本连续的渲染状态发生变化。

以 Mask 为例，UGUI 的 Mask 系统底层依赖 Stencil Buffer 实现裁剪，而 Stencil 状态属于 GPU 渲染状态的一部分。当两个 UI 元素使用不同的 Stencil 配置时，它们绝对无法进入同一个 Batch。

再例如动态修改材质参数：

```
image.material.SetFloat("_Value", value);
```

很多开发者以为只是修改了一个参数，但实际上 Unity 往往会实例化新的 Material。原本所有 UI 共用的共享材质会变成多个独立材质实例，最终直接导致 DrawCall 激增。

这也是为什么很多项目中的 UI 特效看似简单，却会产生严重性能问题。

### 19.1.2 渲染顺序对 Batch 的影响

UGUI 是严格顺序渲染系统，Hierarchy 中的节点顺序会直接决定最终渲染顺序。

假设有三个 UI 元素：

```
A
B
C
```

其中 A 与 C 使用相同材质，而 B 使用不同材质，那么最终 Batch 很可能会变成：

```
Batch 0 -> A
Batch 1 -> B
Batch 2 -> C
```

即使 A 与 C 使用完全相同的材质，它们也无法重新合并，因为 UGUI 不允许为了合批而打乱渲染顺序。

这与 3D Renderer 的动态合批机制存在明显区别。3D 渲染可以在一定程度上调整渲染顺序，而 UGUI 为了保证界面层级正确，必须严格维持 Hierarchy 顺序。

因此，在 UI 系统中，Hierarchy 排列本身也会直接影响最终 DrawCall 数量。

### 19.1.3 Canvas 对批处理的影响

Canvas 是 UGUI 批处理的天然边界，不同 Canvas 之间永远无法合批。

即使两个 UI 使用完全相同的材质、纹理与 Shader，只要它们位于不同 Canvas 下，Unity 也会分别生成独立 Mesh，并产生独立 DrawCall。

因此，不合理拆分 Canvas 往往会导致 DrawCall 数量迅速增加。

但另一方面，如果整个界面只使用一个超大 Canvas，又会导致 Canvas 重建成本过高。因为任何一个子节点发生变化，都可能触发整个 Canvas 重新生成 Mesh。

这也是 UGUI 性能优化中最经典的矛盾之一：Canvas 过多会增加 DrawCall，Canvas 过大会增加重建开销。

因此，合理拆分 Canvas 是 UI 优化中的核心工作。

### 19.1.4 隐式 Batch 中断问题

很多 Batch 中断并不是显式发生的，而是隐藏在一些看似普通的操作中。

例如修改 material.color、开启不同 Shader Keyword、切换 AlphaClip、使用不同 Additional Shader Channels，都会导致最终渲染状态不一致。

尤其是在大型项目中，很多 UI 特效插件会在内部自动修改材质参数，而开发者往往并未意识到这些行为已经破坏了批处理。

因此，UI DrawCall 问题通常具有极强隐蔽性。

很多时候界面看起来只有少量 Image，但 Frame Debugger 中却已经出现数十甚至上百个 DrawCall。

### 19.1.5 Frame Debugger 中的 Batch 分析

Unity Frame Debugger 是分析 UGUI Batch 问题最重要的工具。

在：

```
Window -> Analysis -> Frame Debugger
```

中开启调试后，可以查看每一个 UI DrawCall 的具体信息，并且能够直接看到 Batch 被打断的原因。

例如：

```
Different Material
Different Texture
Different Stencil State
Different Clipping Rect
```

这些信息能够帮助开发者快速定位性能问题来源。

在实际项目中，大多数 UI 性能优化工作，本质上都是在减少 Batch 中断。

### 19.1.6 小结

UGUI 的批处理本质上是渲染状态连续的 Mesh 合并，而不是简单地将多个 UI 一起绘制。任何会改变材质、纹理、Shader、Stencil、裁剪状态或者渲染顺序的行为，都有可能打断 Batch。

而 UI 特效由于天然会修改渲染状态，因此往往是 DrawCall 激增的主要来源。

理解批处理的底层机制后，才能真正理解为什么某些 UI 特效虽然视觉效果很简单，却依然会对性能造成巨大影响。

### 19.2 材质变化

在 UGUI 渲染体系中，材质变化是导致 DrawCall 增加最常见的问题之一。很多开发者在优化 UI 时会优先关注图集、层级和 Mask，但真正导致大量 Batch 断裂的核心原因，往往是材质状态发生了变化。

对于 GPU 而言，Material 并不仅仅只是一个资源对象，它本质上代表一整套渲染状态集合，其中包括 Shader、纹理、Blend 状态、Stencil 配置、Shader Keyword 以及各种 Uniform 参数。只要这些状态中任意一项发生变化，GPU 就必须重新切换渲染状态，而 Unity 也必须中断当前 Batch，重新生成新的 DrawCall。

因此，材质变化本质上就是渲染状态变化。

### 19.2.1 为什么 UGUI 对材质变化极度敏感

UGUI 的核心优化机制是批处理，而批处理要求参与合并的 UI 元素必须拥有完全一致的渲染状态。

假设存在三个 Image：

```
Image A
Image B
Image C
```

如果它们使用相同材质、相同纹理、相同 Shader，并且 Shader Keyword 与 Stencil 状态完全一致，那么 Unity 就能够将它们合并为一次 DrawCall。

但如果其中一个对象切换了材质：

```
Image B -> New Material
```

那么原有 Batch 会立即断裂：

```
Batch 0 -> A
Batch 1 -> B
Batch 2 -> C
```

即使新的材质只是修改了一个颜色参数，也无法继续参与原有 Batch。

因为 GPU 无法在同一次 DrawCall 中同时使用多个不同材质状态。

### 19.2.2 material 与 sharedMaterial 的区别

很多 UI 性能问题都来源于错误使用 material 属性。

在 Unity 中：

```
graphic.material
```

与：

```
graphic.sharedMaterial
```

有本质区别。

其中：

```
graphic.material
```

通常会在运行时创建新的材质实例。

例如：

```
image.material.color = Color.red;
```

很多开发者认为这里只是简单修改颜色，但 Unity 内部往往已经执行了材质实例化。

最终结果会从：

```
Shared Material
```

变成：

```
Material Instance A
Material Instance B
Material Instance C
```

原本能够共享的材质彻底被拆散，最终导致 DrawCall 急剧增加。

因此，在 UI 系统中，频繁访问 material 属性是非常危险的行为。

### 19.2.3 UI 特效为什么容易产生材质实例

大量 UI 特效都会隐式创建新的 Material。

例如：

- Outline
- Shadow
- SoftMask
- Glow
- Blur
- Dissolve
- UIEffect

这些特效通常需要动态修改 Shader 参数、切换 Keyword、设置额外纹理或者修改 Stencil 状态，因此往往会在内部创建新的材质实例。

尤其是很多第三方 UI 特效插件，内部会频繁执行：

```
new Material(baseMaterial)
```

开发者虽然只是给组件勾选了一个特效，但底层实际上已经生成了大量独立 Material。

最终不仅会导致 DrawCall 增加，还会带来额外内存占用与 GC 压力。

### 19.2.4 Shader Keyword 对批处理的影响

即使两个 UI 使用的是同一个 Shader，只要 Shader Keyword 不同，也无法继续合批。

例如：

```
UNITY_UI_CLIP_RECT
UNITY_UI_ALPHACLIP
```

这些 Keyword 会生成不同 Shader Variant。

假设：

```
Image A -> ALPHACLIP ON
Image B -> ALPHACLIP OFF
```

虽然它们使用的是同一个 Shader 文件，但 GPU 实际加载的已经是两个不同 Variant，因此 Unity 会认为它们属于不同材质状态。

最终 Batch 必然中断。

很多 UI 特效插件为了实现裁剪、发光、模糊等效果，会动态切换 Keyword，这也是 UI DrawCall 激增的重要原因之一。

### 19.2.5 Mask 导致的材质变化

UGUI 中最特殊的一类材质变化来源，就是 Mask。

Mask 底层依赖 Stencil Buffer 实现裁剪，而不同层级的 Mask 会生成不同的 Stencil 参数。

例如：

```
Stencil Ref = 1
Stencil Ref = 2
Stencil Ref = 3
```

虽然所有对象使用的是同一个 Shader，但由于 Stencil 状态已经不同，Unity 依然必须创建新的 Material。

因此，复杂 Mask 嵌套会直接导致材质数量增加。

很多 ScrollView 在层级较深时会出现 DrawCall 暴涨，本质原因就是 Stencil 状态不断变化。

### 19.2.6 动态修改材质参数的问题

很多开发者喜欢通过代码动态控制 UI 特效：

```
material.SetFloat("_Glow", value);
```

或者：

```
material.SetColor("_Color", color);
```

这种做法在 3D Renderer 中问题可能不明显，但在 UGUI 中却非常危险。

因为 UI 的核心依赖共享材质进行批处理，一旦参数变化导致材质实例化，那么原有 Batch 会立刻被拆散。

尤其是在商城列表、背包系统、排行榜等高密度 UI 场景中，如果每个元素都拥有独立材质，那么 DrawCall 数量会迅速失控。

### 19.2.7 UI 材质优化策略

UI 材质优化的核心目标，是尽可能维持渲染状态统一。

首先，应尽量共享材质，不要随意创建新的 Material 实例，更不要在 Update 中频繁访问 material 属性。

其次，应减少 Shader Variant 数量，避免频繁切换 Keyword，也不要随意开启 AlphaClip。

再次，应减少运行时 Shader 特效。很多简单效果完全可以提前烘焙到贴图中，而不是依赖实时 Shader 运算。

另外，还需要控制 Mask 层级深度，因为 Stencil 层级越复杂，材质实例数量也会越多。

最后，项目中应尽量统一 UI Shader，避免混用大量不同 UI Shader 导致 Batch 难以合并。

### 19.2.8 Frame Debugger 中的材质分析

Frame Debugger 是分析 UI 材质问题最重要的工具之一。

在：

```
Window -> Analysis -> Frame Debugger
```

中开启调试后，可以直接查看每一个 DrawCall 的 Batch Breaking Reason。

例如：

```
Different Material
Different Shader Keywords
Different Stencil State
```

这些信息能够帮助开发者快速定位到底是哪一种材质状态导致 Batch 中断。

在大型项目中，大部分 UI DrawCall 优化工作，本质上都是在减少材质变化。

### 19.2.9 小结

在 UGUI 中，材质变化本质上是 GPU 渲染状态变化，而任何渲染状态变化都会直接导致 Batch 中断。

很多 UI 特效虽然看起来非常简单，但底层往往已经创建了大量 Material 实例或者 Shader Variant。

因此，UI 性能优化最核心的原则之一，就是尽可能保持材质状态统一，从而维持批处理连续性。

### 19.3 Mask 影响

在 UGUI 所有组件中，Mask 是对渲染性能影响最大的功能之一。很多项目中的 UI 性能问题，看起来像是层级复杂、图片太多或者界面元素数量过大，但真正导致 DrawCall 激增与 GPU 压力上升的根源，往往是 Mask。

因为 Mask 并不仅仅只是一个简单的裁剪功能，它底层实际上会直接修改 GPU 的 Stencil 状态，同时还会影响批处理、材质生成以及 Fragment 渲染流程。

尤其是在 ScrollView、背包系统、聊天窗口、排行榜等高频列表界面中，Mask 往往会成为整个 UI 系统最大的性能瓶颈。

### 19.3.1 Mask 的本质

UGUI 中的 Mask 本质上是基于 GPU 的 Stencil Buffer 实现的。

Stencil Buffer 可以理解为一张额外的标记缓冲区，GPU 会在渲染过程中为像素记录对应的标记值，而 Mask 会利用这些标记值决定哪些像素允许显示，哪些像素需要被裁剪。

当 Mask 开始渲染时，Unity 会先向 Stencil Buffer 写入标记值，然后子节点在渲染时会额外执行一次 Stencil 测试，只有通过测试的像素才允许真正输出到屏幕。

因此，一个普通 Mask 的底层流程实际上包含两部分：

1. 先绘制 Mask 区域并写入 Stencil
2. 再对子节点执行 Stencil Test

这意味着 Mask 本身就会增加额外渲染步骤。

### 19.3.2 Mask 为什么会增加 DrawCall

Stencil 状态属于 GPU 渲染状态的一部分，而 UGUI 的批处理要求所有对象必须拥有完全一致的渲染状态。

只要 Stencil 参数发生变化，当前 Batch 就会立刻中断。

例如：

```
Image A -> 无 Mask
Image B -> Stencil Ref 1
Image C -> 无 Mask
```

最终 DrawCall 很可能变成：

```
Batch 0 -> A
Batch 1 -> B
Batch 2 -> C
```

即使三个对象使用同一个图集、同一个 Shader，也无法继续合批。

因为 GPU 无法在同一次 DrawCall 中同时混用不同 Stencil 状态。

这也是为什么一个看起来非常普通的 Mask，经常会直接打断整段 UI Batch。

### 19.3.3 Mask 嵌套的问题

真正危险的并不是单个 Mask，而是多层嵌套。

例如：

```
Mask A
    Mask B
        Mask C
```

Unity 会不断生成新的 Stencil 层级：

```
Mask A -> Ref 1
Mask B -> Ref 2
Mask C -> Ref 3
```

每增加一层 Mask：

- Stencil 状态都会变化
- 新 Material 会被生成
- Batch 会再次中断

因此 Mask 层级越深，DrawCall 往往越高。

很多复杂 ScrollView 的性能问题，本质上都是由于多层 Mask 嵌套导致。

尤其是：

- 聊天列表
- 无限循环列表
- 多层分页
- 嵌套 ScrollView
- 多层弹窗系统

这些 UI 结构最容易形成复杂 Stencil 链。

### 19.3.4 Mask 对材质的影响

很多开发者认为 Mask 只是修改裁剪区域，但实际上 Mask 会直接生成新的 Material。

UGUI 内部会通过：

```
StencilMaterial.Add()
```

动态创建带有 Stencil 参数的新材质。

例如原始材质：

```
UI/Default
```

经过第一层 Mask 后可能变成：

```
UI/Default + Stencil Ref 1
```

继续进入下一层：

```
UI/Default + Stencil Ref 2
```

虽然 Shader 完全相同，但由于 Stencil 参数已经不同，因此这些材质无法共享。

最终结果就是：

- Material 数量增加
- DrawCall 提升
- 内存占用上升
- Batch 大量断裂

这也是为什么复杂 UI 往往会产生大量 Material Instance。

### 19.3.5 RectMask2D 与 Mask 的区别

UGUI 中存在两套裁剪系统：

- Mask
- RectMask2D

很多开发者会误以为它们只是接口不同，但实际上底层实现完全不同。

普通 Mask 使用的是 Stencil Buffer，而 RectMask2D 使用的是矩形裁剪。

RectMask2D 会直接向 Shader 传递裁剪矩形：

```
_ClipRect
```

随后 Shader 在 Fragment 阶段判断像素是否位于矩形范围内。

由于 RectMask2D 不依赖 Stencil，因此它相比普通 Mask：

- 不需要额外 Stencil Buffer
- 不会产生复杂 Stencil 层级
- 材质变化更少
- DrawCall 更稳定

因此在实际项目中，如果只是普通矩形区域裁剪，应优先使用 RectMask2D。

但 RectMask2D 也存在局限，它只能处理矩形区域，无法支持圆形或者复杂不规则遮罩。

### 19.3.6 Mask 对 Overdraw 的影响

很多开发者认为 Mask 能够减少渲染区域，因此会降低 GPU 开销，但实际上情况并非如此。

因为 Mask 的大部分裁剪发生在 Fragment 阶段，而 GPU 在执行 Fragment 裁剪之前，往往已经完成：

- 顶点处理
- 三角形光栅化
- Fragment 生成

也就是说，很多最终被 Mask 隐藏的像素，其实已经参与了渲染计算。

因此 Mask 并不会真正减少 FillRate，反而可能因为额外 Stencil 测试进一步增加 GPU 压力。

尤其在移动平台上，大量透明 UI 配合多层 Mask，很容易产生严重 Overdraw。

### 19.3.7 ScrollView 为什么容易性能差

ScrollView 是 UGUI 中最典型的性能重灾区。

因为 ScrollView 通常同时具备：

- Mask
- 大量子节点
- 动态布局
- 高频滚动
- Canvas 重建
- 顶点更新

默认 ScrollView 通常会自动携带：

```
Viewport + Mask
```

当列表元素数量增加时：

- DrawCall 会持续增加
- Canvas 会频繁重建
- GPU FillRate 会迅速上升

如果再叠加 Outline、Shadow、渐变、发光等 UI 特效，性能问题会更加严重。

因此 ScrollView 本质上是 UI 性能问题最集中爆发的区域。

### 19.3.8 Mask 优化策略

在实际项目中，Mask 优化通常遵循几个核心原则。

1. 应尽量减少 Mask 数量，很多界面实际上并不需要真正裁剪，只是习惯性挂载了 Mask。
2. 应优先使用 RectMask2D，因为矩形裁剪远比普通 Mask 性能更高。
3. 应避免多层 Mask 嵌套，因为 Stencil 层级越深，材质与 DrawCall 数量越高。
4. 还应减少透明 UI 叠加，因为透明区域本身就会提高 Fragment 开销，而 Mask 会进一步放大这种问题。
5. 对于 ScrollView，则必须使用节点复用机制，避免大量不可见元素长期参与渲染。
6. 还应尽量缩小 Mask 区域面积，因为裁剪区域越大，GPU FillRate 压力越高。

### 19.3.9 Frame Debugger 中观察 Mask

在 Frame Debugger 中，可以非常直观看到 Mask 对渲染流程的影响。

例如：

```
Clear Stencil
Set Stencil
Draw Mask
Draw Child
```

还可以直接看到：

```
Batch Breaking Reason:
Different Stencil State
```

这些信息能够清晰反映：“Mask 是如何破坏批处理的。”

很多 UI DrawCall 问题，在 Frame Debugger 中都会表现为大量 Stencil 状态切换。

### 19.3.10 小结

Mask 本质上是基于 GPU Stencil Buffer 实现的裁剪系统，而 Stencil 状态变化会直接导致 Batch 中断。

Mask 不仅会增加 DrawCall，还会生成新的 Material，并提高 GPU Fragment 阶段开销。

尤其是在多层嵌套 ScrollView 与复杂透明 UI 场景中，Mask 往往会成为整个 UGUI 系统最大的性能瓶颈之一。

因此，在实际项目中必须谨慎使用 Mask，并尽可能减少复杂 Stencil 层级。

### 19.4 顶点膨胀

在 UGUI 性能问题中，很多开发者会重点关注 DrawCall、材质以及 Mask，但却忽略了另一个非常重要的问题，那就是顶点膨胀。

所谓顶点膨胀，本质上是指 UI 特效导致 Mesh 顶点数量急剧增加。很多 UI 特效虽然不会明显增加 DrawCall，但却会大幅提高顶点数量，从而增加 CPU Mesh 构建、Canvas 重建以及 GPU 顶点处理压力。

尤其是在移动平台上，顶点膨胀往往会成为 UI 卡顿的重要来源。

### 19.4.1 UGUI 本质上是 Mesh 渲染系统

很多开发者容易把 UGUI 理解为普通二维界面系统，但实际上 UGUI 本质上仍然是 Mesh 渲染系统。

所有 UI 元素最终都会转换为顶点数据。

例如一个最普通的 Image，本质上只是一个矩形 Mesh：

```
4 个顶点
2 个三角形
```

Text 同样如此，每一个字符都会生成独立四边形：

```
4 个顶点
2 个三角形
```

因此，UGUI 的整个界面系统，本质上就是大量 Mesh 的组合。

而 UI 特效最大的影响之一，就是会直接修改这些 Mesh 数据。

### 19.4.2 顶点膨胀的来源

UGUI 中最常见的顶点膨胀来源包括：

- Outline
- Shadow
- Text
- 渐变特效
- 发光特效
- UIEffect
- 曲面文字
- 自定义 MeshEffect

这些特效通常都会在：

```
ModifyMesh()
```

阶段修改原始顶点数据。

例如 Outline：

```
public override void ModifyMesh(VertexHelper vh)
{
    ApplyShadowZeroAlloc(...);
}
```

它会复制原始顶点，并生成额外偏移顶点。

原本一个普通 Image：

```
4 个顶点
```

开启 Outline 后会直接变成：

```
20 个顶点
```

因为 Unity 会为上下左右四个方向分别复制一次顶点。

Shadow 同样也会额外复制顶点，因此很多 UI 特效本质上其实是在不断复制 Mesh。

### 19.4.3 Text 为什么容易产生大量顶点

Text 是 UGUI 中最容易产生顶点膨胀的组件之一。

因为每一个字符都会生成独立 Mesh。

例如：

```
100 个字符
```

通常意味着：

```
400 个顶点
200 个三角形
```

如果再叠加 Outline：

```
400 × 5 = 2000 顶点
```

再叠加 Shadow：

```
2000 × 2 = 4000 顶点
```

也就是说，一个看起来非常普通的文本，很可能已经拥有数千顶点。

尤其是：

- 聊天系统
- 公告界面
- 排行榜
- 富文本
- 多语言文本

这些系统通常都会产生大量字符，因此 Text 往往是 UI 顶点数量最高的组件之一。

### 19.4.4 顶点膨胀对 CPU 的影响

很多开发者认为顶点多只是 GPU 压力，但实际上 UGUI 中大量顶点问题首先影响的是 CPU。

因为 UGUI 的 Mesh 构建完全发生在 CPU 侧。

例如：

```
OnPopulateMesh()
ModifyMesh()
VertexHelper
Canvas BuildBatch
```

这些流程全部运行在 CPU。

当顶点数量暴涨时：

- VertexHelper 操作成本增加
- List 扩容频繁发生
- Canvas BuildBatch 时间增加
- Mesh Upload 数据量增加

尤其是在 Canvas 频繁重建时，大量顶点会显著提高 CPU 耗时。

因此顶点膨胀不仅仅是 GPU 问题，更是 CPU 问题。

### 19.4.5 顶点膨胀对 GPU 的影响

除了 CPU 之外，GPU 同样需要处理大量顶点。

GPU 在渲染 UI 时，需要执行：

- Vertex Shader
- 三角形装配
- 光栅化
- Fragment 处理

虽然 UI Shader 通常比较简单，但当顶点数量巨大时，Vertex 阶段依然会产生明显压力。

尤其在：

- 大量动态文本
- 特效文字
- 长列表
- 聊天窗口

等场景中，GPU 顶点处理量会迅速增加。

在低端移动设备上，这种问题尤其明显。

### 19.4.6 Canvas 重建为什么害怕顶点膨胀

UGUI 的 Canvas 重建本质上会重新生成整个 Mesh。

例如：

```
SetVerticesDirty()
```

会触发：

```
Canvas Rebuild
```

一旦 Canvas 中存在大量顶点：

- Mesh 重建时间会增加
- 顶点拷贝会增加
- Native Upload 会增加

因此顶点越多，Canvas 重建成本越高。

很多大型 UI 卡顿问题，本质上并不是 DrawCall 太高，而是：

```
Canvas Rebuild + Massive Vertices
```

尤其在 ScrollView 滚动时：

- Layout 变化
- Text 更新
- 顶点重新生成

这些操作叠加后，CPU 开销会迅速上升。

### 19.4.7 Outline 为什么性能差

Outline 是最经典的顶点膨胀案例。

很多开发者认为 Outline 只是一个简单描边效果，但实际上它会：

- 复制顶点
- 生成额外偏移 Mesh
- 增加 FillRate
- 增加 Fragment 数量

普通 Image：

```
4 顶点
```

开启 Outline 后：

```
20 顶点
```

如果是 Text：

```
字符越多
顶点膨胀越严重
```

因此 Outline 是移动端 UI 最危险的特效之一。

很多项目中的 UI 卡顿，本质上都是大量 Text + Outline 导致。

### 19.4.8 自定义 MeshEffect 的风险

很多开发者喜欢继承：

```
BaseMeshEffect
```

实现各种 UI 特效。

例如：

- 波浪文字
- 弧形文字
- 渐变
- 发光
- UV 动画
- 扭曲特效

这些效果通常都会在：

```
ModifyMesh()
```

阶段操作顶点。

如果实现不当：

- 会频繁申请 List
- 会重复复制顶点
- 会产生 GC
- 会导致顶点指数增长

因此 MeshEffect 本质上属于 CPU Mesh 编程，它的性能成本远高于很多开发者想象。

### 19.4.9 顶点优化策略

在实际项目中，顶点优化通常遵循几个核心原则。

首先，应减少 Outline 与 Shadow 使用，尤其不要在大量 Text 上叠加。

其次，应减少动态 Text，因为频繁更新文字会持续触发 Mesh 重建。

再次，应避免复杂富文本，因为富文本会增加字符拆分与顶点生成成本。

另外，还需要控制 ScrollView 中节点数量，大量文本节点会迅速提高顶点规模。

很多描边与发光效果，其实完全可以提前烘焙到贴图中，而不是依赖实时 Shader 或 Mesh 特效。

对于自定义 MeshEffect，则应尽量减少 Update 中频繁修改 Mesh 的行为。

最后，还应合理拆分动态 Canvas，避免少量变化导致整个大 Mesh 重建。

### 19.4.10 Profiler 中观察顶点问题

在 Unity Profiler 中，可以重点观察：

```
Canvas.BuildBatch
Canvas.SendWillRenderCanvases
UI.Rendering
```

如果这些模块耗时较高，通常意味着：

- 顶点数量过大
- Canvas 重建频繁
- Mesh 更新过多

同时在 GameView Stats 中，还可以直接观察：

```
Verts
Tris
```

这些指标能够非常直观地反映 UI 顶点规模。

很多时候即使 DrawCall 不高，但 Vertices 极高，依然会导致明显卡顿。

### 19.4.11 小结

UGUI 本质上是 Mesh 渲染系统，而大量 UI 特效都会直接修改或者复制顶点数据。

顶点膨胀不仅会增加 GPU 顶点处理压力，还会显著提高 CPU Mesh 构建与 Canvas 重建成本。

尤其是在大量 Text、Outline、Shadow 与复杂 ScrollView 场景中，顶点数量往往会迅速失控。

因此，在 UI 性能优化中，除了关注 DrawCall，更必须关注顶点规模与 Mesh 重建成本。

### 19.5 优化策略

在前面的章节中，我们已经分析了 UI 特效对批处理、材质、Mask、顶点数量以及 GPU 渲染带来的影响。很多开发者只有在项目后期出现严重卡顿时，才开始处理 UI 性能问题，但真正有效的优化，往往应该从 UI 架构设计阶段就开始考虑。

因为 UGUI 的性能问题通常并不是某一个组件单独导致的，而是多个系统共同叠加后的结果。例如 Batch 被频繁打断、Material 不断实例化、Mask 层级过深、顶点数量膨胀、Canvas 高频重建以及 Overdraw 过高，这些问题通常都会同时出现在复杂 UI 中。

因此，UI 性能优化本质上是一套系统性工程。

### 19.5.1 优先减少 Canvas 重建

在大多数项目中，UI 卡顿最常见的问题并不是 GPU，而是 Canvas 重建。

因为 UGUI 的核心渲染流程本质上是重新生成 Mesh、重新构建 Batch，并重新上传 GPU 数据，而这些操作全部发生在 CPU 侧。

因此，减少 Canvas Rebuild 永远是 UGUI 优化中的第一优先级。

很多开发者喜欢把整个界面全部放进同一个 Canvas，但这样会导致任意一个节点发生变化时，整个 Canvas 都必须重新生成 Mesh。

例如：

- Text 内容变化
- 血条刷新
- 数字更新
- 动画播放
- Layout 变化

这些操作都会触发 Canvas 重建。

因此，在实际项目中，应当按照“更新频率”拆分 Canvas。

例如：

```
静态背景 Canvas
动态信息 Canvas
高频刷新 Canvas
弹窗 Canvas
```

这样即使某个区域频繁更新，也不会影响整个 UI。

### 19.5.2 减少 DrawCall 数量

DrawCall 优化的核心，本质上是维持批处理连续性。

而 Batch 最容易被以下因素打断：

- Material 不同
- Texture 不同
- Shader 不同
- Stencil 状态不同
- 渲染顺序变化

因此，UI DrawCall 优化通常围绕这些问题展开。

首先，应尽量统一图集，避免 UI 使用大量零散 Texture，因为纹理切换会频繁打断 Batch。

其次，应统一材质与 Shader。项目中不要混用大量 UI Shader，因为即使只是 Shader 不同，也已经无法继续合批。

再次，应减少运行时材质实例化，尤其不要频繁访问：

```
graphic.material
```

因为它极容易生成新的 Material Instance。

最后，应减少复杂 Mask 嵌套，因为 Stencil 状态变化会直接打断 Batch。

### 19.5.3 控制 Mask 使用

Mask 是 UGUI 中最危险的性能组件之一。

因为它不仅会增加 DrawCall，还会生成新的 Material，并提高 GPU Fragment 阶段开销。

因此，能不用 Mask，就尽量不用。

很多界面实际上并不需要真正裁剪，只是开发过程中习惯性挂载了 Mask。

如果只是普通矩形区域裁剪，应优先使用：

```
RectMask2D
```

因为 RectMask2D 不依赖 Stencil Buffer，因此性能远优于普通 Mask。

同时还应避免多层 Mask 嵌套，因为 Stencil 层级越深，DrawCall 往往越高。

对于 ScrollView，则必须配合节点复用机制，否则大量不可见节点会长期参与渲染。

### 19.5.4 控制顶点数量

很多 UI 性能问题并不是 DrawCall 太高，而是顶点数量已经失控。

尤其是：

- Outline
- Shadow
- Text
- 富文本
- MeshEffect

这些组件都容易导致顶点膨胀。

例如一个普通 Text：100 字符 ≈ 400 顶点

如果再叠加 Outline 与 Shadow，顶点数量会迅速翻倍。

因此，移动端应尽量减少 Text 特效。

很多描边与发光效果，其实完全可以提前烘焙到贴图中，而不是运行时动态生成。

同时，还应减少动态 Text 更新，因为每次文字变化都会重新生成 Mesh。

对于 ScrollView，则必须严格控制节点数量，因为长列表中的大量 Text 往往会形成海量顶点。

### 19.5.5 避免 Layout 系统滥用

LayoutGroup 与 ContentSizeFitter 虽然使用方便，但性能成本并不低。

因为 Layout 系统会在：

```
Layout Rebuild
```

阶段递归遍历整个层级。

例如：

- HorizontalLayoutGroup
- VerticalLayoutGroup
- GridLayoutGroup
- ContentSizeFitter

这些组件都会增加 CPU 布局计算压力。

尤其在动态列表中：

- 节点增删
- Text 变化
- 尺寸变化

都会导致整个 Layout 重新计算。

因此，LayoutGroup 更适合编辑器开发效率，而不适合超高频动态界面。

在大型项目中，很多高性能列表都会手动控制坐标，而不是依赖 LayoutGroup。

### 19.5.6 减少透明区域叠加

UGUI 默认使用透明混合渲染。

透明渲染无法利用 Early-Z，因此 GPU 必须逐像素执行 Fragment Shader。

如果多个透明 UI 相互覆盖：

```
UI A
UI B
UI C
```

那么同一个像素可能会被重复渲染多次。

这就是：

```
Overdraw
```

很多项目中的 GPU 压力，其实并不是来自复杂 Shader，而是来自大量透明区域叠加。

因此，应尽量减少：

- 全屏透明 Image
- 半透明背景层层覆盖
- 大型发光特效
- 大面积 Alpha 渐变

这些都能显著降低 GPU FillRate 压力。

### 19.5.7 ScrollView 优化

ScrollView 是 UGUI 中最容易出现性能问题的区域。

因为它通常同时包含：

- Mask
- 大量节点
- 动态布局
- Text
- 高频滚动
- Canvas 重建

因此 ScrollView 优化往往决定整个 UI 性能上限。

ScrollView 优化的核心原则包括：

第一，必须使用节点复用，不要创建几百上千个 Item 长期存在。

第二，避免嵌套 LayoutGroup，因为滚动过程中会频繁触发布局计算。

第三，减少 Item 中的 Text 与特效，尤其不要在列表里大量使用 Outline。

第四，拆分动态区域，避免滚动导致整个 Canvas 重建。

第五，减少 Mask 层级，复杂嵌套 ScrollView 会迅速增加 DrawCall。

### 19.5.8 动画优化

UI 动画同样可能导致严重性能问题。

例如：

- Animator
- DOTween
- Layout 动画
- Scale 动画
- Alpha 动画

这些动画往往会频繁触发：

```
Vertices Dirty
Layout Dirty
Material Dirty
```

从而导致 Canvas 高频重建。

尤其 Layout 动画成本最高。

例如：

```
修改 Width
修改 Height
修改 PreferredSize
```

这些操作都会触发 Layout Rebuild。

因此，UI 动画应尽量只修改 Transform，例如：

- AnchoredPosition
- LocalPosition
- Scale
- Rotation

这些属性对 Layout 系统影响较小。

### 19.5.9 分离静态与动态 UI

这是 UGUI 中最核心的优化原则之一。

因为静态 UI 不应该参与重建。

例如：

- 背景
- 装饰
- 固定框架
- Logo
- 静态按钮

这些内容应放入独立静态 Canvas。

而：

- 血条
- 数字
- 倒计时
- 聊天
- 动态列表

则应放入动态 Canvas。

这样才能避免少量变化导致整个界面重新生成 Mesh。

### 19.5.10 使用工具分析性能

UI 性能优化不能只凭感觉。

Unity 提供了大量工具帮助分析问题，例如：

```
Profiler
Frame Debugger
UI Profiler
Overdraw 模式
Stats 面板
```

其中：

- Profiler 用于分析 CPU
- Frame Debugger 用于分析 DrawCall
- Overdraw 用于分析 GPU FillRate
- Stats 用于观察顶点与批处理数量

很多时候，真正的问题与开发者想象的完全不同，因此性能分析必须依赖工具，而不是经验猜测。

### 19.5.11 小结

UGUI 性能优化并不是单独优化某一个组件，而是对整个渲染体系进行系统性控制。

真正影响 UI 性能的核心问题包括 Canvas 重建、Batch 中断、材质实例化、Mask 嵌套、顶点膨胀、Overdraw 以及 Layout 重建。

而所有优化工作的最终目标，本质上都是减少不必要的重建与渲染。

### 19.6 Overdraw 分析

在所有 UI 性能问题中，Overdraw 是最容易被忽视，同时也是对移动平台影响最大的 GPU 问题之一。很多开发者在优化 UI 时，通常会优先关注 DrawCall、Batch 或 Canvas 重建，但即使这些指标已经控制得很好，项目依然可能出现严重掉帧、发热以及 GPU 占用过高的问题，而这类问题的根源，往往就是 Overdraw。

尤其是在移动设备上，大量透明 UI 的重复覆盖会让 GPU 在同一个像素上反复执行 Fragment Shader，从而导致 FillRate 压力急剧增加，因此 UI 往往是 Overdraw 最严重的区域。

### 19.6.1 什么是 Overdraw

所谓 Overdraw，本质上是指同一个像素被重复渲染多次。

例如：

```
背景
半透明面板
按钮
发光
特效
文字
```

如果这些 UI 互相重叠，那么 GPU 在同一个屏幕像素位置上，可能需要连续执行多次 Fragment Shader。

例如：

```
Pixel A
→ Draw 1
→ Draw 2
→ Draw 3
→ Draw 4
→ Draw 5
```

虽然最终屏幕上只显示最后结果，但 GPU 实际已经执行了多次像素计算，这就是 Overdraw。

### 19.6.2 为什么 UI 特别容易产生 Overdraw

UGUI 默认使用透明混合渲染，而透明渲染最大的特点就是无法使用 Early-Z。

在 3D 场景中，不透明物体通常能够利用深度测试提前剔除被遮挡像素，因此很多不可见区域根本不会进入 Fragment 阶段。

但 UI 不同。

UI 大量使用：

- Alpha Blend
- 半透明贴图
- 渐变
- 发光
- 阴影
- 粒子

这些透明效果无法提前剔除，因此 GPU 必须真正执行 Fragment Shader 后，才能完成透明混合。

这意味着即使某个像素最终完全被遮挡，它依然已经被渲染过，因此 UI 天生就是 Overdraw 高发区域。

### 19.6.3 FillRate 为什么会成为瓶颈

GPU 渲染通常包含：

- Vertex 阶段
- Rasterization
- Fragment 阶段

而 Overdraw 最影响的，就是 Fragment 阶段。

因为每一次 Overdraw，都会额外增加 Fragment 运算量。

例如：1920 × 1080 ≈ 207 万像素

如果平均 Overdraw 为：4x

那么 GPU 实际 Fragment 处理量就会变成：207 万 × 4 ≈ 828 万次 Fragment

如果再叠加 Blur、Glow、Mask 与 Alpha 特效，Fragment Shader 开销还会进一步增加。

因此，Overdraw 本质上是在放大 FillRate 压力，尤其在移动平台上，GPU FillRate 通常远比 PC 更容易达到瓶颈。

### 19.6.4 全屏半透明 UI 的危害

很多项目最典型的 Overdraw 来源，就是全屏半透明背景。

例如：黑色半透明遮罩

看起来只是一个简单 Image，但实际上它会覆盖整个屏幕。

这意味着整个屏幕所有像素都会额外执行一次 Fragment。

如果弹窗层级继续叠加：

```
背景遮罩
模糊层
渐变层
发光层
UI 面板
```

那么 GPU 会不断重复渲染同一区域。

很多项目中的 GPU 发热问题，本质上都是大量全屏透明 UI 导致。

### 19.6.5 Text 与 Overdraw

很多开发者只关注 Text 顶点数量，却忽略了 Text 本身同样会产生严重 Overdraw。

因为字体贴图通常包含大量 Alpha 区域。

例如：

```
字符边缘
抗锯齿区域
阴影区域
```

这些区域都会参与透明混合。

如果再叠加：

- Outline
- Shadow
- Glow

那么同一个字符区域可能会被重复渲染多次。

尤其在聊天窗口、弹幕、富文本以及排行榜等高密度文字区域中，Overdraw 往往非常严重。

### 19.6.6 Mask 为什么会增加 Overdraw

很多开发者认为 Mask 可以减少渲染区域，因此能够降低 GPU 压力，但实际上情况恰恰相反。

因为大部分 Mask 裁剪发生在 Fragment 阶段，而 GPU 在执行裁剪之前，通常已经完成了顶点处理、光栅化以及 Fragment 生成。

也就是说，很多最终被 Mask 隐藏的像素，其实已经参与了 Fragment 运算。

因此 Mask 并不会真正减少 FillRate，反而：

- Stencil Test
- Fragment Discard
- Alpha Blend

还会进一步增加 GPU 压力。

尤其是多层 Mask 嵌套时，Overdraw 往往会迅速上升。

### 19.6.7 特效为什么容易导致 GPU 爆炸

UI 特效通常会同时增加：

- 顶点数量
- Fragment 数量
- Alpha Blend
- Overdraw

例如：

- Blur
- Glow
- Bloom
- 发光描边
- 大面积渐变
- 粒子特效

这些效果最大的特点，就是覆盖面积大、透明区域多、重复混合频繁。

尤其 Blur 最危险，因为 Blur 往往需要：

- 多次采样
- 多 Pass
- 全屏 Fragment

因此，UI Blur 是移动端最昂贵的 UI 特效之一。

很多项目中，一个全屏模糊就足以让 GPU 占用翻倍。

### 19.6.8 Overdraw 查看方式

Unity 提供了专门的 Overdraw 分析模式。

在Scene View -&gt; Draw Mode -&gt; Overdraw中，可以直接查看 Overdraw 分布。

通常：

- 黑色表示 Overdraw 很低
- 蓝色表示较低
- 绿色表示中等
- 红色表示很高
- 白色表示严重爆炸

如果某些 UI 区域已经接近白色，通常意味着大量透明 UI 正在重复覆盖，这也是移动端 GPU 掉帧最典型的表现。

### 19.6.9 降低 Overdraw 的核心策略

降低 Overdraw 的核心目标，本质上是减少无意义 Fragment 渲染。

首先，应减少全屏透明 UI，例如大面积遮罩、半透明背景、全屏渐变以及全屏模糊，这些效果都会显著增加 FillRate。

其次，应减少透明区域叠加。很多 UI 明明已经被遮挡，但依然继续参与渲染，例如被覆盖的 Image、被挡住的粒子以及不可见背景层，这些内容应尽量隐藏或者关闭。

再次，应减少复杂 UI 特效，尤其 Blur、Glow、Bloom 以及多层 Alpha 特效，这些效果对移动 GPU 压力极大。

另外，还应减少大面积 Text 特效，尤其不要在聊天系统中大量使用 Outline 与 Shadow。

最后，还应尽量缩小透明区域面积，因为即使按钮贴图大部分区域完全透明，GPU 依然会处理这些像素，因此透明区域本身也是 FillRate 成本。

### 19.6.10 移动端为什么最怕 Overdraw

PC GPU 通常拥有较高 FillRate 与显存带宽，因此对 Overdraw 的容忍度较高。

但移动端不同。

移动 GPU 通常：

- 带宽有限
- FillRate 较低
- Fragment 能力较弱
- 功耗限制严格

因此移动端 GPU 最怕大量透明 Fragment。

很多项目在 PC 上运行正常，但一到手机上就严重掉帧，本质原因往往并不是 DrawCall，而是 Overdraw。

尤其是全屏 UI、特效界面、商城、聊天窗口以及活动界面，这些区域通常都会形成极高 Fragment 压力。

### 19.6.11 小结

Overdraw 本质上是同一个像素被 GPU 重复渲染多次，而 UI 由于大量透明混合与特效叠加，天然就是 Overdraw 高发区域。

大量半透明背景、Mask、Text 特效、Blur 与 Glow，都会显著提高 GPU FillRate 压力。

尤其在移动平台上，Overdraw 往往比 DrawCall 更容易成为真正性能瓶颈。

因此，在 UI 性能优化中，不仅要关注 CPU 与 Batch，更必须关注 Fragment 渲染成本与 FillRate 压力。

### 本章小结

本章主要围绕 UGUI 中的 UI 特效与性能问题展开分析，深入讨论了批处理、材质变化、Mask、顶点膨胀以及 Overdraw 等核心问题对渲染性能的影响。

首先，UI 特效会直接影响 Batch 合并。无论是材质变化、Stencil 状态切换还是 Shader 不一致，都会导致 DrawCall 增加，而 DrawCall 的本质其实就是 GPU 渲染状态切换成本。

随后，本章分析了 Material 在 UGUI 中的影响。很多开发者误以为材质只是视觉表现层，但实际上 Material 会直接决定 Batch 是否能够继续合并。尤其在运行时频繁实例化材质时，往往会导致大量 Material Instance 与 DrawCall 激增。

接着，本章重点讨论了 Mask 的底层原理。Mask 本质上基于 GPU Stencil Buffer 实现，而 Stencil 状态变化会直接打断 Batch。同时，多层 Mask 嵌套还会导致 Material 数量增加，并显著提高 GPU Fragment 阶段压力，因此 Mask 往往是复杂 UI 性能问题的核心来源之一。

随后，本章进一步分析了顶点膨胀问题。Outline、Shadow、Text 与各种 MeshEffect 本质上都会增加顶点数量，而顶点增长不仅会提高 GPU Vertex 阶段压力，还会显著增加 CPU Mesh 构建与 Canvas Rebuild 成本。尤其在动态列表与大量文本场景中，顶点问题往往远比 DrawCall 更严重。

然后，本章总结了 UGUI 中常见的优化策略，包括拆分 Canvas、减少 Material 实例化、降低 Mask 层级、控制顶点数量、避免 Layout 系统滥用以及 ScrollView 节点复用等。这些优化手段的核心目标，本质上都是减少不必要的重建与渲染。

最后，本章深入分析了 Overdraw 对 GPU FillRate 的影响。由于 UI 大量使用透明混合，因此天然容易形成高 Overdraw，而 Blur、Glow、半透明背景以及大量 Text 特效都会进一步提高 Fragment Shader 压力。尤其在移动平台上，Overdraw 往往比 DrawCall 更容易成为真正性能瓶颈。

通过本章内容可以发现，UGUI 的性能问题并不是单一模块导致的，而是整个渲染体系共同作用后的结果。真正高性能的 UI，并不仅仅只是减少 DrawCall，更重要的是合理控制 Canvas 重建、顶点规模、Fragment 开销以及 GPU FillRate 压力。
