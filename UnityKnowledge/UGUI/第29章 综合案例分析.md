### 29.1 背包系统

背包系统几乎存在于所有 RPG、MMORPG、卡牌、策略以及生存类游戏之中。虽然从玩家视角来看，背包只是一个用于存放物品的界面，但从程序设计角度来看，它实际上是一个典型的数据驱动 UI 系统。

一个完整的背包系统通常涉及以下几个模块：

- 物品数据管理
- 背包数据管理
- UI 展示
- 数据与界面同步
- 物品增删改查
- 拖拽与交换
- 分页与筛选
- 网络同步

其中最核心的问题并不是如何显示格子，而是如何建立数据与界面之间的关系。

很多初学者在开发背包系统时，会直接在 UI 上操作数据。例如点击获得物品后，立即创建一个格子并显示在界面中；删除物品时直接销毁格子对象。这种做法虽然简单，但随着项目规模扩大，很快就会出现维护困难、逻辑混乱以及性能下降等问题。

商业项目通常采用数据驱动模式。

程序首先修改背包数据，然后通知 UI 进行刷新。

整个流程如下：

```
获得物品
    ↓
修改背包数据
    ↓
发送刷新事件
    ↓
刷新UI显示
```

这样可以保证数据始终是唯一可信来源（Single Source of Truth）。

### 29.1.1 背包系统整体架构

一个典型背包系统可以划分为三层。

```
┌─────────────────┐
│      UI层       │
│ InventoryPanel  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│    业务逻辑层    │
│ InventorySystem │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│      数据层      │
│ InventoryData   │
└─────────────────┘
```

数据层负责保存物品。

逻辑层负责增删改查。

UI层负责显示。

这样可以避免界面代码直接修改数据。

### 29.1.4 背包管理器实现

背包系统核心逻辑全部集中到 InventorySystem。

```csharp
using System;
using System.Collections.Generic;

public class InventorySystem
{
    public static InventorySystem Instance { get; } = new();

    public InventoryData Data = new();

    public Action OnInventoryChanged;

    public void AddItem(int itemId, int count)
    {
        ItemData item = Data.Items.Find(x => x.ItemId == itemId);

        if (item != null)
        {
            item.Count += count;
        }
        else
        {
            Data.Items.Add(new ItemData
            {
                ItemId = itemId,
                Count = count
            });
        }

        OnInventoryChanged?.Invoke();
    }

    public void RemoveItem(int itemId, int count)
    {
        ItemData item = Data.Items.Find(x => x.ItemId == itemId);

        if (item == null)
            return;

        item.Count -= count;

        if (item.Count <= 0)
        {
            Data.Items.Remove(item);
        }

        OnInventoryChanged?.Invoke();
    }
}
```

此时已经具备：

- 添加物品
- 删除物品
- 自动刷新通知

三个核心功能。

### 29.1.6 背包界面实现

下面实现背包主界面。

Hierarchy：

```
InventoryPanel
└─ Content
```

Content 挂载 GridLayoutGroup。

```csharp
using System.Collections.Generic;
using UnityEngine;

public class InventoryPanel : MonoBehaviour
{
    [SerializeField]
    private Transform content;

    [SerializeField]
    private InventoryItemView itemPrefab;

    private readonly List<InventoryItemView> itemViews = new();

    private void OnEnable()
    {
        InventorySystem.Instance.OnInventoryChanged += Refresh;

        Refresh();
    }

    private void OnDisable()
    {
        InventorySystem.Instance.OnInventoryChanged -= Refresh;
    }

    private void Refresh()
    {
        foreach (var view in itemViews)
        {
            Destroy(view.gameObject);
        }

        itemViews.Clear();

        foreach (ItemData item in InventorySystem.Instance.Data.Items)
        {
            InventoryItemView view = Instantiate(itemPrefab, content);

            view.Refresh(item);

            itemViews.Add(view);
        }
    }
}
```

运行效果如下：

```
┌─────────────────────┐
│ [药] [药] [剑] [币] │
│                     │
│                     │
└─────────────────────┘
```

### 29.1.7 当前实现存在的问题

假设背包拥有 300 个物品。

每次增加一个物品都会执行：

```
Destroy(全部格子)
重新创建全部格子
```

例如：

```
300个格子
↓
获得1个药水
↓
销毁300个
↓
创建301个
```

这种做法会产生大量：

- GC Alloc
- Layout Rebuild
- Graphic Rebuild

性能极差。

在 Profiler 中经常能够看到：

```
Canvas.BuildBatch
LayoutRebuilder
Instantiate
Destroy
GC.Collect
```

占据大量 CPU 时间。

这也是很多项目打开背包时出现卡顿的根本原因。

### 29.1.8 使用对象池优化

商业项目通常不会反复创建格子。

而是采用对象池。

首先实现一个简单对象池。

```csharp
using System.Collections.Generic;
using UnityEngine;

public class UIObjectPool<T> where T : Component
{
    private readonly Stack<T> pool = new();

    private readonly T prefab;

    private readonly Transform root;

    public UIObjectPool(T prefab, Transform root)
    {
        this.prefab = prefab;
        this.root = root;
    }

    public T Get()
    {
        if (pool.Count > 0)
        {
            T obj = pool.Pop();

            obj.gameObject.SetActive(true);

            return obj;
        }

        return Object.Instantiate(prefab, root);
    }

    public void Release(T obj)
    {
        obj.gameObject.SetActive(false);

        pool.Push(obj);
    }
}
```

刷新逻辑改为：

```
回收旧格子
获取新格子
刷新内容
```

这样即可避免频繁 Instantiate 和 Destroy。

对于拥有数百个格子的背包，性能提升非常明显。

### 29.1.9 实战案例：背包拖拽交换

大多数 RPG 游戏都支持拖拽物品。

例如：

```
原状态

[剑][药][盾]

拖动剑到盾位置

↓

[盾][药][剑]
```

首先扩展背包数据。

```csharp
[Serializable]
public class InventorySlot
{
    public ItemData Item;
}
public class InventoryData
{
    public List<InventorySlot> Slots = new();
}
```

然后增加交换接口。

```csharp
public void Swap(int indexA, int indexB)
{
    var temp = Data.Slots[indexA];

    Data.Slots[indexA] = Data.Slots[indexB];

    Data.Slots[indexB] = temp;

    OnInventoryChanged?.Invoke();
}
```

格子实现拖拽接口。

```csharp
using UnityEngine;
using UnityEngine.EventSystems;

public class InventorySlotView :
    MonoBehaviour,
    IBeginDragHandler,
    IDragHandler,
    IEndDragHandler,
    IDropHandler
{
    public int Index;

    public void OnBeginDrag(PointerEventData eventData)
    {

    }

    public void OnDrag(PointerEventData eventData)
    {

    }

    public void OnEndDrag(PointerEventData eventData)
    {

    }

    public void OnDrop(PointerEventData eventData)
    {
        InventorySlotView source =
            eventData.pointerDrag
            .GetComponent<InventorySlotView>();

        InventorySystem.Instance.Swap(
            source.Index,
            Index);
    }
}
```

这样即可实现最基础的物品交换。

### 29.2 列表优化

在现代游戏开发中，列表界面是出现频率最高的 UI 类型之一。

例如：

- 背包列表
- 邮件列表
- 好友列表
- 排行榜
- 商城列表
- 任务列表
- 聊天记录
- 成就系统
- 公会成员列表
- 战绩记录

从视觉表现来看，这些界面往往只是一个 ScrollRect 加若干 Item 预制体组成的简单结构。

但实际上，列表系统却是 UI 性能问题的重灾区。

许多项目在开发阶段运行流畅，而到了测试阶段，当数据量达到数百甚至上千条时，便会出现明显卡顿、掉帧甚至界面假死。

而这些问题，大多数并不是因为设备性能不足，而是由于列表实现方式存在缺陷。

本节将从 UGUI 的底层运行机制出发，分析传统列表的性能瓶颈，并通过完整案例实现一个高性能 [虚拟列表](https://zhida.zhihu.com/search?content_id=276108659&content_type=Article&match_order=1&q=%E8%99%9A%E6%8B%9F%E5%88%97%E8%A1%A8&zhida_source=entity) （Virtual List）。

### 29.2.1 最容易犯的错误

很多开发者第一次实现排行榜时，通常会采用如下方式。

```csharp
for (int i = 0; i < dataList.Count; i++)
{
    Instantiate(itemPrefab, content);
}
```

假设排行榜拥有 1000 条数据。

那么最终 Hierarchy 会变成：

```
Scroll View
└── Content
    ├── Item
    ├── Item
    ├── Item
    ├── Item
    ├── Item
    ├── ...
    └── Item
```

如果每个 Item 包含：

```
Item
├── Avatar
├── Name
├── Rank
├── Score
├── Button
└── Background
```

那么：

```
1000个Item × 6个Graphic组件 = 6000个Graphic
```

这意味着：

- Canvas Build Batch 增加
- Graphic Rebuild 增加
- Layout Rebuild 增加
- Raycast 检测增加
- DrawCall 合批压力增加

即使玩家只能看到屏幕上的十几个元素，UGUI 仍然需要维护全部 1000 个节点。

这就是最典型的性能浪费。

### 29.2.2 ScrollRect 的工作原理

为了理解优化方案，首先需要了解 ScrollRect 的运行机制。

ScrollRect 的核心结构如下：

```
ScrollRect
├── Viewport
│   └── Content
│       ├── Item
│       ├── Item
│       ├── Item
│       └── ...
```

滚动时实际上并没有移动 Item。

移动的是 Content。

例如：

```
初始状态

Content.y = 0

↓

向下滚动

Content.y = 500
```

所有 Item 都会跟随 Content 一起移动。

因此：

即使玩家只能看到前 10 个元素。

第 500 个元素仍然存在于场景中。

第 1000 个元素仍然存在于场景中。

它们依然参与：

- Transform 更新
- Canvas 更新
- Layout 计算
- Graphic 管理

因此节点数量越多，性能越差。

### 29.2.3 虚拟列表思想

商业项目通常不会创建全部 Item。

而是只创建当前可见区域需要的节点。

例如：

```
总数据1000条
```

屏幕最多显示：

```
10条
```

那么实际创建：

```
10 + 缓冲区5 = 15个Item
```

当玩家滚动时：

```
旧Item
↓
移动位置
↓
刷新数据
↓
显示新内容
```

而不是：

```
销毁旧Item
创建新Item
```

这种技术称为： **Virtual List（虚拟列表）**

也称： **Recycle List（循环列表）**

其核心思想如下：

```
1000条数据

↓

15个UI节点

↓

循环复用
```

这样无论数据有多少条：

```
100条
1000条
10000条
100000条
```

场景中的节点数量始终保持不变。

### 29.2.4 传统列表与虚拟列表对比

假设每个 Item 高度为：

```
100像素
```

Viewport 高度：

```
800像素
```

可见数量：

```
800 ÷ 100

=

8
```

增加缓冲区：

```
8 + 4 = 12
```

最终只需创建：

```
12个Item
```

而不是：

```
10000个Item
```

可以发现虚拟列表的节点数量与数据量无关。

### 29.2.8 虚拟列表实现

下面开始实现核心系统。

```csharp
using System.Collections.Generic;
using UnityEngine;
using UnityEngine.UI;

public class VirtualRankList : MonoBehaviour
{
    [SerializeField]
    private ScrollRect scrollRect;

    [SerializeField]
    private RectTransform content;

    [SerializeField]
    private RankItemView itemPrefab;

    [SerializeField]
    private float itemHeight = 100f;

    private readonly List<RankItemView> items = new();

    private List<RankData> dataList;

    private int visibleCount;

    private int poolCount;

    private void Start()
    {
        dataList = RankDataGenerator.Generate(10000);

        Init();
    }

    private void Init()
    {
        float viewportHeight = scrollRect.viewport.rect.height;

        visibleCount = Mathf.CeilToInt(viewportHeight / itemHeight);

        poolCount = visibleCount + 4;

        for (int i = 0; i < poolCount; i++)
        {
            RankItemView item = Instantiate(itemPrefab, content);

            items.Add(item);
        }

        content.sizeDelta = new Vector2(
                content.sizeDelta.x,
                dataList.Count * itemHeight);

        scrollRect.onValueChanged.AddListener(OnScroll);

        Refresh();
    }

    private void OnScroll(Vector2 value)
    {
        Refresh();
    }

    private void Refresh()
    {
        float scrollY = content.anchoredPosition.y;

        int startIndex = Mathf.FloorToInt(scrollY / itemHeight);

        for (int i = 0; i < poolCount; i++)
        {
            int dataIndex = startIndex + i;

            if (dataIndex >= dataList.Count)
            {
                items[i].gameObject.SetActive(false);
                continue;
            }

            items[i].gameObject.SetActive(true);

            RectTransform rt = items[i].GetComponent<RectTransform>();

            rt.anchoredPosition = new Vector2(0, -dataIndex * itemHeight);

            items[i].Refresh(dataList[dataIndex]);
        }
    }
}
```

### 29.2.9 实现原理分析

假设当前滚动位置：

```
Content.y = 0
```

此时：

```
Item0 → 数据0
Item1 → 数据1
Item2 → 数据2
...
```

滚动到：

```
Content.y = 500
```

则：

```
startIndex = 5
```

刷新后：

```
Item0 → 数据5
Item1 → 数据6
Item2 → 数据7
...
```

注意：

这里并没有创建新的 Item。

仅仅是：

```
修改位置
+
修改数据
```

因此运行成本极低。

### 29.2.10 进一步优化：对象池

上面的实现已经避免滚动过程中创建节点。

但初始化时仍然执行：

```csharp

```

对于大型项目：

通常会配合对象池统一管理。

例如：

```csharp

itemPool.Release(item);
```

这样可以避免界面频繁打开关闭时产生 GC。

### 29.2.11 进一步优化：避免频繁刷新

很多项目中经常出现：

```csharp
scrollRect.onValueChanged
```

每帧触发几十次。

然后：

```csharp
Refresh();
```

刷新全部 Item。

实际上很多时候根本没有跨行。

例如：

```
当前位置

500px

↓

501px
```

此时：

```
startIndex仍然等于5
```

根本无需刷新。

优化方案如下：

```csharp
private int currentStartIndex = -1;
int startIndex = Mathf.FloorToInt(scrollY / itemHeight);

if (startIndex == currentStartIndex)
{
    return;
}

currentStartIndex = startIndex;

Refresh();
```

这样只有跨越新的 Item 时才刷新。

CPU 开销会进一步下降。

区别仅在于：

### 29.2.13 小结

列表优化是 UGUI 性能优化中收益最高的方向之一。

因为 ScrollRect 往往承载着项目中最多的 UI 节点。

普通实现的复杂度为：

节点数量与数据量直接相关。

而虚拟列表的复杂度为：

```
O(VisibleCount)
```

节点数量仅与屏幕可见区域有关。

通过本节案例，我们完成了一个支持一万条数据的高性能排行榜，并掌握了虚拟列表的核心思想：

- 数据驱动刷新
- 避免动态创建销毁
- 减少 Layout 与 Canvas 重建

在大型项目中，一个设计良好的虚拟列表系统，往往能够将 ScrollRect 的性能提升数十倍甚至上百倍。

### 29.3.5 打开界面

实现 Open 方法。

```csharp
using UnityEngine;

public partial class UIManager
{
    [SerializeField]
    private Transform popupLayer;

    public T Open<T>(T prefab)
        where T : UIPanel
    {
        string key = typeof(T).Name;

        if (openedPanels.TryGetValue(
            key,
            out UIPanel panel))
        {
            return panel as T;
        }

        T instance =
            Instantiate(
                prefab,
                popupLayer);

        instance.OnOpen();

        openedPanels.Add(
            key,
            instance);

        return instance;
    }
}
```

使用方式：

```csharp
UIManager.Instance.Open(settingPanelPrefab);
```

此时：

```csharp
PopupLayer
└── SettingPanel(Clone)
```

自动创建完成。

### 29.3.6 关闭界面

实现关闭逻辑。

```csharp
public void Close<T>()
    where T : UIPanel
{
    string key = typeof(T).Name;

    if (!openedPanels.TryGetValue(
        key,
        out UIPanel panel))
    {
        return;
    }

    panel.OnClose();

    Destroy(panel.gameObject);

    openedPanels.Remove(key);
}
```

调用方式：

```csharp
UIManager.Instance.Close<SettingPanel>();
```

此时：

```

└── 空
```

界面被正确销毁。

```csharp
using System;
using TMPro;
using UnityEngine;
using UnityEngine.UI;

public class ConfirmPanel : UIPanel
{
    [SerializeField]
    private TMP_Text messageText;

    [SerializeField]
    private Button confirmButton;

    [SerializeField]
    private Button cancelButton;

    private Action onConfirm;

    private Action onCancel;

    public void Show(
        string message,
        Action confirm,
        Action cancel = null)
    {
        messageText.text = message;

        onConfirm = confirm;

        onCancel = cancel;

        confirmButton.onClick.RemoveAllListeners();

        cancelButton.onClick.RemoveAllListeners();

        confirmButton.onClick.AddListener(
            OnConfirmClick);

        cancelButton.onClick.AddListener(
            OnCancelClick);
    }

    private void OnConfirmClick()
    {
        onConfirm?.Invoke();

        UIManager.Instance.Close<ConfirmPanel>();
    }

    private void OnCancelClick()
    {
        onCancel?.Invoke();

        UIManager.Instance.Close<ConfirmPanel>();
    }
}
```

### 29.3.11 对象池优化

```
↓
```

这样可以显著减少卡顿。

在前面的章节中，我们分别分析了 UGUI 的渲染机制、事件系统、布局系统、背包系统、列表优化以及弹窗系统。

```csharp
    private UILayerRoot layerRoot;

    private readonly Dictionary<Type, UIPanel>
        openedPanels = new();

    private void Awake()
    {
        Instance = this;
    }
}
```

### 29.4.8 实战案例：统一打开界面

实现通用 Open 方法。

```csharp
public T Open<T>()
    where T : UIPanel
{
    Type type = typeof(T);

    if (openedPanels.TryGetValue(
        type,
        out UIPanel panel))
    {
        panel.OnOpen();

        return (T)panel;
    }

    UIConfig config =
        UIConfigManager.Get(type.Name);

    GameObject prefab =
        Resources.Load<GameObject>(
            config.Path);

    Transform parent =
        layerRoot.GetLayer(config.Layer);

    GameObject go =
        Instantiate(
            prefab,
            parent);

    T instance =
        go.GetComponent<T>();

    instance.OnCreate();

    instance.OnOpen();

    openedPanels.Add(
        type,
        instance);

    return instance;
}
```

使用方式：

```csharp
UIManager.Instance.Open<LoginPanel>();
```

此时框架自动完成：

```
读取配置
↓
加载资源
↓
找到层级
↓
实例化
↓
调用生命周期
```

### 29.4.9 实战案例：统一关闭界面

实现关闭逻辑。

```csharp
public void Close<T>()
    where T : UIPanel
{
    Type type = typeof(T);

    if (!openedPanels.TryGetValue(
        type,
        out UIPanel panel))
    {
        return;
    }

    panel.OnClose();

    panel.gameObject.SetActive(false);
}
```

注意这里没有销毁。

原因是后面会使用缓存。

### 29.4.10 界面缓存机制

商业项目中很少直接销毁界面。

例如：

```
打开背包
关闭背包
打开背包
关闭背包
打开背包
```

如果每次都：

```csharp
Instantiate
Destroy
```

会产生大量 GC。

因此一般采用缓存。

修改 Open。

```csharp
if (openedPanels.TryGetValue(
    type,
    out UIPanel panel))
{
    panel.gameObject.SetActive(true);

    panel.OnOpen();

    return (T)panel;
}
```

关闭时：

```csharp
panel.gameObject.SetActive(false);
```

这样再次打开时无需重新创建。

### 29.4.11 实战案例：界面间通信

很多项目中会出现如下需求。

```
背包界面
↓
装备发生变化
↓
角色界面同步刷新
```

错误做法：

```csharp
FindObjectOfType<RolePanel>()
```

或者：

```csharp
GetComponent<RolePanel>()
```

这种方式会产生模块耦合。

商业项目通常采用事件系统。

定义事件中心。

```csharp
using System;

public static class UIEventCenter
{
    public static Action OnEquipChanged;
}
```

背包界面发送事件。

```csharp
UIEventCenter.OnEquipChanged?.Invoke();
```

角色界面监听事件。

```csharp
private void OnEnable()
{
    UIEventCenter.OnEquipChanged += Refresh;
}
private void OnDisable()
{
    UIEventCenter.OnEquipChanged -= Refresh;
}
```

这样两个界面无需直接引用。

很多界面打开时需要携带参数。

例如：

```csharp
public class ItemInfoPanel :
```

```csharp
public T Open<T>(object data)
```

### 29.4.13 实战案例：异步加载界面

随着项目规模扩大。

UI 资源可能达到数 GB。

此时不能全部预加载。

需要异步加载。

以 Addressables 为例。

```csharp
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;
public async void OpenAsync<T>()
    where T : UIPanel
{
    UIConfig config = UIConfigManager.Get(typeof(T).Name);

    AsyncOperationHandle<GameObject> handle = Addressables.LoadAssetAsync<GameObject>(
            config.Path);

    await handle.Task;

    GameObject prefab = handle.Result;

    Instantiate(prefab);
}
```

这样可以避免界面打开时卡顿。

### 29.5 性能分析

在 UGUI 开发过程中，绝大多数性能问题都不是因为某个 API 本身执行速度慢，而是由于开发者不了解 UGUI 的运行机制，从而导致大量不必要的重建、布局计算以及渲染开销。

很多项目在功能开发阶段运行正常，但随着界面数量不断增加，最终会出现以下现象：

- 打开界面卡顿
- 滚动列表掉帧
- 动态刷新产生 GC
- Draw Call 数量暴增
- CPU 占用过高
- 手机上帧率不稳定

这些问题往往不会在代码审查阶段暴露，而是在运行过程中逐渐积累。

因此，一个成熟的 UI 开发者不仅需要会编写界面，还必须具备分析性能问题的能力。

本节将结合 Unity 提供的分析工具以及 UGUI 底层原理，系统讲解 UI 性能问题的定位方法与优化思路。

### 29.5.1 性能分析的基本原则

很多开发者遇到卡顿时第一反应是：是不是代码写得不好？

实际上：

```
不要猜测
不要凭感觉
不要主观判断
```

而应该：

```
先定位
再分析
最后优化
```

性能优化最忌讳的就是没有问题先优化，或者不知道问题在哪就修改代码。

正确流程应该是：

```
发现卡顿
    ↓
定位瓶颈
    ↓
分析原因
    ↓
制定方案
    ↓
验证结果
```

### 29.5.2 Profiler 简介

Unity 最重要的性能分析工具就是 Profiler。

打开方式：Window→Analysis→Profiler

Profiler 可以监控：

- CPU
- GPU
- Memory
- Rendering
- Audio
- Physics
- UI

对于 UGUI 开发而言，最常用的是：CPU Usage

以及：UI Details

当界面发生卡顿时，首先观察 CPU Usage。

例如：

```
PlayerLoop
└── UI
    ├── Canvas.BuildBatch
    ├── LayoutRebuilder
    └── GraphicRaycaster
```

这些模块通常就是 UI 性能问题的主要来源。

### 29.5.3 Canvas.BuildBatch

在大型项目中最常见的性能热点之一就是：

```
Canvas.BuildBatch
```

例如：

```
CPU Usage

Canvas.BuildBatch

12ms
```

出现这种情况通常意味着：Canvas发生重建

回顾前面的章节：

当以下内容发生变化时：

- RectTransform
- Image
- Text
- LayoutGroup
- Mask

UGUI 都会标记脏节点。

随后执行：

```
CanvasUpdateRegistry
        ↓
Graphic Rebuild
        ↓
Canvas.BuildBatch
```

重新生成批处理数据。

例如：

```csharp
hpText.text = hp.ToString();
```

看似只修改了一个文本。

实际上可能触发：

```
Graphic Rebuild
Canvas Rebuild
Batch Rebuild
```

如果界面元素很多，代价会非常高。

### 29.5.4 案例：频繁刷新文本

下面是一种常见错误。

```csharp
private void Update()
{
    hpText.text = playerHp.ToString();
}
```

即使血量没有变化。

每帧仍然执行：

```
SetVerticesDirty
SetMaterialDirty
Graphic Rebuild
```

Profiler 中经常出现：

```
Canvas.BuildBatch
```

持续占用 CPU。

正确做法：

```csharp
private int lastHp;

private void Update()
{
    if (lastHp == playerHp)
        return;

    lastHp = playerHp;

    hpText.text = playerHp.ToString();
}
```

只有数据变化时才刷新。

### 29.5.5 Layout Rebuilder

另一个常见热点是：

```
LayoutRebuilder
```

例如：

```
LayoutRebuilder

6ms
```

通常意味着布局系统正在频繁重算。

最典型场景如下。

```
VerticalLayoutGroup
    ↓
ContentSizeFitter
    ↓
大量动态节点
```

例如聊天系统。

```
聊天消息
↓
创建消息
↓
重新计算高度
↓
重新计算布局
```

消息越多：

```
Layout Rebuild
```

越频繁。

### 29.5.6 案例：聊天系统卡顿

错误实现：

```
ScrollView
└── Content
    ├── Message
    ├── Message
    ├── Message
    └── ...
```

Content 同时挂载：

```
VerticalLayoutGroup
ContentSizeFitter
```

每新增一条消息：

```
重新计算所有节点
```

当消息达到数百条时：

```
LayoutRebuilder

10ms+
```

并不少见。

优化方案：

- 使用虚拟列表
- 预计算高度
- 减少 LayoutGroup
- 手动设置位置

例如：

```csharp
itemRect.anchoredPosition = new Vector2(0, y);
```

替代：

```
VerticalLayoutGroup
```

性能提升十分明显。

### 29.5.7 GraphicRaycaster

UGUI 默认会对可交互组件进行射线检测。

例如：

```
Button
Image
Text
RawImage
```

只要开启：

```
Raycast Target
```

就会参与检测。

很多项目存在如下情况。

```
排行榜
1000个Item
```

每个 Item：

```
Image
Text
Text
Text
```

全部勾选：

```
Raycast Target
```

最终：

```
4000次射线检测
```

实际上玩家只需要点击按钮。

其余文本根本不需要参与射线检测。

优化方式：

取消：

```
Raycast Target
```

例如：

```
头像
关闭

名称
关闭

分数
关闭

背景
关闭
```

只保留真正需要点击的组件。

### 29.5.8 案例：排行榜优化

优化前：

```
1000个Item
```

每个：

```

```

总计：

```
5000 Graphic
```

全部开启射线检测。

Profiler：

```

```

优化后：


无法合批。

### Mask 打断批次

会产生额外 Draw Call。

### 特效材质

### 29.5.12 内存分析

很多 UI 卡顿并非 CPU 问题。

而是 GC 问题。

查看方式：Profiler→Memory

常见现象：

```
GC Alloc
```

持续增长。

例如：

```csharp
private void Update()
{
    text.text = string.Format("HP:{0}", hp);
}
```

每帧产生字符串。

导致：

```
GC Alloc
```

持续增加。

随后：

```
GC.Collect
```

触发卡顿。

优化方式：

```csharp
if(lastHp != hp)
{
    lastHp = hp;

    text.text = hp.ToString();
}
```

避免无意义字符串创建。

### 29.5.13 案例：界面频繁打开关闭

错误实现：

```
Instantiate(panel);
Destroy(panel);
```

频繁执行：

```
GC Alloc
GC Collect
```

Profiler：

```
GC Alloc

500KB/s
```

优化后：

```
panelPool.Get();
panelPool.Release();
```

Profiler：

```
GC Alloc

0
```

几乎消除垃圾回收。

### 29.5.16 UI 优化检查清单

项目上线前通常会进行一次全面检查。

### Canvas

- 是否存在超大 Canvas
- 是否频繁重建
- 是否需要拆分 Canvas

### Layout

- 是否滥用 LayoutGroup
- 是否滥用 ContentSizeFitter
- 是否可以手动布局

### ScrollRect

- 是否使用虚拟列表
- 是否存在大量隐藏节点

### Graphic

- 是否关闭无用 Raycast Target
- 是否减少 Graphic 数量

### Rendering

- 是否统一图集
- 是否统一材质
- 是否减少 Mask

### Memory

- 是否存在频繁 GC
- 是否使用对象池
- 是否频繁创建字符串

### Resource

- 是否异步加载
- 是否及时释放资源
