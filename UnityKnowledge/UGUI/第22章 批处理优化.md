在 [UGUI](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 渲染体系中，批处理机制是连接 UI 数据与 GPU 绘制之间最关键的一环。无论 UI 的结构如何复杂，最终进入渲染管线的，都必须通过批处理阶段进行组织与合并，从而生成 [DrawCall](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=DrawCall&zhida_source=entity) 。

从系统角度来看，批处理的目标非常明确：在不改变视觉结果的前提下，尽可能减少渲染指令的数量。DrawCall 的多少，直接决定了 CPU 提交开销与 GPU 状态切换成本，因此批处理能力的优劣，直接影响 UI 的整体性能表现。

UGUI 的批处理并不是一个独立模块，而是贯穿 CanvasRenderer 与 Canvas 系统的核心流程。在 Canvas 执行 BuildBatch 阶段时，系统会收集所有子 UI 的 Mesh 数据，并根据材质、纹理以及渲染状态进行归类与合并。只有满足合批条件的 UI 元素，才会被组合成同一个 DrawCall。

然而，在实际项目中，批处理往往难以达到理想状态。UI 结构设计、资源组织方式以及渲染状态变化，都会影响最终的合批结果。即使 UI 数量并不庞大，如果合批条件被频繁打断，DrawCall 依然会显著增加。

因此，批处理优化的核心，并不是简单减少 UI 数量，而是围绕“如何维持渲染状态一致性”展开。通过控制材质使用、纹理组织以及层级结构，可以最大化合批效率，从根本上降低渲染开销。

本章将围绕 UGUI 的批处理机制展开分析，重点讨论图集管理、材质策略、合批规则以及动态 UI 对批处理的影响，从系统层面建立对 DrawCall 优化的完整认知体系。

**22.1 图集管理**

在 UGUI 的批处理体系中，图集（ [SpriteAtlas](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=SpriteAtlas&zhida_source=entity) ）是影响 DrawCall 数量最关键的资源组织方式之一。它的核心作用，是将多个 Sprite 合并到同一张纹理中，从而减少纹理切换带来的合批中断问题，提高批处理效率。

从渲染机制来看，UGUI 的合批条件之一是“纹理一致”。如果多个 UI 元素使用不同的纹理，即使它们的材质完全相同，也无法被合并为同一个 DrawCall。因此，图集的存在，本质上是为了解决“纹理碎片化”导致的批处理失败问题。

**22.1.1 图集在批处理中的作用**

在没有图集的情况下，每个 Sprite 通常对应一张独立纹理。当 UI 中同时使用多个 Sprite 时，会产生频繁的纹理绑定切换，而每一次切换都会打断当前批次，从而生成新的 DrawCall。

引入图集之后，多个 Sprite 被打包到同一纹理中，UI 在渲染时只需要绑定一次纹理，即可绘制多个元素。这使得多个 UI 元素具备了合批的基础条件。

从本质上来看，图集并不直接减少 UI 数量，而是通过减少“渲染状态变化”，来提升批处理的连续性。

**22.1.2 图集对合批条件的影响**

UGUI 的合批依赖于多个条件同时满足，其中纹理一致性是最关键的一项。图集正是通过统一纹理来源，使多个 UI 元素满足这一条件。

具体影响包括：

一、减少纹理切换

多个 Sprite 使用同一图集后，GPU 只需绑定一次纹理，从而避免频繁状态切换。

二、提高批次连续性

相同图集内的 UI 元素更容易在渲染顺序上保持连续，从而提升合批成功率。

三、降低 DrawCall 数量

当多个 UI 元素满足合批条件时，会被合并为同一个 DrawCall，从而减少 CPU 与 GPU 的交互次数。

需要注意的是，图集只解决“纹理一致性”问题，并不能解决材质差异、Mask 以及层级穿插等其他合批中断因素。

**22.1.3 图集使用中的常见问题**

在实际开发中，图集使用不当也可能导致性能问题，甚至削弱其优化效果。

一、图集过多

如果项目中存在大量小图集，而不是集中管理，会导致纹理切换依然频繁，无法有效提升合批效率。

二、图集过大

单个图集过大可能导致内存占用增加，同时在加载和管理时带来额外开销。尤其在移动平台上，需要平衡内存与性能。

三、跨图集使用

如果 UI 元素分散在多个图集中，并且在层级中交替出现，会导致合批被频繁打断，从而增加 DrawCall。

四、动态加载图集不合理

运行时频繁加载或卸载图集，会导致纹理状态变化，进而破坏批处理连续性。

**22.1.4 图集与 UI 结构的关系**

图集优化不仅是资源层面的工作，也与 UI 结构设计密切相关。

在理想情况下，相同界面或同一功能模块的 UI 元素，应尽量使用同一图集，并保持在层级结构中的连续排列。这种设计方式可以最大化合批效果。

相反，如果 UI 层级中频繁交错使用不同图集资源，即使每个图集内部都是优化良好的，也可能因为渲染顺序被打断而无法合批。

因此，图集设计必须与 UI 架构设计同步考虑，而不能作为独立优化手段。

**22.1.5 图集优化原则**

在实际项目中，可以遵循以下原则进行图集管理：

- 优先将同一模块 UI 资源集中到同一图集
- 避免过度拆分图集，减少纹理切换
- 控制图集大小，避免极端的大图集或小图集
- 保证 UI 排列顺序与图集使用尽量连续
- 结合帧调试工具验证合批效果

从系统角度来看，图集的本质，是“通过纹理统一降低渲染状态变化”。它并不会改变 UI 的结构，但会显著影响批处理的效率。

因此，在批处理优化体系中，图集管理属于基础但关键的一环，是决定 DrawCall 能否有效降低的核心因素之一。

**22.2 材质策略**

在 UGUI 的批处理体系中，材质是影响 DrawCall 合并能力的第二个关键因素，仅次于纹理一致性。如果说图集解决的是“纹理碎片化问题”，那么材质策略解决的就是“渲染状态碎片化问题”。

从批处理机制来看，UGUI 要求参与合批的 UI 元素必须满足材质一致这一条件。即使两个 UI 使用相同的纹理，只要材质不同，仍然无法合并为同一个 DrawCall。因此，材质的组织方式，直接决定了批处理的上限。

**22.2.1 材质在合批中的作用**

在渲染流程中，材质不仅仅是一个资源引用，它实际上代表了一整套渲染状态，包括 Shader、贴图采样方式、混合模式以及部分 GPU 参数。

当 Canvas 执行批处理时，会以材质为单位对 UI 进行分组处理。每一个不同的材质，都会生成独立的渲染批次。这意味着：

- 材质数量越多，DrawCall 通常越多
- 材质差异越小，合批效率越高

因此，材质统一性，是控制批处理质量的核心前提之一。

**22.2.2 默认 UI 材质与合批基础**

UGUI 默认使用统一的 [UI Shader](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=UI+Shader&zhida_source=entity) （通常为 UI/Default）。在理想情况下，所有 UI 元素如果都使用该默认材质，并且满足纹理一致条件，就可以实现最大程度的合批。

这也是为什么在基础 UI 设计中，通常建议尽量复用默认 UI 材质，而不是为每个 UI 元素单独创建材质实例。

然而在实际项目中，由于特效需求、颜色变化或特殊效果，材质往往会被频繁修改，从而破坏合批条件。

**22.2.3 [材质实例化](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=%E6%9D%90%E8%B4%A8%E5%AE%9E%E4%BE%8B%E5%8C%96&zhida_source=entity) 问题**

材质策略中最常见的性能问题，是“材质实例化”。

当开发者在运行时修改材质属性（例如颜色、透明度、描边参数等）时，如果直接修改 sharedMaterial，Unity 可能会自动生成材质实例。这会导致：

- 原本可合批的 UI 被拆分为多个独立材质
- DrawCall 数量增加
- 材质管理复杂度上升

尤其是在 Image 或 Text 组件中频繁修改材质属性时，这种问题会更加明显。

因此，在 UI 开发中必须明确区分 sharedMaterial 与 material 的使用场景，避免无意中破坏合批结构。

**22.2.4 多材质 UI 的代价**

在某些复杂 UI 中，例如带有描边、发光、溶解等效果的界面，通常会使用多个材质或多 Pass Shader。

这种设计虽然可以实现视觉效果，但会带来明显的性能代价：

一、DrawCall 增加

每增加一个材质，就可能增加一个渲染批次。

二、批处理被打断

即使 UI 在同一个 Canvas 中，也无法跨材质合批。

三、状态切换成本上升

GPU 需要频繁切换 Shader 状态与渲染参数。

因此，在设计 UI 特效时，需要在视觉效果与批处理效率之间进行权衡。

**22.2.5 材质共享策略**

为了提升批处理效率，在实际项目中通常采用“材质共享策略”。

其核心思想是：尽可能减少材质种类，而不是减少 UI 数量。

具体做法包括：

- 统一使用基础 UI Shader
- 通过参数而非材质区分视觉差异
- 将可变效果转移到顶点或 UV 层实现
- 避免为单个 UI 创建独立材质实例
- 集中管理特效材质，避免分散使用

这种方式可以显著降低材质碎片化程度，从而提升合批成功率。

**22.2.6 材质策略与 UI 架构的关系**

材质策略不仅是渲染层面的优化问题，也与 UI 架构设计密切相关。

如果 UI 系统缺乏统一的材质管理机制，就很容易出现以下问题：

- 同类 UI 使用不同材质
- 不同模块重复创建相同 Shader
- 特效逻辑分散导致材质不可控

因此，在较大型项目中，通常会引入统一的 UI 材质管理层，对所有 UI 材质进行集中控制，从源头减少不必要的变化。

**22.2.7 小结**

材质策略的本质，是对“渲染状态一致性”的控制。它决定了 UI 是否能够在 Canvas 层级中形成连续的批处理链路。

如果说图集解决的是“纹理层面的合批问题”，那么材质策略解决的则是“渲染状态层面的合批问题”。两者共同构成了 UGUI 批处理优化的基础结构。

在实际优化过程中，只有同时控制图集与材质两个维度，才能真正稳定降低 DrawCall，并维持高效的 UI 渲染性能。

**22.3 合批优化**

在 UGUI 的渲染体系中，合批优化是批处理机制的最终目标。前面的图集管理与材质策略，分别从纹理与渲染状态两个维度减少了 DrawCall 的产生条件，而合批优化则是在系统行为层面，进一步控制 UI 如何真正进入批处理流程。

从本质上来看，UGUI 的合批并不是“自动最大化”的过程，而是一个严格依赖条件匹配与执行顺序的线性组织过程。任何一个条件被打断，都会导致当前批次结束并生成新的 DrawCall。因此，合批优化的核心，是尽可能维持渲染状态的连续性。

**22.3.1 合批的执行逻辑**

在 Canvas 执行 BuildBatch 阶段时，系统会按照一定顺序遍历所有 UI 元素，并根据其渲染状态进行分组。

合批的基本逻辑可以概括为：

- 相同材质、相同纹理、相同渲染状态的 UI 元素被归为同一批次
- 一旦条件发生变化，当前批次立即结束并生成 DrawCall
- 新的状态将开启新的批次

这一机制决定了合批是“顺序敏感”的。因此，UI 在层级中的排列方式，直接影响最终的批次数量。

**22.3.2 影响合批的关键因素**

在实际运行中，以下因素会直接影响合批结果：

一、材质差异

不同材质必然导致批次拆分，这是最基础的限制条件。

二、纹理差异

即使材质相同，只要使用不同纹理（未使用图集），也无法合批。

三、Mask 与裁剪状态

Mask 会引入 Stencil 状态变化，使渲染批次被强制拆分。

四、Canvas 分割

不同 Canvas 之间无法进行合批，这是系统级边界限制。

五、层级顺序打断

当 UI 元素在层级中交错使用不同状态资源时，会导致批处理被频繁中断。

这些因素共同决定了 UI 最终的 DrawCall 数量。

**22.3.3 合批连续性的关键原则**

合批优化的核心目标，是提升“批次连续性”。所谓连续性，是指在渲染顺序中，尽可能长时间维持相同的渲染状态。

在实际设计中，可以遵循以下原则：

一、同类资源连续排列

使用相同图集与材质的 UI 元素应尽量在层级中连续出现，避免交错。

二、减少状态切换

避免在 UI 中频繁插入不同材质或不同特效元素，以减少批次中断。

三、控制 Mask 层级

Mask 应尽量局部使用，避免跨多个 UI 模块造成批次分裂。

四、合理规划 Canvas

将高频变化区域与静态区域分离，减少跨区域批处理干扰。

这些原则的核心思想，是“减少渲染状态变化频率”。

**22.3.4 合批失败的典型场景**

在实际项目中，合批失败通常表现为以下几种情况：

- UI 元素数量不多，但 DrawCall 偏高
- 图集已使用，但仍无法合批
- 界面结构复杂，但性能异常波动

这些问题通常并非资源问题，而是由以下结构性原因导致：

- 材质实例过多
- 图集使用不连续
- Mask 嵌套过深
- UI 层级交错排列
- Canvas 拆分不合理

这些因素都会导致批处理被频繁打断，从而降低合批效率。

**22.3.5 合批优化的工程策略**

在工程实践中，合批优化通常需要从 UI 架构层进行设计，而不是单点优化。

常见策略包括：

一、统一资源规范

所有 UI 元素尽量使用统一 Shader 与标准图集体系，减少状态差异。

二、结构分组设计

将 UI 按功能模块分组，并保持组内资源连续性。

三、减少动态插入

避免在高频 UI 中动态插入不同类型元素，以防止批次中断。

四、控制特效边界

将特效 UI 与普通 UI 分离，避免影响主批次结构。

五、结合帧调试验证

使用 Frame Debugger 检查每一个 DrawCall 的来源，反向优化结构。

**22.3.6 合批优化的本质**

从系统角度来看，合批优化并不是“减少 DrawCall”的简单操作，而是对 UI 渲染状态组织方式的重构。

其本质可以概括为：在不改变视觉结果的前提下，尽可能延长相同渲染状态的连续执行区间。换句话说，合批优化的核心不是“减少 UI”，而是“减少状态变化”。只有当 UI 在结构设计阶段就具备良好的状态连续性，合批机制才能发挥最大效率。

总体来看，合批优化是 UGUI 批处理体系中最接近底层执行逻辑的一环。它直接决定了 Canvas 最终生成的 DrawCall 数量，也是 UI 性能优化中最需要系统性设计的一部分。

**22.4 [动态图集](https://zhida.zhihu.com/search?content_id=275647182&content_type=Article&match_order=1&q=%E5%8A%A8%E6%80%81%E5%9B%BE%E9%9B%86&zhida_source=entity)**

在 UGUI 的批处理体系中，图集的核心价值在于统一纹理来源，从而减少 DrawCall 的拆分。然而，在实际项目中，静态图集往往无法完全覆盖所有 UI 使用场景，特别是在资源动态加载、运营活动界面以及可扩展 UI 系统中，动态图集（Dynamic Atlas）成为一种重要的补充方案。

动态图集的本质，是在运行时将分散的 Sprite 按需打包到同一纹理中，从而在不依赖预制图集的情况下，维持合批条件的一致性。

**22.4.1 动态图集的工作机制**

动态图集通常在运行时维护一张或多张共享纹理。当 UI 需要使用某个 Sprite 时，系统会判断该 Sprite 是否已经存在于图集中：

- 如果存在，则直接复用
- 如果不存在，则将其打包进图集，并更新 UV 映射关系

在渲染层面，UI 实际使用的仍然是同一张纹理，只是不同 Sprite 在图集中对应不同的 UV 区域。

从批处理角度来看，这种方式的关键价值在于：将“多纹理问题”转化为“单纹理 + UV 切分问题”，从而满足合批的纹理一致性条件。

**22.4.2 动态图集对合批的影响**

动态图集对 DrawCall 的影响主要体现在以下几个方面：

一、减少纹理切换

多个 Sprite 被统一到同一纹理后，可以显著减少 GPU 纹理绑定次数，从而提升合批成功率。

二、提升批处理连续性

原本分散的资源在运行时被整合，使 UI 更容易形成连续的渲染批次。

三、改善资源碎片化问题

在未使用图集的情况下，UI 往往存在大量小纹理资源，动态图集可以有效减少这种碎片化。

但需要注意的是，动态图集并不能解决所有合批问题，例如材质差异、Mask 以及 Canvas 分割仍然会导致批次中断。

**22.4.3 动态图集的成本与风险**

尽管动态图集在提升合批方面具有明显优势，但其本身也带来一定的系统成本：

一、运行时打包开销

Sprite 在首次加入图集时，需要进行纹理拷贝与 UV 重映射，这一过程会消耗 CPU 时间。

二、内存增长不可控

动态图集随着运行时间不断扩展，如果缺乏回收机制，可能导致内存占用持续增长。

三、纹理空间浪费

由于图集采用固定或半固定分配策略，可能存在一定的空白区域浪费。

四、加载时性能波动

在 UI 初次加载或资源集中出现时，可能出现短时间的 CPU 峰值。

因此，动态图集并不是完全替代静态图集的方案，而是一种“补充型优化机制”。

**22.4.4 使用场景分析**

动态图集通常适用于以下场景：

- 运行时生成或加载的 UI 资源
- 无法提前确定内容的运营活动界面
- 插件化或模块化 UI 系统
- 本地资源较为分散的项目结构

而在以下场景中，则不建议依赖动态图集：

- 核心主界面 UI（应使用静态图集保证稳定性）
- 高频更新 UI（避免运行时打包开销）
- 对启动性能要求极高的场景

合理区分使用场景，是发挥动态图集价值的关键。

**22.4.5 动态图集与静态图集的关系**

在完整的 UI 架构中，动态图集与静态图集并不是替代关系，而是互补关系。

静态图集的优势在于：

- 构建阶段完成打包
- 运行时零开销
- 合批稳定性高

动态图集的优势在于：

- 运行时灵活扩展
- 适配不确定资源
- 减少碎片化纹理

在实际工程中，通常采用混合策略：

- 核心 UI 使用静态图集
- 扩展 UI 或动态内容使用动态图集

这种组合方式可以在性能与灵活性之间取得平衡。

**22.4.6 动态图集的设计原则**

为了合理使用动态图集，可以遵循以下原则：

- 控制图集数量，避免无限增长
- 设置合理的容量上限，防止内存膨胀
- 优先缓存高频使用 Sprite
- 避免频繁释放与重建图集
- 与资源管理系统协同设计

从系统角度来看，动态图集的核心价值并不在于“减少 DrawCall”，而在于“降低运行时纹理碎片化”，从而为合批提供更稳定的基础条件。

总体而言，动态图集是 UGUI 批处理体系中的重要补充机制。它通过运行时资源整合，增强了 UI 系统的灵活性，但也引入了额外的管理成本。在实际项目中，需要结合场景谨慎使用，才能真正发挥其优化价值。

**本章小结**

本章围绕 UGUI 批处理机制展开，从图集管理、材质策略、合批优化到动态图集四个层面，系统分析了 DrawCall 形成与控制的关键路径。

其次，在材质策略中，可以看到材质一致性是另一个核心限制条件。即使纹理完全一致，只要材质不同，批处理仍然会被拆分。因此，材质统一与共享策略，是控制 DrawCall 数量的重要手段。同时也需要注意材质实例化问题，避免在运行时无意破坏合批结构。

在合批优化部分，本质问题被进一步抽象为“渲染状态连续性”的控制。UGUI 的批处理是严格顺序依赖的，一旦材质、纹理或渲染状态发生变化，就会立即打断当前批次。因此，优化的核心并不是减少 UI 数量，而是减少状态切换频率，使相同状态尽可能连续出现。

最后，在动态图集部分，扩展了图集体系在运行时的应用能力。动态图集通过动态合并 Sprite，实现了对不确定资源的合批支持，在一定程度上缓解了资源碎片化问题。但其运行时开销与内存增长问题，也决定了它更适合作为静态图集的补充，而非替代方案。

总体来看，本章的核心可以归纳为一个关键结论：UGUI 的批处理优化，本质是对“纹理一致性、材质一致性以及渲染状态连续性”的统一控制。

只有同时从资源组织、材质设计与UI结构三个层面进行协同优化，才能真正稳定降低 DrawCall，并发挥 UGUI 批处理机制的最大性能优势。
