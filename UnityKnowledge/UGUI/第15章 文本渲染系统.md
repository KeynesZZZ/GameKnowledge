在 [UGUI](https://zhida.zhihu.com/search?content_id=275431483&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 的整个渲染体系中，文本渲染是最特殊的一类 UI 组件。与 Image 或 RawImage 不同，文本并不是基于固定贴图进行简单映射，而是一个由“语义数据”驱动的动态几何生成过程。也就是说，文本的本质并不是“图像”，而是“字符到网格的实时转换”。

从系统架构来看，文本渲染系统位于 Graphic 体系之下，与 Image、RawImage 并列，但其内部实现复杂度远高于普通 UI 元素。它不仅需要处理字符解析，还需要进行字体映射、排版计算、换行处理以及 Mesh 动态生成，是 UGUI 中最接近“轻量排版引擎”的模块。

在 UGUI 中，文本渲染主要由两套体系构成：基于内置 Text 组件的旧文本系统，以及基于 [TextMeshPro](https://zhida.zhihu.com/search?content_id=275431483&content_type=Article&match_order=1&q=TextMeshPro&zhida_source=entity) 的增强文本系统。前者结构简单但性能有限，后者则引入更复杂的字体图集与排版模型，提供更高质量的文本渲染能力。

从渲染流程来看，文本系统的核心链路可以概括为：字符串输入 → 字符解析 → 字形索引 → UV映射 → Mesh生成 → CanvasRenderer提交。与 Image 的“单四边形映射”不同，文本的 Mesh 是由大量字符逐个构建而成，因此在 CPU 侧具有显著更高的计算成本。

同时，文本系统还需要处理大量与“排版”相关的问题，例如字符间距、行距、对齐方式、富文本解析等。这些逻辑使得文本渲染不仅仅是图形生成问题，更接近一个简化版的排版引擎。

本章将从字体与字形图集开始，逐步分析 UGUI Text 的完整渲染流程，并深入对比 TextMeshPro 的实现方式，最终建立对 Unity 文本渲染体系的完整理解。

**15.1 字体与字形图集**

文本渲染系统的本质，是一个将“字符”转换为“可渲染图形”的过程。

在 UGUI 中，无论是传统的 Text 组件，还是 TextMeshPro，其底层都不会直接渲染字符串本身。

字符串在进入渲染流程后，会被逐字符解析，并转换为对应的字形（Glyph）数据，随后再构建为 Mesh，最终提交给 GPU 完成绘制。

因此，真正参与渲染的并不是文本，而是字符对应的图形数据。

而这一整套文本渲染机制的基础，便是字体（Font）与字形图集（Glyph Atlas）。

**15.1.1 字体文件的本质**

字体文件本质上是一套用于描述字符形状的数据集合。

常见的字体格式包括：

- TrueType Font（TTF）
- OpenType Font（OTF）

这些字体文件内部通常包含以下信息：

- 字符轮廓数据（Outline）
- 字距信息（Spacing）
- 行高信息（Line Height）
- Kerning 数据
- 字形索引表
- Unicode 映射关系

以字符 “A” 为例，在字体文件中，它并不是一张已经绘制好的图片，而是一组用于描述轮廓的矢量曲线数据。

这些轮廓通常由贝塞尔曲线（Bezier Curve）构成。

因此，字体本身并不依赖固定分辨率。

无论字符被放大还是缩小，其轮廓都可以重新计算并生成对应像素，从理论上讲能够始终保持平滑边缘。

这也是矢量字体（Vector Font）与位图字体（Bitmap Font）之间最大的区别。

- 矢量字体保存的是数学曲线
- 位图字体保存的是固定像素图像

因此：

- 矢量字体具备良好的缩放能力
- 位图字体在放大后容易出现锯齿与模糊

在 Unity 中，导入字体文件时，并不会立即生成整套字符纹理。

Unity 实际上只是保存了字体资源的引用以及相关字体信息。

真正的字符光栅化（Rasterization）与纹理生成，通常发生在后续的字形图集（Glyph Atlas）构建阶段。

也就是说： **字体文件负责“描述字符”，字形图集负责“提供可渲染纹理”** 。

当文本系统首次遇到某个字符时，Unity 会根据字体文件中的矢量轮廓，将字符转换为对应的位图字形，并写入字形图集纹理中，供后续渲染直接使用。

这一过程，也是整个文本渲染系统中最核心的基础环节之一。

**15.1.2 什么是字形**

在文本渲染系统中，“字符（Character）”与“字形（Glyph）”并不是同一个概念。

很多初学者容易将二者混为一谈，但实际上，它们分别属于文本系统中的两个不同层级。

字符（Character）是文本层面的逻辑概念。

它本质上是一个 Unicode 编码对应的符号，例如：

- 字符 “A”
- 字符 “中”
- 字符 “你”

这些字符仅仅代表“文本含义”，并不包含真正的图形信息。

换句话说： **Character 负责“表示内容”， Glyph 负责“显示内容”。**

而字形（Glyph），则是真正用于渲染的图形数据。

同一个字符，在不同字体下，往往会对应完全不同的字形外观。

例如，同样都是字符 “A”：

- 宋体中的 “A”
- 黑体中的 “A”
- Arial 中的 “A”

它们虽然属于同一个 Unicode 字符，但实际显示出来的形状却并不相同。

其原因就在于： **字符是抽象编码，字形才是具体图形。**

因此，在真正的文本渲染过程中，参与绘制的并不是 Character，而是 Glyph。

一个完整的字形数据，通常会包含以下信息：

- Glyph Index（字形索引）
- 字形纹理区域（Glyph Rect）
- 字形宽高（Size）
- Bearing 偏移
- Advance 前进距离
- UV 坐标

其中，最核心的数据之一便是 **Advance** 。

Advance（前进距离）用于描述：当前字符绘制完成后，光标应该向前移动多少距离。

例如：当系统绘制完字符 “A” 后，并不会立即在原位置继续绘制下一个字符，而是会根据当前字形的 Advance 值，将绘制位置向右推进，再开始绘制下一个字符。

因此，文本排版的本质，其实就是一个不断累积 Advance 的过程。

最终，所有字符依次排列，形成完整文本。

需要特别注意的是：Advance 并不等同于字形实际宽度。

某些字符虽然图形较窄，但可能会占据较大的 Advance；

而某些特殊字形甚至可能出现负 Bearing 或特殊间距调整。

因此：文本渲染并不是简单的“图片拼接”，而是一套完整的排版计算系统。

**15.1.3 字形图集的作用**

虽然字体文件中保存的是矢量曲线数据，但 GPU 并不能直接渲染这些曲线。

因此，在真正进入渲染阶段之前，文本系统必须先将字形转换为实际的像素图像。

这个过程称为字形栅格化（Rasterization）。

所谓栅格化，本质上就是将字体中的矢量轮廓转换为二维像素。

例如字符 “A” 在字体文件中原本是一组贝塞尔曲线，而经过栅格化之后，便会生成对应的灰度位图。

只有变成位图后，GPU 才能够对其进行纹理采样与渲染。

为了减少纹理切换与渲染开销，Unity 不会为每个字符单独创建纹理，而是会将多个字符生成后的位图统一打包到同一张纹理中。

这张专门用于存储字符位图的纹理，就称为字形图集（Glyph Atlas）。

例如：

- 字符 “A” 可能位于图集左上区域
- 字符 “B” 可能位于图集中部区域
- 字符 “中” 可能位于图集右下区域

每个字符都会在图集中占据一块独立区域。

当文本系统需要渲染字符时，并不会重新生成字符图像，而是直接通过 UV 坐标从图集中找到对应区域，然后完成字符绘制。

因此，文本系统在渲染阶段真正关心的，并不是字符本身，而是：

- 当前字符在图集中的 UV 区域
- 字符对应的顶点坐标
- 当前使用的材质与纹理

随后系统会为字符生成对应的 Mesh 数据，并提交给 GPU。

实际上，字符在底层最终都会被转换为普通的 Quad 面片。

在 UGUI 中，一个字符通常会生成：

- 4 个顶点
- 2 个三角形
- 6 个索引

因此，文本字符数量越多，生成的 Mesh 顶点数量也会越大。

例如：

- 100 个字符，大约对应 400 个顶点
- 1000 个字符，大约对应 4000 个顶点

这也是为什么大量动态文本往往会带来明显的 Canvas 性能压力。

因为文本数量增加后，不仅会提升 GPU 顶点处理开销，还会增加：

- Mesh 构建成本
- Canvas Rebuild 开销
- Batch 合批压力
- 动态顶点上传成本

所以在 UGUI 性能优化中，文本系统始终是一个非常关键的优化方向。

**15.1.4 静态字体图集**

早期的文本渲染系统，大多采用静态字体图集（Static Font Atlas）方案。

所谓静态图集，本质上就是在编辑阶段提前完成字符烘焙，并将所有需要使用的字符一次性生成到图集中。

例如项目在构建字体资源时，可能会提前指定：

- ASCII 字符集
- 常用中文字符集
- 指定 Unicode 范围
- 游戏内固定文本内容

随后文本系统会将这些字符统一栅格化，并生成固定的字体图集。

生成完成后，图集内容在运行期间不会再发生变化。

也就是说运行时不会再动态添加新字符。

这种方案最大的特点，就是“预生成”。

所有字符都已经在编辑阶段准备完成，因此运行时几乎不需要额外的字体生成开销。

静态字体图集的第一个优点，就是运行时性能稳定。

由于字符已经提前烘焙完成，因此文本系统在运行时无需再执行：

- 字形栅格化
- 动态纹理写入
- 图集扩容
- 字符重新布局

GPU 可以直接从现有图集中采样字符纹理进行绘制。

因此整体运行成本较低。

第二个优点，是不会产生动态图集重建问题。

由于字符集合固定，因此图集大小也是固定的。

不会因为运行时出现新字符而导致：

- 图集扩容
- 图集重新打包
- 材质纹理更新
- 文本重新生成 Mesh

这意味着渲染行为更加稳定，也更容易控制性能波动。

第三个优点，是内存更加容易管理。

因为图集大小在编辑阶段已经确定，因此：

- 显存占用可预估
- 内存预算更容易控制
- 平台兼容性更稳定

对于资源受限的平台来说，这一点非常重要。

例如：

- 主机平台
- 掌机平台
- 嵌入式设备
- 低端移动设备

都更倾向于使用可控的静态资源方案。

不过，静态字体图集也存在非常明显的局限性。

尤其是在中文环境下，这种问题会更加突出。

英文字符数量非常少。

即使包含大小写、数字以及符号，常用字符通常也只有数百个。

因此英文项目非常适合静态图集。

但中文字符数量却极其庞大。

例如：

- 常用汉字约 3500 个
- GB2312 字符集超过 6000 个字符
- Unicode CJK 字符集甚至包含数万个汉字

如果一次性将所有中文字符全部烘焙进图集，那么字体纹理会变得非常巨大。

这会直接导致：

- 图集尺寸暴增
- 显存占用增加
- 加载时间增长
- 内存压力提升

甚至可能超过硬件允许的最大纹理尺寸。

因此在实际项目中，静态字体图集通常更适用于以下场景：

- 英文项目
- 字符集合固定的项目
- 控制台 UI
- 小型嵌入式系统
- 文本内容可完全预知的游戏

而对于需要动态输入、大量中文文本、聊天系统、多语言切换的现代项目而言，单纯依赖静态字体图集通常已经难以满足需求。

**15.1.5 动态字体图集**

为了解决字符数量过于庞大的问题，现代文本渲染系统普遍开始采用动态字体图集（Dynamic Font Atlas）方案。

Unity 中的 Dynamic Font，本质上就属于这一类型。

与静态图集不同，动态字体图集不会在编辑阶段一次性生成全部字符。

它的核心思想是：只有当字符真正被使用时，系统才会动态生成对应字形。

也就是说，字符的生成过程被延迟到了运行阶段。

例如，当文本中第一次出现字符 “你” 时，Unity 会立即执行一套动态生成流程。

首先，系统会从字体文件中读取该字符对应的矢量轮廓数据。

随后，文本系统会对字符进行栅格化，将矢量曲线转换为实际像素。

接着，生成后的字符位图会被写入当前字体图集中，并为其分配对应的纹理区域。

完成后，系统会记录：

- 当前字符在图集中的 UV 坐标
- 字形尺寸信息
- Advance 数据
- Glyph Index
- 字符缓存映射关系

这样，当下一次再次使用字符 “你” 时，系统便无需重新生成字形。

而是可以直接从图集中复用已有字符数据。

因此，动态字体图集最大的优势，就是显著降低了初始资源占用。

因为系统只会生成真正使用到的字符，所以：

- 未使用字符不会占用图集空间
- 初始纹理尺寸更小
- 内存占用更加灵活
- 更适合大型字符集语言

这一点对于中文、日文、韩文等 CJK 语言尤其重要。

因为这些语言的字符数量远远超过英文字符集。

如果仍然使用静态图集，往往会导致字体纹理极其庞大。

而动态字体方案则可以根据实际文本内容逐步生成字符，从而避免一次性加载海量字形。

因此，现代 UI 系统、聊天系统、多语言系统，通常都会优先采用动态字体方案。

不过，动态字体图集虽然提升了灵活性，但也引入了新的运行时问题。

因为字符是在运行过程中动态生成的，所以系统需要不断处理：

- 字形栅格化
- 图集写入
- 图集扩容
- UV 更新
- 字符缓存维护

这些操作都可能带来额外的 CPU 与 GPU 开销。

尤其是在首次出现大量新字符时，性能波动会更加明显。

因此，动态字体图集本质上是一种：“以内存换灵活性”的方案。

**15.1.6 图集扩容与重建**

动态字体图集虽然能够有效降低初始内存占用，但它也存在一个非常核心的问题：图集空间始终是有限的。

例如，一张大小为 1024×1024 的字体图集，能够容纳的字符数量并不是无限的。

随着运行过程中不断出现新的字符，图集中的可用区域会越来越少，最终整张纹理会被完全填满。

当图集空间不足时，文本系统通常会采用两种处理方式。

第一种方式是图集扩容。

例如：

- 从 1024×1024 扩展到 2048×2048
- 或继续扩展到更大的纹理尺寸

扩容后，系统可以继续向新的纹理空间中写入字符。

但纹理尺寸越大，意味着：

- 显存占用越高
- 纹理上传成本越大
- GPU 采样压力越高

并且纹理尺寸本身还受到硬件最大纹理限制。

因此扩容并不能无限进行。

第二种方式，则是图集重建（Atlas Rebuild）。

所谓重建，本质上就是：重新整理整个字体图集内容。

系统会重新分配字符位置，并重新生成整张图集中的字符数据。

这意味着：原有字符的 UV 区域很可能全部发生变化。

因此，所有依赖旧 UV 的文本 Mesh 都必须重新更新。

而这也是动态图集最容易引发性能抖动的地方。

因为无论是扩容还是重建，背后都会伴随大量运行时操作。

例如：

- 字形栅格化
- 图集纹理更新
- UV 重新计算
- 文本 Mesh 重建
- GPU 顶点重新上传
- 材质纹理同步更新

这些操作往往会在同一帧内集中发生。

尤其是在首次出现大量新字符时，问题会更加明显。

例如中文聊天系统就是一个典型场景。

当玩家第一次打开聊天界面时，系统可能会瞬间遇到大量此前从未出现过的中文字符。

此时动态字体系统会开始连续生成：

- 中文字形
- 图集纹理
- 字符缓存
- 文本 Mesh

如果短时间内生成字符数量过多，就很容易导致明显卡顿。

因此，很多项目中首次打开聊天窗口、邮件系统、任务系统时出现的瞬时掉帧，本质原因往往并不是 UI 本身复杂。

而是动态字体系统正在首次生成大量字符。

**15.1.7 [Font Texture Rebuild](https://zhida.zhihu.com/search?content_id=275431483&content_type=Article&match_order=1&q=Font+Texture+Rebuild&zhida_source=entity) 机制**

在 Unity 传统 Text 文本系统中，动态字体并不仅仅只是“生成字符”这么简单。

当字体图集发生变化时，整个文本渲染系统还会触发一套非常重要的更新机制。

这个机制就是 Font Texture Rebuild。

在动态字体模式下，只要新的字符被写入图集，就意味着字体纹理内容已经发生变化。

而一旦字体纹理发生变化，原有文本 Mesh 中记录的 UV 坐标，就有可能失效。

因为字符在图集中的位置可能已经改变。

例如：

原本字符 “你” 位于图集左上区域。

但在图集扩容或重建后，它可能被重新排列到了新的位置。

这样一来，旧 Mesh 中保存的 UV 坐标便无法继续正确采样。

如果不重新生成 Mesh，那么字符就会出现：

- 显示错误
- UV 错乱
- 采样异常
- 字体错位

因此，当动态图集更新后，Unity 必须通知所有正在使用该字体的 Text 组件重新生成文本数据。

Unity 内部通过 Font.textureRebuilt 事件完成这一机制。

其整体调用流程大致如下：

```
Font.textureRebuilt
    ↓
FontUpdateTracker
    ↓
Text.FontTextureChanged()
    ↓
SetAllDirty()
    ↓
重新生成 Mesh
```

首先，当字体图集发生变化后，Unity 会触发 Font.textureRebuilt 事件。

随后，FontUpdateTracker 会找到所有正在引用该字体的 Text 组件。

接着，每个 Text 都会调用 FontTextureChanged()。

在该函数内部，Text 会执行 SetAllDirty()。

而 SetAllDirty() 的作用，则是将当前 UI 标记为：

- Layout Dirty
- Vertices Dirty
- Material Dirty

之后 Canvas 系统便会重新执行：

- 文本布局计算
- 顶点生成
- UV 重建
- Mesh 更新

最终重新上传新的顶点数据到 GPU。

因此，动态图集的变化，实际上会影响所有共享该字体的 Text 组件。

这也是为什么：当动态字体频繁生成新字符时，整个 Canvas 往往会出现大量重建。

尤其是在多个 Text 共用同一个动态字体时，这种影响会更加明显。

因为只要字体图集发生一次更新，所有引用该字体的 Text 都可能被迫重新生成 Mesh。

如果界面中存在大量文本组件，那么一次字体重建，就可能引发：

- Canvas Rebuild 激增
- CPU 峰值抖动
- Batch 重新合并
- GPU 顶点重新上传

因此在实际项目中动态字体并不仅仅影响文本系统本身，它还会间接影响整个 UGUI 的渲染性能结构。

**15.1.8 字形缓存机制**

在动态字体系统中，字符生成本身是一项成本较高的操作。

因为每当系统首次遇到一个新字符时，都需要执行：

- 字形查找
- 矢量轮廓解析
- 字形栅格化
- 图集写入
- UV 分配
- 字符信息记录

如果每次出现字符都重新执行这些流程，那么文本系统的性能开销将会非常巨大。

因此，为了避免重复生成字符，现代文本系统通常都会维护一套字形缓存机制（Glyph Cache）。

所谓字形缓存，本质上就是：

将已经生成过的字形数据保存下来，供后续直接复用。

当某个字符第一次生成完成后，系统会将对应的字形信息记录到缓存中。

缓存内容通常包括：

- Glyph Index
- 字形 UV 区域
- 字形尺寸信息
- Advance
- Bearing
- 图集位置映射

这样，当同一个字符再次出现时，系统便无需重新进行栅格化。

而是可以直接从缓存中读取已有字形数据。

这意味着：

- 不需要重新生成字符纹理
- 不需要重新写入图集
- 不需要重新计算 UV
- 不需要重新创建 Glyph

因此，字形缓存命中率越高，文本系统运行就越稳定。

尤其是在大量动态文本场景中，缓存机制的重要性会更加明显。

例如：

- 聊天系统
- 战斗飘字
- 数值刷新
- 任务文本
- 弹幕系统

这些场景都会频繁重复使用相同字符。

如果缓存命中率较高，那么绝大部分字符都可以直接复用已有图集数据，从而避免频繁触发动态图集更新。

因此，在大型项目中，通常都会主动进行字体预热（Font Warmup）。

所谓字体预热，本质上就是：

在游戏正式运行前，提前生成常用字符。

这样做的目的，是避免游戏运行过程中首次出现字符时触发动态图集构建。

常见的预热内容包括：

- 数字字符
- UI 常用英文
- 常见中文
- 聊天系统字符
- 属性面板文本

例如很多项目会提前生成：

```
0123456789
HP
MP
LEVEL
START
EXIT
```

因为这些字符在游戏中几乎一定会被频繁使用。

提前缓存后，后续文本渲染便可以直接复用已有字形数据，而不会再触发运行时字形生成。

这不仅能够减少运行时 CPU 开销，还可以有效避免：

- Font Texture Rebuild
- Canvas 频繁重建
- 动态图集扩容
- 首次打开界面卡顿

因此，字体预热本质上也是一种非常重要的 UI 性能优化手段。

**15.1.9 [SDF 字体图集](https://zhida.zhihu.com/search?content_id=275431483&content_type=Article&match_order=1&q=SDF+%E5%AD%97%E4%BD%93%E5%9B%BE%E9%9B%86&zhida_source=entity)**

TextMeshPro 所使用的字体图集，与传统 UGUI Text 使用的普通位图字体并不相同。

传统字体图集保存的，通常是字符最终的像素颜色信息。

而 TextMeshPro 使用的，则是 Signed Distance Field（SDF，有符号距离场）字体。

SDF 字体最大的特点在于：

纹理中保存的并不是字符颜色，而是“距离信息”。

具体来说，图集中每一个像素都会记录：

当前像素距离字符边缘还有多远。

例如：

- 位于字符内部的像素，会记录为内部距离
- 位于字符边缘附近的像素，会记录为接近边界
- 位于字符外部的像素，则会记录为外部距离

因此，SDF 纹理本质上保存的是一种距离场数据，而不是普通位图。

在真正渲染时，Shader 会根据这些距离信息，动态计算字符边缘的位置。

随后再实时决定：

- 哪些区域应该显示
- 哪些区域应该透明
- 边缘应该如何过渡
- 抗锯齿如何处理

由于字符边缘并不是直接写死在纹理中的，因此即使字体被大幅放大，Shader 依然能够重新计算清晰边缘。

这也是 SDF 字体最核心的优势。

传统位图字体在放大后，往往会出现：

- 模糊
- 锯齿
- 边缘失真
- 像素颗粒感

因为普通位图本质上只是固定分辨率的像素图像。

一旦超出原始尺寸，像素便会被强行拉伸。

而 SDF 字体由于保存的是距离场，因此即使放大很多倍，字符边缘依然能够保持相对锐利。

这也是为什么 TextMeshPro 的文本清晰度明显高于传统 UGUI Text。

尤其是在：

- 高清 UI
- 4K 分辨率
- 世界空间文本
- 大字号标题
- 摄像机缩放场景

中，这种差异会更加明显。

除此之外，SDF 字体还有一个非常重要的优势：

大量字体效果都可以直接在 Shader 中实时计算。

例如：

- 描边（Outline）
- 阴影（Shadow）
- 发光（Glow）
- Softness 柔化
- 边缘羽化
- 外轮廓扩展

这些效果都不需要重新生成字体纹理。

Shader 可以直接基于距离场信息动态完成计算。

因此：

同一套字体图集，便能够支持大量不同视觉效果。

这不仅提升了文本质量，也大幅降低了字体资源数量。

因为传统字体如果需要：

- 描边版本
- 阴影版本
- 粗体版本

往往需要额外生成不同纹理。

而 SDF 则可以通过材质参数动态调整。

因此，TextMeshPro 不仅拥有更高的文本清晰度，同时也具备更强的字体表现能力。

这也是它逐渐取代传统 UGUI Text 的核心原因之一。

**15.1.10 字体图集与 [DrawCall](https://zhida.zhihu.com/search?content_id=275431483&content_type=Article&match_order=1&q=DrawCall&zhida_source=entity)**

字体图集本质上仍然是一张普通纹理。

因此，对于 GPU 来说，文本渲染与普通图片渲染并没有本质区别。

文本最终同样会：

- 使用材质
- 绑定纹理
- 提交 Mesh
- 执行 DrawCall

所以，文本是否能够成功合批，本质上依然取决于渲染状态是否一致。

其中最关键的一项，就是字体图集是否相同。

如果两个 Text 组件同时满足以下条件：

- 使用相同材质
- 使用相同字体图集
- 使用相同 Shader
- 渲染状态一致

那么它们就有机会进入同一个 Batch。

GPU 便可以一次性完成绘制，从而减少 DrawCall 数量。

但只要其中任意一项发生变化，都可能导致 Batch 被打断。

例如：

- 使用不同字体
- 使用不同材质
- 使用不同字体图集
- 使用不同 Shader 参数

都会导致文本无法继续合批。

尤其是在动态字体系统中，这种问题会更加明显。

因为动态图集并不是固定不变的。

当运行过程中不断生成新字符时，字体系统可能会发生：

- 图集扩容
- 图集重建
- 字符重新排列
- 材质纹理更新

而这些变化，都可能影响原有 Batch 结构。

例如在图集重建后：原本使用的字体纹理可能已经发生变化。

此时：

- 材质引用可能更新
- 字体纹理可能重新绑定
- Batch 状态可能被重新拆分

最终导致额外 DrawCall 增加。

因此，动态字体不仅仅会带来 CPU 侧的文本重建问题，同时还会影响 GPU 侧的批处理效率。

这也是为什么：文本系统往往是 UGUI DrawCall 的高频来源之一。

尤其是在大量动态文本场景中，这种问题会更加突出。

例如：

- 聊天系统
- 排行榜
- 弹幕系统
- 战斗飘字
- 实时数值刷新
- 多语言动态文本

这些系统都会频繁更新文本内容。

而文本一旦频繁变化，就可能不断触发：

- Mesh 重建
- Canvas Rebuild
- Batch 拆分
- 动态字体更新

最终导致：

- DrawCall 增加
- CPU 开销上升
- GPU 提交压力增大

因此在实际项目中，文本系统往往是 UI 性能优化中的重点区域。

很多复杂 UI 的性能瓶颈，最终都并不是图片，而是大量动态文本。

**15.2 动态字体机制**

动态字体机制，是 Unity 文本系统中最重要的底层功能之一。

它解决了“大字符集无法一次性全部烘焙”的问题。

尤其是在中文、日文、韩文等 Unicode 字符数量极大的语言环境下，动态字体几乎成为现代 UI 系统的标准方案。

传统静态字体图集通常只能适用于英文界面与固定字符系统，而在实际项目中，文本内容往往是动态变化的。

例如玩家昵称、聊天系统、公告内容、本地化语言、网络文本与热更新文本。

这些内容在开发阶段通常无法完全预知。

因此 Unity 引入了 Dynamic Font 机制，它允许系统在运行时动态生成字符，并自动写入字体图集。

**15.2.1 Dynamic Font 的本质**

Unity 的动态字体，本质上仍然基于 TrueType 或 OpenType 字体文件。

例如 Arial.ttf、simhei.ttf、SourceHanSans.otf。

这些字体文件内部保存的是矢量曲线数据。

Unity 并不会直接渲染这些曲线，而是在真正需要某个字符时，才进行字符生成。

整个流程包括读取 Glyph、字符栅格化、写入字体图集、生成 UV 信息以及更新文本 Mesh。

因此动态字体最核心的特点，就是按需生成字符。

这种机制大幅降低了初始内存占用。

因为系统不会一次性生成全部字符。

**15.2.2 Font.RequestCharactersInTexture**

Unity 传统 Text 系统中，动态字体生成的核心 API 是 Font.RequestCharactersInTexture。

它的作用是主动请求字体生成指定字符。

例如：

```csharp
font.RequestCharactersInTexture("你好世界");
```

系统会检查字符是否已经存在于图集中。

如果不存在，则会自动生成对应字形。

字符生成完成后，即可通过 Font.GetCharacterInfo 获取字符信息。

例如：

```csharp
CharacterInfo info;
font.GetCharacterInfo('你', out info);
```

CharacterInfo 内部保存了字符 UV、顶点坐标、Advance 与字形尺寸等数据。

Text 组件最终就是基于这些数据生成文本 Mesh。

**15.2.3 字符请求流程**

当 Text 组件需要渲染字符串时，会逐字符检查。

例如字符串：

```csharp
HELLO
```

系统内部会依次处理：

```csharp
H
E
L
L
O
```

对于每一个字符，系统都会先查询缓存。

如果字符不存在于图集，则会请求生成。

生成完成后，再返回对应的 CharacterInfo 数据。

因此首次出现新字符时，通常会产生额外性能开销。

而已经缓存过的字符，则只需要直接读取。

这也是为什么首次打开聊天界面会卡顿，而后续会变得流畅。

因为第一次需要生成大量字符，后续则全部命中缓存。

**15.2.4 字体图集生成过程**

动态字体图集的生成，本质上是一个字符打包过程。

Unity 内部会维护一张字体纹理。

例如：

```
1024 × 1024
```

当字符生成后，会被依次放入空闲区域。

例如：

```
[A][B][C]
[D][E][F]
[中][文][字]
```

系统内部通常会使用矩形打包算法。

其核心目标，是尽可能减少纹理浪费。

字符越多，空闲区域越少。

Packing 难度也会不断增加。

因此中文项目中的字体图集通常增长极快。

**15.2.5 图集扩容机制**

当字体图集空间不足时，Unity 会尝试扩容。

例如：

```csharp
512 × 512
    ↓
1024 × 1024
    ↓
2048 × 2048
```

扩容后，需要创建新的纹理对象。

随后系统会拷贝旧字符、重新写入 Glyph、更新 UV，并重新上传 GPU。

这是一个开销非常昂贵的过程。

尤其在移动平台中，CPU 与 GPU 都会产生明显负担。

因此大型项目通常会限制字体图集最大尺寸，避免字体无限扩容。

**15.2.6 Font.textureRebuilt 事件**

动态字体最核心的问题之一，是图集变化后所有 UV 都可能失效。

因此 Unity 提供了 Font.textureRebuilt 事件。

当字体纹理发生变化时，所有引用该字体的 Text 组件，都需要重新生成 Mesh。

Unity 内部调用链如下：

```
Font.textureRebuilt
    ↓
FontUpdateTracker
    ↓
Text.FontTextureChanged()
    ↓
SetAllDirty()
    ↓
CanvasUpdateRegistry
    ↓
Rebuild()
```

因此动态字体更新，本质上会触发 UI 系统级联刷新。

这也是 UGUI 中非常典型的性能问题来源。

**15.2.7 Dynamic Font 与 Canvas 重建**

动态字体与 Canvas 系统耦合非常严重。

因为 Text 属于 Graphic。

而 Graphic 一旦 Dirty，就会触发 Layout 重建、Vertex 重建以及 Material 重建。

因此新增一个字符，最终甚至可能导致整个 Canvas 重建。

特别是在大型聊天界面、滚动排行榜、弹幕系统以及实时日志窗口中。

这些场景都会频繁触发动态字体更新。

因此实际项目中，经常需要拆分 Canvas。

动态文本区域通常也会被单独放置。

否则一次字体更新，可能导致整个 UI 全部刷新。

**15.2.8 字符缺失机制**

当请求字符时，字体文件未必一定存在对应 Glyph。

例如英文字体请求中文字符，或者普通字体请求 Emoji。

如果字体中不存在对应字形，则会出现空白字符或者 Missing Glyph。

Unity 内部通常会返回一个占位字符。

例如：

```
□
```

这也是为什么本地化系统中，经常需要多字体回退机制。

否则部分语言可能无法正确显示。

**15.2.9 字体回退机制**

为了解决字符缺失问题，现代文本系统通常支持 Font Fallback。

例如：

```
主字体：
Arial

回退字体：
思源黑体
Emoji 字体
日文字体
韩文字体
```

字符查询流程通常会先从主字体查找。

若失败，则继续查找 Fallback 字体。

如果仍然失败，则继续向后查找。

最终若全部失败，才返回 Missing Glyph。

TextMeshPro 在这方面支持更加完整。

其内部已经实现了 Font Asset Fallback、Dynamic Atlas 与多级字体回退机制。

而传统 UGUI Text 对这部分支持相对较弱。

**15.2.10 动态字体的性能问题**

动态字体虽然灵活，但其代价也非常明显。

第一个问题是字符首次生成卡顿。

因为首次生成字符时，需要进行字体解析、Glyph 栅格化以及 Texture Upload。

因此 CPU 与 GPU 都会产生瞬时性能开销。

第二个问题是图集重建。

当图集空间不足时，纹理扩容、UV 更新与 Mesh 重建都会引发性能波动。

第三个问题是 Canvas Rebuild。

字体纹理变化后，所有 Text 都需要重新生成顶点数据。

Canvas 也需要重新 Batch。

这是 UGUI 中非常高频的性能问题。

第四个问题是内存增长。

中文项目字符数量巨大。

动态字体长期运行后，图集会越来越大。

显存占用也会持续增长。

尤其聊天系统最容易出现这种问题。

**15.2.11 动态字体预热**

为了避免运行时首次生成字符导致卡顿，很多项目都会进行字体预热。

例如游戏启动时：

```csharp
font.RequestCharactersInTexture(
    "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ你好世界开始退出"
);
```

这样可以提前生成常用字符。

从而减少后续运行时卡顿。

大型项目通常会预热数字字符、常用 UI 文本以及聊天常用字。

尤其数字字符的收益最明显。

因为 HP、MP、Damage、Time 与 FPS 都属于高频变化文本。

提前缓存能够有效降低运行时开销。

**15.2.12 动态字体机制的局限性**

传统 Dynamic Font 机制虽然解决了字符数量问题。

但其底层架构存在天然缺陷。

例如图集频繁变化、UV 不稳定、Canvas 级联刷新以及文本质量有限。

尤其字体放大后，还容易出现模糊问题。

因此 Unity 后期逐渐转向 TextCore 与 TextMeshPro。

尤其 TextMeshPro 的 SDF 技术，能够提供更加稳定、更加清晰的文本渲染效果。

目前现代 Unity 项目中，传统 UGUI Text 已经越来越少作为主文本方案使用。

它更多作为兼容性组件存在。

**15.3 UGUI Text 渲染流程**

UGUI Text 的渲染流程，本质上是一个“字符串转换为 UI Mesh”的过程。

很多开发者认为 Text 只是简单绘制文字，但实际上整个流程会经历文本解析、字体请求、字符排版、顶点生成、Mesh 构建、Canvas 合批以及 GPU 渲染等多个阶段。

最终字符并不是直接绘制到屏幕，而是先转换为 Quad Mesh，再交由 CanvasRenderer 提交给 GPU。

同时 Text 也不是一个独立渲染系统。

它本质上仍然属于 UGUI Graphic 体系的一部分。

因此它会完整参与 Canvas 的刷新与重建流程。

这也是 Text 性能问题频繁出现的根本原因。

**15.3.1 Text 继承结构**

UGUI Text 的核心类为 UnityEngine.UI.Text。

其继承结构如下：

```
UIBehaviour
    ↓
Graphic
    ↓
MaskableGraphic
    ↓
Text
```

这里最重要的是 Graphic。

因为这意味着 Text 本质上也是一个 Mesh 生成器。

它最终仍然需要生成 Vertex、Triangle、UV 与 Color 数据。

然后再提交给 CanvasRenderer。

因此 Text 与 Image 在底层渲染结构上其实非常相似。

区别仅仅在于 Image 使用 Sprite，而 Text 使用字体图集。

**15.3.2 Text 内容修改流程**

当修改 Text.text 时：

```csharp
text.text = "Hello";
```

内部实际上会进入 Text 的 Setter。

其核心逻辑如下：

```csharp
m_Text = value;
SetVerticesDirty();
SetLayoutDirty();
```

这里有两个非常关键的 Dirty 标记。

第一个是 Vertex Dirty。

它表示文本顶点需要重新生成。

第二个是 Layout Dirty。

它表示文本布局尺寸可能发生变化。

例如从单字符变为长文本时，RectTransform 尺寸可能需要重新计算。

因此 Text 修改不仅仅只是更新字符。

它甚至可能引发整个 UI Layout 系统重建。

**15.3.3 Dirty 延迟刷新机制**

当 Text 被标记 Dirty 后，并不会立刻重新生成 Mesh。

UGUI 使用的是延迟刷新机制。

Text 会被注册到 CanvasUpdateRegistry，等待下一帧统一刷新。

其核心流程如下：

```
SetVerticesDirty()
    ↓
CanvasUpdateRegistry.RegisterCanvasElementForGraphicRebuild()
```

这样做可以避免一帧内重复刷新。

例如：

```csharp
text.text = "A";
text.text = "B";
text.text = "C";
```

如果没有延迟机制，系统会连续生成三次 Mesh。

而现在只会在下一帧统一刷新一次。

**15.3.4 Rebuild 阶段**

当 Canvas 进入更新阶段时，会调用 ICanvasElement.Rebuild。

对于 Text 来说，最终会进入：

```csharp
Graphic.Rebuild()
```

随后继续进入：

```csharp
UpdateGeometry()
```

真正开始生成文本 Mesh。

**15.3.5 TextGenerator 的作用**

UGUI Text 的核心文本生成器为 TextGenerator。

它负责字符解析、文本排版、换行计算、顶点生成以及 UV 生成。

TextGenerator 是传统 UGUI 文本系统最核心的类。

其内部会调用 Unity 原生文本引擎。

例如：

```csharp
cachedTextGenerator.Populate()
```

Populate 的作用，是将字符串转换为可渲染顶点数据。

这是整个文本渲染最核心的一步。

**15.3.6 Populate 内部流程**

Populate 内部会进行大量文本计算。

首先系统会逐字符解析 Unicode。

例如：

```csharp
H
E
L
L
O
```

随后系统会检查字符是否存在于字体图集中。

若不存在，则会请求动态字体生成。

之后系统开始进行排版计算。

其中包括行高、Advance、Kerning、对齐方式以及自动换行。

排版完成后，系统会开始生成顶点数据。

每个字符最终都会生成一个 Quad。

例如字符：

```
A
```

最终会生成：

```
4 个顶点
2 个三角形
6 个索引
```

之后系统再根据字体图集中的字符区域生成对应 UV。

最后再将 Text.color 写入 Vertex Color。

**15.3.7 每个字符的 Mesh 结构**

UGUI Text 中，每个字符本质上都是一个 Quad。

结构如下：

```
v0 ----- v1
 |     / |
 |   /   |
 | /     |
v2 ----- v3
```

对应两个三角形：

```
v0 v1 v2
v2 v1 v3
```

因此：

- 1 个字符等于 4 个顶点
- 100 个字符等于 400 个顶点
- 1000 个字符等于 4000 个顶点

这也是为什么大量文本会迅速增加 Canvas 顶点数量。

**15.3.8 VertexHelper 构建**

TextGenerator 生成的数据，最终会写入 VertexHelper。

VertexHelper 本质上是一个 Mesh 构建辅助器。

它内部维护：

- Vertices
- UV
- Color
- Indices

随后再统一生成 Mesh。

整体流程如下：

```
TextGenerator
    ↓
VertexHelper
    ↓
Mesh
    ↓
CanvasRenderer
```

**15.3.9 CanvasRenderer 提交**

当 Mesh 生成完成后，最终会提交给 CanvasRenderer。

CanvasRenderer 是 UGUI 与底层渲染系统之间的桥梁。

它负责保存 Mesh、保存 Material、保存 Texture，并最终提交 DrawCall。

随后进入 Canvas Batch 阶段，由 Canvas 系统统一进行合批。

**15.3.10 Text 合批机制**

Text 是否能够合批，核心取决于是否使用同一字体图集、同一材质以及同一 Shader。

例如：

```
Text A → Arial
Text B → Arial
```

通常可以进入同一个 Batch。

但如果：

```csharp
Text A → Arial
Text B → 宋体
```

则一定无法合批。

因为字体图集不同。

此外动态字体发生重建时，也可能导致 Batch 被打断。

因此文本系统往往是 UI DrawCall 的主要来源之一。

**15.3.11 Font.textureRebuilt 的影响**

当动态字体图集发生变化时，Unity 会触发 Font.textureRebuilt。

随后所有引用该字体的 Text 都会收到通知。

然后执行：

```
Text.FontTextureChanged()
```

内部最终会调用：

```
SetAllDirty()
```

这意味着：

- Layout Dirty
- Vertex Dirty
- Material Dirty

都会全部触发。

最终导致整个文本重新生成。

如果多个 Text 共用同一个动态字体。

一次字符生成，甚至可能导致整个 Canvas 重建。

**15.3.12 Best Fit 的性能问题**

UGUI Text 提供了 Best Fit 功能。

它的作用是自动调整字体大小。

例如文本超出区域时，系统会自动减小字体。

但其内部实现开销非常大。

因为它会不断循环尝试不同字体大小。

例如：

```
size = 40
size = 39
size = 38
...
```

每次尝试都会重新 Populate。

直到文本能够放入 RectTransform。

因此 Best Fit 会导致多次排版、多次顶点生成以及多次布局计算。

这是 UGUI Text 中非常典型的高开销功能。

大型项目通常都会禁用。

**15.3.13 Rich Text 解析**

UGUI Text 支持 Rich Text。

例如：

```csharp
<b>Hello</b>
<i>World</i>
<color=red>Text</color>
```

在 Populate 阶段，系统会先进行 Tag 解析。

随后再修改字符样式。

例如 Bold、Italic、Color 与 Size。

但 Rich Text 会增加额外解析开销。

因为系统需要维护 Tag Stack、样式状态以及字符样式切换。

尤其复杂嵌套标签，会明显增加 CPU 消耗。

**15.3.14 UGUI Text 的核心局限**

传统 UGUI Text 的底层架构存在明显局限性。

第一个问题是 CPU 开销较高。

因为文本排版与顶点生成全部在 CPU 完成。

第二个问题是动态图集不稳定。

字体纹理变化后，会导致大量 Text 重建。

第三个问题是字体放大后容易模糊。

因为传统 Text 本质上属于位图字体。

第四个问题是大文本性能较差。

字符越多，顶点数量增长越快。

因此在现代 Unity 项目中，UGUI Text 已经越来越少作为主文本方案使用。

更多情况下，项目会使用 TextMeshPro 替代传统 Text。

**15.4 TextMeshPro 渲染机制**

TextMeshPro 是 Unity 现代文本系统的核心方案。

相比传统 UGUI Text，TextMeshPro 拥有更高的文本质量、更稳定的动态图集以及更完整的排版能力。

传统 Text 基于旧版 TextGenerator。

而 TMP 则完全重构了整个文本系统。

其底层基于：

- TextCore
- SDF 字体
- Shader 文本渲染
- 动态字体资产
- 高级排版系统

因此 TMP 并不仅仅只是“更清晰的 Text”。

它本质上是一整套现代化文本渲染框架。

**15.4.1 TMP 的核心架构**

TextMeshPro 的核心类为 TMP\_Text。

其主要继承结构如下：

```
UIBehaviour
    ↓
MaskableGraphic
    ↓
TMP_Text
    ↓
TextMeshProUGUI
```

其中最核心的是 TMP\_Text。

它内部封装了：

- 文本解析
- 字符缓存
- 排版系统
- 字形查询
- Mesh 生成
- 富文本解析
- SubMesh 管理

相比传统 Text。

TMP 拥有更加完整且独立的文本渲染体系。

**15.4.2 TMP 的本质**

TextMeshPro 的本质，仍然是 Mesh 渲染。

它并不会直接绘制字符串，而是将每一个字符转换为 Quad Mesh。

最终再提交给 CanvasRenderer。

因此：

- 每个字符仍然是 4 个顶点
- 每个字符仍然是 2 个三角形
- 本质上仍然属于 UI Mesh

但 TMP 与传统 Text 最大的区别，在于字体渲染方式。

传统 Text 使用普通位图字体，而 TMP 使用 Signed Distance Field 字体。

**15.4.3 SDF 字体原理**

SDF 的全称为 Signed Distance Field，即“有符号距离场”。

普通位图字体保存的是颜色信息，而 SDF 字体保存的是距离信息。

它记录当前像素距离字符边缘有多远。

例如：

```
内部区域 → 正值
边缘区域 → 0
外部区域 → 负值
```

Shader 会根据距离动态计算字符边缘。

因此即使字体被大幅放大，边缘依然能够保持锐利。

相比普通位图字体：

- 普通字体放大会模糊
- SDF 字体放大后依然清晰

这也是 TMP 文本质量远高于传统 Text 的根本原因。

**15.4.4 SDF Shader 渲染**

TMP 的文本渲染高度依赖 Shader。

字符边缘、描边、阴影、Glow 等效果，全部由 Shader 实时计算。

例如 Outline。

传统 Text 通常需要：

- 多层文本
- 多次渲染
- 偏移绘制

而 TMP 只需要：

```
Outline Width
Outline Color
```

即可在 Shader 中完成描边。

因此 TMP 能够以极低 DrawCall 成本实现复杂文本效果。

**15.4.5 TMP\_FontAsset**

TextMeshPro 并不直接使用 Font，而是使用 TMP\_FontAsset。

它本质上是字体资源的封装对象。

内部包含：

- Glyph Table
- Character Table
- Atlas Texture
- Font Metrics
- Kerning Data
- Fallback 信息

因此 TMP 实际使用的是 Font Asset，而不是原始字体文件。

**15.4.6 Font Asset 构建流程**

当导入字体时，TMP 会先读取字体文件。

随后：

1. 提取 Glyph
2. 栅格化字符
3. 生成 SDF 数据
4. 打包 Atlas
5. 构建 Character Table
6. 保存 Font Asset

最终形成 TMP\_FontAsset。

因此 TMP 字体本质上是一套预处理资源，而不是运行时即时生成。

**15.4.7 TMP 文本更新流程**

当修改 TMP\_Text.text 时：

```csharp
tmp.text = "Hello";
```

内部会触发：

```csharp
SetVerticesDirty()
SetLayoutDirty()
```

随后 TMP 会进入：

```csharp
GenerateTextMesh()
```

这是 TMP 最核心的函数。

它的作用是将字符串转换为最终 Mesh 数据。

**15.4.8 GenerateTextMesh 流程**

GenerateTextMesh 内部流程非常复杂。

但核心可以分为几个阶段。

首先系统会解析 RichText 标签。

例如：

```
<b>
<i>
<color>
<size>
<link>
<sprite>
```

随后 TMP 会维护 Style Stack。

之后系统开始逐字符查找 Character Table 与 Glyph Table。

如果字符不存在，则会继续查找 Fallback Font Asset。

然后系统开始进行排版计算。

其中包括：

- Kerning
- Word Wrapping
- Alignment
- Baseline
- Ascender
- Descender

随后 TMP 开始生成顶点数据。

每个字符最终会生成：

```
4 Vertices
2 Triangles
UV
Color
Normal
Tangent
```

相比传统 Text。

TMP 会生成更多 Vertex 数据。

因为 Shader 需要更多信息。

**15.4.9 TMP 的字符缓存机制**

TMP 内部维护了大量缓存。

包括：

- Glyph Cache
- Character Lookup
- Material Reference
- Mesh Info

因此绝大多数字符查询都属于 O(1)。

相比传统 Text 的频繁字体查询。

TMP 的缓存体系更加稳定。

**15.4.10 Dynamic Atlas 机制**

TMP 同样支持动态图集。

当字符不存在时。

系统会：

1. 从字体文件读取 Glyph
2. 生成 SDF
3. 写入 Atlas
4. 更新 Character Table

这套机制与传统 Dynamic Font 类似。

但 TMP 的动态图集系统更加稳定。

因为其 Atlas 管理与 TextCore 深度整合。

**15.4.11 Multi Atlas 机制**

传统 UGUI Text 通常只有一张字体图集。

而 TMP 支持 Multi Atlas。

例如：

```
Atlas 0
Atlas 1
Atlas 2
```

当一张图集空间不足时，TMP 会自动创建新图集，而不是强制重建整个 Atlas。

这极大降低了动态图集带来的性能波动。

**15.4.12 TMP Sprite Asset**

TMP 支持在文本中插入 Sprite。

例如：

```
Hello <sprite=0>
```

系统会自动将 Sprite 转换为 Quad，并插入文本 Mesh。

因此 Emoji、表情与 Icon 都可以直接嵌入文本。

这是传统 UGUI Text 很难实现的功能。

**15.4.13 TMP\_SubMeshUI**

当文本中存在不同材质时，TMP 会自动生成 TMP\_SubMeshUI。

例如：

- Sprite 使用独立材质
- Fallback Font 使用独立材质
- 特殊效果使用独立材质

每个 SubMesh 都会形成独立 DrawCall。

因此复杂文本仍然可能增加渲染开销。

**15.4.14 TMP 自动换行系统**

TMP 内部拥有完整自动换行算法。

其核心会结合：

- 单词边界
- Unicode Line Breaking
- 空格规则
- CJK 换行规则

因此 TMP 在中英文混排时，效果明显优于传统 Text，尤其中文自动换行更加稳定。

**15.4.15 TMP 的性能优势**

相比传统 Text。

TMP 最大优势包括：

**一、更高的文本质量**

SDF 字体即使大字号依然清晰。

**二、更稳定的动态图集**

TMP 支持 Multi Atlas，避免整张图集频繁重建。

**三、更完整的 RichText**

TMP 支持更多标签与样式。

**四、更先进的排版系统**

TMP 拥有完整 Typography 支持。

**五、更强的 Shader 效果**

Outline、Glow 与 Underlay 都可以实时计算。

**15.4.16 TMP 的性能问题**

虽然 TMP 优势明显，但它也并非完全没有代价。

第一个问题是 Vertex 数据更多。

TMP 通常比传统 Text 顶点结构更加复杂。

第二个问题是 Shader 更重。

SDF Shader 的消耗通常高于普通 UI Shader。

第三个问题是复杂 RichText 开销明显。

尤其大量标签嵌套时，CPU 解析压力会增加。

第四个问题是 SubMesh 增多。

多材质文本会增加 DrawCall。

**15.4.17 TextCore 架构**

Unity 后期已经将 TMP 底层逐渐整合进 TextCore。

TextCore 属于 Unity 新一代文本框架。

目前：

- TMP
- UI Toolkit
- 新文本系统

都在逐渐统一到 TextCore 架构下。

这意味着 TMP 已经不再只是一个插件，而是 Unity 官方现代文本系统的重要组成部分。

**15.4.18 为什么现代项目更倾向 TMP**

现代 Unity 项目中，绝大部分 UI 系统已经默认使用 TextMeshPro。

因为相比传统 Text。

TMP 拥有：

- 更稳定的动态图集
- 更强的文本质量
- 更完整的排版系统
- 更先进的 Shader 文本效果
- 更成熟的国际化支持

因此传统 UGUI Text 已经逐渐退出主流项目。

更多情况下，它仅用于兼容旧工程。

**15.5 字符排版与换行算法**

字符排版，是文本渲染系统中最核心的组成部分之一。

因为文本系统不仅仅只是“绘制字符”。

它还需要决定字符放在哪里、行如何排列、什么时候换行以及不同语言如何布局。

因此文本排版本质上属于一套布局算法，无论是传统 UGUI Text，还是 TextMeshPro。

最终都必须经过排版阶段，才能生成最终 Mesh。

**15.5.1 文本排版的本质**

文本排版的核心，本质上是计算每个字符的最终坐标。

例如字符串：

```
HELLO
```

系统最终需要得到：

```
H → x=0
E → x=20
L → x=40
L → x=60
O → x=80
```

随后再根据坐标生成 Quad Mesh。

因此文本排版，本质上其实是字符坐标计算。

**15.5.2 Advance 前进距离**

字符排版最核心的数据之一，就是 Advance。

Advance 表示当前字符绘制结束后，光标应该向前移动多少距离。

例如：

```
A → Advance = 20
B → Advance = 18
C → Advance = 19
```

文本系统会不断累加 Advance。

例如：

```csharp
CursorX += Advance
```

从而确定下一个字符位置。

因此文本横向布局，本质上就是不断累加字符宽度。

**15.5.3 Bearing 偏移**

除了 Advance，排版系统还需要考虑 Bearing。

Bearing 表示字符相对于基线的偏移。

例如：

```
g
p
y
```

这些字符通常会低于 Baseline。

而：

```
A
H
T
```

则主要位于 Baseline 上方。

因此字符真正绘制时。

都会包含：

- X Bearing
- Y Bearing

否则字符会出现上下错位。

**15.5.4 Baseline 基线**

Baseline 是文本排版中的核心概念。

它表示字符对齐的参考线。

例如：

```
HELLO
world
```

虽然字符大小不同，但它们会共享同一条 Baseline。

这样字符才能正确对齐。

例如：

- 英文大写
- 英文小写
- 中文
- 数字

都需要围绕 Baseline 布局。

**15.5.5 Ascender 与 Descender**

排版系统中，字符通常分为：

- 上升区域
- 基线区域
- 下降区域

例如：

```
Ascender
Baseline
Descender
```

其中：

- h
- k
- l

通常拥有较高 Ascender。

而：

- g
- p
- y

通常拥有 Descender。

文本系统需要根据这些数据计算行高与垂直布局。

否则字符会互相重叠。

**15.5.6 Kerning 字距调整**

Kerning 表示字符之间的特殊间距调整。

例如：

```
AV
```

如果直接使用默认 Advance。

视觉上会显得距离过大。

因此字体会提供 Kerning 数据。

使字符之间产生特殊偏移。

例如：

```
A 向右缩进
V 向左靠近
```

最终让字符组合更加自然。

优秀字体通常会包含大量 Kerning Pair。

例如：

- AV
- To
- WA
- Yo

这些组合都会进行特殊处理。

**15.5.7 Word Wrapping 自动换行**

自动换行，是文本排版中最核心的算法之一。

其目标是保证文本不会超出显示区域。

例如：

```
Rect Width = 200
```

当文本长度超出宽度时，系统就需要换行。

最基础的换行算法通常如下：

1. 逐字符累加宽度
2. 判断是否超出区域
3. 若超出，则寻找换行点
4. 移动到下一行

例如：

```
HELLO WORLD
```

可能最终变成：

```
HELLO
WORLD
```

**15.5.8 英文换行规则**

英文排版通常基于单词换行。

例如：

```
HELLO WORLD UNITY
```

系统通常不会：

```
HELLO WO
RLD UNITY
```

因为这会破坏单词结构。

因此英文换行系统通常会记录空格位置。

当文本超出宽度时，系统会回退到最近空格进行换行。

**15.5.9 中文换行规则**

中文排版与英文不同。

因为中文通常没有空格。

例如：

```
你好世界欢迎来到Unity
```

因此中文通常允许任意字符换行。

例如：

```
你好世界
欢迎来到
Unity
```

但中文同样存在特殊规则。

例如：

- 标点不能出现在行首
- 左括号不能出现在行尾
- 右括号不能出现在行首

例如：

```
（Hello
```

这是非法排版。

因此现代文本系统通常都会实现 Unicode Line Breaking Rule。

**15.5.10 Unicode 换行规则**

Unicode 官方定义了一整套换行规则。

其中包括：

- 禁止行首字符
- 禁止行尾字符
- CJK 换行规则
- 空格规则
- Emoji 规则

现代文本系统通常都会实现这些规范。

尤其 TextMeshPro，其内部已经支持 Unicode Line Breaking。

因此相比传统 Text，TMP 的中英文混排效果更加稳定。

**15.5.11 RichText 对排版的影响**

RichText 会显著增加排版复杂度。

例如：

```
<size=40>Hello</size>
```

此时：

- 行高会变化
- Baseline 会变化
- Advance 会变化

因此系统不仅需要处理字符。

还需要维护样式状态。

例如：

- 当前字号
- 当前字体样式
- 当前颜色
- 当前材质

因此 RichText 的排版复杂度远高于普通文本。

**15.5.12 Alignment 对齐算法**

文本排版完成后，系统还需要进行整体对齐。

例如：

- Left
- Center
- Right
- Justified

例如居中对齐：

```csharp
OffsetX = (RectWidth - LineWidth) / 2
```

系统会整体移动当前行。

而两端对齐则更加复杂。

因为系统需要：

- 计算空格数量
- 重新分配间距
- 拉伸单词间隔

因此对齐系统实际上属于二次布局过程。

**15.5.13 行高计算**

文本系统还需要计算 Line Height。

例如：

```
LineHeight = Ascender - Descender + LineSpacing
```

不同字体：

- Ascender 不同
- Descender 不同
- Font Size 不同

因此行高通常属于动态计算。

尤其 RichText 混合大字号时，行高计算会更加复杂。

**15.5.14 Auto Size 与换行**

部分文本系统支持：

- Auto Size
- Best Fit

例如：文本太长自动缩小字号

其核心逻辑通常如下：

1. 尝试当前字号
2. 计算排版
3. 判断是否超出
4. 若超出，则缩小字号
5. 重新排版

因此 Auto Size 本质上会触发多次排版计算。

这也是其性能开销较大的原因。

**15.5.15 双向文本排版**

部分语言属于双向文本。

例如：

- 阿拉伯语
- 希伯来语

它们会同时存在：

- 从左到右
- 从右到左

文本系统需要实现 Bidi Algorithm，即 Bidirectional Algorithm。

否则字符顺序会完全错误。

因此国际化文本系统的复杂度远高于普通英文系统。

**15.5.16 Emoji 与组合字符**

现代文本系统还需要处理：

- Emoji
- 组合字符
- Unicode Surrogate Pair

例如：

```
👨‍👩‍👧‍👦
```

虽然看起来是一个字符。

但实际上可能由多个 Unicode 组合而成。

因此现代文本系统不能简单按 char 遍历，而需要基于 Unicode Grapheme Cluster 进行处理。

**15.5.17 TMP 排版系统优势**

相比传统 Text，TMP 的排版系统更加先进。

其优势包括：

- Unicode 换行支持
- Kerning 支持
- 多语言支持
- 更完整的 RichText
- 更稳定的自动换行
- 更准确的 Baseline 计算

尤其在：

- 中英文混排
- 多语言 UI
- 富文本系统

中优势非常明显。

**15.5.18 排版系统的性能问题**

文本排版本质上属于 CPU 密集型计算。

因为系统需要：

- 遍历字符
- 计算宽度
- 计算换行
- 计算对齐
- 生成顶点

字符越多，CPU 开销越高。

尤其：

- RichText
- Auto Size
- 多语言
- Emoji
- 动态换行

都会显著增加排版复杂度。

因此大型项目通常会尽量避免频繁更新长文本。

否则文本排版本身就可能成为 UI 性能瓶颈。

**15.6 两套系统对比**

Unity 目前主要存在两套文本系统。

第一套是传统 UGUI Text。

第二套是 TextMeshPro。

虽然两者最终都会生成 UI Mesh 并提交 GPU 渲染。

但它们在底层架构、字体系统、排版能力以及性能特性上，存在非常明显的差异。

理解这两套系统的区别，对于 UI 架构设计与性能优化非常重要。

**15.6.1 底层架构对比**

传统 Text 基于旧版 TextGenerator。

其内部大量依赖 Unity 早期文本系统。

整个架构偏向简单、轻量以及旧工程兼容。

而 TMP 基于现代 TextCore。

其内部拥有独立排版系统、高级字体系统、SDF 渲染体系、多级字体回退以及 Unicode 排版支持。

因此 TMP 本质上属于现代化文本框架，而传统 Text 更像 Legacy 文本系统。

**15.6.2 字体渲染方式对比**

传统 Text 使用普通位图字体，字符最终会被栅格化到字体图集，放大后容易模糊。

尤其大字号下，边缘锯齿会非常明显。

而 TMP 使用 Signed Distance Field 字体，即 SDF 字体。

SDF 保存的是距离信息，而不是颜色信息。

因此字符边缘可以由 Shader 动态计算。

即使字体被大幅放大，边缘依然能够保持清晰。

这也是 TMP 文本质量远高于传统 Text 的根本原因。

**15.6.3 字体图集机制对比**

传统 Text 通常只有一张动态图集，当图集空间不足时，系统通常会扩容图集或者重建图集。

这会导致：

- UV 变化
- Mesh 重建
- Canvas 刷新

而 TMP 支持 Multi Atlas，当一张图集不足时，TMP 会自动创建新图集，而不是强制重建整张 Atlas。

因此 TMP 的动态图集更加稳定。

**15.6.4 动态字体机制对比**

传统 Text 的动态字体依赖：

```csharp
Font.RequestCharactersInTexture()
```

字符首次出现时。

系统会：

1. 栅格化字符
2. 写入字体图集
3. 更新 UV
4. 重新生成 Mesh

如果大量字符同时生成，非常容易产生卡顿，而 TMP 同样支持动态字体。

但其 Atlas 管理更加先进，尤其结合 Multi Atlas 后，动态图集重建频率会明显降低。

**15.6.5 排版系统对比**

传统 Text 的排版系统相对简单。

虽然支持自动换行、对齐以及 RichText。

但整体能力有限。

尤其：

- 中英文混排
- Unicode 换行
- 多语言排版

支持并不完善。

而 TMP 内部拥有完整 Typography 系统。

其支持：

- Unicode Line Breaking
- Kerning
- Baseline
- Ascender
- Descender
- 多语言规则

因此 TMP 的排版质量明显更高。

**15.6.6 RichText 对比**

传统 Text 支持部分 RichText 标签。

例如：

```
<b>
<i>
<color>
<size>
```

但支持范围有限。

复杂嵌套时也容易出现问题。

而 TMP 支持更加完整的 RichText 系统。

例如：

```
<link>
<sprite>
<mark>
<gradient>
```

并且 TMP 内部拥有完整 Style Stack 管理。

因此复杂富文本表现更加稳定。

**15.6.7 Shader 能力对比**

传统 Text 基本依赖普通 UI Shader。

因此：

- 描边能力较弱
- Glow 效果有限
- 阴影依赖额外组件

很多效果需要额外 DrawCall。

而 TMP 基于 SDF Shader。

大量文字效果都可以实时完成。

例如：

- Outline
- Underlay
- Glow
- Softness

这些效果都可以在 Shader 中动态计算。

因此 TMP 在文本特效方面优势极大。

**15.6.8 DrawCall 对比**

传统 Text 是否能够合批，高度依赖字体图集是否一致。

动态图集重建后，还可能导致 Batch 中断。

而 TMP 虽然也依赖 Font Asset，但其图集管理更加稳定。

因此 Batch 稳定性通常更高。

但 TMP 存在一个问题。

如果文本中包含：

- Sprite
- 多材质
- Fallback Font

则会生成 TMP\_SubMeshUI。

每个 SubMesh 都会产生额外 DrawCall。

因此复杂文本下，TMP 的 DrawCall 也可能增加。

**15.6.9 CPU 开销对比**

传统 Text 的 CPU 开销主要来自：

- TextGenerator
- 动态字体更新
- Mesh 重建
- Canvas Rebuild

尤其字体纹理变化时，很容易触发整个 Canvas 刷新。

而 TMP 的 CPU 开销主要来自：

- RichText 解析
- 高级排版
- Style Stack
- SubMesh 管理

因此 TMP 虽然更先进，但其单次文本生成成本通常更高。

**15.6.10 GPU 开销对比**

传统 Text 的 Shader 较轻，因此 GPU 开销通常较低。

而 TMP 的 SDF Shader 更复杂。

尤其：

- Outline
- Glow
- Underlay

都会增加 GPU 运算压力。

因此低端移动平台中，大量 TMP 文本也可能带来 GPU 压力。

**15.6.11 内存占用对比**

传统 Text 的字体图集通常较小。

但中文项目中，动态图集可能持续增长。

而 TMP 的 Font Asset 通常更大。

因为：

- SDF 数据更复杂
- Character Table 更多
- Glyph 信息更多

但 TMP 的 Multi Atlas 管理更加稳定。

因此长期运行下，内存波动通常更小。

**15.6.12 国际化支持对比**

传统 Text 对国际化支持较弱。

尤其：

- Emoji
- 阿拉伯语
- Unicode 组合字符
- 多语言回退

支持并不完善。

而 TMP 内部已经支持：

- Fallback Font
- Unicode 排版
- Emoji
- 多语言文本系统

因此现代国际化项目几乎都会使用 TMP。

**15.6.13 编辑器工作流对比**

传统 Text 使用简单，拖入字体即可使用，学习成本较低。

而 TMP 需要：

- 创建 Font Asset
- 配置 Atlas
- 配置 Fallback
- 配置 Material

初期工作流相对复杂，但 TMP 的可扩展性更强。

**15.6.14 兼容性对比**

传统 Text 属于 Unity 最早期 UI 系统。

大量旧项目仍然依赖它，因此兼容性极强。

而 TMP 在 Unity 后期才逐渐成为官方方案。

虽然目前已经深度集成，但旧项目迁移时。

仍然可能存在：

- 字体兼容问题
- RichText 差异
- 布局差异

因此很多旧项目仍然保留部分传统 Text。

**15.6.15 Unity 为什么最终选择 TMP**

Unity 后期已经将 TMP 深度整合进 TextCore。

其核心原因在于传统 Text 的底层架构已经难以满足现代文本需求。

尤其：

- 高清 UI
- 国际化
- Emoji
- 富文本
- 多语言
- 高质量排版

这些功能都要求更加现代化的文本系统。

而 TMP 正好满足这些需求。

因此 Unity 最终选择 TMP 作为官方主文本方案。

**15.6.16 实际项目中的选择**

目前现代 Unity 项目中。

绝大多数 UI 系统已经默认使用 TMP。

尤其：

- 商业项目
- 国际化项目
- 中大型项目
- 高分辨率 UI
- 富文本系统

几乎都会优先使用 TMP。

而传统 Text 更多用于：

- 旧项目兼容
- 简单 Demo
- 极轻量 UI
- 历史遗留系统

因此从长期来看，TMP 已经成为 Unity 文本系统的主流方案。

**本章小结**

本章围绕 UGUI 文本渲染系统展开，从字体资源结构到动态字体机制，再到文本渲染流程与排版算法，最终扩展到 TextMeshPro 的整体实现方案，对 Unity 中文本系统的完整链路进行了系统性拆解。

首先，在字体与字形图集部分可以看到，文本渲染的基础并不是“字符本身”，而是“字符到 Glyph 再到 Atlas”的映射体系。字体负责语义层面的字符定义，而字形图集负责提供可供 GPU 采样的视觉资源，两者共同构成文本渲染的数据基础。

在动态字体机制中，系统进一步引入了运行时 Glyph 生成能力，使字体系统从“预生成模型”扩展为“按需生成模型”。这一机制解决了多语言与未知字符输入的问题，但也带来了运行时内存增长与首次加载成本上升的代价。

UGUI Text 渲染流程则完整展示了从字符串到 Mesh 的转换过程，包括字符串解析、字符映射、排版计算以及顶点生成。可以看到，Text 的本质是一个运行时排版系统，其核心成本集中在 CPU 侧的逐字符计算与 Mesh 重建。

在 TextMeshPro 部分，文本系统被进一步重构为基于 SDF 字体的高级渲染引擎。它不仅在字体层面引入距离场表示，还在排版系统与 Shader 层进行了全面增强，使文本渲染具备更高的视觉质量与更强的扩展能力。

字符排版与换行算法则揭示了文本系统中最关键的布局逻辑，本质上是一个基于空间约束的逐字符计算过程，负责将线性字符串转换为具有二维结构的字符集合，并为 Mesh 生成提供精确的位置依据。

最后，通过 UGUI Text 与 TextMeshPro 的对比可以明确，两者并非简单的功能升级关系，而是两种完全不同的设计体系：前者偏向轻量级 UI 组件实现，后者则是结构化文本渲染引擎。它们在字体模型、排版系统、Mesh生成方式以及 GPU 参与程度上均存在本质差异。

总结来看，本章构建了 UGUI 文本系统的完整认知链路，从底层字体数据结构到上层渲染流程，再到不同文本系统的设计对比，为理解 Unity UI 文本渲染机制奠定了完整的技术基础。
