在 [UGUI](https://zhida.zhihu.com/search?content_id=275722848&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 的工程实践中，UI 架构设计往往决定了系统的上限，而不仅仅是某个功能模块的实现方式。当项目规模逐渐扩大，UI 数量增加、交互复杂度提升、状态来源多样化之后，单纯依赖组件堆叠或局部优化已经无法保证系统的稳定性，此时架构层面的设计能力就会直接影响整体性能表现与开发效率。

与前面章节关注的单点机制不同，UI 架构设计更强调“整体组织方式”。它不仅涉及界面如何被创建与销毁，还涉及 UI 如何被分层管理、如何响应数据变化，以及如何在复杂交互下保持结构清晰与性能可控。

从系统角度来看，一个成熟的 UI 架构通常需要解决三个核心问题：结构如何组织、职责如何划分，以及数据如何驱动界面更新。如果这些问题在设计阶段没有被清晰定义，后续的 Layout 优化、批处理优化甚至 Canvas 优化，都只能在局部层面进行修补，而无法从根本上解决系统复杂度上升带来的性能与维护问题。

因此，本章将从 UGUI 实际工程出发，围绕 UI 架构的 [分层设计](https://zhida.zhihu.com/search?content_id=275722848&content_type=Article&match_order=1&q=%E5%88%86%E5%B1%82%E8%AE%BE%E8%AE%A1&zhida_source=entity) 、模块组织方式、生命周期管理以及状态驱动模型展开分析，逐步建立一套适用于复杂项目的 UI 架构思维模型。

### 25.1 UI 分层与模块化

在大型项目中，UI 系统的复杂度往往随着功能增长呈指数级上升。如果缺乏清晰的结构设计，极易出现代码耦合严重、复用性差、维护成本高等问题。因此，引入“分层”与“模块化”的设计思想，是构建可扩展 UI 架构的基础。

### 25.1.1 分层设计的核心目的

UI 分层的本质，是将不同职责的逻辑进行隔离，使系统具备以下特性：

1. 职责单一 每一层只处理特定类型的问题，避免逻辑混杂。
2. 降低耦合 上层不依赖下层的具体实现细节，仅通过接口交互。
3. 提高复用性 通用逻辑可以在不同 UI 模块之间复用。
4. 易于维护与扩展 修改某一层的实现，不会影响整体结构。

从工程角度看，分层并不是强制的规则，而是一种“约束设计”，用于控制复杂度。

### 25.1.2 UI 常见分层结构

在实际项目中，UGUI 常见的分层结构可以划分为以下几层：

一、表现层（View）

负责 UI 的显示与交互，直接依赖 UGUI 组件。

其职责包括：

1. 控制界面显示与隐藏
2. 响应用户输入（点击、拖拽等）
3. 更新 UI 元素（Text、Image 等）
4. 转发事件给逻辑层

典型特点：

1. 不包含业务逻辑
2. 不直接操作数据源
3. 尽量保持“被动更新”

示例：

```csharp
public class LoginView : MonoBehaviour
{
    public Button loginButton;
    public InputField accountInput;
    public InputField passwordInput;

    private void Awake()
    {
        loginButton.onClick.AddListener(OnClickLogin);
    }

    private void OnClickLogin()
    {
        string account = accountInput.text;
        string password = passwordInput.text;

        LoginController.Instance.Login(account, password);
    }

    public void SetLoading(bool active)
    {
        // 控制加载动画
    }
}
```

二、控制层（Controller / Presenter）

负责 UI 与业务逻辑之间的桥梁。

其职责包括：

1. 接收 View 的事件
2. 调用业务逻辑（Model / Service）
3. 更新 View 状态

典型特点：

1. 不直接操作 UI 组件
2. 不负责数据存储
3. 作为中间协调者

示例：

```csharp
public class LoginController
{
    public static LoginController Instance = new LoginController();

    private LoginView view;

    public void BindView(LoginView view)
    {
        this.view = view;
    }

    public void Login(string account, string password)
    {
        view.SetLoading(true);

        // 模拟请求
        bool success = account == "admin" && password == "123";

        view.SetLoading(false);

        if (success)
        {
            // 登录成功逻辑
        }
        else
        {
            // 提示错误
        }
    }
}
```

三、数据层（Model）

负责数据管理与状态维护。

其职责包括：

1. 存储数据（玩家信息、配置等）
2. 提供数据访问接口
3. 处理数据变更通知

典型特点：

1. 与 UI 无关
2. 可被多个 UI 复用
3. 可独立测试

示例：

```csharp
public class UserModel
{
    public string Account;
    public string Token;
}
```

四、服务层（Service）

负责与外部系统交互，如网络、文件、数据库等。

其职责包括：

1. 网络请求封装
2. 数据持久化
3. 资源加载

示例：

```csharp
public class LoginService
{
    public async Task<bool> Login(string account, string password)
    {
        // 模拟网络请求
        await Task.Delay(1000);
        return account == "admin" && password == "123";
    }
}
```

### 25.1.3 分层之间的依赖关系

合理的依赖关系应遵循“单向依赖原则”：

View → Controller → Model / Service

需要注意：

1. View 不应直接访问 Model
2. Model 不应依赖 View
3. Service 不应依赖 UI

错误示例：

```csharp
// 错误：Model 直接操作 UI
public class UserModel
{
    public void UpdateUI(Text text)
    {
        text.text = "更新数据";
    }
}
```

正确做法应通过 Controller 转发更新。

### 25.1.4 模块化设计

模块化是在分层基础上的进一步抽象，其核心思想是：将 UI 拆分为“功能独立的单元”。

一个 UI 模块通常包含：

1. View
2. Controller
3. 相关资源（Prefab、Sprite 等）
4. 配置数据

例如，一个“背包系统”可以作为独立模块：

```csharp
InventoryModule
 ├── InventoryView
 ├── InventoryController
 ├── InventoryItemView
 ├── InventoryModel
 └── Prefabs
```

模块之间应具备以下特性：

1. 高内聚 模块内部逻辑紧密相关
2. 低耦合 模块之间通过接口通信
3. 可独立加载与卸载 支持按需加载 UI

### 25.1.5 UI 模块通信方式

模块之间通信是架构设计中的关键问题，常见方式包括：

一、直接引用（不推荐）

模块 A 直接调用模块 B

缺点：

1. 强耦合
2. 难以维护

二、事件系统

通过事件进行解耦：

```csharp
EventBus.Publish("OnLoginSuccess");
```

优点：

1. 解耦
2. 灵活扩展

三、接口调用

定义接口规范：

```csharp
public interface ILoginModule
{
    void OnLoginSuccess();
}
```

优点：

1. 类型安全
2. 结构清晰

### 25.1.6 模块化与 UGUI 的关系

UGUI 本身并不提供模块化机制，但其组件化设计天然支持模块划分：

1. 每个 UI Prefab 可作为模块载体
2. Canvas 可作为模块容器
3. 子 Canvas 可实现局部更新

结合资源管理系统（如 Addressables），可以实现：

1. UI 动态加载
2. 模块按需实例化
3. 降低内存占用

### 25.1.7 常见设计误区

一、View 中写大量业务逻辑

导致 UI 与逻辑强耦合，难以复用。

二、Controller 过于庞大

承担过多职责，变成“上帝类”。

三、模块边界不清晰

多个模块互相调用，形成循环依赖。

四、忽略数据层设计

所有数据直接写在 UI 中，难以维护。

### 25.1.8 小结

UI 分层与模块化的核心，并不在于形式，而在于“控制复杂度”。

分层解决的是职责问题，模块化解决的是结构问题。两者结合，可以构建出：

1. 易扩展的 UI 系统
2. 易维护的代码结构
3. 高复用的功能模块

### 25.2 UI 管理系统设计

在完成 UI 分层与模块化之后，下一步需要解决的是“如何统一管理这些 UI 模块”。如果缺乏统一的管理系统，UI 的创建、销毁、层级控制、状态维护等都会变得混乱，从而导致性能问题与逻辑错误。

UI 管理系统的本质，是对 UI 生命周期与显示规则的统一调度。

### 25.2.1 UI 管理系统的核心职责

一个完整的 UI 管理系统通常需要承担以下职责：

1. UI 的创建与销毁 负责实例化 UI Prefab，并控制其生命周期。
2. UI 的显示与隐藏 控制界面的打开与关闭行为。
3. UI 层级管理 确保 UI 在正确的显示层级（如弹窗覆盖主界面）。
4. UI 状态管理 管理当前打开的 UI 状态，避免重复打开或冲突。
5. 资源加载与缓存 控制 UI 资源加载策略（同步或异步）。
6. UI 栈管理（可选） 支持返回、关闭栈顶界面等行为。

从架构角度看，UI 管理系统通常位于 Controller 层之上，作为全局调度入口。

### 25.2.2 UI 分层结构设计（显示层级）

为了控制 UI 的显示顺序，通常会对 UI 进行层级划分。一个常见的分层结构如下：

一、Background 层

用于背景 UI，例如主界面底图。

二、Normal 层

普通界面，例如功能面板（背包、任务）。

三、Popup 层

弹窗界面，例如提示框、确认框。

四、Top 层

最顶层 UI，例如引导、提示信息。

五、System 层

系统级 UI，例如加载界面、全局遮罩。

在 Unity 中，通常通过多个子 Canvas 或节点进行组织：

```csharp
UIRoot
 ├── Background
 ├── Normal
 ├── Popup
 ├── Top
 └── System
```

每个层级可以挂载独立的 Canvas，以减少不必要的重建与批处理影响。

### 25.2.3 UI 打开与关闭流程

UI 管理系统需要统一 UI 的打开流程。一个标准流程如下：

一、打开 UI

1. 判断 UI 是否已存在
2. 若不存在，则加载 Prefab
3. 实例化并挂载到对应层级
4. 初始化（Bind、数据设置）
5. 播放打开动画（可选）
6. 加入 UI 管理列表或栈

二、关闭 UI

1. 播放关闭动画（可选）
2. 从管理列表或栈中移除
3. 销毁或隐藏 UI
4. 释放资源（按策略）

示例：

```csharp
public class UIManager : MonoBehaviour
{
    private Dictionary<string, GameObject> uiDict = new Dictionary<string, GameObject>();

    public Transform normalLayer;

    public GameObject OpenUI(string name, GameObject prefab)
    {
        if (uiDict.ContainsKey(name))
        {
            return uiDict[name];
        }

        GameObject ui = Instantiate(prefab, normalLayer);
        uiDict.Add(name, ui);

        return ui;
    }

    public void CloseUI(string name)
    {
        if (!uiDict.ContainsKey(name)) return;

        GameObject ui = uiDict[name];
        Destroy(ui);

        uiDict.Remove(name);
    }
}
```

### 25.2.4 UI 缓存策略

频繁创建与销毁 UI 会带来较高的性能开销，因此需要设计合理的缓存策略。

常见策略包括：

一、完全销毁

关闭 UI 时直接销毁对象。

适用于：

1. 使用频率低
2. 占用内存较大

二、隐藏缓存

关闭 UI 时仅 SetActive。

适用于：

1. 高频 UI（如背包）
2. 打开关闭频繁

三、对象池复用

通过对象池管理 UI 实例。

适用于：

1. 动态生成 UI（列表项）
2. 大量重复元素

示例：

```csharp
public void CloseUI(string name)
{
    if (!uiDict.ContainsKey(name)) return;

    GameObject ui = uiDict[name];
    ui.SetActive(false);
}
```

### 25.2.5 UI 栈（Stack）机制

为了支持“返回上一界面”等逻辑，通常会引入 UI 栈。

其核心规则：

1. 新 UI 入栈
2. 当前 UI 可选择隐藏或保持
3. 关闭 UI 时弹出栈顶
4. 恢复上一个 UI 状态

示例：

```csharp
private Stack<GameObject> uiStack = new Stack<GameObject>();

public void PushUI(GameObject ui)
{
    if (uiStack.Count > 0)
    {
        uiStack.Peek().SetActive(false);
    }

    uiStack.Push(ui);
    ui.SetActive(true);
}

public void PopUI()
{
    if (uiStack.Count == 0) return;

    GameObject top = uiStack.Pop();
    Destroy(top);

    if (uiStack.Count > 0)
    {
        uiStack.Peek().SetActive(true);
    }
}
```

### 25.2.6 UI 生命周期设计

为了规范 UI 行为，通常需要定义统一的生命周期接口。

常见生命周期包括：

1. OnInit 初始化，仅执行一次。
2. OnOpen 每次打开 UI 时调用。
3. OnClose UI 关闭时调用。
4. OnDestroy UI 被销毁时调用。

示例：

```csharp
public abstract class UIBase : MonoBehaviour
{
    public virtual void OnInit() { }
    public virtual void OnOpen() { }
    public virtual void OnClose() { }
    public virtual void OnDestroyUI() { }
}
```

UIManager 在控制 UI 时，应主动调用这些接口，而不是依赖 Unity 生命周期。

### 25.2.7 异步加载与资源管理

在实际项目中，UI 通常不会全部预加载，而是按需加载。

结合资源系统（如 Addressables）可以实现：

1. 异步加载 UI Prefab
2. 加载完成后再实例化
3. 释放未使用资源

示例流程：

```csharp
public async Task<GameObject> OpenUIAsync(string path)
{
    var handle = Addressables.LoadAssetAsync<GameObject>(path);
    await handle.Task;

    GameObject prefab = handle.Result;
    return Instantiate(prefab);
}
```

需要注意：

1. 避免重复加载
2. 管理引用计数
3. 控制释放时机

### 25.2.8 UI 管理系统扩展方向

一个成熟的 UI 管理系统通常还会扩展以下能力：

1. UI 动画统一管理
2. UI 音效控制
3. 多语言支持
4. 分辨率适配
5. 输入拦截（遮罩管理）

这些功能通常不直接耦合在 UIManager 中，而是通过子系统或组件扩展实现。

### 25.2.9 常见问题分析

一、UI 重复打开

未做存在性判断，导致多个实例。

二、层级混乱

未统一管理父节点，导致遮挡错误。

三、内存泄漏

UI 未正确销毁或资源未释放。

四、频繁 Rebuild

UI 频繁激活/隐藏，触发 Canvas 重建。

### 25.2.10 小结

UI 管理系统是 UI 架构的核心调度中心，其设计直接影响：

1. UI 系统的可维护性
2. 性能表现
3. 扩展能力

一个良好的 UI 管理系统，应具备：

1. 清晰的层级结构
2. 完整的生命周期控制
3. 合理的缓存与加载策略
4. 可扩展的架构设计

在此基础上，下一节将进一步深入 UI 生命周期管理机制，从执行时序角度分析 UI 行为。

### 25.3 界面生命周期管理

在 UI 管理系统中，生命周期管理是保证界面行为一致性与可控性的关键。若缺乏统一的生命周期规范，不同界面之间将出现初始化混乱、重复执行逻辑、资源未释放等问题。

界面生命周期的本质，是对 UI 从“创建到销毁”的全过程进行标准化定义与调度。

### 25.3.1 生命周期设计的核心目标

生命周期设计的核心目标在于：

1. 明确阶段边界 将 UI 的不同执行阶段进行划分，避免逻辑混杂。
2. 控制执行时机 确保逻辑在正确的时间点执行。
3. 避免重复初始化 区分“一次性初始化”与“多次打开逻辑”。
4. 统一管理资源 在合适阶段加载与释放资源。
5. 提高可维护性 所有 UI 遵循统一规范，降低理解成本。

### 25.3.2 生命周期阶段划分

一个完整的 UI 生命周期通常包含以下阶段：

一、创建阶段（Create）

UI 实例被创建，但尚未初始化。

特点：

1. 仅进行 GameObject 实例化
2. 不执行业务逻辑
3. 不访问外部数据

二、初始化阶段（Init）

执行一次性初始化逻辑。

典型操作包括：

1. 缓存组件引用
2. 注册事件（按钮点击等）
3. 初始化内部结构

该阶段只执行一次。

三、打开阶段（Open）

UI 被显示时触发。

典型操作包括：

1. 设置数据
2. 刷新 UI 显示
3. 播放打开动画

该阶段可能被多次调用。

四、激活阶段（Show）

UI 进入可见状态。

区别于 Open：

1. Open 偏向逻辑初始化
2. Show 偏向显示状态控制

在一些架构中，Open 与 Show 可以合并。

五、隐藏阶段（Hide）

UI 被临时隐藏，但未销毁。

典型操作：

1. 停止动画
2. 暂停逻辑
3. 取消输入响应

六、关闭阶段（Close）

UI 被关闭，准备退出。

典型操作：

1. 播放关闭动画
2. 移除事件监听
3. 保存状态（如需要）

七、销毁阶段（Destroy）

UI 被彻底释放。

典型操作：

1. 释放资源
2. 清理引用
3. 取消所有外部绑定

### 25.3.3 生命周期接口定义

为了统一行为，通常定义一个基类作为生命周期规范：

```csharp
public abstract class UIBase : MonoBehaviour
{
    public virtual void OnCreate() { }
    public virtual void OnInit() { }
    public virtual void OnOpen(object param = null) { }
    public virtual void OnShow() { }
    public virtual void OnHide() { }
    public virtual void OnClose() { }
    public virtual void OnDestroyUI() { }
}
```

需要注意：

1. 不直接依赖 Unity 的 Start / OnEnable
2. 所有生命周期由 UIManager 主动驱动
3. 保证执行顺序一致

### 25.3.4 生命周期执行顺序

标准执行流程如下：

一、首次打开 UI

Create → Init → Open → Show

二、再次打开 UI（缓存模式）

Open → Show

三、关闭 UI（隐藏模式）

Hide

四、关闭 UI（销毁模式）

Close → Destroy

通过这种方式，可以明确区分：

1. 一次性逻辑（Init）
2. 多次执行逻辑（Open）

### 25.3.5 UIManager 中的生命周期调度

UI 管理系统需要负责驱动生命周期的执行。

示例：

```csharp
public class UIManager : MonoBehaviour
{
    private Dictionary<string, UIBase> uiDict = new Dictionary<string, UIBase>();

    public T OpenUI<T>(string name, GameObject prefab) where T : UIBase
    {
        UIBase ui;

        if (!uiDict.TryGetValue(name, out ui))
        {
            GameObject go = Instantiate(prefab);
            ui = go.GetComponent<T>();

            ui.OnCreate();
            ui.OnInit();

            uiDict.Add(name, ui);
        }

        ui.OnOpen();
        ui.OnShow();

        ui.gameObject.SetActive(true);

        return ui as T;
    }

    public void CloseUI(string name, bool destroy = false)
    {
        if (!uiDict.ContainsKey(name)) return;

        UIBase ui = uiDict[name];

        if (destroy)
        {
            ui.OnClose();
            ui.OnDestroyUI();

            Destroy(ui.gameObject);
            uiDict.Remove(name);
        }
        else
        {
            ui.OnHide();
            ui.gameObject.SetActive(false);
        }
    }
}
```

### 25.3.6 生命周期与动画系统的配合

UI 动画通常贯穿生命周期的多个阶段：

1. Open 阶段 播放进入动画
2. Show 阶段 动画完成后进入可交互状态
3. Close 阶段 播放退出动画

需要注意：

1. 动画结束后再执行真正关闭
2. 避免在动画过程中销毁 UI
3. 使用回调或协程控制流程

示例：

```csharp
public IEnumerator CloseWithAnimation(UIBase ui)
{
    yield return ui.PlayCloseAnimation();

    ui.OnClose();
    ui.OnDestroyUI();

    Destroy(ui.gameObject);
}
```

### 25.3.7 生命周期与数据更新

UI 的数据更新应绑定在合适的生命周期阶段：

1. Init 初始化静态数据结构
2. Open 设置或刷新数据
3. Show 最终显示修正

避免在以下阶段更新数据：

1. Awake / Start（不可控）
2. 每帧 Update（性能浪费）

推荐方式：

```csharp
public override void OnOpen(object param = null)
{
    var data = param as UserData;
    Refresh(data);
}
```

### 25.3.8 生命周期与资源管理

资源加载与释放需要与生命周期匹配：

1. Init 加载常驻资源
2. Open 加载动态资源（如头像）
3. Close / Destroy 释放资源

需要注意：

1. 避免重复加载
2. 控制引用计数
3. 避免内存泄漏

### 25.3.9 常见设计问题

一、生命周期混乱

不同 UI 使用不同逻辑，导致不可维护。

二、重复初始化

在 Open 中重复执行 Init 逻辑。

三、依赖 Unity 生命周期

使用 Start / OnEnable 导致执行顺序不可控。

四、关闭逻辑不完整

未清理事件或引用，导致异常。

### 25.3.10 小结

界面生命周期管理，是 UI 架构稳定运行的基础。

通过标准化生命周期，可以实现：

1. 明确的执行流程
2. 可控的资源管理
3. 清晰的逻辑划分

在实际工程中，应始终遵循：

1. 生命周期由 UIManager 驱动
2. 不依赖 Unity 默认生命周期
3. 严格区分各阶段职责

在此基础上，可以进一步构建复杂的 UI 状态机与动画系统，实现更加完整的 UI 架构设计。
