随着项目规模不断扩大， [UGUI](https://zhida.zhihu.com/search?content_id=275704417&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 的性能问题会越来越明显。大量 UI 节点、复杂布局、动态列表以及频繁刷新，都会导致 CPU 开销增加、Canvas 重建频繁以及渲染效率下降。

在大型项目中，UI 往往不仅仅是界面展示层，还承担了数据刷新、交互逻辑、动画播放以及资源管理等职责。因此，大规模 UI 优化并不是简单减少节点数量，而是需要从整体架构层面进行设计。

一个优秀的 UI 系统，不仅需要保证功能完整与交互流畅，还需要在复杂场景下维持稳定的性能表现。尤其是在移动端项目中，UI 往往会成为 CPU 与渲染性能的重要瓶颈，因此建立合理的 UI 优化体系十分重要。

本章将从架构设计、缓存复用、虚拟化以及实际项目优化等多个角度，系统分析大型项目中的 UGUI 优化方案。

**24.1 大规模 UI 架构设计**

在大型项目中，很多 UI 性能问题并不是某一个组件导致的，而是整体架构设计不合理所引起的。

当界面数量不断增加后，即使单个界面的性能开销并不高，大量界面同时刷新、频繁切换以及复杂交互，依然会逐渐形成严重的性能压力。

因此，大规模 UI 优化的核心并不仅仅是优化某一个界面，而是建立一套稳定、高效且可扩展的 UI 架构体系。

一个合理的 UI 架构通常需要满足以下目标。

- 降低 Canvas 重建频率
- 减少无意义刷新
- 控制层级复杂度
- 降低对象创建与销毁开销
- 提高界面复用能力
- 保持模块解耦
- 方便后期维护与扩展

很多项目在开发初期并不会出现明显问题，但随着功能不断增加，UI 系统会逐渐暴露出大量性能隐患。

例如，所有界面共用一个 Canvas、频繁切换 SetActive、大量动态创建节点、ScrollView 不做复用以及 Layout 组件过度嵌套等，都会导致 CPU 开销持续升高。

因此，在项目初期建立合理的 UI 架构十分重要。

**24.1.1 UI 分层设计**

大型项目通常会对 UI 进行分层管理，不同类型的界面会放置在不同 Canvas 中，从而降低 UI 重建影响范围。

常见的 UI 分层包括：

- 底层主界面
- 功能窗口层
- 弹窗层
- 顶层提示层
- 新手引导层
- Loading 层
- 调试层

通过 UI 分层，可以避免某个界面刷新时导致整个 UI 系统一起重建。

例如，背包界面中的物品刷新，不应该影响顶部货币栏与主界面的静态区域。如果所有界面都位于同一个 Canvas 中，那么任意一个 Text 变化，都可能导致整个界面重新生成 Mesh。

因此，大型项目通常会将不同刷新频率的界面进行隔离。

一种常见的 UI 层级结构如下。

```csharp
UIRoot
├── BottomLayer
├── WindowLayer
├── PopupLayer
├── TopLayer
├── GuideLayer
└── LoadingLayer
```

这种分层结构不仅能够降低重建范围，还能够统一界面管理逻辑。

**24.1.2 Canvas 拆分原则**

Canvas 是 UGUI 中最核心的性能节点之一。

当 Canvas 内任意 Graphic 发生变化时，整个 Canvas 都可能重新进行顶点重建、Batch 计算以及渲染排序。

因此，大型项目通常不会将所有 UI 放在同一个 Canvas 中，而是按照刷新频率进行拆分。

通常可以按照以下原则进行 Canvas 拆分。

**静态区域独立 Canvas**

变化频率极低的 UI，例如背景、固定装饰、静态按钮等，应尽量放置在独立 Canvas 中。

这样可以避免动态区域刷新时影响静态内容。

**高频刷新区域独立 Canvas**

血条、倒计时、聊天窗口、实时排行榜等高频刷新区域，应单独使用 Canvas。

因为这些区域会频繁修改 Text、Image 或 Layout，如果与静态区域混合，将导致大量无意义重建。

**ScrollView 独立 Canvas**

大型 ScrollView 通常会频繁刷新 Item，因此建议独立 Canvas。

否则列表滚动时，可能导致整个界面重新计算 Batch。

**动画区域独立 Canvas**

大量 Tween 动画、透明渐变以及频繁位移的 UI，也应尽量独立。

因为 RectTransform 的变化同样可能触发 Canvas 更新。

**24.1.3 UI 模块化设计**

大型项目中的 UI 数量通常非常庞大，因此模块化设计十分重要。

一个完整的 UI 模块通常会包含以下部分。

- View
- Controller
- Model
- 配置数据
- 资源管理
- 事件系统

通过模块化设计，可以降低界面之间的耦合程度，同时提高后期维护效率。

一个典型的 UI 模块结构如下。

```csharp
BagModule
├── BagView
├── BagController
├── BagModel
├── BagItem
├── BagConfig
└── BagData
```

这种结构能够有效解决以下问题。

1. UI 与业务逻辑混乱耦合
2. 界面之间相互依赖
3. 数据刷新逻辑混乱
4. 后期维护困难
5. 热更新扩展困难

此外，模块化架构还能够方便后续进行动态加载、资源卸载、界面复用以及热更新管理，这也是大型商业项目中最常见的 UI 架构方案之一。

**24.1.4 UI [生命周期管理](https://zhida.zhihu.com/search?content_id=275704417&content_type=Article&match_order=1&q=%E7%94%9F%E5%91%BD%E5%91%A8%E6%9C%9F%E7%AE%A1%E7%90%86&zhida_source=entity)**

很多项目中的 UI 性能问题，并不是渲染造成的，而是生命周期管理混乱导致的。

例如：

- 重复打开同一界面
- 界面关闭后事件未注销
- 长时间持有无效引用
- 动态注册大量监听器
- 多次重复初始化

因此，大型项目通常会建立统一的 UI 生命周期管理机制。

一个标准 UI 生命周期通常包括：

```csharp
Create
Initialize
Show
Refresh
Hide
Destroy
```

不同阶段负责不同逻辑，可以有效降低重复初始化与资源泄漏问题。

例如：

- Create 阶段负责实例化
- Initialize 阶段负责绑定事件
- Show 阶段负责显示动画
- Refresh 阶段负责刷新数据
- Hide 阶段负责隐藏界面
- Destroy 阶段负责资源释放

合理的生命周期管理，能够显著降低复杂 UI 系统中的逻辑混乱问题。

**24.2 缓存与复用**

在大型 UI 系统中，频繁创建与销毁对象会产生大量性能开销，同时还会带来严重的 GC 问题。

尤其是在 ScrollView、聊天窗口、背包系统以及排行榜等高频动态界面中，如果持续实例化与销毁 UI 节点，CPU 与内存压力会迅速增加。

因此，缓存与复用是大型 UI 优化中非常重要的一部分。

缓存的核心思想并不是“减少创建”，而是尽可能重复利用已经存在的对象，从而降低运行期开销。

**24.2.1 UI 缓存机制**

很多 UI 界面在关闭时，并不会真正销毁，而是进入缓存状态。

当界面再次打开时，系统直接从缓存中恢复，而不是重新实例化。

这样可以避免：

- 重复加载资源
- 重复创建节点
- 频繁 GC
- 界面打开卡顿

因此，大型项目通常会维护一套 UI 缓存池，用于管理已经创建过的界面。

对于频繁打开的窗口，例如背包、商城、任务界面等，通常都会采用缓存机制。

一个简单的缓存流程如下。

```csharp
首次打开界面
        ↓
实例化 UI
        ↓
关闭界面
        ↓
进入缓存池
        ↓
再次打开
        ↓
直接复用
```

这种方案能够显著提升界面打开速度。

**24.2.2 [对象池](https://zhida.zhihu.com/search?content_id=275704417&content_type=Article&match_order=1&q=%E5%AF%B9%E8%B1%A1%E6%B1%A0&zhida_source=entity) 复用**

对象池是 UI 复用中最常见的优化方案之一。

其核心思想是提前创建一定数量的对象，在需要时直接取出使用，不再使用时重新回收到池中，而不是直接销毁。

这种方案非常适合：

- ScrollView Item
- 聊天气泡
- 飘字提示
- 列表节点
- 动态按钮
- 特效 UI

通过对象池，可以大幅降低 Instantiate 与 Destroy 的调用次数。

同时，还能够有效减少 GC Alloc 的产生。

下面是一个简单的 UI 对象池示例。

```csharp
using System.Collections.Generic;
using UnityEngine;

public class UIObjectPool<T> where T : Component
{
    private Queue<T> pool = new Queue<T>();

    private T prefab;

    private Transform parent;

    public UIObjectPool(T prefab, int count, Transform parent)
    {
        this.prefab = prefab;
        this.parent = parent;

        for (int i = 0; i < count; i++)
        {
            T obj = GameObject.Instantiate(prefab, parent);

            obj.gameObject.SetActive(false);

            pool.Enqueue(obj);
        }
    }

    public T Get()
    {
        if (pool.Count > 0)
        {
            T obj = pool.Dequeue();

            obj.gameObject.SetActive(true);

            return obj;
        }

        return GameObject.Instantiate(prefab, parent);
    }

    public void Release(T obj)
    {
        obj.gameObject.SetActive(false);

        pool.Enqueue(obj);
    }
}
```

**24.2.3 ScrollView 节点复用**

ScrollView 是大型 UI 中最容易出现性能问题的组件之一。

很多开发者会直接根据数据量生成全部 Item，当数据达到几百甚至上千时，界面会出现明显卡顿。

实际上，用户在同一时间真正能够看到的节点数量通常非常有限。

因此，大型项目通常会采用“节点复用”方案。

例如，屏幕中只能显示 10 个 Item，那么系统通常只会创建 10 到 15 个节点。

当列表滚动时，离开屏幕的节点会被重新移动到另一侧，并刷新数据内容，而不是重新创建新的节点。

这种方案本质上就是 [虚拟列表](https://zhida.zhihu.com/search?content_id=275704417&content_type=Article&match_order=1&q=%E8%99%9A%E6%8B%9F%E5%88%97%E8%A1%A8&zhida_source=entity) 的核心思想。

通过节点复用，可以显著降低：

- Hierarchy 节点数量
- Canvas 重建压力
- 内存占用
- GC 开销

这也是大型商业项目中 ScrollView 优化的核心方案之一。

**24.2.4 资源 [异步加载](https://zhida.zhihu.com/search?content_id=275704417&content_type=Article&match_order=1&q=%E5%BC%82%E6%AD%A5%E5%8A%A0%E8%BD%BD&zhida_source=entity) 与缓存**

大型 UI 系统中的另一个重要优化点，是资源异步加载。

如果 UI 打开时同步加载大量 Sprite、Prefab 或 Atlas，很容易导致主线程卡顿。

因此，商业项目通常会采用以下方案。

**异步加载 UI 资源**

通过 Addressables、AssetBundle 或自定义资源系统进行异步加载。

避免主线程长时间阻塞。

**图集缓存**

对于频繁使用的 UI 图集，可以常驻内存。

避免频繁卸载与重新加载。

**延迟加载**

部分非关键 UI 可以在界面打开后逐步加载。

例如：

- 商城推荐图
- 玩家头像
- 排行榜数据
- 大尺寸预览图

这样能够明显降低界面首次打开时的卡顿。

**24.3 虚拟列表实现**

在大型项目中，ScrollView 往往是 UI 性能问题最严重的区域之一。

很多开发者会根据数据量直接生成全部列表节点，当数据达到几百甚至上千时，界面会出现明显卡顿。

这是因为大量 Item 会带来以下问题。

- Hierarchy 节点数量过多
- Canvas 重建压力增加
- Layout 计算耗时上升
- Draw Call 增加
- 内存占用升高
- GC 频繁产生

但实际上，用户在同一时间能够看到的节点数量通常非常有限。

例如，一个背包界面即使有上千个物品，屏幕中真正显示的节点可能只有十几个。

因此，虚拟列表的核心思想就是“只创建可见区域中的节点”。

**24.3.1 虚拟列表原理**

虚拟列表不会一次性创建全部 Item，而是只创建当前可见区域所需要的节点。

当列表滚动时，已经离开屏幕的节点不会销毁，而是重新移动到新的位置，并刷新对应数据。

也就是说，列表中的节点数量通常是固定的，而变化的只是节点显示的数据内容。

例如：

- 屏幕一次显示 10 个节点
- 系统实际创建 12 到 15 个节点
- 滚动时循环复用这些节点

这样即使数据量达到上万条，真正存在于场景中的 UI 节点依然非常少。

**24.3.2 节点循环复用**

虚拟列表最核心的部分就是节点循环复用。

当节点离开可视区域后，系统会立即修改它的位置，并重新绑定新的数据。

整个过程中不会产生：

- Instantiate
- Destroy
- 大量 GC

因此，列表滚动会非常流畅。

大型项目中的背包、商城、排行榜、邮件列表以及聊天系统，几乎都会使用这种方案。

**24.3.3 数据与显示分离**

虚拟列表通常会将“数据”与“显示节点”完全分离。

列表真正保存的是数据集合，而 UI 节点只是数据的显示载体。

当节点滚动到新的位置时，只需要重新刷新数据即可。

这种设计能够有效降低 UI 系统复杂度，同时提高列表扩展能力。

此外，当数据发生变化时，也不需要重新生成整个列表，只更新对应节点即可。

**24.3.4 虚拟列表优化要点**

实现虚拟列表时，还需要注意以下问题。

首先，应尽量避免使用大量 Layout 组件，因为 Layout 会增加额外计算开销。

其次，节点刷新时应减少频繁修改 Text、Image 等组件，避免触发大量 Canvas 重建。

另外，列表节点内部应尽量减少复杂层级与动画系统，否则即使节点数量减少，也依然可能产生较高 CPU 开销。

在实际项目中，一个成熟的虚拟列表系统通常还会支持：

- 动态尺寸
- 多列布局
- 不规则排列
- 自动回收
- 异步加载
- 增量刷新

这些功能也是大型商业项目中常见的虚拟列表扩展方案。

**24.3.5 动态高度列表优化**

很多商业项目中的列表 Item 高度并不固定。

例如：

- 聊天消息
- 邮件内容
- 富文本公告
- 评论系统

这种情况下，传统固定高度虚拟列表已经无法满足需求。

因此，通常需要维护：

- Item 高度缓存
- 累积高度表
- 可视区域快速查找

大型项目中通常会采用二分查找来快速定位当前可见区域的起始索引。

否则在超长列表中，滚动时可能产生额外 CPU 开销。

**24.4 实战案例**

下面通过一个“背包虚拟列表示例”，完整演示大型 UI 系统中的优化方案。

该案例将实现：

- 大数据量 ScrollView
- Item 节点复用
- 对象池缓存
- 虚拟列表滚动
- 动态数据刷新

最终实现一个支持上千条数据依然流畅滚动的背包系统。

**24.4.1 案例需求**

假设当前背包系统存在 1000 条物品数据。

如果直接实例化 1000 个 Item，会导致：

- Hierarchy 节点数量暴增
- Canvas 重建开销增加
- Layout 计算耗时上升
- 内存占用过高
- 滚动明显卡顿

因此，本案例将采用虚拟列表方案。

核心思想如下。

1. 只创建可视区域中的节点
2. 节点离开屏幕后循环复用
3. 不再频繁 Instantiate 与 Destroy
4. 使用对象池管理 Item

**24.4.2 UI 结构**

ScrollView 结构如下。

```csharp
ScrollView
├── Viewport
│   ├── Content
│       ├── Item(Clone)
```

其中：

- Viewport 用于裁剪显示区域
- Content 用于承载 Item
- Item 为背包节点预制体

Item 结构如下。

```csharp
Item
├── Icon
├── NameText
├── CountText
```

**24.4.3 背包数据结构**

首先定义背包数据。

```csharp
public class BagItemData
{
    public int Id;
    public string Name;
    public int Count;
}
```

下面模拟生成 1000 条数据。

```csharp
private List<BagItemData> dataList = new List<BagItemData>();

private void CreateData()
{
    for (int i = 0; i < 1000; i++)
    {
        BagItemData data = new BagItemData();

        data.Id = i;
        data.Name = "物品_" + i;
        data.Count = Random.Range(1, 999);

        dataList.Add(data);
    }
}
```

**24.4.4 Item 节点脚本**

下面编写背包节点脚本。

```csharp
using UnityEngine;
using UnityEngine.UI;

public class BagItem : MonoBehaviour
{
    public Text nameText;
    public Text countText;

    public void Refresh(BagItemData data)
    {
        nameText.text = data.Name;
        countText.text = data.Count.ToString();
    }
}
```

该脚本只负责数据刷新，不负责列表逻辑。

这样能够降低耦合程度。

**24.4.5 虚拟列表实现**

下面开始实现虚拟列表核心逻辑。

```csharp
using System.Collections.Generic;
using UnityEngine;
using UnityEngine.UI;

public class BagVirtualList : MonoBehaviour
{
    [Header("UI")]
    public ScrollRect scrollRect;
    public RectTransform content;
    public BagItem itemPrefab;

    [Header("Item Setting")]
    public float itemHeight = 100f;

    // 数据
    private List<BagItemData> dataList = new List<BagItemData>();

    // 可复用Item
    private List<BagItem> itemList = new List<BagItem>();

    // 可见数量
    private int visibleCount;

    // 上一次起始索引
    private int lastStartIndex = -1;

    private void Start()
    {
        scrollRect.movementType = ScrollRect.MovementType.Clamped;

        CreateData();

        InitList();

        scrollRect.onValueChanged.AddListener(OnScroll);
    }

    private void OnDestroy()
    {
        scrollRect.onValueChanged.RemoveListener(OnScroll);
    }

    private void CreateData()
    {
        for (int i = 0; i < 1000; i++)
        {
            BagItemData data = new BagItemData();

            data.Id = i;
            data.Name = "物品_" + i;
            data.Count = Random.Range(1, 999);

            dataList.Add(data);
        }
    }

    private void InitList()
    {
        visibleCount = Mathf.CeilToInt
            (
                scrollRect.viewport.rect.height
                / itemHeight
            ) + 2;

        for (int i = 0; i < visibleCount; i++)
        {
            BagItem item = Instantiate(itemPrefab, content);

            RectTransform rect = item.GetComponent<RectTransform>();

            rect.sizeDelta =
                new Vector2
                (
                    rect.sizeDelta.x,
                    itemHeight
                );

            itemList.Add(item);
        }

        content.sizeDelta =
            new Vector2
            (
                content.sizeDelta.x,
                dataList.Count * itemHeight
            );

        RefreshItems(true);
    }

    private void OnScroll(Vector2 pos)
    {
        RefreshItems();
    }

    private void RefreshItems(bool force = false)
    {
        float scrollY = content.anchoredPosition.y;

        scrollY = Mathf.Max(0, scrollY);

        int startIndex = Mathf.FloorToInt(scrollY / itemHeight);

        if (!force && startIndex == lastStartIndex)
            return;

        lastStartIndex = startIndex;

        for (int i = 0; i < itemList.Count; i++)
        {
            int dataIndex = startIndex + i;

            if (dataIndex < 0 || dataIndex >= dataList.Count)
            {
                itemList[i].gameObject.SetActive(false);
                continue;
            }

            BagItem item = itemList[i];

            item.gameObject.SetActive(true);

            RectTransform rect = item.GetComponent<RectTransform>();

            rect.anchoredPosition =
                new Vector2
                (
                    0,
                    -dataIndex * itemHeight
                );

            item.Refresh(dataList[dataIndex]);
        }
    }
}
```

**24.4.6 核心优化分析**

该方案的核心优势主要体现在以下几个方面。

第一，只创建少量节点。

即使拥有 1000 条数据，场景中真正存在的 Item 数量也只有十几个。

第二，不再频繁创建与销毁对象。

整个滚动过程中不会持续产生 GC。

第三，大幅降低 Canvas 重建压力。

因为真正参与渲染的节点数量非常有限。

第四，Hierarchy 保持稳定。

避免大量节点导致编辑器与运行时性能下降。

第五，降低 Layout 计算开销。

因为绝大多数节点根本不会被创建。

**24.4.7 最终效果**

通过虚拟列表方案，即使数据量达到上千条，ScrollView 依然能够保持流畅滚动。

这也是大型商业项目中最常见的列表优化方案之一。

在实际项目中，还可以继续扩展：

- 多列布局
- 动态高度
- 异步加载
- 自动回收
- 增量刷新
- 分页加载

这些功能本质上都建立在“节点复用”这一核心思想之上。
