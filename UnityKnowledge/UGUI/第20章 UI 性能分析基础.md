在 [UGUI](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=UGUI&zhida_source=entity) 的整体体系中，性能问题往往并不来源于单一模块，而是由渲染、布局、更新机制等多个系统协同作用的结果。相比于功能实现，性能分析更强调对“整体数据流与执行路径”的理解。本章将从宏观角度出发，建立 UI 性能分析的基础认知，为后续优化策略提供理论支撑。

UI 性能的本质，是 CPU 与 GPU 在处理 UI 数据时所消耗的时间与资源。在运行过程中，UGUI 需要完成布局计算、顶点生成、批处理组织以及最终绘制等一系列步骤。这些步骤分布在不同的系统之中，并在不同阶段对性能产生影响。

从执行流程来看，UI 的性能消耗主要集中在两个阶段：一是 CPU 侧的数据处理，包括 [Layout 计算](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=Layout+%E8%AE%A1%E7%AE%97&zhida_source=entity) 、Graphic 重建以及 Mesh 生成；二是 GPU 侧的渲染开销，包括 [DrawCall](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=DrawCall&zhida_source=entity) 执行、像素填充以及混合计算。这两个阶段相互独立又彼此关联，共同决定了 UI 的整体性能表现。

在 CPU 层面，性能问题通常表现为频繁的重建操作。例如，当 UI 发生属性变化时，系统会通过 Dirty 标记触发重建流程。如果重建范围过大或触发频率过高，就会导致明显的 CPU 开销上升。尤其是在复杂 UI 结构中，Layout 系统与 Graphic 系统的联动，往往会放大这一问题。

在 GPU 层面，性能瓶颈则更多体现在 DrawCall 数量与 [Overdraw](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=Overdraw&zhida_source=entity) 上。UI 由于其半透明特性，通常需要进行混合计算，这使得像素填充成本显著高于不透明物体。当 UI 元素层级较多、覆盖关系复杂时，Overdraw 会迅速增加，从而成为 GPU 的主要压力来源。

与 3D 渲染相比，UI 渲染具有其特殊性。一方面，UI 更依赖批处理来降低 DrawCall；另一方面，UI 更新频率较高，容易产生频繁的重建操作。因此，UI 性能优化不仅需要关注渲染阶段，还需要从数据更新机制入手进行整体分析。

此外，Canvas 在性能体系中占据核心地位。由于 UI 的重建与渲染提交均以 Canvas 为单位进行，Canvas 的划分方式将直接影响性能表现。过大的 Canvas 会导致重建范围扩大，而过多的 Canvas 又会增加 DrawCall 数量，因此需要在两者之间进行权衡。

从分析方法来看，UI 性能问题通常需要借助工具进行定位，例如帧调试器用于分析 DrawCall 结构，性能分析器用于观察 CPU 开销。这些工具能够帮助开发者从执行结果反推系统行为，从而找到性能瓶颈所在。

本章将围绕 UI 性能的基本构成展开，分别从 CPU 与 GPU 开销、Rebuild 成本、DrawCall 分析以及 Overdraw 四个方面进行系统讲解。通过对这些核心问题的拆解，可以建立对 UGUI 性能机制的整体认知，并为后续章节中的优化实践打下基础。

**20.1 CPU 与 GPU 开销**

在 UI 性能分析中，最基础也是最核心的划分方式，是将性能开销拆分为 CPU 与 GPU 两个部分。二者分别对应“数据处理”与“图形绘制”两个阶段，共同决定了 UI 的最终性能表现。

理解 CPU 与 GPU 的职责边界，是分析 UGUI 性能问题的前提。只有明确每一类开销发生的位置与原因，才能在优化过程中做出正确的决策。

**20.1.1 CPU 开销来源**

CPU 侧的主要职责，是完成 UI 数据的构建与更新。在 UGUI 中，这一过程贯穿了从布局计算到网格生成的完整流程。

首先是 Layout 系统的计算开销。当 UI 层级中存在 LayoutGroup、ContentSizeFitter 等组件时，系统需要根据子节点信息反复计算尺寸与位置。这一过程具有明显的递归特性，一旦某个节点发生变化，就可能触发整棵 UI 树的重新计算。

其次是 Graphic 系统的重建开销。每当 UI 元素的显示属性发生变化（如颜色、尺寸、Sprite 等），都会触发顶点数据的重新生成。该过程通过 OnPopulateMesh 完成，本质上是将 UI 转换为可渲染的 Mesh 数据。

再次是 Dirty 标记驱动的更新机制。UGUI 并不会在每一帧都完全重建 UI，而是通过 SetVerticesDirty、SetLayoutDirty 等接口标记需要更新的节点。但如果标记范围过大或触发频繁，仍然会带来显著的 CPU 消耗。

此外，CanvasUpdateRegistry 的调度过程也属于 CPU 开销的一部分。系统会在特定阶段统一处理所有待重建的 UI 元素，这一批量处理虽然提高了整体效率，但在数据量较大时仍然会形成性能峰值。

综合来看，CPU 开销的本质，是“UI 数据从逻辑状态到渲染数据的转换成本”。其特点是对结构变化敏感，尤其容易受到层级复杂度与更新频率的影响。

**20.1.2 GPU 开销来源**

GPU 侧的主要职责，是执行实际的图形绘制。相比 CPU 的数据处理，GPU 更关注绘制指令的执行效率与像素计算成本。

最直接的开销来源是 DrawCall。每一次 DrawCall 都代表一次渲染状态的提交，包括材质、纹理以及顶点数据等。当 DrawCall 数量过多时，会增加 GPU 的调度压力，同时也会带来 CPU 与 GPU 之间的通信开销。

其次是顶点处理开销。UI 的每一个元素最终都会被转换为三角形网格，GPU 需要对这些顶点进行变换与处理。虽然单个 UI 元素的顶点数量较少，但在大规模 UI 场景中，累计开销依然不可忽视。

更为关键的是像素填充开销，即 Overdraw。由于 UI 通常使用半透明材质，像素在被最终显示之前，可能会被多次绘制与混合。当多个 UI 元素在屏幕上重叠时，同一像素会被重复计算，从而显著增加 GPU 负担。

此外，混合（Blend）操作本身也是 GPU 的重要开销来源。UI 默认采用透明混合模式，这意味着 GPU 在绘制每一个像素时，都需要读取当前帧缓冲区的数据并进行计算，这一过程远比不透明渲染更加昂贵。

因此，GPU 开销的本质，可以理解为“绘制指令执行与像素计算的成本”。其特点是对覆盖关系与绘制顺序敏感，尤其容易受到 UI 层级深度与透明度的影响。

**20.1.3 CPU 与 GPU 的协同关系**

在实际运行中，CPU 与 GPU 并不是独立工作的，而是通过渲染管线形成协同关系。CPU 负责准备数据并提交绘制命令，GPU 负责执行这些命令并完成最终渲染。

当 CPU 成为瓶颈时，通常表现为帧率受限于脚本执行与 UI 重建过程；而当 GPU 成为瓶颈时，则表现为渲染耗时过高，即使 CPU 空闲也无法提升帧率。

需要注意的是，DrawCall 同时影响 CPU 与 GPU。过多的 DrawCall 不仅增加 GPU 的绘制压力，也会提高 CPU 在提交渲染指令时的开销。因此，在 UI 优化中，控制 DrawCall 数量具有双重意义。

另一方面，Overdraw 主要影响 GPU，但其根源往往来自 UI 的结构设计。例如过多的层级叠加、不必要的半透明区域等，都会导致像素重复计算。因此，优化 GPU 开销也需要从 UI 设计层面入手。

总体而言，CPU 与 GPU 开销构成了 UI 性能分析的两个基本维度。前者关注“是否频繁重建”，后者关注“是否高效绘制”。只有在这两个方面同时达到合理状态，才能实现稳定且高效的 UI 渲染表现。

**20.2 Rebuild 成本**

在 UGUI 性能体系中，Rebuild 是 CPU 开销的核心来源之一。几乎所有 UI 的动态变化，最终都会转化为某种形式的重建操作。因此，理解 Rebuild 的触发机制、执行流程以及成本来源，是分析 UI 性能问题的关键。

从本质上来看，Rebuild 是“UI 从逻辑状态重新生成渲染数据”的过程。它包含布局计算、顶点重建以及渲染数据更新等多个阶段。每一次 Rebuild，都会消耗一定的 CPU 时间，而当 Rebuild 频繁发生或范围过大时，就会成为性能瓶颈。

**20.2.1 Rebuild 的触发机制**

UGUI 并不会在每一帧主动重建所有 UI，而是采用基于 Dirty 标记的延迟更新机制。当 UI 元素的某些属性发生变化时，系统会标记该元素为“需要重建”，并在后续统一执行。

常见的触发方式包括以下几种：

1. 布局相关变化。例如 RectTransform 尺寸改变、父子层级变动、LayoutGroup 参数修改等，都会触发 Layout 重建。这类变化通常具有传播性，可能影响整个 UI 子树。
2. 显示属性变化。例如颜色、Sprite、材质等发生改变时，会触发 Graphic 重建，从而重新生成顶点数据。
3. 显式调用接口。例如通过 SetVerticesDirty、SetLayoutDirty、SetAllDirty 等方法手动标记节点，也会加入重建队列。

这些触发操作并不会立即执行，而是被记录在 CanvasUpdateRegistry 中，等待统一调度。这种延迟机制避免了重复计算，但也意味着多个变化可能在同一帧集中爆发。

**20.2.2 Rebuild 的执行阶段**

Rebuild 并不是单一过程，而是被划分为多个阶段，并在特定顺序下执行。其核心流程由 CanvasUpdateRegistry 统一调度，主要包括 Layout 阶段与 Graphic 阶段。

在 Layout 阶段，系统会计算 UI 元素的尺寸与位置。这一过程通常分为两步：先计算布局输入（如最小尺寸、首选尺寸），再进行实际排布。由于布局依赖父子关系，因此该阶段需要按照层级顺序进行遍历。

在 Graphic 阶段，系统会调用各个 Graphic 组件的 OnPopulateMesh 方法，重新生成顶点数据。这些顶点随后会被提交到 CanvasRenderer，等待后续的批处理。

需要注意的是，Layout 与 Graphic 是两个相互独立但又紧密关联的阶段。布局变化往往会引发顶点重建，而顶点重建本身并不会反向影响布局。

**20.2.3 Rebuild 成本来源分析**

Rebuild 的性能开销，主要来源于以下几个方面：

一、层级遍历成本

Layout 计算需要遍历 UI 树结构。当层级较深或节点数量较多时，这一过程的时间复杂度会显著增加。尤其是在嵌套 LayoutGroup 的情况下，重复计算问题会更加明显。

二、重复计算问题

在复杂 UI 中，一个节点的变化可能引发多次布局计算。例如 ContentSizeFitter 与 LayoutGroup 组合使用时，可能产生多轮尺寸调整，从而放大 CPU 开销。

三、顶点生成开销

Graphic 重建需要重新生成 Mesh 数据，包括顶点、UV、颜色等信息。对于 Text 或复杂 Image（如九宫格）而言，顶点数量较多，生成成本也更高。

四、批量执行带来的峰值问题

由于 Rebuild 是延迟执行的，多个 Dirty 标记可能在同一帧集中处理，从而形成 CPU 峰值。这种“瞬时压力”在 UI 频繁变化的场景中尤为明显。

**20.2.4 Canvas 对 Rebuild 的影响**

Canvas 在 Rebuild 体系中扮演着“更新边界”的角色。所有 UI 的重建，都是以 Canvas 为单位进行组织的。

当某个 UI 元素被标记为 Dirty 时，其所属 Canvas 会在更新阶段重新处理相关数据。这意味着，如果一个 Canvas 包含大量 UI 元素，那么即使只修改其中一个节点，也可能导致较大范围的重建。

因此，Canvas 的粒度直接影响 Rebuild 成本：

- Canvas 过大，会导致重建范围扩大，增加不必要的计算开销。
- Canvas 过多，虽然可以减少单次重建范围，但会增加 DrawCall 数量，从而对 GPU 产生压力。

如何在两者之间取得平衡，是 UI 性能优化中的核心问题之一。

**20.2.5 Rebuild 的优化方向**

从性能角度来看，降低 Rebuild 成本的关键，在于减少“触发频率”与“影响范围”。

在触发频率方面，应尽量避免频繁修改 UI 属性。例如避免在 Update 中持续改变布局或文本内容，减少不必要的 Dirty 标记。

在影响范围方面，可以通过合理拆分 Canvas，将高频更新的 UI 与静态 UI 分离，从而限制重建范围。

此外，应尽量避免复杂的 Layout 嵌套结构，减少重复计算问题。在必要情况下，可以使用代码手动控制布局，而非完全依赖自动布局系统。

总体而言，Rebuild 成本的本质，是“UI 数据重建的 CPU 消耗”。它既受到系统机制的影响，也与 UI 结构设计密切相关。只有在理解其内部流程的基础上，才能有效控制其对性能的影响。

**20.3 DrawCall 分析**

在 GPU 性能分析中，DrawCall 是最直观且最关键的指标之一。对于 UGUI 而言，DrawCall 的数量不仅直接影响 GPU 的渲染效率，同时也会对 CPU 的提交开销产生影响。因此，对 DrawCall 的生成机制与影响因素进行深入分析，是 UI 性能优化的重要基础。

从本质上来看，DrawCall 是一次完整的渲染指令提交过程。它包含了顶点数据、材质状态以及纹理绑定等信息。GPU 在执行渲染时，会按照 DrawCall 的顺序逐条处理这些指令，从而完成图像的绘制。

**20.3.1 DrawCall 的生成机制**

在 UGUI 中，DrawCall 并不是由单个 UI 元素直接产生的，而是由 Canvas 在批处理阶段统一生成。所有 Graphic 组件生成的 Mesh 数据，都会先提交给各自的 CanvasRenderer，随后由 Canvas 在 BuildBatch 阶段进行整合。

在这一过程中，系统会根据一系列条件对 Mesh 进行合批处理。当多个 UI 元素满足合批规则时，它们会被合并为一个 DrawCall；反之，则会被拆分为多个独立的绘制指令。

因此，可以将 DrawCall 的生成理解为“批处理结果的最终体现”。其数量取决于 UI 元素之间是否能够成功合并，而不是单纯由元素数量决定。

**20.3.2 合批的基本规则**

UGUI 的合批机制，本质上是基于渲染状态一致性的合并策略。只有当多个 UI 元素在渲染所需的关键状态上完全一致时，才能被合并为一个 DrawCall。

主要规则包括以下几个方面：

一、材质一致

UI 元素必须使用相同的材质实例。即使 Shader 相同，但如果材质参数不同，也无法合批。

二、纹理一致

材质所使用的主纹理必须相同。 [SpriteAtlas](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=SpriteAtlas&zhida_source=entity) 的使用，正是为了解决这一问题，通过将多个 Sprite 合并到同一纹理中，从而提高合批率。

三、渲染顺序连续

UI 的绘制顺序必须保持连续。如果中间插入了无法合批的元素，会导致批处理被打断。

四、相同的渲染状态

包括裁剪状态（如 Mask）、混合模式、Stencil 设置等。这些状态一旦发生变化，就必须产生新的 DrawCall。

需要注意的是，UGUI 的合批是严格有序的。这意味着系统不会为了合批而改变 UI 的绘制顺序，因此在层级结构设计不合理时，很容易破坏合批条件。

**20.3.3 DrawCall 增加的常见原因**

在实际项目中，DrawCall 数量过多通常由以下几个因素导致：

一、纹理分散

未使用图集或图集拆分不合理，会导致频繁切换纹理，从而增加 DrawCall。

二、材质实例过多

例如对单个 UI 元素单独修改材质参数，或使用不同的材质实例，都会破坏合批条件。

三、Mask 与裁剪

Mask 会引入额外的 Stencil 状态变化，从而打断批处理。尤其是在多层嵌套 Mask 时，DrawCall 数量会明显增加。

四、层级穿插

不同类型 UI 元素在层级中交错排列，会导致合批被频繁中断。例如多个图集之间的 UI 元素交替出现。

五、子 Canvas 的使用

每一个 Canvas 都会独立进行批处理，因此多个 Canvas 会产生各自的 DrawCall。虽然可以降低重建成本，但也会增加渲染开销。

**20.3.4 DrawCall 与性能的关系**

DrawCall 对性能的影响，主要体现在两个方面。

首先是 CPU 提交开销。每一个 DrawCall 都需要 CPU 向 GPU 提交渲染指令，这一过程涉及状态切换与命令组织。当 DrawCall 数量较多时，CPU 端的负担会明显增加。

其次是 GPU 执行开销。虽然 GPU 擅长并行处理，但频繁的状态切换仍然会影响其执行效率。特别是在移动设备上，DrawCall 数量往往是限制性能的重要因素。

需要注意的是，DrawCall 并不是越少越好。在某些情况下，为了减少 Rebuild 范围或提高结构清晰度，适当增加 DrawCall 是可以接受的。因此，优化的关键在于“合理控制”，而非单纯追求最小值。

**20.3.5 DrawCall 分析方法**

在实际分析过程中，通常需要借助工具对 DrawCall 进行观察与拆解。

通过帧调试器，可以逐条查看每一个 DrawCall 的来源，包括对应的 UI 元素、材质以及渲染状态。这有助于定位合批失败的具体原因。

通过性能分析器，可以观察 DrawCall 数量的变化趋势，以及其对帧时间的影响。在 UI 频繁变化的场景中，这一点尤为重要。

此外，还可以结合 UI 结构进行静态分析。例如检查图集使用情况、材质复用情况以及层级组织方式，从源头上优化合批条件。

总体来看，DrawCall 的本质是“渲染指令的数量体现”。它既受到资源组织方式的影响，也与 UI 结构密切相关。通过合理设计图集、材质以及层级结构，可以有效控制 DrawCall 数量，从而提升 UI 渲染性能。

**20.4 Overdraw**

在 UI 渲染性能分析中，Overdraw 是影响 GPU 性能的核心因素之一。相比 DrawCall，Overdraw 更加隐蔽，但其带来的性能消耗往往更为显著，尤其是在移动设备上。

从本质上来看，Overdraw 指的是同一屏幕像素在一帧内被重复绘制的次数。当多个 UI 元素在空间上发生重叠时，位于上层的像素会覆盖下层的像素，但 GPU 在渲染过程中，仍然需要对所有被覆盖的像素执行完整的绘制与混合计算。

**20.4.1 Overdraw 的产生原因**

UGUI 的渲染默认采用透明混合模式，这意味着大多数 UI 元素都无法利用深度测试进行提前剔除。GPU 无法像处理不透明物体那样跳过被遮挡的像素，而是必须按照绘制顺序逐层计算。

因此，Overdraw 的产生主要来源于以下几个方面：

一、层级叠加

UI 元素按照层级顺序进行绘制，后绘制的元素会覆盖先绘制的内容。当层级较深时，同一像素可能被多次重复计算。

二、大面积透明区域

许多 UI 图片（如带透明背景的 Sprite）在视觉上只有局部区域可见，但其 Mesh 通常仍是完整矩形。这会导致大量“不可见但仍被绘制”的像素参与计算。

三、全屏覆盖元素

例如背景图、遮罩层、弹窗蒙版等，会对整个屏幕进行绘制。当这些元素叠加出现时，会显著增加 Overdraw。

四、Mask 与裁剪

Mask 会引入额外的绘制过程。在某些实现中，被裁剪区域仍然参与计算，只是在最终阶段被丢弃，从而增加像素处理成本。

**20.4.2 Overdraw 的性能影响**

Overdraw 直接影响 GPU 的像素填充率（Fill Rate）。每一次像素绘制，都需要执行顶点插值、纹理采样以及混合计算等操作。当同一像素被多次处理时，这些开销会线性叠加。

在移动设备上，Fill Rate 往往是性能瓶颈之一。相比顶点处理，像素处理的成本更高，尤其是在高分辨率屏幕上，Overdraw 的影响会被进一步放大。

此外，透明混合操作需要读取帧缓冲区中的已有像素值，并与当前像素进行计算。这种读写操作增加了内存带宽压力，使 GPU 更容易达到性能上限。

需要强调的是，Overdraw 与 DrawCall 是两个不同维度的问题。即使 DrawCall 数量较少，如果存在大量像素重复绘制，仍然会导致 GPU 性能下降。

**20.4.3 Overdraw 的可视化分析**

在实际开发中，Overdraw 通常需要借助工具进行可视化分析。Unity 提供了 Overdraw 视图模式，可以通过颜色深浅直观显示像素被绘制的次数。

一般情况下：

颜色越浅，表示像素绘制次数较少；

颜色越深，表示存在较严重的 Overdraw。

通过该视图，可以快速定位 UI 中的高 Overdraw 区域，例如多层叠加的面板、全屏遮罩以及复杂列表区域。

此外，也可以结合 GPU 性能分析工具，从填充率与片元着色器执行时间的角度，进一步验证 Overdraw 是否成为瓶颈。

**20.4.4 降低 Overdraw 的方法**

降低 Overdraw 的核心思路，是减少“无意义的像素绘制”。具体方法包括以下几个方面：

一、减少层级叠加

避免不必要的 UI 覆盖关系，例如隐藏被完全遮挡的元素，或在切换界面时及时关闭不可见 UI。

二、优化图片资源

尽量减少大面积透明区域，例如使用紧凑裁剪的 Sprite，或采用多边形网格替代默认矩形，以降低无效像素填充。

三、控制全屏元素数量

减少全屏背景、遮罩层的叠加使用。在设计 UI 时，应尽量避免多个全屏半透明元素同时存在。

四、合理使用 Mask

避免过度嵌套 Mask，必要时可使用 [RectMask2D](https://zhida.zhihu.com/search?content_id=275541058&content_type=Article&match_order=1&q=RectMask2D&zhida_source=entity) 替代，以减少额外的像素处理开销。

五、分层设计 UI

通过合理划分 UI 层级与结构，使高频变化区域与静态区域分离，从而减少重叠区域的复杂度。

**20.4.5 Overdraw 与 UI 设计的关系**

与 DrawCall 不同，Overdraw 很大程度上来源于 UI 设计本身，而不仅仅是技术实现问题。界面视觉效果越复杂，往往意味着更多的层级叠加与透明区域，从而带来更高的 Overdraw。

因此，在进行 UI 设计时，需要在视觉表现与性能之间取得平衡。例如减少不必要的阴影、渐变与半透明叠加，合理控制界面层次结构，都是降低 Overdraw 的有效手段。

总体来看，Overdraw 的本质是“像素级重复计算”。它直接决定了 GPU 在片元阶段的工作量，是 UI 渲染性能的重要影响因素之一。只有在理解其产生机制的基础上，结合 UI 结构与资源设计进行优化，才能有效降低其对性能的影响。

**本章小结**

本章围绕 UI 性能分析的基础展开，从整体角度建立了 UGUI 性能问题的核心认知框架。

首先，从 CPU 与 GPU 两个维度出发，明确了 UI 性能的基本构成。CPU 主要负责布局计算、顶点生成以及重建调度等数据处理工作，而 GPU 则负责 DrawCall 执行与像素绘制。两者相互协同，共同决定最终的帧表现。

在 Rebuild 成本部分，分析了 UI 重建的触发机制与执行流程。通过 Dirty 标记与 CanvasUpdateRegistry 的调度，UGUI 实现了延迟更新机制，但频繁或大范围的重建仍然会带来显著的 CPU 开销。Canvas 作为更新边界，其粒度划分对性能具有重要影响。

在 DrawCall 分析中，重点说明了批处理机制对渲染性能的决定作用。DrawCall 的数量本质上是合批结果的体现，而材质、纹理、渲染状态以及层级顺序，都会直接影响合批效果。合理组织资源与 UI 结构，是控制 DrawCall 的关键。

在 Overdraw 部分，则从像素层面分析了 GPU 的性能瓶颈。由于 UI 采用透明混合渲染，像素往往会被多次重复绘制，从而增加填充开销。层级叠加、透明区域以及全屏元素，是 Overdraw 的主要来源，其优化需要结合 UI 设计与结构进行综合考虑。

总体来看，UI 性能问题可以归纳为两个核心方向：一是减少不必要的数据重建，降低 CPU 负担；二是提高渲染效率，控制 DrawCall 与 Overdraw，从而减轻 GPU 压力。两者相互影响，需要在系统设计层面进行整体权衡。至此，UI 性能分析的基础模型已经建立。
