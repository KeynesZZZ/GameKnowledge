# Prompt与上下文管理

> Prompt 的核心不是“写得玄”，而是让 AI 在正确边界内做正确的事。

## 高质量 Prompt 结构

```text
角色：
- 你是一个熟悉 Unity 架构和 C# 性能优化的工程师。

目标：
- 修复资源加载模块中的重复加载问题。

上下文：
- 关键文件：
- 已知问题：
- 现有限制：

约束：
- 不改变公开 API
- 不引入新依赖
- 保持现有日志格式

验收标准：
- 现有测试通过
- 新增重复加载用例
- 输出变更摘要
```

## 上下文分层

| 层级 | 内容 | 什么时候提供 |
|------|------|--------------|
| 任务层 | 目标、范围、验收标准 | 每次任务都提供 |
| 项目层 | 架构、目录、编码规范 | 新项目或长期会话开始时 |
| 文件层 | 关键代码、错误日志、测试结果 | 具体实现前提供 |
| 历史层 | 过去决策、踩坑记录、禁区 | 涉及架构或重构时提供 |

## 不同阶段的 Prompt

### 探索阶段

```text
请只阅读和分析，不要修改文件。
请告诉我这个问题涉及哪些模块、当前实现路径、潜在风险和建议的修改顺序。
```

### 实现阶段

```text
请按你刚才的方案实现。
限制修改范围在这些文件内：
- 

实现后请运行可用验证，并说明结果。
```

### Review 阶段

```text
请以代码评审视角检查这次改动。
优先找 bug、回归风险、边界条件和缺失测试。
请按严重程度排序，并引用具体文件和行号。
```

### 复盘阶段

```text
请把这次任务沉淀成知识库笔记。
输出：
- 问题背景
- 根因
- 解决方案
- 可复用检查清单
- 下次避免方式
```

## 常见反模式

| 反模式 | 问题 | 改进 |
|--------|------|------|
| 只说“优化代码” | 目标不可验证 | 明确性能指标或代码质量目标 |
| 一次塞入太多任务 | 上下文混乱 | 拆成分析、实现、验证、复盘 |
| 不给文件边界 | 容易大范围改动 | 指定模块、文件、禁止事项 |
| 不要求验证 | 难以判断完成 | 写明测试、构建、检查命令 |
