# 文档创建规范执行指南

> 本指南说明如何使用工具确保文档创建符合 `00_元数据与模板/` 目录下的规范

## 快速开始

### 1. 创建新文档

在 Claude Code 中使用 `/create-doc` Skill：

```bash
/create-doc "对象池性能优化"
```

Skill 将引导你完成：
1. 选择文档类型（8种）
2. 选择合适的模板
3. 填写 YAML Frontmatter
4. 验证合规性

### 2. 检查文档合规性

```bash
# 检查所有文档
python tools/check_docs_compliance.py

# 检查特定目录
python tools/check_docs_compliance.py UnityKnowledge/10_架构设计
```

## 工具说明

### `/create-doc` Skill

**位置**: `.claude/skills/create-doc.md`

**功能**:
- 引导选择文档类型和模板
- 验证 YAML Frontmatter 必填字段
- 检查标签数量和类别
- 检查标题前缀
- 检查 category 路径
- 检查 description 长度

**使用场景**:
- 创建新的知识库文档
- 确保文档符合规范

### `check_docs_compliance.py` 脚本

**位置**: `tools/check_docs_compliance.py`

**功能**:
- 批量检查所有 Markdown 文档
- 报告不合规文档
- 按严重程度分类问题
- 提供修正建议

**检查项**:
1. ✅ YAML Frontmatter 必填字段
2. ✅ title 类型前缀
3. ✅ tags 数量（≥2个）
4. ✅ 技术领域标签
5. ✅ category 标准路径
6. ✅ description 长度（≤50字）
7. ✅ 时间格式
8. ✅ 文档结构（文档定位、相关链接章节）

**输出示例**:

```

总文档数: 206

❌ 不合规文档: 193 (93.7%)
```

## 文档创建流程

### 步骤 1: 确定文档类型

根据内容选择合适的文档类型：

| 文档类型 | 前缀 | 核心问题 | 示例 |
|---------|------|---------|------|
| 设计原理 | 【设计原理】 | Why | 为什么用对象池 |
| 教程 | 【教程】 | How to use | 如何使用对象池 |
| 代码片段 | 【代码片段】 | Quick copy | 对象池完整代码 |
| 性能数据 | 【性能数据】 | How fast | 对象池性能测试 |
| 最佳实践 | 【最佳实践】 | Best practice | 对象池使用建议 |
| 踩坑记录 | 【踩坑记录】 | Pitfalls | 对象池常见问题 |
| 架构决策 | 【架构决策】 | Which | 对象池vs实例化 |
| 系统架构 | 【系统架构】 | What | 对象池系统设计 |

### 步骤 2: 选择模板

根据文档类型选择模板：

| 文档类型 | 推荐模板 |
|---------|---------|
| 设计原理、教程、架构决策、系统架构 | `模板-通用编程笔记.md` |
| 代码片段 | `模板-代码片段.md` |
| 踩坑记录、Bug记录 | `模板-Bug记录.md` |
| 性能数据、性能优化 | `模板-性能分析.md` |

### 步骤 3: 填写 YAML Frontmatter

```yaml
---
title: 【代码片段】对象池通用实现
tags: [C#, Unity, 架构, 性能优化, 代码片段]  # 至少2个
category: 架构设计/代码片段
created: 2024-03-05 14:30
updated: 2024-03-05 14:30
description: C#泛型对象池的基础实现，减少GC分配  # ≤50字
unity_version: 2021.3+
---
```

**必填字段**:
- `title`: 包含类型前缀
- `tags`: 至少2个标签，包含技术领域标签
- `category`: 标准分类路径
- `created` / `updated`: 时间戳
- `description`: 简洁描述

### 步骤 4: 完成文档结构

```markdown
# 【代码片段】对象池通用实现

## 文档定位

本文档提供开箱即用的对象池代码实现。

**设计原理**：参见 [[设计原理-对象池本质]]
**性能数据**：参见 [[../30_性能优化/32_内存管理/【性能数据】对象池vs实例化]]

## 一、完整代码

...
```

## 常见问题修正

### 问题 1: 缺少 YAML Frontmatter

**错误**: `缺少 YAML Frontmatter`

**修正**: 在文档开头添加：

```yaml
---
title: 【文档类型】标题
tags: [标签1, 标签2]
category: 分类路径
created: YYYY-MM-DD HH:mm
updated: YYYY-MM-DD HH:mm
description: 描述
---
```

### 问题 2: title 缺少类型前缀

**错误**: `title 缺少类型前缀: 对象池通用实现`

**修正**: 添加类型前缀

```yaml
title: 【代码片段】对象池通用实现
```

### 问题 3: tags 数量不足

**错误**: `tags 数量不足，当前: 1，要求: ≥2`

**修正**: 添加更多标签，包括技术领域标签

```yaml
tags: [C#, Unity, 架构, 性能优化, 代码片段]
```

### 问题 4: 缺少技术领域标签

**错误**: `tags 缺少技术领域标签`

**修正**: 添加技术领域标签（架构、渲染、物理、动画、UI、音频、网络、AI、性能优化等）

```yaml
tags: [Unity, 架构, 性能优化, 对象池]
```

### 问题 5: category 路径不规范

**错误**: `category 格式不正确，应为 "一级分类/二级分类"`

**修正**: 使用标准分类路径

```yaml
category: 架构设计/代码片段
```

### 问题 6: description 过长

**错误**: `description 过长，当前: 65 字，建议: ≤50 字`

**修正**: 精简描述

```yaml
description: C#泛型对象池的基础实现，减少GC分配
```

## 批量修正现有文档

### 1. 查看所有不合规文档

```bash
python tools/check_docs_compliance.py > compliance_report.txt
```

### 2. 按优先级修正

1. **CRITICAL 级别**: 必须修正（缺少 Frontmatter）
2. **HIGH 级别**: 应该修正（缺少类型前缀）
3. **MEDIUM 级别**: 建议修正（标签问题）
4. **LOW 级别**: 可选修正（description 过长、缺少章节）

### 3. 修正后验证

```bash
python tools/check_docs_compliance.py
```

## Git 集成

### 提交前检查

将合规性检查添加到 Git Hook（可选）：

```bash
# .git/hooks/pre-commit
#!/bin/bash
python tools/check_docs_compliance.py

if [ $? -ne 0 ]; then
    echo "❌ 文档合规性检查失败，请修正后再提交"
    exit 1
fi
```

### 跳过检查（不推荐）

```bash
git commit --no-verify -m "提交信息"
```
